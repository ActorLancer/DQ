## 2. 当前本地开发目标

本地环境至少必须支持：

- `platform-core` 的主链路开发
- `portal-web / console-web` 联调
- PostgreSQL 升级/降级验证
- Kafka/outbox/dead-letter 基本链路
- 搜索投影同步
- Mock PaymentProvider 支付演练（作为 `local` 下的 `mocks` 子场景）
- Fabric 测试网络的最小交互验证

### 4.1 本地直接运行的业务进程

- `platform-core`
- `portal-web`
- `console-web`
- `fabric-adapter`

### 4.2 容器内运行的基础设施和辅助进程

- `PostgreSQL`
- `Redis`
- `Kafka`
- `MinIO`
- `OpenSearch`
- `Keycloak`
- `Prometheus`
- `Alertmanager`
- `Grafana`
- `Loki`
- `Tempo`
- `mock-payment-provider`（仅支付/回执联调或 `demo`）

- `fabric-event-listener / search-indexer / data-processing-worker / notification-worker` 既可以本地跑，也可以容器跑。

- 当前阶段优先保证 `platform-core + PostgreSQL + Kafka + Redis + MinIO + OpenSearch + Keycloak` 先跑通；支付/回执联调时再叠加 `Mock PaymentProvider`。

## 7. 启动顺序

推荐顺序：

1. `PostgreSQL`
2. `Redis`
3. `Kafka`
4. `MinIO`
5. `OpenSearch`
6. `Keycloak`
7. `Prometheus / Alertmanager / Grafana / Loki / Tempo`
8. `mock-payment-provider`（仅支付/回执联调或 `demo`）
9. `platform-core`
10. `search-indexer`
11. `fabric-adapter`
12. `fabric-event-listener`
13. `portal-web`
14. `console-web`

原因：

- `platform-core` 启动前要先拿到数据库、缓存、消息总线和认证中心
- 前端启动前最好已有 `platform-core`

## 8.2 数据库

- compose / 容器 bootstrap：`POSTGRES_DB`、`POSTGRES_USER`、`POSTGRES_PASSWORD`、`POSTGRES_PORT`
- 应用 / 脚本运行时入口：`DATABASE_URL`
- `psql / libpq` 临时调试兼容：`PGHOST`、`PGPORT`、`PGDATABASE`、`PGUSER`、`PGPASSWORD`
- 冻结说明：
  - 本地 PostgreSQL 与 Keycloak bootstrap 继续使用 `POSTGRES_*`
  - 应用、seed、SQLx、smoke 的正式数据库入口统一使用 `DATABASE_URL`
  - `PG_HOST / PG_PORT / PG_DATABASE / PG_USER / PG_PASSWORD` 不再作为正式主配置名

## 9. 当前开发模式冻结

正式运行模式仍只有三种：`local`、`staging`、`demo`。

本文件作为本地环境说明，只描述：

- `local` 的基础姿态
- `local` 下的 staging-like 联调姿态
- `demo` 在本地环境中的演示姿态

其中 staging-like 联调姿态不是第四套正式 mode，不得单独扩张为新的 `APP_ENV` 或测试矩阵主维度。

## 9.1 `local`

特点：

- 默认先跑 `core` 基础栈
- 可关闭 Fabric 写入，只保留接口占位
- 可用本地数据库和本地对象存储
- 通过 profile 控制是否叠加更多外围依赖；支付/回执联调时通过 `mocks` 子 profile 启用 `Mock PaymentProvider`

## 9.2 `local` 下的 staging-like 联调姿态（非正式运行模式）

特点：

- 开启 Kafka/outbox/retry/dead-letter
- 开启搜索同步
- 开启 Fabric 测试网络写入
- 只是在本地打开更接近 `staging` 的依赖组合与校验链路
- 不新增第四套正式 mode，不单独引入新的 `APP_ENV`

## 9.3 `demo`

特点：

- 允许演示数据
- 使用 Mock PaymentProvider
- 可开启公链展示模拟，但不进入真实生产结算

## 10. 本地 Mock 策略

## 10.1 必须 Mock 的能力

- 真实支付渠道
- 部分 KYC/KYB 第三方
- 外部签章或发票服务

## 10.2 建议不 Mock 的能力

- 数据库迁移
- outbox / Kafka
- 搜索索引同步
- 权限校验
- 审计留痕

原因：

- 这些是平台自身核心骨架，本地就必须真实演练

## 11. 本地联调检查清单

本地联调前至少确认：

- 数据库迁移执行通过
- `platform-core` 能连接 PostgreSQL / Redis / Kafka / MinIO / OpenSearch / Keycloak
- `portal-web / console-web` 能拿到登录态
- 商品创建、SKU 创建、订单创建接口可用
- 如需支付/回执联调，已先启用 `mocks` 或 `demo`，且 Mock PaymentProvider 支付成功和失败回调都可用
- 搜索同步至少跑通一次
- 审计联查接口可查到订单链路

## 13. 当前不建议本地抢跑的内容

当前阶段不建议在本地一开始就加入：

- `OPA` 正式运行时
- `SecretFlow / OpenFL / Kata Containers`
- 真实生产支付通道
- 多链/跨链生产能力
- 复杂外部数据空间连接器

## 15. 一句话结论

当前本地开发环境应优先围绕：

**`platform-core + PostgreSQL + Kafka + Redis + MinIO + OpenSearch + Keycloak`**
先跑通 `V1-Core` 主链路；支付/回执联调时再通过 `mocks` 子 profile 叠加 `Mock PaymentProvider`，之后逐步接入搜索同步、Fabric 测试网络和其他外围 Worker。
