# 统一错误码字典正式版

## 1. 文档定位

本文件用于冻结当前阶段平台统一错误码体系，服务于以下工作：

- 后端异常返回
- 前端错误提示与状态页
- OpenAPI 错误响应定义
- 回调与外部集成错误归一
- 测试断言
- 联调排障

本文件不替代业务规则，不替代日志，不替代审计。

## 2. 冻结原则

### 2.1 编码规则

统一采用：

`<DOMAIN>_<SCENARIO>_<RESULT>`

示例：

- `AUTH_INVALID_CREDENTIAL`
- `SKU_TYPE_INVALID`
- `PAYMENT_CORRIDOR_BLOCKED`

### 2.2 返回原则

- 所有业务错误必须返回稳定 `code`
- `message` 允许面向用户本地化
- `details` 允许携带结构化补充字段
- 禁止仅返回裸字符串错误

### 2.3 复用原则

- 同一错误语义在全平台只保留一个错误码
- 不允许不同模块用不同错误码表达同一冻结规则
- 不允许把日志文本直接冒充错误码

## 3. 通用错误码

| 错误码 | 含义 |
|---|---|
| `OK` | 成功 |
| `BAD_REQUEST` | 请求格式错误 |
| `UNAUTHORIZED` | 未认证 |
| `FORBIDDEN` | 已认证但无权限 |
| `NOT_FOUND` | 对象不存在 |
| `CONFLICT` | 状态冲突或资源冲突 |
| `TOO_MANY_REQUESTS` | 限流命中 |
| `INTERNAL_ERROR` | 未分类内部错误 |
| `SERVICE_UNAVAILABLE` | 下游或服务不可用 |
| `IDEMPOTENCY_KEY_REQUIRED` | 缺少幂等键 |
| `IDEMPOTENCY_CONFLICT` | 幂等键冲突 |
| `STEP_UP_REQUIRED` | 需要二次认证 |
| `STEP_UP_EXPIRED` | 二次认证已过期 |

## 17. 统一响应建议

错误响应统一为：

```json
{
  "code": "SKU_TYPE_INVALID",
  "message": "invalid sku_type",
  "request_id": "req_01J...",
  "details": {}
}
```

补充规则：

- `message` 可本地化
- `details` 必须结构化，不要塞长文本堆栈
- 原始异常和堆栈只进入日志与审计，不直接返回给前端

## 18. 当前不进入本文件的内容

当前不在本文件中展开：

- HTTP 状态码到业务错误码的最终映射矩阵
- 每个错误码的 UI 展示文案
- 多语言文案资源文件

这些内容可在前后端联调阶段继续细化。

## 19. 一句话结论

当前阶段应统一理解为：

**所有接口、回调、Worker 回写和联调断言都必须优先依赖稳定错误码，而不是依赖 message 文本或日志描述。**
