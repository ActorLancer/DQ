# 仓库拆分与目录结构建议

## 1. 文档定位

本文件用于冻结当前阶段代码仓库拆分策略、目录结构建议、共享契约位置和语言边界落位。

本文件服务于以下工作：

- 初始化代码仓库
- 统一目录结构
- 避免多语言代码混堆
- 固定共享契约与迁移脚本位置
- 为后续 CI、构建、联调做准备

本文件不替代服务边界文档，不替代部署文档。

上位文档：

- [服务清单与服务边界正式版.md](../开发准备/服务清单与服务边界正式版.md)
- [接口清单与OpenAPI-Schema冻结表.md](../开发准备/接口清单与OpenAPI-Schema冻结表.md)
- [事件模型与Topic清单正式版.md](../开发准备/事件模型与Topic清单正式版.md)
- [本地开发环境与中间件部署清单.md](../开发准备/本地开发环境与中间件部署清单.md)

## 2. 当前仓库策略结论

### 2.1 当前推荐策略

当前阶段推荐采用：

- **单代码仓 + 多应用目录**

原因：

- 当前是个人开发项目
- 主体仍是模块化单体，不是全面微服务
- 需要共享大量领域对象、接口契约、迁移脚本、测试数据和文档

### 2.2 当前不推荐策略

当前不推荐：

- 一开始就拆成多个独立 Git 仓库
- Rust、Go、Python、前端分别各自独立仓库
- 把数据库迁移脚本和接口契约散落到各服务目录中

## 3. 仓库顶层建议

建议最终代码仓根目录结构如下：

```text
repo-root/
  apps/
  packages/
  infra/
  scripts/
  docs/
  tests/
  tools/
  .github/
```

## 4. 顶层目录职责

## 4.1 `apps/`

存放可独立构建和运行的应用。

当前建议：

- `apps/platform-core`
- `apps/portal-web`
- `apps/console-web`
- `apps/fabric-adapter`
- `apps/fabric-event-listener`
- `apps/search-indexer`
- `apps/data-processing-worker`
- `apps/notification-worker`
- `apps/mock-payment-provider`

## 4.2 `packages/`

存放共享代码、共享契约和共享 schema。

当前建议：

- `packages/api-contracts`
- `packages/event-contracts`
- `packages/domain-types`
- `packages/test-fixtures`

## 4.3 `infra/`

存放基础设施编排和运行配置。

当前建议：

- `infra/docker`
- `infra/k8s`
- `infra/monitoring`
- `infra/fabric`
- `infra/keycloak`

## 4.4 `scripts/`

存放本地开发、迁移、初始化、联调辅助脚本。

## 4.5 `docs/`

存放正式开发文档的工程镜像版本。

建议放：

- 接口冻结文件
- 服务边界文件
- 环境部署文件
- ADR

## 4.6 `tests/`

存放跨应用、跨接口、端到端测试。

## 4.7 `tools/`

存放非主业务运行时工具，例如：

- 数据生成器
- schema 生成器
- 契约校验工具

## 5. `apps/` 目录建议

## 5.1 `apps/platform-core`

语言：

- `Rust`

建议结构：

```text
apps/platform-core/
  src/
    app/
    shared/
    modules/
      iam/
      party/
      access/
      catalog/
      contract_meta/
      listing/
      provider_ops/
      inquiry/
      order/
      contract/
      authorization/
      delivery/
      billing/
      dispute/
      audit/
      consistency/
      fairness/
      search/
      recommend/
      developer/
      ops/
      integration/
  migrations/
  config/
  tests/
```

冻结要求：

- 所有业务主状态只允许在 `platform-core` 内定义和推进
- 各模块内必须分：
  - `api`
  - `application`
  - `domain`
  - `infrastructure`

## 5.2 `apps/portal-web`

语言：

- `TypeScript`

建议结构：

```text
apps/portal-web/
  app/
  components/
  features/
  lib/
  tests/
```

## 5.3 `apps/console-web`

语言：

- `TypeScript`

建议结构与 `portal-web` 保持一致。

## 5.4 `apps/fabric-adapter`

语言：

- `Go`

职责：

- Gateway 调用
- 链码交互
- CA 管理接口

建议结构：

```text
apps/fabric-adapter/
  cmd/
  internal/
    api/
    service/
    fabric/
    config/
  tests/
```

## 5.5 `apps/fabric-event-listener`

语言：

- `Go`

职责：

- 监听 chaincode event
- 监听 commit 状态
- 输出 Kafka 或内部回写

## 5.6 `apps/search-indexer`

语言：

- 可优先用 `Rust`

职责：

- 消费搜索同步事件
- 写 `OpenSearch`
- 回写索引状态

## 5.7 `apps/data-processing-worker`

语言：

- `Python`

职责：

- 数据清洗
- 标准化
- 质量分析
- 报表与离线任务

建议结构：

```text
apps/data-processing-worker/
  worker/
  jobs/
  pipelines/
  tests/
```

## 5.8 `apps/notification-worker`

语言：

- `Rust` 或 `Go`

优先建议：

- 若复用 `platform-core` 契约方便，则用 `Rust`

## 5.9 `apps/mock-payment-provider`

语言：

- `Rust` 或 `Go`

优先建议：

- 若与支付编排契约共用较多，优先用 `Rust`

## 6. `packages/` 目录建议

## 6.1 `packages/api-contracts`

存放：

- OpenAPI 草案
- JSON Schema
- 错误码字典镜像
- 接口枚举定义

当前建议：

```text
packages/api-contracts/
  openapi/
    v1/
    v2/
    v3/
  json-schema/
  errors/
```

## 6.2 `packages/event-contracts`

存放：

- 事件 schema
- Topic 清单镜像
- 事件版本定义

当前建议：

```text
packages/event-contracts/
  topics/
  schemas/
    v1/
    v2/
    v3/
```

## 6.3 `packages/domain-types`

存放跨应用共享枚举和只读契约，例如：

- `sku_type`
- `rights_type`
- `delivery_mode`
- `pricing_mode`
- 生命周期对象状态枚举

冻结要求：

- `packages/domain-types` 只放契约，不放业务逻辑

## 6.4 `packages/test-fixtures`

存放：

- 测试数据
- Mock 事件
- 示例请求响应
- 标准 SKU 测试样本

## 7. 数据库与迁移脚本位置建议

当前建议：

- 数据库设计文档继续留在现有文档目录
- 真正执行用 SQL 迁移脚本在代码仓内放一份镜像

建议结构：

```text
apps/platform-core/migrations/
  v1/
  v2/
  v3/
```

补充规则：

- 文档目录中的 `数据库设计/` 仍是上位设计源
- 代码仓中的迁移目录是执行镜像版本
- 任何 SQL 变更必须同步更新两侧或建立明确单向生成机制

## 8. `infra/` 目录建议

建议结构：

```text
infra/
  docker/
    docker-compose.local.yml
    docker-compose.test.yml
  fabric/
  keycloak/
  monitoring/
  opensearch/
  k8s/
```

说明：

- 本地环境优先使用 `infra/docker`
- 后续测试/生产环境再进入 `infra/k8s`

## 9. `tests/` 目录建议

建议结构：

```text
tests/
  integration/
  e2e/
  api/
  event/
  fixtures/
```

说明：

- 模块内测试放各应用目录
- 跨应用测试放根目录 `tests/`

## 10. 文档与代码同步规则

以下文件变化后，必须同步更新代码仓镜像：

- 服务边界
- 接口冻结表
- 事件 Topic 清单
- 错误码字典
- 测试矩阵

建议做法：

- 在 `docs/engineering-freeze/` 下放镜像版

## 11. 当前不建议的目录做法

当前不建议：

- 在 `apps/platform-core` 里混放前端代码
- 在各语言目录里各自复制一份接口契约
- 在 `fabric-adapter` 中定义订单、合同、结算主状态
- 在 `data-processing-worker` 中定义交易主对象

## 12. 当前仓库演进路径

当前阶段：

- 单仓多应用

后续若出现明确瓶颈，可优先独立拆出：

1. `search-indexer`
2. `recommend`
3. `audit`
4. `billing`
5. `fabric-*`

最晚拆：

- `order`
- `contract`
- `authorization`
- `delivery`

## 13. 一句话结论

当前阶段最稳妥的代码组织方式是：

**单仓多应用，`platform-core` 作为唯一业务主应用，前端、链适配、索引同步、数据处理、通知和 Mock 支付作为独立应用目录存在；共享契约、事件 schema、错误码和测试样本统一收敛到 `packages/`。**
