## 2. 测试冻结原则

### 2.1 当前最小测试目标

当前测试必须至少证明：

- `V1-Core` 可完成首笔标准交易闭环
- 八个标准 SKU 的主链路可独立跑通
- 生命周期对象可独立落库、联查、审计
- 支付、链确认、搜索、审计和一致性副作用不会破坏主状态

### 2.2 当前测试分层

统一按以下层次组织：

1. 单元测试
2. 模块集成测试
3. 主链路联调测试
4. 异步与一致性测试
5. 权限与安全测试
6. 运维与恢复测试

### 2.3 当前不允许缺失的测试面

以下测试面不得省略：

- 订单主状态机
- `Authorization / Delivery / Settlement / Dispute`
- 支付回调幂等
- outbox / dead letter / replay
- 搜索投影同步
- 审计包导出
- step-up 高风险动作

## 3. 测试环境分层

## 3.1 本地开发测试

覆盖：

- 单元测试
- 模块集成测试
- Mock PaymentProvider

## 3.2 联调环境测试

## 3.3 冻结验收环境测试

覆盖：

- 关键异常分支
- 一致性修复
- 回放与审计导出

## 16. 上线前最低通过标准

上线前至少必须满足：

- 八个标准 SKU 主链路全部跑通
- 支付回调幂等通过
- `Authorization / Delivery / Settlement / Dispute` 四类对象均可独立联查
- 搜索同步、dead letter、replay、reconcile 至少各通过一轮正反向用例
- 审计导出、step-up、权限阻断均通过

## 17. 一句话结论

当前阶段测试必须围绕：

**八个标准 SKU、七个生命周期对象、支付幂等、事件一致性、搜索投影、审计回放和高风险权限控制**  
建立最小但完整的自动化与联调测试矩阵。
