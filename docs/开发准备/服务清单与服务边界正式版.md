## 2. 架构冻结结论

### 2.1 当前正式架构风格

当前正式采用：

- 前后端分离
- `Rust` 模块化单体主应用
- 少量外围适配进程与 Worker
- 独立基础设施

当前不采用：

- 全面微服务
- 多个业务主服务同时持有交易主状态
- 多门语言重复实现同一条交易主链路

### 2.2 当前正式进程清单

## 3. 服务边界总原则

### 3.1 主状态原则

- `PostgreSQL` 是业务运营主状态权威。
- `Fabric` 是可信确认与摘要证明层，不是订单主状态机。
- `Kafka` 是事件总线，不是业务权威。
- `OpenSearch` 是搜索读模型，不是目录主数据源。
- `Redis` 是缓存与短期辅助状态，不是业务真相源。

### 3.2 生命周期对象原则

以下对象是跨版本正式生命周期主对象：

- `Order`
- `Digital Contract`
- `Authorization`
- `Delivery`
- `Settlement`
- `Dispute`
- `Billing Event`

任何模块、进程、Worker 或外部适配器都不得绕开这组对象另建第二套主生命周期。

### 3.3 事务原则

在 `platform-core` 内，关键业务动作必须在同一业务事务内同时写入：

- 主业务对象
- 审计事件
- outbox 事件

不允许只写主对象不写审计，不允许只发事件不落主对象。

### 3.4 适配器原则

外围进程可以：

- 提供外部系统适配
- 执行长任务
- 消费事件并回写外部事实

外围进程不得：

- 直接定义订单主状态
- 直接定义合同生效
- 直接定义授权发放结果
- 直接定义结算完成结果

## 5. 主应用 `platform-core`

## 5.1 定位

`platform-core` 是当前阶段唯一业务主应用，使用 `Rust` 实现，承载全部业务主状态和主编排。

其职责包括：

- 目录与商品管理
- 交易编排
- 合同、授权、交付、验收、结算、争议
- 权限判断
- 支付编排
- 审计与一致性编排
- 搜索查询编排
- 推荐查询编排
- 开发者调试能力编排

## 7. 调用边界与集成规则

## 7.1 同步调用

同步调用仅适用于：

- 前端请求 `platform-core`
- `platform-core` 调用 `Keycloak`
- `platform-core` 调用外围适配器的管理接口

禁止：

- 前端直连 `Kafka`
- 前端直连 `OpenSearch`
- 外围进程之间相互形成业务主链路调用

## 7.2 异步调用

以下能力必须优先异步：

- 搜索索引同步
- 推荐行为回流
- 通知
- 审计扩展动作
- 链上摘要提交后的确认回写
- 重试、对账、回放

## 7.3 数据访问规则

- 模块只能通过自身仓储访问自身主表。
- 跨模块读取优先走应用服务或只读投影。
- 不允许跨模块直接写对方主表。
- SQL 视图、投影表、搜索文档都不能替代主对象表。

## 8. 状态归属冻结

## 8.1 主对象与状态归属

| 对象 | 主归属模块 | 主状态说明 |
|---|---|---|
| `Order` | `order` | `current_state` 为默认业务主状态 |
| `Digital Contract` | `contract` | 合同生效、签署、冻结、失效状态 |
| `Authorization` | `authorization` | 授权发放、到期、断权、撤权状态 |
| `Delivery` | `delivery` | 交付准备、开通、完成、失败、撤回状态 |
| `Settlement` | `billing` | 结算、分账、打款、退款、赔付、对账状态 |
| `Dispute` | `dispute` | 争议受理、处理中、裁决、关闭状态 |
| `Billing Event` | `billing` | 事件型计费记录与计量状态 |

## 8.2 一致性状态字段族

以下字段是正式一致性状态族：

- `authority_model`
- `proof_commit_policy`
- `proof_commit_state`
- `external_fact_status`
- `reconcile_status`
- `idempotency_key`
- `request_id`
- `trace_id`
- `status_version`

适用对象：

- `Order`
- `Digital Contract`
- `Authorization`
- `Delivery`
- `Settlement`
- `Dispute`
- `Billing Event`

## 8.3 外围进程回写边界

外围进程允许回写：

- `proof_commit_state`
- `external_fact_status`
- `reconcile_status`
- 技术回执记录

外围进程不得直接回写：

- `Order.current_state`
- 合同生效主状态
- 授权主状态最终裁决
- 争议裁决主状态

## 9. 数据库归属建议

## 9.1 schema 归属建议

建议按下列方向归属：

- `identity / access / party`：`iam + party + access`
- `catalog / listing / contract_meta`：商品与供给域
- `trade / contract / authorization / delivery`：交易主链路域
- `billing / dispute`：结算与争议域
- `audit / consistency / ops`：审计、一致性与运维域
- `search / recommend`：投影与发现域
- `integration`：适配、外部回执、第三方引用记录

## 9.2 不允许的归属方式

- 把 `Authorization / Delivery / Settlement / Dispute` 全塞进 `Order` 附表而不保留正式主对象
- 把 `Billing Event` 当成 `Settlement` 替代物
- 把搜索投影表当成商品真相源
- 把审计事件表当成业务状态表

## 11. 后续拆服务路径

当前不拆微服务，但未来若出现明显瓶颈，优先考虑拆出：

1. `search`
2. `recommend`
3. `audit`
4. `billing`
5. `fabric integration`
6. `data-processing`

最后才考虑拆出：

- `order`
- `contract`
- `authorization`
- `delivery`

原因是这四个模块当前共同承载交易核心状态机，过早拆出会直接放大一致性成本。

## 12. 一句话结论

当前阶段的正式工程边界应理解为：

**以 `platform-core` 作为唯一业务主应用，在其中冻结模块边界和生命周期对象；将 Fabric 交互、链事件监听、索引同步、数据处理、通知和 Mock 支付作为外围进程隔离出去；以 `PostgreSQL + Kafka + MinIO + OpenSearch + Redis + Keycloak + Fabric + 可观测性栈` 作为统一底座推进 `V1` 开发。**
