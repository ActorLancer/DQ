## 11.2 事件驱动一致性

所有关键业务操作均采用“先写链上权威状态，再由事件同步器驱动数据库投影”的思路。例如，订单创建由应用服务调用 OrderEscrow 创建链上订单；链返回 tx_hash；事件同步器监听 OrderCreated 事件后更新 order_main 状态为 created。这样能够避免链下先写成功、链上写失败导致的双写不一致。

**一致性设计要点**

| **要点** | **说明**                                                          |
|----------|-------------------------------------------------------------------|
| 幂等键   | 每次链调用都必须带 request_id，数据库投影也要以 request_id 防重。 |
| 事件回放 | 事件同步器应支持按 block range 补拉，避免漏事件。                 |
| 补偿事务 | 链失败时回滚链下临时状态，链成功但投影失败时通过重放恢复。        |
| 对账任务 | 定时扫描数据库状态与链状态差异，并生成对账报告。                  |

## 11.3 状态机细则

**订单状态迁移约束**

| **当前状态**   | **允许操作**       | **下一状态**                  |
|----------------|--------------------|-------------------------------|
| created        | 买方锁定资金       | buyer_locked                  |
| buyer_locked   | 卖方提交交付承诺   | seller_delivering / delivered |
| delivered      | 买方确认收货       | accepted                      |
| delivered      | 买方拒收/申诉      | rejected / dispute_opened     |
| accepted       | 结算               | settled                       |
| dispute_opened | 自动裁决或人工裁决 | settled / closed              |
| 任意可超时状态 | 超时关闭           | expired / closed              |
