# 13. 安全、合规、审计与应急响应设计

## 13.1 安全控制基线

**安全控制矩阵**

| **控制域**     | **最低要求**                             | **实施说明**               |
|----------------|------------------------------------------|----------------------------|
| 身份认证       | 多因素认证、组织实名、服务到服务鉴权     | 高风险操作必须二次校验。   |
| 数据保护       | 传输加密、对象加密、密钥轮换、脱敏       | 默认不传明文，不存明文。   |
| 链安全         | 成员准入、证书吊销、链码审计、背书策略   | 严禁直接在生产链调试合约。 |
| 主机与容器安全 | 镜像扫描、最小镜像、补丁管理、运行时防护 | 链节点和网关优先加固。     |
| 日志审计       | 统一结构化审计、不可篡改存储、长期归档   | 证据保留策略需制度化。     |
| 漏洞与应急     | SAST/DAST、依赖扫描、演练和响应流程      | 上线前完成安全基线检查。   |

## 13.2 合规策略设计

合规设计不应写死为单一法域规则，而应通过“数据分类分级 + 策略模板 + 审计保留 + 监管接口”的方式参数化实现。最基本的合规能力包括：数据分类分级、敏感数据标记、使用目的限制、授权时限、导出限制、证据保留、删除与失效策略、监管审计留存。

**推荐合规模型字段**

| **字段**          | **说明**         | **示例**                                          |
|-------------------|------------------|---------------------------------------------------|
| sensitivity_level | 数据敏感等级     | public / internal / restricted / highly_sensitive |
| origin_region     | 数据来源地域     | CN / EU / SG                                      |
| allowed_region    | 允许使用地域     | CN only                                           |
| usage_purpose     | 允许用途         | analysis, internal_use_only                       |
| retention_days    | 保留天数         | 365                                               |
| exportable        | 是否允许导出结果 | false                                             |
| regulatory_tags   | 监管标签         | financial, pii, health                            |

## 13.3 审计证据链

审计证据链必须覆盖从目录到交易再到争议、任务、跨链和风控的全流程。推荐每个关键操作都生成统一的 audit_event，同时将关键证据对象的哈希写链。证据链最少应包含：操作主体、对象标识、时间戳、客户端指纹、请求摘要、结果摘要、链上 tx_hash、证据对象哈希。

- 证据对象包括：交付回执、下载日志、申诉材料、人工裁决单、训练轮次摘要、跨链回执。

- 审计日志应与业务日志物理隔离，权限隔离，且具备保留策略。

- 任何覆盖式更新都需要先保留旧版本摘要，防止审计断链。

## 13.4 应急响应流程

19. 告警触发：来自监控、风控、监管通知或人工发现。

20. 分级处置：按 P1/P2/P3 级别确定响应时限和值班角色。

21. 隔离影响：必要时冻结主体、商品、任务或下载令牌，阻断进一步扩散。

22. 证据保全：立即锁定相关 audit_event、对象版本、链上区块高度和网关日志。

23. 根因分析：由研发、安全、运维和业务联合复盘，形成纠正措施。

24. 恢复与通知：验证恢复后逐步解除冻结，向受影响方和监管方通知。

25. 复盘改进：更新规则、监控、合约或流程，并纳入回归测试。
