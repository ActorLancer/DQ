# 14. 部署架构、容量规划与持续交付

## 14.1 环境规划

**环境建议**

| **环境** | **用途**                     | **说明**                                 |
|----------|------------------------------|------------------------------------------|
| Dev      | 单人/小组开发联调            | 可采用简化链拓扑和小规模对象存储。       |
| Test     | 接口、合约、集成测试         | 覆盖链节点、数据库、对象存储、消息总线。 |
| Pre-Prod | 灰度演练、性能测试、升级演练 | 拓扑尽量接近生产。                       |
| Prod     | 正式运行                     | 多可用区、多节点、审计隔离。             |

## 14.2 生产拓扑建议

- 区块链网络：至少 3 个组织、每组织至少 2 个 peer 或等价节点；排序服务建议 3 节点及以上；证书服务高可用。

- 业务服务：API 网关、目录、订单、交付、争议、信誉、审计、风控等服务至少双实例部署。

- 数据库：采用主备或多副本方案，并配置只读副本承载报表查询。

- 对象存储：至少开启多副本和版本化，重要证据对象跨可用区复制。

- 消息总线：部署集群并启用持久化和死信队列。

- 监控与日志：独立部署监控、日志检索与告警系统，避免与业务同一故障域。

## 14.3 容量与性能规划（建议基线）

**容量规划示例**

| **维度**      | **Phase 1**    | **Phase 2**             | **Phase 3**                   |
|---------------|----------------|-------------------------|-------------------------------|
| 日订单量      | 1,000 ~ 10,000 | 5,000 ~ 20,000          | 10,000+                       |
| 对象存储增长  | 按数据量主导   | 数据 + 模型文件同步增长 | 数据 + 模型 + 证据 + 跨链回执 |
| 链上 TPS 需求 | 中低           | 中等                    | 中等偏高（含跨链与风控事件）  |
| 异步任务压力  | 中             | 高（训练任务）          | 高（图分析、跨链、监管）      |

容量规划时要牢记：本平台性能瓶颈大概率不在链本身，而在对象存储吞吐、网关下载带宽、联邦训练通信、风控计算和审计写入。链上只承担控制流，链下承担数据流和计算流。

## 14.4 持续交付与版本治理

- 后端服务使用 CI/CD 自动构建、自动测试、镜像签名和安全扫描。

- 智能合约版本必须遵循兼容性策略：新增字段优先向后兼容，破坏性变更通过新合约地址或新链码版本实现。

- 数据库变更采用迁移脚本管理，不允许手工改表上线。

- 对象元数据 schema 需版本化，保证旧商品与新商品可同时运行。
