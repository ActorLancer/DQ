# 3. 信任模型、威胁模型与总体技术路线

## 3.1 信任模型

本系统采用“弱信任 + 可验证约束”模型。平台不假设买方、卖方、运营方天然可信，而是通过链上状态、签名、哈希、保证金、评价与审计共同建立交易信任。系统信任边界分为三层：链上共识层、链下服务层和组织边界层。链上共识层以联盟链共识和证书体系为基本信任；链下服务层通过服务鉴权、KMS 和日志确保操作可追踪；组织边界层通过企业实名、证书签发、合同与监管规则约束。

**信任边界**

| **边界**       | **默认信任程度** | **控制手段**                                           |
|----------------|------------------|--------------------------------------------------------|
| 链上共识与账本 | 高               | 联盟链成员准入、证书、共识、合约版本治理。             |
| 平台后端服务   | 中               | 服务鉴权、审计日志、双人审批、代码发布控制。           |
| 买卖双方节点   | 低到中           | 实名认证、签名、保证金、信誉、限权与速率控制。         |
| 链下存储与传输 | 低到中           | 对象加密、哈希校验、一次性密钥、下载令牌、有效期控制。 |
| 跨链与预言机   | 低               | 多签确认、消息签名、重放保护、仲裁回执。               |

## 3.2 威胁模型

**主要威胁与缓解措施**

| **威胁**     | **表现**                                       | **缓解策略**                                                   |
|--------------|------------------------------------------------|----------------------------------------------------------------|
| 恶意卖方     | 上传伪造数据、交付与描述不一致、拖延交付。     | 样本哈希、版本签名、保证金、超时处罚、信誉扣减、第三方抽检。   |
| 恶意买方     | 下载后拒付、恶意拒收、虚假投诉、刷差评。       | 支付锁定、下载凭证、日志证据、买方保证金、加权评分与限诉机制。 |
| 平台内部滥用 | 擅自访问数据、篡改订单状态、泄露密钥。         | 零信任访问、最小权限、KMS 托管、敏感操作双审、WORM 审计。      |
| 链下文件泄露 | 下载链接被转发、对象存储暴露、密文被离线分析。 | 短期令牌、一次性密钥封装、客户端解密、水印与访问日志。         |
| 模型投毒     | 联邦学习节点上传恶意更新、拖垮收敛。           | 节点信誉、异常梯度检测、鲁棒聚合、贡献奖励与惩罚。             |
| 跨链攻击     | 重放、中间人、伪造回执、桥接节点串谋。         | 消息 nonce、时间戳、椭圆曲线签名、多签确认、回执上链。         |
| 风控绕过     | 串谋评分、洗订单、频繁转售、异常下载。         | 图分析、规则引擎、阈值告警、冻结与人工复核。                   |

## 3.3 总体技术路线

总体路线采用“许可链 + 业务微服务 + 对象存储 + 审计与风控 + 可插拔隐私计算/联邦学习”的组合架构。链上层负责高可信状态、合约执行和审计凭证；业务服务层负责目录、订单、支付、争议、评分、风控、监管视图和工作流；数据层负责对象存储、关系数据库、缓存、搜索索引和日志仓；二期引入联邦学习协调器与策略引擎；三期引入跨链网关、预言机、图分析引擎和监管节点。

| **链选型建议：**Phase 1 与 Phase 2 推荐优先使用联盟链平台（如 Hyperledger Fabric 或 FISCO BCOS 一类的许可链框架），原因是其更容易满足成员准入、隐私通道、证书管理、可审计和合约治理要求。最终选型应结合团队语言栈、合约模型、运维经验和生态工具评估。 |
