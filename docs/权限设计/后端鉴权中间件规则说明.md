# 数据交易平台后端鉴权中间件规则说明

## 1. 文档说明

本说明定义后端统一鉴权中间件的职责、处理顺序和返回规范。  
目标是让：

- 网关
- BFF
- 业务服务
- 异步任务消费者

都使用同一套鉴权判断逻辑。

## 2. 鉴权链路总原则

每个请求都必须按以下顺序判断：

1. 身份解析
2. 主体状态校验
3. 基础权限校验
4. 作用域校验
5. 业务对象状态校验
6. 合同/策略校验
7. 环境/网络/配额校验
8. 合规/风控阻断校验
9. 高风险附加校验
10. 审计记录

## 3. 中间件分层

建议分 6 层：

### 3.1 Identity Middleware

职责：

- 解析用户令牌、应用令牌、连接器证书、服务签名
- 提取：
  - `actor_type`
  - `actor_id`
  - `tenant_id`
  - `role_set`
  - `app_id`
  - `cert_sn`

失败返回：

- `AUTH_INVALID_TOKEN`
- `AUTH_CERT_REVOKED`
- `AUTH_SIGNATURE_INVALID`

### 3.2 Subject Status Middleware

职责：

- 校验 Tenant/Organization 状态
- 校验 User / Application / Connector / Environment 是否有效

失败返回：

- `AUTH_TENANT_DISABLED`
- `AUTH_USER_SUSPENDED`
- `AUTH_APP_REVOKED`
- `AUTH_CONNECTOR_DISABLED`

### 3.3 Permission Middleware

职责：

- 根据接口绑定的权限点做基础判定
- 支持单权限或多权限任一满足

失败返回：

- `PERM_DENIED`

### 3.4 Scope Middleware

职责：

- 校验作用域是否覆盖目标对象

示例：

- 订单接口要求 `tenant + order`
- 商品接口要求 `tenant + product`
- 资产对象接口要求 `tenant + asset_version`
- 版本发布接口要求 `tenant + asset`
- 应用接口要求 `tenant + application`
- 监管接口要求 `regulator_scope + object`

失败返回：

- `PERM_SCOPE_MISMATCH`

### 3.5 Policy Middleware

职责：

- 校验合同、Usage Policy、时间窗、地域、导出限制、输出边界
- 校验共享授权、订阅策略、模板白名单和结果边界
- 校验数据契约是否存在、是否与订单快照一致、是否满足验收口径和责任边界

失败返回：

- `POLICY_NOT_EFFECTIVE`
- `POLICY_REGION_BLOCKED`
- `POLICY_SHARE_NOT_ALLOWED`
- `POLICY_SUBSCRIPTION_NOT_ALLOWED`
- `POLICY_TEMPLATE_NOT_ALLOWED`
- `POLICY_OUTPUT_FORBIDDEN`
- `POLICY_USAGE_FORBIDDEN`
- `POLICY_DATA_CONTRACT_MISSING`
- `POLICY_DATA_CONTRACT_MISMATCH`

### 3.6 Risk Guard Middleware

职责：

- 校验风控阻断、冻结状态、黑名单、监管限制
- 校验 recipient / subscriber / template caller 是否仍处于允许状态

失败返回：

- `RISK_SUBJECT_FROZEN`
- `RISK_PRODUCT_BLOCKED`
- `RISK_SHARE_RECIPIENT_BLOCKED`
- `RISK_SUBSCRIPTION_BLOCKED`
- `RISK_REQUEST_BLOCKED`
- `RISK_EXPORT_BLOCKED`

## 4. 高风险动作附加校验

以下动作进入 `HighRiskGuard`：

- 合规阻断
- 风险冻结/解冻
- 退款/赔付
- 算法白名单放行
- 结果导出
- 分润执行/冻结
- 凭证吊销
- 跨链终止/补偿
- 监管材料导出
- 互认关系变更
- 搜索别名切换
- 搜索全量重建
- 共享对象撤销
- 模板查询授权变更
- 订阅规则变更

附加校验项：

- 二次认证状态
- 最近一次 MFA 时间
- 双人复核标记
- 审批票据/治理票据

失败返回：

- `AUTH_MFA_REQUIRED`
- `PERM_SECOND_REVIEW_REQUIRED`
- `GOV_TICKET_REQUIRED`

## 4.1 搜索请求专项规则

- 搜索引擎结果只用于召回与排序，不直接作为最终业务可见性事实
- 搜索接口返回前应再次校验 PostgreSQL 中的商品状态、卖方状态和风险状态
- 搜索运维接口属于平台运维高风险接口，必须强审计
- OpenSearch 或 Redis 异常时允许切换 PostgreSQL fallback 搜索，但不得跳过权限和可见性校验

## 4.2 数据原样处理与产品化加工专项规则

- 原始接入批次、对象清单、格式识别结果、抽取任务和预览工件必须绑定同一资产归属链。
- 当请求涉及原始对象或预览工件写操作时，必须额外校验对象哈希、存储指针、遮罩策略和可编辑状态。
- 平台不得把“存在原始对象引用”视为“允许平台自由读取明文”，任何解密、抽取、预览生成都必须走责任链和审计。

## 4.3 数据商品元信息与数据契约专项规则

- 商品详情、下单、验收、争议和审计接口都必须引用同一份数据契约快照。
- 当请求涉及字段结构、质量报告或加工责任链写操作时，必须额外校验资产版本归属和版本可编辑状态。
- 数据契约发布前必须校验元信息档案、质量报告和字段结构摘要已经齐备。

## 4.4 数据商品分层存储专项规则

- `raw` 区对象不得直接作为正式商品交付对象。
- 任何对外可售对象都必须落在正式交付区或受控执行入口，并绑定存储策略与生命周期策略。
- 预览对象必须位于 `preview` 区或等价受控预览路径，不得直接引用原始对象地址。

## 5. 异步任务鉴权

### 5.1 任务生产

任务入队时必须携带：

- `request_id`
- `actor_id`
- `actor_type`
- `tenant_id`
- `permission_code`
- `scope_snapshot`

### 5.2 任务消费

消费者不得默认信任队列消息，仍需校验：

- 服务身份是否合法
- 作用域快照是否仍有效
- 对象状态是否允许继续执行

适用场景：

- 公链锚定
- 分润执行
- 跨链补偿
- 风控冻结同步
- 联邦聚合
- 版本发布与订阅分发
- 只读共享开通 / revoke
- 模板查询任务执行

## 6. 机器身份鉴权规则

### 6.1 Application

- 校验应用凭证
- 校验应用归属 Tenant
- 校验绑定的 SKU / API / 模型 / 模板查询范围
- 校验 IP / 来源 / 配额

### 6.2 Connector

- 校验证书与签名
- 校验连接器版本与状态
- 校验绑定环境与资源范围

### 6.3 Cross-Chain Gateway / Witness

- 校验网关或见证服务独立身份
- 校验签名来源与轮换状态
- 不得使用平台管理员口令替代

## 7. 审计要求

所有鉴权结果都建议写审计，但至少以下必须记录：

- 鉴权成功的高风险动作
- 鉴权失败的高风险动作
- 监管查询
- 跨链治理动作
- 凭证签发/吊销
- 分润执行/冻结

建议审计字段：

- `request_id`
- `actor_id`
- `permission_code`
- `scope_snapshot`
- `object_type`
- `object_id`
- `result_code`
- `risk_flags`

## 8. 错误码返回规范

### 8.1 HTTP 建议

- `401`：身份无效
- `403`：权限不足或策略阻断
- `409`：状态冲突
- `423`：对象被冻结

### 8.2 业务错误码分组

- `AUTH_*`
- `PERM_*`
- `POLICY_*`
- `RISK_*`
- `GOV_*`

## 9. 接口注册建议

建议每个接口在代码中通过元数据声明：

```text
route
permission_codes
scope_type
high_risk
audit_required
policy_required
risk_guard_required
```

## 10. 数据对象交付专项补充

### 10.1 只读共享接口

- 鉴权顺序应为：身份 -> 主体状态 -> `delivery.share.enable/read` -> 订单作用域 -> 共享对象归属 -> recipient 合法性 -> 共享协议与到期策略 -> 风控 -> 审计
- 不得仅凭订单状态放行，必须再次校验 `asset_object_binding.object_kind = share_object`

### 10.2 版本订阅接口

- 鉴权顺序应为：身份 -> 主体状态 -> `delivery.subscription.manage/read` -> 订单作用域 -> 订阅型 SKU 校验 -> 版本发布策略校验 -> 周期与历史 revision 访问范围校验 -> 审计
- 周期更新任务消费时，必须带 `revision_subscription_id` 与 `request_id`

### 10.3 模板查询接口

- 鉴权顺序应为：身份 -> 主体状态 -> `delivery.template_query.enable/use` -> 订单作用域 -> 模板白名单 -> 参数校验 -> 输出边界 -> 风控 -> 审计
- 模板执行成功不等于可导出，导出仍需经过结果边界与最小披露校验

### 10.3A 查询面与查询运行接口

- 查询面配置接口应按：身份 -> 主体状态 -> `delivery.query_surface.manage` -> 商品/资产作用域 -> 读取分层区域合法性 -> 执行环境合法性 -> 输出边界校验 -> 审计。
- 查询模板发布接口应按：身份 -> 主体状态 -> `delivery.query_template.manage` -> 查询面作用域 -> 参数 schema 校验 -> analysis rule 校验 -> 导出策略校验 -> 审计。
- 查询运行记录接口应按：身份 -> 主体状态 -> `delivery.query_run.read` -> 订单或授权作用域 -> 最小披露 -> 结果边界再校验 -> 审计。

### 10.3B SKU 配置与模板绑定接口

- SKU 创建/更新接口应按：身份 -> 主体状态 -> `catalog.sku.create/update` -> 商品作用域 -> 标准 SKU 枚举校验 -> `sku_code` 唯一性校验 -> `trade_mode` 合法性校验 -> 模板兼容性预校验 -> 审计。
- `sku_type` 必须识别为标准 SKU 真值，不得把商业套餐编码写入 `sku_type`。
- `FILE_SUB` 若显式绑定授权模板，只允许命中 `LICENSE_FILE_USE_V1`；`SBX_STD / RPT_STD` 若显式绑定授权模板，只允许命中 `LICENSE_SANDBOX_USE_V1 / LICENSE_RESULT_USE_V1`。
- `API_SUB / API_PPU` 只能绑定 `SUB / PPU` 细分模板族，不得回退到未区分计费模式的通用 `API` 模板名。
- `SHARE_RO` 接口必须额外校验共享协议、recipient 范围、撤权策略。
- `QRY_LITE` 接口必须额外校验模板白名单、参数 schema、输出边界。
- `SBX_STD` 接口必须额外校验执行环境、会话策略、导出规则，不得误落到模板查询授权路径。

### 10.3C 生命周期对象摘要接口

- 若接口返回授权、交付、结算或争议摘要，鉴权链除订单作用域外，还必须校验对象归属与最小披露。
- 摘要字段必须使用对象化命名，例如 `authorization_status / delivery_status / settlement_status / dispute_status`，不得返回无语义上下文的裸 `status`。
- 若返回一致性信息，字段名必须使用统一状态字段族：`authority_model / proof_commit_policy / proof_commit_state / external_fact_status / reconcile_status / status_version`。

示例：

```text
POST /api/v2/profitshare/rewards/{id}/execute
permission_codes = ["profitshare.reward.execute"]
scope_type = "reward"
high_risk = true
audit_required = true
policy_required = false
risk_guard_required = true
```

## 10. 一句话结论

后端鉴权中间件的核心不是“检查有没有某个角色”，而是把身份、权限、作用域、策略、风控和审计连成统一链路。只有这样，前端菜单、按钮、接口和异步任务的权限语义才能一致。

## 11. 存储与信任边界扩展规则

### 11.1 新增校验点

所有涉及数据本体、模型文件、结果导出和跨平台托管的请求，除现有鉴权链外，还必须校验：

- 当前 `storage_mode`
- 当前 `payload_location_type`
- 平台是否允许触达明文
- 是否存在有效连接器/API 绑定
- 是否存在有效信任证据
- 当前导出是否超出 `result_export_policy`

### 11.2 建议错误码

- `TRUST_STORAGE_MODE_MISMATCH`
- `TRUST_PLATFORM_PLAINTEXT_FORBIDDEN`
- `TRUST_CONNECTOR_NOT_READY`
- `TRUST_EVIDENCE_MISSING`
- `TRUST_EXPORT_BOUNDARY_BLOCKED`

## 12. 支付、资金流与轻结算扩展规则

### 12.1 新增校验点

- `payment_intent` 必须校验订单状态、费用快照、渠道能力和幂等键。
- `payment_intent` 必须校验付款司法辖区、收款司法辖区、起步司法辖区和命中的走廊策略。
- `webhook` 必须校验来源、签名、事件唯一性和重复回调。
- `payout` 必须校验结算状态、风控结果、二次确认和审计要求。
- `mock payment` 只允许在非 production 环境执行。

### 12.2 建议错误码

- `PAYMENT_PROVIDER_NOT_AVAILABLE`
- `PAYMENT_JURISDICTION_UNSUPPORTED`
- `PAYMENT_CORRIDOR_BLOCKED`
- `PAYMENT_PAYOUT_PREFERENCE_INVALID`
- `PAYMENT_SIGNATURE_INVALID`
- `PAYMENT_DUPLICATE_WEBHOOK`
- `PAYMENT_STATUS_MISMATCH`
- `PAYMENT_RECONCILIATION_DIFF_OPEN`
- `PAYMENT_MOCK_FORBIDDEN_IN_PRODUCTION`

## 13. 身份认证、注册登录与会话管理扩展规则

### 13.1 认证链路

后端鉴权必须额外支持：

- 本地登录凭证校验
- 企业 OIDC / SAML 外部身份校验
- `MFA` / `step-up` challenge 校验
- `user_session` / `refresh_token_family` / `trusted_device` 状态校验
- Fabric 身份、证书状态与平台主体绑定关系校验

### 13.2 高风险动作放行条件

以下动作除权限外还必须校验最近一次强认证时间：

- 退款、赔付、人工打款
- 走廊策略、SSO 配置、角色边界修改
- Fabric 身份签发、证书吊销
- 高敏下载、训练放行、跨域导出

### 13.3 建议错误码

- `AUTH_SESSION_REVOKED`
- `AUTH_DEVICE_REVOKED`
- `AUTH_STEP_UP_REQUIRED`
- `AUTH_STEP_UP_EXPIRED`
- `AUTH_SSO_CONNECTION_INVALID`
- `AUTH_FABRIC_IDENTITY_NOT_BOUND`
- `AUTH_CERTIFICATE_REVOKED`
- `AUTH_CERTIFICATE_EXPIRED`

## 14. 审计、证据链与回放扩展规则

### 14.1 新增校验点

- `audit.trace` 只能返回摘要视图，除非显式命中原文读取权限
- `audit.raw.read`、`audit.unmasked_view.read`、`audit.package.export`、`audit.replay.execute`、`audit.legal_hold.manage` 必须校验 step-up
- 回放接口默认只允许 `dry_run = true`
- `break-glass` 必须校验理由、审批状态和事后复核标记
- 监管导出必须校验脱敏级别与导出范围

### 14.2 审计访问再审计

所有以下动作都必须二次写入审计链：

- 查看原始证据
- 查看未脱敏视图
- 导出证据包
- 发起回放
- 创建/解除 `legal hold`
- 执行 `break-glass`

### 14.3 建议错误码

- `AUDIT_RAW_VIEW_FORBIDDEN`
- `AUDIT_UNMASKED_VIEW_REQUIRES_STEP_UP`
- `AUDIT_REPLAY_DRY_RUN_ONLY`
- `AUDIT_LEGAL_HOLD_ACTIVE`
- `AUDIT_BREAKGLASS_REASON_REQUIRED`
- `AUDIT_EXPORT_SCOPE_FORBIDDEN`

## 15. 双层权威模型与一致性扩展规则

### 15.1 基本规则

- 后端不得把 Kafka 消息、Fabric 提交结果或外部回执中的任一者单独当作全局唯一主状态。
- 业务接口默认读取数据库业务状态。
- 对外证明接口必须同时展示 `proof_commit_state`。
- 外部事实型对象必须校验 `external_fact_status` 与来源回执。

### 15.2 高风险动作

以下动作除权限外还必须校验 step-up 与审计：

- `ops.consistency.reconcile`
- `ops.dead_letter.reprocess`
- 任何会改写 `reconcile_status`
- 任何会触发补偿重放或人工覆盖确认的动作

### 15.3 建议错误码

- `CONSISTENCY_MISMATCH_DETECTED`
- `CONSISTENCY_REPLAY_DRY_RUN_ONLY`
- `CONSISTENCY_EXTERNAL_FACT_NOT_CONFIRMED`
- `CONSISTENCY_PROOF_NOT_CONFIRMED`
- `OUTBOX_EVENT_LOCKED`
- `DEAD_LETTER_REPROCESS_FORBIDDEN`

## 16. 推荐与个性化发现补充规则

### 16.1 基本规则

- 推荐服务可以读取搜索投影、行为聚合、画像快照和候选召回结果，但不能绕过 PostgreSQL 最终业务校验。
- 外部推荐引擎或开源中间件只允许作为候选召回或排序增强器，不得被当作主状态权威。
- 推荐请求、推荐结果、解释码、曝光和点击必须进入审计链。

### 16.2 高风险动作

以下动作除权限外还必须校验 step-up 与审计：

- `ops.recommendation.manage`
- `ops.recommend_rebuild.execute`
- `ops.recommend_model.manage`
- `ops.recommend_experiment.manage`
- `ops.recommend_page_optimize.manage`

### 16.3 建议错误码

- `RECOMMENDATION_NOT_ALLOWED`
- `RECOMMENDATION_RESULT_STALE`
- `RECOMMENDATION_CANDIDATE_BLOCKED`
- `RECOMMENDATION_PROFILE_DISABLED`
- `RECOMMENDATION_REBUILD_FORBIDDEN`

## 17. 日志、可观测性与告警补充规则

### 17.1 基本规则

- 应用日志、访问日志、任务日志和安全日志的查询统一走 `ops.log.query`，不得借由审计权限绕过。
- 原始审计证据、未脱敏证据原文继续走 `audit.*` 权限，不与日志查询权限复用。
- Trace 联查默认只返回结构化索引、关键 span 与业务关联对象，不返回未授权的敏感正文。
- 告警规则变更、事件工单关闭、SLO 调整、观测后端配置变更必须进入审计链。

### 17.2 高风险动作

以下动作除权限外还必须校验 step-up 与审计：

- `ops.log.export`
- `ops.alert.manage`
- `ops.incident.manage`

## 18. 敏感数据受控交付专项规则

- 命中敏感或高敏分类时，必须先读取 `SensitiveHandlingPolicy`，再决定是否允许下单、开通、执行或导出。
- 敏感商品没有 `LegalBasisEvidence`、`SafePreviewArtifact` 或等价快照时，不得进入正式可售状态。
- 任何敏感结果导出前，必须先校验输出边界、结果披露审查状态和审批票据。
- 到期、撤权、冻结、争议或监管指令后，后端必须阻断继续执行，并要求补齐销毁/保留证明。

## 19. 交易链监控、公平性与信任安全规则

### 19.1 基本规则

- 交易链查询接口默认只返回与调用方作用域匹配的检查点、回执、事件和缺口对象。
- 监控对象读取必须遵守“业务最小披露 + 审计留痕 + 敏感内容掩码”原则。
- 任何修改监控策略、确认回执、处理公平性事件、关闭投影缺口的动作都必须再次写入审计链。

### 19.2 高风险动作

以下动作除权限外还必须校验 step-up 与审计：

- `ops.trade_monitor.manage`
- `ops.external_fact.manage`
- `ops.monitor_policy.manage`
- `ops.projection_gap.manage`
- `risk.fairness_incident.handle`

- `TRADE_MONITOR_SCOPE_FORBIDDEN`
- `TRADE_CHECKPOINT_NOT_FOUND`
- `EXTERNAL_FACT_CONFIRM_REQUIRES_STEP_UP`
