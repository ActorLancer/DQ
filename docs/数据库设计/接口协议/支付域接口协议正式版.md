# 《支付域接口协议正式版》

> 本文档是支付域的正式接口协议基线，供后续正式技术方案、开发文档、OpenAPI 细化稿、联调说明和测试用例直接引用。  
> 本文档对应当前目录下的原始 PRD、正式 PRD、业务流程、页面说明书、权限设计、领域模型和数据库设计。  
> 当前起步司法辖区固定为 `Singapore / SG`。

## 1. 文档目标

本协议用于固定以下内容：

- 支付域对外与对内接口边界
- `V1 / V2 / V3` 的接口分期
- 请求头、鉴权、幂等、签名、回调、错误码规范
- 司法辖区、走廊策略、收款偏好、支付意图、退款、打款、对账、结算路由的对象协议
- `MockPaymentProvider` 的测试协议

本协议刻意不展开：

- 具体技术框架选型
- 最终 OpenAPI YAML/JSON 细稿
- 渠道 SDK 封装细节
- 内部微服务拆分细节

## 2. 设计定位

支付域不是“订单服务里的一堆支付接口”，而是独立子域，负责：

- 收费规则与费用快照
- 起步司法辖区与走廊策略
- 支付编排
- 渠道回调验签与幂等
- 退款与打款编排
- 内部账务镜像联动
- 对账与差异处理
- `V2` 的自动打款、分账、周期扣费
- `V3` 的结算路由、FX 快照、数字资产结算轨道

补充约束：

- 支付域读取的商品收费模式统一来自 `pricing_mode` 快照，不再并列引入正式 `billing_mode` 主字段。
- 账单归集、履约计量与结算计算统一由 `Billing Event` 与计费模板驱动。

## 4. 基础约定

### 4.1 Base Path

- `V1`: `/api/v1`
- `V2`: `/api/v2`
- `V3`: `/api/v3`

### 4.2 内容类型

- 请求：`application/json`
- 回调：`application/json`
- 对账文件导入：`multipart/form-data` 或预签名文件引用

### 4.3 时间与金额

- 时间：`RFC 3339 / ISO-8601`
- 金额类型：字符串化十进制或高精度 numeric
- 币种：`ISO 4217` 为主，数字资产用平台内部 `asset_code`

### 4.4 通用响应结构

```json
{
  "code": "OK",
  "message": "success",
  "request_id": "req_01J...",
  "data": {}
}
```

错误结构：

```json
{
  "code": "PAYMENT_CORRIDOR_BLOCKED",
  "message": "real payment is blocked by launch corridor policy",
  "request_id": "req_01J...",
  "details": {
    "launch_jurisdiction_code": "SG",
    "payer_jurisdiction_code": "US",
    "payee_jurisdiction_code": "SG"
  }
}
```

## 5. 鉴权与请求头

### 5.1 用户/平台接口

必须支持：

- `Authorization: Bearer <token>`
- `X-Request-Id: <uuid-or-snowflake>`

推荐支持：

- `X-Idempotency-Key: <opaque-key>`
- `X-Trace-Id: <trace-id>`
- `X-Step-Up-Token: <short-lived-token>`：退款、赔付、人工打款、自动打款、走廊策略修改等高风险动作必需

### 5.2 渠道回调接口

渠道回调不使用普通用户 Bearer Token，必须走：

- `provider path parameter`
- 渠道签名头
- 时间戳头
- 原文摘要校验
- 事件唯一键去重

推荐统一头：

- `X-Provider-Event-Id`
- `X-Provider-Timestamp`
- `X-Provider-Signature`

### 5.3 服务间接口

平台内部服务间调用建议支持：

- 服务身份令牌
- mTLS 或网关签发身份
- `X-Service-Name`
- `X-Request-Id`

## 6. 幂等与一致性

### 6.1 必须幂等的接口

- 创建支付意图
- 取消支付意图
- 退款执行
- 人工打款执行
- 自动打款执行
- 分账指令创建
- 结算路由保存
- 数字资产结算执行
- 所有 webhook 回调处理

### 6.3 与身份域联动

- 支付意图创建属于高风险支付前置动作，必须携带 `X-Step-Up-Token` 或等价高风险认证结果后才能进入普通支付链路。
- 退款执行、人工打款、自动打款、走廊策略修改、渠道高危配置修改必须携带 `X-Step-Up-Token` 或等价高风险认证结果。
- 渠道管理员、财务结算专员、自动打款运营员等高风险角色必须已完成 `MFA`。

### 6.2 幂等键规则

- 幂等键字段：`idempotency_key`
- 默认建议：
  - `pay:<order_id>:<intent_type>:<attempt_no>`
  - `refund:<refund_id>`
  - `payout:<settlement_id>:<beneficiary_subject_id>`
  - `crypto:<transfer_id>`
- 服务端必须保证：
  - 同幂等键重复提交返回同一业务对象
  - 已进入终态的操作不得被重复执行

### 6.3 主状态与镜像状态

支付域要同时维护：

- 支付状态
- 账务镜像状态
- 订单支付状态
- 交付放行状态
- 对账状态

原则：

- 支付成功前不得放行交付
- 回调未验签不得改订单状态
- 对账差异未解决时不得把高风险打款记为已完成

### 7.1 JurisdictionProfile

用途：起步司法辖区和后续扩展司法辖区能力边界。

```json
{
  "jurisdiction_code": "SG",
  "jurisdiction_name": "Singapore",
  "regulator_name": "MAS",
  "launch_phase": "launch_active",
  "supports_fiat_collection": true,
  "supports_fiat_payout": true,
  "supports_crypto_settlement": false,
  "jurisdiction_status": "active",

    "launch_scope": "initial_production",
    "price_currency": "USD"
  }

```

`POST /api/v1/payments/intents`

用途：

- 为订单创建支付意图

权限：

- `payment.intent.create`

请求头：

- `Authorization`
- `X-Request-Id`
- `X-Idempotency-Key`
- `X-Step-Up-Token` 或等价高风险认证结果

请求体：

```json
{
  "order_id": "uuid",
  "intent_type": "order_payment",
  "provider_key": "paypal",
  "provider_account_id": "uuid",
  "payer_subject_type": "organization",
  "payer_subject_id": "uuid",
  "payee_subject_type": "organization",
  "payee_subject_id": "uuid",
  "payer_jurisdiction_code": "SG",
  "payee_jurisdiction_code": "SG",
  "launch_jurisdiction_code": "SG",
  "corridor_policy_id": "uuid",
  "fee_preview_id": "uuid",
  "payment_amount": "1000.00000000",
  "price_currency_code": "USD",
  "currency_code": "SGD",
  "payment_method": "wallet",
  "expire_at": "2026-04-08T12:00:00Z",
  "metadata": {}
}
```

服务端必须校验：

- 订单状态允许支付
- 费用快照有效
- 渠道能力存在
- 起步司法辖区存在
- 走廊策略存在且允许真实支付
- 幂等键未冲突

### 9.5 查询支付意图

`GET /api/v1/payments/intents/{payment_intent_id}`

权限：

- `payment.intent.read`

返回：

- `PaymentIntent`
- 最近交易事实摘要
- 回调处理摘要

### 9.6 取消支付意图

`POST /api/v1/payments/intents/{payment_intent_id}/cancel`

权限：

- `payment.intent.cancel`

只允许状态：

- `created`
- `pending`

### 9.7 接收渠道回调

`POST /api/v1/payments/webhooks/{provider}`

权限：

- 系统接口

请求体示例：

```json
{
  "provider_event_id": "evt_123",
  "event_type": "payment.succeeded",
  "provider_transaction_no": "txn_123",
  "payment_intent_id": "uuid",
  "transaction_amount": "1000.00000000",
  "currency_code": "SGD",
  "provider_status": "SUCCESS",
  "occurred_at": "2026-04-08T10:00:00Z",
  "raw_payload": {}
}
```

服务端必须执行：

1. 验签
2. 事件唯一性去重
3. payment intent 状态校验
4. 写入 `payment_webhook_event`
5. 写入 `payment_transaction`
6. 幂等更新订单支付状态与账务镜像

### 9.8 导入对账单

`POST /api/v1/payments/reconciliation/import`

权限：

- `payment.reconciliation.import`

请求参数：

- `provider_key`
- `provider_account_id`
- `statement_date`
- `statement_type`
- `file`

## 13. 回调签名建议

正式 OpenAPI 与渠道适配细稿应固定以下签名流程：

1. 读取原始 body
2. 读取渠道时间戳头
3. 读取渠道签名头
4. 用渠道密钥或证书验签
5. 生成内部标准化 `webhook_event`
6. 用 `(provider_key, provider_event_id)` 做幂等

## 14. 审计要求

- 管理司法辖区

- 自动打款执行
