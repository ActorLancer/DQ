# 一致性与事件接口协议正式版

## 1. 文档说明

本协议用于定义双层权威模型下的：

- 一致性联查
- outbox 查询
- dead letter 重处理
- 一致性修复

适用范围：

- `V1 / V2 / V3`

---

## 2. 总体原则

- 数据库是业务运营主状态机
- Fabric 是关键承诺与可信证明确认层
- 外部支付 / CA / 存储回执是来源事实
- Kafka 是事件分发总线，不承担主状态权威
- `ops.event_route_policy` 是 `V1` outbox 路由的唯一运行时权威源
- 正式 outbox 事件必须由应用层 canonical outbox writer 写入，禁止再以 `schema.table -> target_topic` 自动派生作为正式默认规则
- `common.tg_write_outbox()` 已退役，不再作为正式主链路事件生产入口

---

## 3. 统一返回字段

关键响应应支持：

- `request_id`
- `trace_id`
- `authority_model`
- `business_state`
- `proof_commit_state`
- `proof_commit_policy`
- `external_fact_status`
- `reconcile_status`

---

## 4. V1 canonical outbox 规则

### 4.1 最小 envelope / 路由字段

`ops.outbox_event.payload` 顶层至少应支持：

- `event_id`
- `event_type`
- `event_version`
- `occurred_at`
- `producer_service`
- `aggregate_type`
- `aggregate_id`
- `request_id`
- `trace_id`
- `idempotency_key`
- `event_schema_version`
- `authority_scope`
- `source_of_truth`
- `proof_commit_policy`
- `payload`

`ops.outbox_event` 路由侧字段至少应支持：

- `target_bus`
- `target_topic`
- `partition_key`
- `ordering_key`
- `payload_hash`

补充约束：

- `event_name` 不再作为正式 outbox payload 顶层字段
- 若 `(aggregate_type, event_type)` 在 `ops.event_route_policy` 中不存在激活记录，写入必须失败

---

## 5. V1 接口

### 5.1 一致性联查

`GET /api/v1/ops/consistency/{refType}/{refId}`

返回：

- 业务对象主状态
- Fabric 摘要状态
- 外部事实状态
- 最近 outbox 事件
- 最近 dead letter
- 最近审计链路

### 5.2 outbox 查询

`GET /api/v1/ops/outbox`

过滤建议：

- `outbox_status`
- `event_type`
- `target_topic`
- `request_id`
- `trace_id`
- `aggregate_type`
- `idempotency_key`
- `authority_scope`
- `source_of_truth`
- `proof_commit_policy`

返回建议字段：

- `outbox_event_id`
- `event_type`
- `status`
- `target_bus`
- `target_topic`
- `partition_key`
- `ordering_key`
- `payload_hash`
- `request_id`
- `trace_id`

### 5.3 dead letter 查询

`GET /api/v1/ops/dead-letters`

过滤建议：

- `reprocess_status`
- `failure_stage`
- `request_id`
- `trace_id`

### 5.4 dead letter 重处理

`POST /api/v1/ops/dead-letters/{id}/reprocess`

要求：

- 默认 dry-run
- step-up
- 审计

### 5.5 一致性修复

`POST /api/v1/ops/consistency/reconcile`

补充说明：

- `V1` 的 `reconcile` 是控制面动作，不单列正式 `reconcile_job` 表
- 持久化查询对象以 `ops.chain_projection_gap` 为准，修复建议与执行记录由后续控制面审计对象承接

请求建议字段：

- `ref_type`
- `ref_id`
- `mode`
- `dry_run`
- `reason`

---

## 6. V2 / V3 增量

- `V2`：训练、证明、分润对象接入同一联查与修复接口
- `V3`：跨链请求、合作伙伴回执、跨平台结算确认接入同一联查与修复接口
