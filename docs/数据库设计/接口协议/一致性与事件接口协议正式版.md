# 一致性与事件接口协议正式版

## 1. 文档说明

本协议用于定义双层权威模型下的：

- 一致性联查
- outbox 查询
- dead letter 重处理
- 一致性修复

适用范围：

- `V1 / V2 / V3`

---

## 2. 总体原则

- 数据库是业务运营主状态机
- Fabric 是关键承诺与可信证明确认层
- 外部支付 / CA / 存储回执是来源事实
- Kafka 是事件分发总线，不承担主状态权威

---

## 3. 统一返回字段

关键响应应支持：

- `request_id`
- `trace_id`
- `authority_model`
- `business_state`
- `proof_commit_state`
- `proof_commit_policy`
- `external_fact_status`
- `reconcile_status`

---

## 4. V1 接口

### 4.1 一致性联查

`GET /api/v1/ops/consistency/{refType}/{refId}`

返回：

- 业务对象主状态
- Fabric 摘要状态
- 外部事实状态
- 最近 outbox 事件
- 最近 dead letter
- 最近审计链路

### 4.2 outbox 查询

`GET /api/v1/ops/outbox`

过滤建议：

- `outbox_status`
- `target_topic`
- `request_id`
- `trace_id`
- `aggregate_type`

### 4.3 dead letter 查询

`GET /api/v1/ops/dead-letters`

过滤建议：

- `reprocess_status`
- `failure_stage`
- `request_id`
- `trace_id`

### 4.4 dead letter 重处理

`POST /api/v1/ops/dead-letters/{id}/reprocess`

要求：

- 默认 dry-run
- step-up
- 审计

### 4.5 一致性修复

`POST /api/v1/ops/consistency/reconcile`

请求建议字段：

- `ref_type`
- `ref_id`
- `mode`
- `dry_run`
- `reason`

---

## 5. V2 / V3 增量

- `V2`：训练、证明、分润对象接入同一联查与修复接口
- `V3`：跨链请求、合作伙伴回执、跨平台结算确认接入同一联查与修复接口
