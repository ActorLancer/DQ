# 数据交易平台数据库设计总说明

## 1. 设计范围

本说明对应当前全部文档的统一数据库实现基线，覆盖以下核心域：

1. 身份与主体
2. 角色、权限与作用域
3. 商品、版本、模板、策略
4. 订单、交付、验收、账单、争议
5. 支付、资金流、收费规则、对账与清结算
6. 保证金、质押、惩罚、奖励、分润
7. 搜索、推荐与个性化发现
8. 审计、证据、日志、开发调试
9. 模型、算法、受控计算、联邦任务、证明
10. 联盟链摘要、公链锚定、凭证
11. 跨链请求、见证、ack、补偿
12. 图风控、监管穿透、冻结治理、生态互联

## 2. 数据库分层

建议使用以下 schema：

- `common`：公共函数与通用触发器
- `core`：组织、用户、部门、应用、证书、环境、连接器
- `authz`：角色、权限、角色权限、主体角色绑定
- `catalog`：数据资产、版本、可交付对象、商品、SKU、标签、结构化小样本
- `contract`：模板、合同、数据契约、策略、签署与授权
- `trade`：询单、订单、订单行、状态历史
- `delivery`：对象、交付、令牌、回执、共享授权、revision 订阅、API 凭证、沙箱、模板查询、报告
- `billing`：账单、托管、钱包、台账、保证金、退款、赔付、惩罚、奖励、分润
- `payment`：支付渠道、司法辖区、走廊策略、收款偏好、支付意图、交易、Webhook、打款、对账、结算路由
- `support`：争议、证据、裁决
- `risk`：评分、信誉、黑名单、风险告警、图风控
- `audit`：审计事件、证据包、监管查询
- `chain`：链事件投影、公链锚定、凭证
- `ml`：模型、算法、任务、轮次、证明、执行结果
- `crosschain`：跨链请求、ack、见证、网关、补偿
- `ecosystem`：合作伙伴、连接器版本、互认关系
- `developer`：测试应用、测试钱包、mock 绑定
- `search`：搜索投影、倒排与向量检索文档、排序配置、索引同步任务
- `recommend`：推荐位、行为事件、画像快照、候选关系、推荐结果、实验与页面编排
- `ops`：outbox、dead letter、作业、系统日志

## 3. 设计原则

- 业务运营主状态在链下数据库
- Fabric 负责关键承诺与可信证明确认层
- 支付 / CA / 对象存储等外部系统负责其来源事实回执
- 链上存摘要、签名、承诺、状态变更和关键证据哈希
- 全部高增长表都预留分区策略或独立索引策略
- PostgreSQL 保存搜索投影与回退检索能力，OpenSearch 保存商品/卖方/模型/伙伴搜索读模型
- 搜索同步沿用 `outbox + Kafka + reindex/reconcile`，Redis 只做搜索缓存
- 推荐同步沿用 `outbox + Kafka + recall/re-rank/reconcile`，Redis 只做推荐缓存与去重集合
- PostgreSQL 全文检索 + `pg_trgm` 用于 fallback 与索引重建基线
- 向量检索采用 `pgvector`
- 图风控优先使用关系边表建模，必要时外接图计算引擎
- 商品 SKU 建模必须区分三层语义：
  - `sku_type`：标准 SKU 真值
  - `sku_code`：实现型商业套餐编码
  - `trade_mode`：交付/计费执行路径
- `V1` 的标准 SKU 真值仅允许：`FILE_STD / FILE_SUB / SHARE_RO / API_SUB / API_PPU / QRY_LITE / SBX_STD / RPT_STD`

### 3.1 工程 Schema 冻结规则

- 数据库对象名、列名、索引名、约束名、触发器名统一使用英文 `snake_case`
- 中文只允许出现在文档、页面文案、合同模板与审计展示层，不进入数据库对象命名
- 数据库持久化字段与逻辑字段允许存在一层显式映射，但不得多层转义或同义重复
- 当前已冻结的关键映射包括：
  - `current_state -> trade.order_main.status`
  - `order_amount -> trade.order_main.amount`
  - `delivery_mode -> catalog.product.delivery_type`
  - `pricing_mode -> catalog.product.price_mode`
  - `metered_quantity -> billing.billing_event.units`
- `bind_* / allowed_* / supported_*` 一类字段只允许作为投影、缓存或快照，不得充当关系事实源
- 关系事实源必须落在显式关系表、版本表、授权表或事件表中
- `current_version` 只表示当前生效版本指针，不承载审批、对账或状态推进职责
- `status_version` 只用于并发控制、顺序校验和幂等防重，不作为业务展示字段
- 任何新增字段都必须先判断其归属：业务字段、控制字段、审计字段或集成字段，不允许混合语义命名
- `Data Product` 是目录、审核、搜索和聚合展示事实源；`Product SKU` 是下单、合同、授权、交付、验收、账单与结算事实源
- `product.delivery_mode` 仅作为目录默认值或聚合展示值；进入交易时一律以下游 `sku` 快照为准
- `product.sku_template_id` 仅作为 SKU 模板族默认值或批量生成入口；最终模板绑定事实源应落到 SKU 绑定表或模板绑定表
- `usage_mode` 仅表达推荐使用姿态，不得替代正式 `rights_type`
- `pricing_mode` 是对外收费模式逻辑字段；账单归集、履约计量与结算计算统一由 `billing_event` 与计费模板驱动
- 若数据库仍保留 `trade_mode` 或 `billing_mode` 一类实现字段，只能视为内部执行路径或计量模式投影，不得替代 `sku_type / delivery_mode / pricing_mode`

## 4. 关键对象落库策略

### 4.1 大对象与训练数据

- 原始交易数据、密文数据集、模型文件、证据原文、训练日志原文默认存对象存储
- PostgreSQL 保存：
  - 元数据
  - 原始接入批次与对象清单
  - 格式识别结果
  - 抽取/预览任务摘要
  - 存储命名空间与存储策略
  - 查询面定义与模板版本
  - 查询执行运行记录
  - 数据契约
  - 字段结构摘要
  - 质量报告
  - 加工责任链
  - 哈希
  - 路径
  - 索引字段
  - 生命周期状态
  - 审计与访问记录

### 4.2 小型结构化数据

对于小型交易数据集、样本集、标签集、统计维表，可在 PostgreSQL 中落以下模式：

- `catalog.asset_structured_dataset`
- `catalog.asset_structured_row`

用于：

- 小样本展示
- 轻量聚合查询
- 训练样本预览
- 规则验证

## 5. 搜索与向量设计

### 5.1 倒排检索

主要使用：

- PostgreSQL `tsvector`
- `GIN`
- `pg_trgm`
- OpenSearch 全文检索与聚合

索引对象：

- 商品目录
- 服务商品目录
- 卖方主体目录
- 模型目录
- 审计联查摘要
- 风险告警摘要

### 5.2 向量检索

使用 `vector(1536)` 作为默认 embedding 维度占位，可按部署时实际模型调整。

建议索引对象：

- 商品语义搜索
- 模型语义搜索
- 风险文本摘要聚类

### 5.3 搜索同步与排序治理

- `search.product_search_document`：商品/服务搜索投影主表
- `search.seller_search_document`：卖方搜索投影主表
- `search.search_signal_aggregate`：曝光、点击、成交、热度等排序特征聚合
- `search.ranking_profile`：综合排序权重与过滤规则
- `search.index_alias_binding`：OpenSearch 读写别名与活动索引绑定
- `search.index_sync_task`：索引同步、重建与修复任务
- `search.model_search_document`：V2 模型搜索投影
- `search.synonym_rule`：V2 同义词与搜索运营规则
- `search.partner_search_document`：V3 伙伴/生态搜索投影

## 6. 触发器与自动化

建议统一使用以下触发器：

- `updated_at` 自动更新时间
- 搜索文档自动同步
- 关键状态变更写历史表
- 关键业务表写 outbox_event
- `ops.outbox_event`、`ops.dead_letter_event` 生产环境建议按 `created_at` 归档或分区

## 8. 命名规范

- 主键统一使用 `uuid`
- 主表命名使用单数语义，便于和领域对象直接对应
- 逻辑层优先使用对象化状态名，例如 `current_state / payment_status / replay_status / alert_status`
- 持久化层若历史表已使用 `status`，必须在表字典和接口协议中声明其逻辑语义，不得裸露为无上下文通用状态
- 时间戳统一使用 `timestamptz`
- JSON 扩展字段统一使用 `jsonb`
- 所有业务主表统一包含：
  - `created_at`
  - `updated_at`

## 9. 一句话结论

这套数据库设计不是把 `V1` 做成一套短期模型，而是用 `V1` 先落稳定骨架，让 `V2/V3` 在相同 schema 体系和对象主键体系上自然扩展，不再返工。

### 14.2 核心字段增强

- `trade.order_main` / `contract.digital_contract` / `delivery.delivery_record` / `payment.payment_intent` / `billing.settlement_record` 增加：
  - `authority_model`
  - `proof_commit_state`

  - `external_fact_status`
  - `reconcile_status`
- `ops.outbox_event` 增加：
  - `request_id`
  - `trace_id`
  - `target_bus`
  - `target_topic`
  - `payload_hash`
  - `max_retries`
- `ops.dead_letter_event` 增加：
  - `failure_stage`
  - `reprocess_status`

### 14.3 设计原则

- Kafka 是事件总线，不承担主状态权威。
- 数据库事务内必须同时写业务对象、审计事件和 outbox。
- 外部事实型对象必须保留来源回执映射。

### 14.4 状态字段工程规则

- `trade.order_main.status` 承接逻辑字段 `current_state`，是订单流程推进唯一主状态。
- `payment_status / delivery_status / acceptance_status / settlement_status / dispute_status / invoice_status` 必须作为业务域子状态单独维护。
- `proof_commit_state / external_fact_status / reconcile_status` 必须作为技术与审计镜像状态单独维护。
- `status_version` 只用于并发控制、顺序校验和重放保护，不得承接业务版本语义。
- 任何自动结算、自动放行、自动锚定都必须同时校验主状态、领域子状态和技术镜像状态，不得只看其中任一字段。

### 15.4 设计原则

- PostgreSQL 保存推荐行为、画像、配置和推荐结果主数据。
- OpenSearch 继续承担候选召回与相似检索读模型。
- Redis 仅承担推荐缓存与已看去重状态，不作为权威存储。
- 推荐域同步继续走 `outbox + Kafka + aggregator/indexer + reconcile`。
- 推荐结果返回前必须回 PostgreSQL 做最终业务校验。

- 审计域与可观测域分离：`audit.*` 负责证据与回放，`ops.*` 负责日志镜像、trace 索引、告警与工单。
- 原始日志、原始 trace 和时序指标不回灌 PostgreSQL；PostgreSQL 只存控制面和关键镜像。
- 告警规则、事件工单、SLO 目标和后端配置的变更必须进入审计链。

## 17. 敏感数据治理落库策略

### 17.1 V1 核心对象

- `catalog.sensitive_handling_policy`
- `contract.legal_basis_evidence`
- `catalog.safe_preview_artifact`
- `delivery.sensitive_execution_policy`
- `delivery.attestation_record`
- `delivery.result_disclosure_review`
- `delivery.destruction_attestation`

### 17.2 V2 增量对象

- `delivery.privacy_budget_ledger`

### 17.3 设计原则

- 敏感数据交易不以“原始副本交付”作为默认前提，而以“受控使用权、受控查询权、受控计算权、结果获取权”建模。
- 数据库必须显式表达处理依据、安全预览、执行边界、结果披露审查与销毁/保留证明，避免只靠工单文本或备注字段。
- `V1` 先落最小闭环，`V2` 再补差分隐私预算与高阶证明能力，`V3` 复用既有对象做跨平台映射。

## 18. 交易链监控、公平性与信任安全落库策略

### 18.1 V1 核心对象

- `ops.trade_lifecycle_checkpoint`
- `ops.external_fact_receipt`
- `risk.fairness_incident`
- `ops.monitoring_policy_profile`
- `ops.chain_projection_gap`

### 18.2 V2 增量

- 复用 `V1` 对象，并把 `task / result / reward / payout / environment` 纳入引用范围。

### 18.3 V3 增量

- 复用 `V1` 对象，并把 `cross_chain_request / ack / partner / crypto_settlement_transfer / compensation` 纳入引用范围。

### 18.4 设计原则

- 生命周期检查点必须显式表达“应发生什么、何时应发生、实际是否发生、由什么证据证明发生”。
- 外部事实回执必须独立建模，不允许只塞进 `metadata` 或备注字段。
- 公平性事件必须能追溯到检查点、外部事实或链投影缺口来源。
- 链投影缺口必须独立记录，以支撑一致性联查、修复和审计回放。
