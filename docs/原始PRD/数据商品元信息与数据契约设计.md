# 数据商品元信息与数据契约设计

- 文档名称：数据商品元信息与数据契约设计
- 适用范围：基于区块链技术的数据交易平台 `V1 / V2 / V3`
- 版本：D1
- 日期：2026-04-10

---

## 1. 文档目标

本设计用于回答四个问题：

1. 一个合格的数据商品，最少必须带哪些元信息。
2. 平台如何把“商品说明”升级为可执行的“数据契约”。
3. 卖方、加工方、平台在清洗、脱敏、加工、交付中的责任边界如何表达。
4. 这些要求如何落入商品建模、页面、权限、验收、审计和数据库。

本文件的结论是：

**数据商品元信息不是一个说明页字段集合，而是 `Data Product Metadata + Data Contract + Delivery Policy + Processing Provenance` 的组合。**

---

## 2. 基础定义

### 2.1 数据商品元信息

数据商品元信息用于说明：

- 卖的是什么
- 适合谁买
- 数据结构如何理解
- 质量达到什么水平
- 是否合规
- 怎么交付
- 如何验收
- 谁负责

### 2.2 数据契约

数据契约不是订单合同本身，而是商品级、版本级、SKU 级的承诺集合。  
它用于把以下内容固化为可审核、可快照、可验收、可争议处理的对象：

- 结构承诺
- 质量承诺
- 合规承诺
- 交付承诺
- 使用权利承诺
- 验收标准
- 责任矩阵

### 2.3 加工责任链

加工责任链用于表达：

- 原始数据由谁提供
- 谁完成清洗、脱敏、标注、标准化、建模
- 平台是直接处理方、编排方，还是仅登记与校验方
- 生成的结果资产与原始资产之间是什么血缘关系

---

## 3. 设计结论

### 3.1 统一采用 10 个元信息域

每个数据商品至少应由以下 10 个元信息域组成：

1. 商业元信息
2. 数据内容元信息
3. 结构元信息
4. 质量元信息
5. 安全与合规元信息
6. 交付与访问元信息
7. 版本与生命周期元信息
8. 授权与交易元信息
9. 责任与证据元信息
10. 加工与血缘元信息

### 3.2 数据契约必须单独建模

不能把全部承诺都塞进 `product.metadata`。  
至少需要显式区分：

- 目录展示元信息
- 字段级结构定义
- 质量报告
- 加工责任链
- 数据契约定义

### 3.3 平台可提供加工处理能力，但不是默认接管全部原始数据

平台可以提供：

- 清洗
- 脱敏
- 标准化
- 标注
- 特征加工
- 模型或结果封装

但加工模式必须显式声明，不能默认等同于“平台已经拿到并可任意使用原始明文数据”。

## 4. 十大元信息域

## 4.1 商业元信息

最少包含：

- 商品名称
- 商品副标题
- 商品摘要
- 价值主张
- 适用行业
- 适用场景
- 目标买方
- 禁止用途
- 计费模式
- 标准 SKU 列表

## 4.2 数据内容元信息

最少包含：

- 数据对象类型
- 数据主题域
- 覆盖时间范围
- 覆盖地域范围
- 覆盖主体范围
- 数据粒度
- 主键或业务唯一键
- 时间字段
- 增量字段 / 水位字段

## 4.3 结构元信息

最少包含：

- schema 版本
- 字段清单
- 字段类型
- 字段说明
- 是否可空
- 主键字段
- 时间字段
- 分区字段
- 编码规则
- 枚举规则
- 单位 / 精度

## 4.4 质量元信息

最少包含：

- 缺失率
- 重复率
- 异常率
- 覆盖率
- 更新频率
- 延迟 / 时效性
- 采样方式
- 质量评估时间
- 质量评估责任方
- 质量报告摘要

## 4.5 安全与合规元信息

最少包含：

- 是否含个人信息
- 是否含敏感个人信息
- 是否含重要数据
- 脱敏状态
- 脱敏方法摘要
- 使用限制
- 地域限制
- 行业限制
- 是否允许导出
- 是否允许训练
- 是否允许再分发

## 4.6 交付与访问元信息

最少包含：

- 交付方式
- 访问入口摘要
- 访问前置条件
- 调用 / 下载 / 查询上限
- 到期策略
- revoke 策略
- 是否支持样例或预览
- 样例代表性说明
- 验收方式

## 4.7 版本与生命周期元信息

最少包含：

- asset version
- product revision
- 发布日期
- 生效日期
- 变更说明
- 是否包含历史版本
- 是否包含未来更新
- 保留策略
- 销毁策略
- 订阅策略

## 4.8 授权与交易元信息

最少包含：

- 可售权利
- 默认禁止动作
- 授权期限
- 可授权主体
- 许可范围
- 并发限制
- 调用上限
- 验收标准
- 退款条件
- 赔付边界

## 4.9 责任与证据元信息

最少包含：

- 原始提供者
- 加工者
- 平台角色
- 第三方评估方
- 质量证明
- 合规证明
- 存证摘要
- 审计证据引用
- 争议归责原则

## 4.10 加工与血缘元信息

最少包含：

- 原始资产引用
- 加工模式
- 加工任务引用
- 处理责任方
- 转换规则版本
- 脱敏规则版本
- 标注规则版本
- 模型 / 算法版本
- 处理时间
- 输出资产版本
- 血缘摘要
- 是否可回溯重建

## 5. 数据契约最小结构

平台应把数据契约定义为以下九类承诺的组合：

1. 业务承诺
2. 结构承诺
3. 质量承诺
4. 合规承诺
5. 交付承诺
6. 版本承诺
7. 验收承诺
8. 权利承诺
9. 责任承诺

对 `V1` 来说，数据契约至少要做到：

- 可绑定到商品或 SKU
- 可版本化
- 可生成 digest
- 可在下单时快照到订单/合同
- 可驱动验收、拒收、争议和审计

## 6. 清洗、脱敏、加工责任分工

### 6.1 默认分工原则

- 卖方负责原始数据来源合法、初步清洗、基础质量和权利声明
- 第三方加工方可负责深度清洗、标注、脱敏、标准化、建模
- 平台负责规则、工具、流程编排、证据留痕与校验
- 高敏感数据优先采用不出域处理，而不是默认复制给平台

### 6.2 平台加工能力预留模式

平台后续可支持以下四种加工模式：

1. `seller_self_managed`
   - 卖方自处理，平台只登记与校验
2. `third_party_processor`
   - 第三方加工，平台编排并留痕
3. `platform_managed_processing`
   - 平台实际处理，平台成为明确处理参与方
4. `in_domain_processing_orchestrated_by_platform`
   - 平台提供规则与编排，但加工发生在卖方域内

当前结论：

- `V1` 只要求把这四种模式建模并保留字段
- 不要求现在就完整设计平台加工流程
- 后续专题应单独设计“平台加工处理能力”

## 7. 对不同交付模式的影响

### 7.1 文件快照

- 必须强调结构承诺、质量承诺、哈希承诺、样例说明
- 验收重点是：字段、批次、哈希、可解密性、样例一致性

### 7.2 只读共享

- 必须强调共享对象说明、recipient、revoke、共享协议和可见版本
- 验收重点是：授权生效、访问成功、对象范围与契约一致

### 7.3 API / 服务

- 必须强调接口说明、字段契约、错误码、配额、SLA、调用边界
- 验收重点是：鉴权成功、字段契约匹配、SLA 满足

### 7.4 模板查询 / clean room lite

- 必须强调模板白名单、参数说明、输出边界、导出限制
- 验收重点是：模板执行成功、输出字段与边界符合契约

### 7.5 结果产品

- 必须强调生成逻辑摘要、输入范围、结果口径、可用边界
- 验收重点是：结果摘要、格式、指标口径与契约一致

## 8. V1 / V2 / V3 分阶段要求

### 8.1 V1

`V1` 必须完成：

- 十大元信息域的对象建模
- 字段级结构定义
- 质量报告对象
- 数据契约对象
- 加工模式与责任链字段预留
- 上架页、详情页、审核页、验收页引用这些对象

`V1` 不要求完成：

- 平台加工处理流程编排
- 复杂自动质量评估流水线
- 跨平台数据契约交换
- 完整 clean room 与模型加工契约

### 8.2 V2

`V2` 必须补齐：

- 模型 / 计算 / clean room 的数据契约
- 更完整的加工责任链和结果证明绑定
- 处理任务、训练任务、输出结果的契约快照

### 8.3 V3

`V3` 必须补齐：

- 跨平台契约交换
- 跨平台责任链与证据互认
- 跨域加工与交付责任穿透

## 9. 数据库落库基线

数据库至少应补齐以下对象：

1. `catalog.product_metadata_profile`
2. `catalog.asset_field_definition`
3. `catalog.asset_quality_report`
4. `catalog.asset_processing_job`
5. `catalog.asset_processing_input`
6. `contract.data_contract`

同时应补齐以下快照字段：

- `contract.digital_contract.data_contract_id`
- `contract.digital_contract.data_contract_digest`
- `catalog.product.product_subtitle`
- `catalog.product.industry_tags`
- `catalog.product.use_case_tags`
- `catalog.product.target_buyer_tags`
- `catalog.product.prohibited_use_tags`
- `catalog.asset_version.processing_mode`
- `catalog.asset_version.lineage_hash`

## 10. 页面与权限基线

平台至少应新增或扩展以下页面能力：

- 产品编辑页：十大元信息域输入
- 元信息与数据契约页：字段结构、质量报告、加工责任链、契约定义
- 产品详情页：面向买方展示可披露元信息和契约摘要
- 验收页：按数据契约加载验收项
- 审核页：按数据契约和质量报告联审

至少应新增以下权限能力：

- 质量报告读写
- 加工责任链读写
- 数据契约读写

---

## 11. 结论

这套设计的核心不是给商品再多加几个字段，而是把：

- 商品说明
- 结构说明
- 质量说明
- 合规说明
- 交付说明
- 验收说明
- 权利说明
- 责任说明
- 加工血缘说明

统一收敛为平台的正式交易对象。

只有这样，数据商品才不是“描述一下就能卖”的目录项，而是一个可审核、可交付、可验收、可争议处理、可审计回放的数据产品。
