# 《支付、资金流与轻结算设计》

> 包含平台收费规则与外部支付对接设计。本文档中的“轻结算”按业务含义统一指向平台的支付、资金流、结算、对账与清结算能力设计，不单指某一种技术方案。

## 1. 设计目标

本专题用于解决以下问题：

- 平台如何接入真实外部支付渠道而不把订单逻辑与具体支付渠道写死。
- 平台如何定义货款、保证金、退款、赔付、打款、分润等多类资金流。
- 平台如何形成可调参数、可版本化的平台收费规则，而不是只写死一个抽成比例。
- 平台如何在开发测试阶段提供可调试、可模拟的支付接口，而不依赖真实货币。
- 平台如何把支付、账务、对账、审计和交易状态闭环起来。

## 2. 核心结论

1. 平台必须新增独立的“支付、资金流与清结算子系统”，不能把支付逻辑直接写进订单模块。
2. 平台不应默认自建真实资金池；真实资金优先由持牌支付机构、银行或合规托管方承接，平台维护业务主状态、账务镜像、结算指令和审计证据。
3. 平台必须同时支持真实支付渠道与内部 `MockPaymentProvider`，开发调试不能依赖真实货币。
4. 平台收费规则必须版本化、参数化，冻结的是“收费公式和参数模型”，不是一开始就写死费率。
5. 支付状态、账务状态、订单状态、交付状态、争议状态必须形成闭环，禁止孤立实现。
6. 平台起步司法辖区固定为新加坡；`V1` 生产范围内的真实支付、真实收款、真实打款和真实对账，必须先落在新加坡起步路由下。
7. 全球扩展通过“司法辖区配置 + 走廊策略 + 结算路由 + 持牌伙伴接入”实现；平台不以技术中间件替代牌照要求。

## 3. 总体原则

- 支付域与交易域解耦。
- 支付渠道通过统一 Provider 适配层接入。
- 账务镜像与真实支付结果分离，但必须可对账。
- 所有高风险动作必须留痕，包括支付确认、退款、赔付、打款、分账、回调重放、对账差异处理。
- Mock、Sandbox、Production 三种支付模式必须统一抽象，不允许业务代码到处写渠道分支。
- 数字货币支付、交易所结算、多币种跨境结算属于后续增强，不得反向污染 V1 主路径。
- 起步司法辖区为新加坡，默认只开放新加坡白名单内的真实支付走廊；其他地区只能在对应走廊被显式启用后进入生产。
- 平台负责合规路由与结算编排，不负责用技术手段突破被监管禁止的支付、换汇或数字资产限制。

## 4. 分层架构

### 4.1 定价与收费规则层

负责：

- 商品价格模型
- 平台服务费
- 渠道手续费
- 保证金规则
- 增值服务费
- 分润规则绑定
- 收费版本冻结与生效时间

### 4.2 支付编排层

负责：

- 创建支付意图
- 选择支付渠道
- 发起支付/支付确认
- 取消支付
- 退款编排
- 打款编排
- 渠道回调解析与验签
- 支付状态汇总

### 4.3 内部账务与台账层

负责：

- 应收/应付镜像
- 保证金锁定/释放/扣罚
- 平台收入镜像
- 渠道手续费镜像
- 卖方待结算金额
- 赔付、退款、分润结果镜像

### 4.4 外部支付渠道适配层

负责：

- 支付宝
- 微信支付
- 银联
- PayPal
- 线下对公转账确认
- 未来数字货币/交易所结算适配器

统一能力接口至少包括：

- `createPaymentIntent`
- `confirmPayment`
- `queryPayment`
- `cancelPayment`
- `refundPayment`
- `createPayout`
- `queryPayout`
- `parseWebhook`
- `verifyWebhook`
- `getCapabilities`

### 4.5 司法辖区策略与全球支付路由层

负责：

- 维护起步司法辖区配置
- 维护付款地/收款地走廊策略
- 管理允许的付款币种、收款币种与收款方式
- 管理是否允许 FX、是否允许数字资产、是否需要人工复核
- 管理全球扩展时的持牌伙伴接入白名单

## 5. 支付对象与资金类型

平台至少要覆盖以下资金对象：

- 订单货款
- 买方保证金
- 卖方履约保证金
- 风险托管金额
- 平台服务费
- 渠道手续费
- 退款金额
- 赔付金额
- 分润金额
- 发票相关金额
- 加急/专线/额外存储/定制报告等外围服务费

## 6. 真实支付渠道设计

### 6.1 V1 支持的支付方式

起步司法辖区：`Singapore / SG`

说明：

- `V1` 的生产支付路径以新加坡为唯一起步司法辖区。
- 商品建议统一以 `USD` 计价；买方实际付款币种和卖方实际收款币种必须受新加坡起步走廊策略约束。
- 真实支付渠道的“是否可用”不由文档枚举决定，而由新加坡起步走廊白名单和持牌伙伴能力决定。

- 在线支付
  - 支付宝
  - 微信支付
  - 银联
  - PayPal
- 线下对公转账
- 人工确认收款
- Mock 测试支付

### 6.2 V1 不强制实现的能力

- 渠道原生自动多方分账
- 自动化复杂子商户体系
- 数字货币支付
- 交易所直连结算
- 多币种跨境清算

### 6.3 V2 / V3 承接

- `V2`：自动打款、自动分润、子商户/分账扩展、周期扣费。
- `V3`：多币种、跨境、合规前提下的数字货币结算与交易所适配。

### 6.4 全球扩展原则

- `V1`：以新加坡为唯一真实生产起步司法辖区。
- `V2`：在新加坡主体和新加坡合作伙伴框架下，逐步扩展受控的跨境法币走廊。
- `V3`：在明确走廊规则、伙伴资质、风控与监管要求后，扩展多币种、跨境和数字资产结算路由。
- 平台不默认自营换汇、跨境出金或数字资产兑换；真实收款、换汇、出金、off-ramp 由持牌伙伴执行。

## 7. 平台收费规则设计

### 7.1 收费规则分层

平台收费规则应拆成：

1. 平台服务费
2. 渠道手续费
3. 保证金规则
4. 增值服务费
5. 分润规则

### 7.2 收费公式应参数化

收费引擎必须支持：

- 固定金额
- 百分比
- 阶梯比例
- 基础费 + 超额费
- 按周期收费
- 按任务/按次收费
- 按风险等级调整保证金

### 7.3 V1 推荐收费框架

- 文件/报告：按单收平台服务费。
- API/订阅：按周期基础费 + 超额用量费。
- 沙箱：按席位/时长/周期收费。
- 增值项单独计费。
- 渠道手续费承担方作为配置项，不写死。

## 8. 资金流设计

### 8.1 一次性文件/报告交易

```text
下单
  -> 价格快照
  -> 支付意图
  -> 买方支付货款
  -> 如需保证金则同步锁定
  -> 交付
  -> 验收
  -> 结算计算
  -> 扣平台费/渠道费
  -> 形成卖方待打款金额
  -> 打款
  -> 归档
```

### 8.2 API / 订阅 / 沙箱

```text
下单/续费
  -> 生成支付意图
  -> 预付或周期账单
  -> API 开通/沙箱开通
  -> 产生账单事件
  -> 到期结算或超额结算
  -> 退款/赔付（如命中）
```

### 8.3 保证金

保证金必须独立建模，不与货款混账：

- 买方保证金
- 卖方履约保证金
- 风险托管余额

状态至少包括：

- `pending_lock`
- `locked`
- `partially_penalized`
- `released`
- `reserved_for_decision`
- `returned`

## 9. 支付状态与结算状态

### 9.1 支付意图状态

`created -> pending -> processing -> succeeded / failed / canceled / expired`

### 9.2 退款状态

`pending -> processing -> succeeded / failed / canceled`

### 9.3 打款状态

`pending_review -> ready -> processing -> succeeded / failed / returned`

### 9.4 对账状态

`pending_import -> imported -> reconciling -> matched / mismatched / resolved`

## 10. Mock PaymentProvider 设计

### 10.1 必须支持的调试场景

- 支付成功
- 支付失败
- 支付超时
- 取消支付
- 重复 webhook
- 延迟回调
- 部分退款
- 全额退款
- 打款成功
- 打款失败
- 对账差异

### 10.2 建议开放的 Mock 接口

- `POST /mock/payments/intents`
- `POST /mock/payments/{id}/simulate-success`
- `POST /mock/payments/{id}/simulate-fail`
- `POST /mock/payments/{id}/simulate-timeout`
- `POST /mock/refunds/{id}/simulate-success`
- `POST /mock/payouts/{id}/simulate-success`
- `POST /mock/webhooks/{provider}`

### 10.3 Mock 接口必须支持的参数

- `payment_amount`
- `currency`
- `provider`
- `payment_method`
- `delay_seconds`
- `duplicate_webhook`
- `partial_refund_amount`

## 11. 对账与清结算

平台必须支持：

- 渠道账单导入
- 平台账务与渠道账单自动比对
- 差异记录与工单
- 人工核账与重放
- 结算批次
- 打款批次
- 失败重试

禁止出现：

- 支付成功但平台未记账且无告警
- 支付失败但订单进入交付
- 退款执行成功但账务未冲销
- 打款失败但结算状态误标为完成

## 12. 合规与风险边界

- 平台应尽量避免自持真实资金池。
- 外部渠道密钥、证书、Webhook 验签材料必须纳入密钥治理。
- 支付回调必须幂等，禁止重复回调导致重复放款或重复记账。
- 高风险退款、赔付、打款、分账必须二次确认并进入审计包。
- 数字货币支付与交易所适配必须在明确司法辖区、牌照边界和风控要求后再进入生产范围。

## 15. 闭环性检查

加入本专题后，平台主链路应形成：

```text
商品定价
  -> 费用预估
  -> 司法辖区与走廊校验
  -> 支付意图
  -> 渠道支付/Mock支付
  -> 回调验签与幂等更新
  -> 账务镜像
  -> 交付与验收
  -> 结算计算
  -> 退款/赔付/打款/分润
  -> 对账与差异处理
  -> 审计固化
```

结论：

- 逻辑可闭环。
- 可以无缝接入当前订单、交付、保证金、赔付、审计和开发调试体系。
- 不会与“平台不是默认明文托管者”的新边界冲突。

## 15. 与身份认证、注册登录与会话管理的联动补充

- 支付意图创建、退款、赔付、人工打款、保证金扣罚、走廊策略修改都属于高风险动作，必须绑定 step-up。
- 平台管理员、平台风控结算员、租户管理员等高风险角色必须强制启用 MFA。
- 企业 SSO、会话撤销、设备撤销和风险登录命中后，必须能联动阻断支付、退款和打款接口。
- 支付回调验签使用的渠道证书、服务密钥和 webhook 签名材料必须纳入统一密钥与证书治理。

## 16. 与双层权威模型的一致性补充

- `payment.payment_intent.status` 承接逻辑字段 `payment_status`，是平台支付流程镜像，不等同于渠道事实结果。
- 支付、退款、打款是否最终成立，以支付渠道/银行回执为来源权威。
- `proof_commit_state` 用于表达支付、保证金、退款、赔付、结算摘要是否已写入 Fabric。
- `external_fact_status` 用于表达渠道事实是否已确认。
- Kafka 只承担支付事件分发、对账、通知和审计联动，不承担支付主状态权威。
