# 敏感数据处理与受控交付设计

- 文档定位：原始 PRD 增补专题稿
- 目标：定义敏感/高敏数据在“不信任平台、原始数据不宜出域、需要可证明交付”的前提下，如何形成完整交易链闭环
- 适用阶段：`V1 / V2 / V3`

---

## 1. 设计目标

本文件回答七个问题：

1. 敏感数据是否还能交易，如果可以，应交易什么而不是交易什么。
2. 卖方不信任平台、平台不持有明文时，如何证明“卖方确有数据且大体符合约定”。
3. 买方在不拿到原始副本的情况下，如何完成预验收、正式交付、正式验收。
4. 平台如何控制敏感数据的查询、计算、导出、二次使用和撤权。
5. 交易达成后，如何让计费、审计、证据、赔付和链上锚定闭环。
6. `V1 / V2 / V3` 分别承诺到什么程度，避免在 `V1` 过度承诺完整隐私计算平台。
7. 哪些开源/商用实践可借鉴，但不直接变成当前固定技术选型。

---

## 2. 核心结论

对于敏感数据，平台默认不应把“原始文件副本交付”作为主路径，而应把“受控使用权、受控查询权、受控计算权、结果获取权”作为主产品。

敏感数据闭环的关键不是“把所有问题都交给 ZKP / 区块链 / TEE”，而是形成多层证据栈：

1. 数据承诺与版本承诺
2. 机器可读数据契约
3. 分类分级与处理依据证明
4. 安全预览/样例/试算
5. 受控执行策略
6. 输出边界与导出审查
7. 执行回执、结果摘要与证据包
8. 审计主链与联盟链锚定

也就是说，平台卖的不是“明文副本”，而是：

- 在什么边界下可被使用
- 由谁执行
- 可以得到什么结果
- 结果是否符合约定
- 双方是否都留下可追责证据

## 3. 适用范围与默认限制

### 3.1 适用对象

本设计适用于以下对象：

- 含个人信息的数据产品
- 含敏感个人信息的数据产品
- 医疗、金融、工业、政务等高敏明细数据
- 含商业秘密或重大经营机密的数据产品
- 重要数据或监管重点数据
- 买方只需要结果、不需要原始明文的数据产品

### 3.2 默认禁止

以下场景默认禁止，除非未来法规、牌照和专项治理能力发生重大变化：

- 原始敏感个人信息副本下载交付
- 医疗/金融高敏明细自由下载
- 未说明处理依据的个人信息类商品上架
- 跨境主体直接申请高敏副本下载
- 公链展示原始明文、实名身份、合同正文、敏感日志

### 3.3 默认优先路径

敏感数据应按以下优先级选择交付路径：

1. 固定报告 / 结果产品
2. 白名单模板查询 / 受限结果导出
3. 卖方自持 API / 评分服务
4. 受控执行 / clean room / TEE
5. 联邦分析 / 联邦训练
6. MPC / PSI / 交集求解
7. 仅在合规允许且低敏时考虑文件副本

## 4. 敏感数据产品形态矩阵

### 4.1 `V1` 可正式支持的敏感数据产品

`V1` 不承诺完整 clean room、完整 MPC 或完整 TEE 平台，但可以正式支持以下敏感数据商品：

1. `report_result`
   - 只交付聚合结果、固定报告、统计摘要、特征结果
2. `template_query_lite`
   - 仅允许白名单参数化模板
   - 仅导出受限结果
3. `sandbox_query`
   - 在受控 workspace 中执行受限模板
   - 不开放底库自由查询
4. `seller_hosted_api`
   - 卖方自持数据，平台只开通调用权、配额和审计

### 4.2 `V2` 正式补齐的高敏能力

`V2` 正式支持：

- 完整 `clean room / analysis rule / TVF / output policy`
- `TEE C2D`
- 联邦分析 / 联邦训练
- MPC / PSI 结果产品
- 差分隐私预算治理
- 结果激活/导出审批

### 4.3 `V3` 跨平台扩展能力

`V3` 支持：

- 跨平台 clean room 互认
- 跨平台受控执行回执
- 跨域销毁/撤权证明
- 跨平台敏感结果审计联查

## 5. 敏感数据完整交易链闭环

## 5.1 交易前闭环

### 第 1 步：主体准入与高敏资格审查

卖方、买方、算法服务方、隐私计算服务方都必须完成更高等级准入：

- 主体真实性
- 行业资质
- 数据处理角色
- 处理依据与合法性声明
- 特殊行业/特殊数据额外资质
- 高风险动作必须支持 step-up

### 第 2 步：数据登记与权利链登记

必须登记：

- 数据来源与采集方式
- 采集时间范围
- 原始控制人/加工者/受托方
- 是否含 PI / SPI / 重要数据
- 是否存在第三方受限条款
- 是否允许训练、推断、商业输出、再分发

### 第 3 步：分类分级与字段级敏感识别

平台必须同时做：

- 资产级分类分级
- 字段级敏感识别
- 主体级/地域级限制识别
- 用途冲突识别
- 交付模式适配判断

至少形成：

- `data_classification`
- `contains_pi`
- `contains_spi`
- `masking_status`
- `allowed_delivery_modes`
- `required_execution_boundary`

### 第 4 步：交付路径决策

根据敏感等级自动收口：

- 低敏：可文件/共享/API
- 敏感：优先 API / 模板查询 / 结果产品
- 高敏：优先 report_result / clean room / TEE / FL / MPC
- 禁止：直接阻断

这里的关键不是“让人工记住规则”，而是生成正式策略快照：

- 允许的查询面
- 允许的模板类型
- 是否允许结果导出
- 是否允许明细导出
- 是否必须先试算
- 是否必须 attestation / proof

### 第 5 步：数据承诺与可持有证明

卖方不把原始数据交给平台时，平台仍需获得以下证据：

1. 版本承诺
   - 对资产版本生成 `content_hash / Merkle root`
   - 写入联盟链摘要
2. 结构承诺
   - schema、字段、时间范围、行数范围、质量阈值写入数据契约
3. 可用性证明
   - 平台托管：对象存在性与加密上传校验
   - 卖方自持：接口连通性、受控试算、健康检查
   - 第三方可信存储：托管证明、保留/销毁承诺
4. 高级可持有证明
   - `V2/V3` 可引入 PDP/POR 类证明或定期挑战式存在性校验

### 第 6 步：安全预览与预验收包

敏感数据不能用“给一份小样本文件”作为默认预览方式，必须按安全预览策略输出：

- schema 摘要
- 字段字典
- 行数/覆盖率/更新频率
- 质量报告摘要
- 脱敏样例
- 合成样例
- 受限模板试算结果
- query preview / report preview

预览策略必须区分：

- 可展示字段
- 必须掩码字段
- 不可预览字段
- 仅可展示统计摘要字段

### 第 7 步：受控模板/算法/环境审核

如果不是文件型商品，而是查询/计算型敏感商品，则必须审核：

- 查询模板版本
- 参数 schema
- 输出 schema
- join / project / filter 规则
- 最小聚合阈值
- 结果导出限制
- 算法镜像摘要
- 运行镜像签名
- 执行环境白名单

### 第 8 步：正式上架

敏感商品上架时，目录中只展示：

- 商品说明
- 使用场景
- 权利边界
- 可执行模板/服务能力摘要
- 结果形态
- 试算/预览方式
- 验收方式

不展示：

- 原始敏感样本
- 底层连接凭据
- 可逆敏感字段
- 公链原文证明

---

## 5.2 交易中闭环

### 第 9 步：买方下单前资格校验

下单前必须校验：

- 买方主体类型
- 买方用途声明
- 地域限制
- 商业输出权限
- 是否允许模型训练
- 是否满足高敏使用资格

不满足则：

- 自动阻断
- 或转人工复核

### 第 10 步：合同与执行策略固化

敏感订单签约时，必须把以下内容快照到合同和订单：

- 处理依据摘要
- 用途限制
- 查询/计算方式
- 输出边界
- 结果导出限制
- 撤权条件
- 验收条件
- 赔付触发条件
- 审计与链上摘要策略

### 第 11 步：支付锁定与公平性控制

敏感商品更需要“公平交付”，因此：

- 买方支付货款 / 保证金先锁定
- 卖方履约保证金按策略锁定
- 平台仅在交付条件满足后释放
- 对于分阶段交付，采用里程碑分段释放

公平性不是密码学绝对公平，而是：

- 支付状态可追踪
- 交付状态可验证
- 验收状态可回放
- 争议状态可冻结

---

## 5.3 交易后闭环

### 第 12 步：正式激活

按交付路径不同，正式激活方式不同：

1. 模板查询
   - 开通 `QueryGrant`
   - 绑定允许模板、参数、配额、结果边界
2. 沙箱
   - 开通 `SandboxWorkspace / Session`
   - 注入最小权限账号
3. API
   - 开通 `API Credential`
   - 生效配额、IP 白名单、用途限制
4. TEE / clean room / C2D
   - 校验执行环境身份
   - 校验 attestation
   - 满足后再释放密钥/数据访问
5. FL / MPC
   - 邀请参与方
   - 建立任务
   - 固化各方输入输出边界

### 第 13 步：受控执行

执行阶段必须记录：

- 谁发起
- 用了哪个订单
- 用了哪个模板/算法/任务
- 输入参数摘要
- 输出摘要
- 风控命中
- 审批结果
- 结果对象地址
- 结果 hash
- 运行环境摘要

### 第 14 步：输出控制

敏感结果默认必须经过输出控制，不允许“查询成功就等于可导出”。

至少支持以下控制：

- 最小聚合阈值
- 行数上限
- 列白名单
- disallowed output columns
- join / project 限制
- 参数范围限制
- 频率限制
- 重复相近查询阻断
- 差分隐私预算
- 导出审批
- 结果水印与指纹

说明：

- `V1` 先做阈值、模板、导出限制、次数配额、人工审批。
- `V2` 正式补齐差分隐私预算、analysis rule、join/project 细粒度规则。

### 第 15 步：结果预验收与正式验收

敏感商品的验收不能用“下载成功”替代。

应按商品类型分别验收：

1. 模板查询 / clean room
   - 首次成功执行
   - 输出 schema 匹配
   - 输出边界未越界
2. TEE / C2D
   - 环境证明通过
   - 任务执行成功
   - 结果模板匹配
3. API
   - 首次有效调用成功
   - 返回码/字段/延迟满足 SLA
4. FL / MPC
   - 任务完成
   - 结果摘要/聚合结果生成
   - 证明材料和轮次日志齐全
5. 报告 / 结果产品
   - 报告摘要与约定一致
   - 无越界导出

### 第 16 步：结算、赔付与撤权

满足验收后：

- 释放卖方收款
- 释放或调整双方保证金
- 平台抽成入账
- 若触发分润则再按规则拆分

到期、拒收、争议、违规时：

- 吊销 QueryGrant / API Credential / Sandbox Session
- 失效导出令牌
- 冻结后续结果导出
- 进入争议和赔付链路

### 第 17 步：归档、保留与销毁

敏感商品必须管理：

- 结果对象保留期限
- 中间结果清理策略
- 沙箱会话销毁
- 导出对象销毁
- 外部合作方销毁/保留回执
- `legal hold` 例外

### 第 18 步：审计与链上锚定

链上只锚定摘要，不锚定原始敏感正文。

建议锚定：

- 资产版本承诺
- 合同摘要
- QueryGrant 摘要
- 模板/算法摘要
- 执行回执摘要
- 结果对象 hash
- 验收摘要
- 结算摘要
- 争议/冻结摘要

## 6. 四条正式交付路径

## 6.1 路径 A：模板查询 / clean room

适合：

- 买方需要查询能力
- 不需要原始明文
- 支持 join / 聚合 / 受限导出

参考实践：

- AWS Clean Rooms analysis rules / analysis templates
- BigQuery clean rooms analysis rules / query templates
- Snowflake Data Clean Rooms templates / join-project-export controls

产品闭环：

`QuerySurface -> QueryTemplate -> QueryGrant -> QueryExecutionRun -> ResultArtifact`

## 6.2 路径 B：TEE Compute-to-Data

适合：

- 高敏原始数据不出域
- 需要运行指定算法或指定程序
- 需要对执行环境做证明

关键机制：

- 镜像摘要
- 远程证明
- attestation-based key release
- 结果模板校验

## 6.3 路径 C：联邦学习 / 联邦分析

适合：

- 多方联合建模
- 本地训练、本地评估
- 只交换参数、梯度或聚合结果

关键机制：

- 参与方资格
- 模型/计划摘要
- 轮次审计
- 聚合策略
- 贡献度与奖励

## 6.4 路径 D：MPC / PSI / 结果产品

适合：

- 交集求解
- 联合统计
- 多方评分
- 只需要结果不需要明文

关键机制：

- 输入边界
- 输出模板
- 结果摘要
- 成本计费
- 各方授权

## 7. 可证明性与证据栈

## 7.1 最小证据栈

每个高敏订单至少形成以下证据：

1. `AssetCommitment`
   - 版本 hash / Merkle root
2. `LegalBasisEvidence`
   - 处理依据、授权、许可、委托关系
3. `DataContractSnapshot`
   - schema、质量、用途、输出边界、验收条件
4. `SafePreviewArtifact`
   - 预览样例、试算结果、schema 摘要
5. `ExecutionPolicySnapshot`
   - 模板/算法/环境/导出限制
6. `AttestationOrProof`
   - TEE 证明、ZKP 证明、MPC 结果证明等
7. `ExecutionReceipt`
   - 执行回执、结果摘要、结果对象 hash
8. `AcceptanceEvidence`
   - 验收结果、拒收原因、自动验收事件
9. `SettlementEvidence`
   - 结算、赔付、冻结、退款回执

## 7.2 ZKP 的边界

ZKP 适合证明：

- 某个承诺与提交一致
- 某个结果由承诺输入计算得出
- 某个资格/属性成立但不暴露全部细节
- 某个阈值条件成立

ZKP 不应被误用为：

- 数据真实性兜底
- 权属合法性兜底
- 语义质量兜底
- 实际业务价值兜底

所以 ZKP 在本平台中属于增强证明层，而不是单点闭环方案。

---

## 8. 权限与审计基线

敏感数据链路必须比普通商品多一层“敏感使用治理”：

- 谁可看结构摘要
- 谁可看脱敏预览
- 谁可发起敏感查询
- 谁可发起高敏导出
- 谁可审批结果导出
- 谁可查看 attestation / proof
- 谁可查看未脱敏审计证据

所有敏感动作必须记录：

- `actor`
- `org`
- `role`
- `session_id`
- `device_id`
- `step_up_status`
- `order_id`
- `query_run_id / task_id`
- `result_object_id`
- `masked_level`
- `export_scope`
- `approval_ticket_id`

---

## 9. 推荐对象模型

建议新增或强化以下对象：

1. `SensitiveHandlingPolicy`
   - 定义不同敏感等级允许的交付模式、模板类型、导出规则、审批规则
2. `LegalBasisEvidence`
   - 记录个人信息/重要数据的处理依据、授权依据、许可依据
3. `SafePreviewArtifact`
   - 记录脱敏预览、合成预览、试算预览、结构预览
4. `SensitiveExecutionPolicy`
   - 记录模板、算法、环境、导出、证明要求
5. `AttestationRecord`
   - 记录 TEE / 环境证明摘要
6. `PrivacyBudgetLedger`
   - `V2` 差分隐私预算与消耗
7. `ResultDisclosureReview`
   - 记录结果导出审批、脱敏级别、导出范围
8. `DestructionAttestation`
   - 记录到期销毁、会话销毁、结果对象销毁回执

## 10. 分阶段实现建议

## 10.1 `V1`

必须完成：

- 高敏副本下载阻断
- 敏感等级驱动的交付路径决策
- 脱敏预览 / query preview / report preview
- `template_query_lite + sandbox_query + report_result + seller_hosted_api`
- 结果导出限制
- 敏感订单的增强审计与证据包
- 资产承诺、合同快照、结果摘要锚定

不承诺：

- 通用 clean room 平台
- 通用 TEE 平台
- 通用 MPC / PSI 编排平台
- 差分隐私预算全量引擎

## 10.2 `V2`

正式补齐：

- clean room
- analysis rules
- TVF / query templates
- TEE attestation
- C2D 任务
- 联邦训练 / 联邦评估
- MPC / PSI 结果产品
- 差分隐私预算

## 10.3 `V3`

扩展：

- 跨平台敏感查询互认
- 跨平台结果导出回执
- 外部销毁/保留证明
- 跨域监管联查与回放

## 11. 可参考的技术与实践

以下内容用于设计参考，不等于当前已锁定技术选型：

- AWS Clean Rooms：analysis rule、analysis template、受控 SQL/PySpark
- BigQuery Data Clean Rooms：analysis rules、query templates、TVF
- Snowflake Data Clean Rooms：template、join/project/export control、differential privacy
- OpenFL：联邦训练 / 联邦评估
- SecretFlow：MPC / HE / TEE / PSI / 混合隐私计算
- PySyft：remote data science / datasite
- Nitro Enclaves / Confidential Space / SGX 体系：attestation-based key release

---

## 12. 一句话结论

敏感数据交易的正确设计，不是“把数据交给平台再想办法保护”，而是把整条链路改造成：

**敏感数据不轻易出域 + 使用方式先被定义 + 执行过程可验证 + 输出结果被治理 + 结算与争议有证据**

只有这样，平台才能同时满足：

- 卖方不信任平台也能上架
- 买方不拿明文也能获得价值
- 平台不过度持有高敏原始数据
- 审计、赔付、撤权、锚定和监管都可闭环
