# 商品推荐与个性化发现设计

## 1. 目标

本设计用于补齐平台在“发现与转化”层面的能力缺口。推荐模块的目标不是替代搜索，也不是替代交易主状态机，而是在**严格遵守可见性、权限、合同、Usage Policy、风控与审计边界**的前提下，提升商品发现效率、商品服务配套转化率和高质量供方曝光效率。

平台推荐正式目标：

- 为数据商品、服务商品、卖方主体提供统一的推荐编排能力。
- 保持与双层权威模型一致，不让推荐系统成为新的业务真相源。
- 支持 `V1` 规则型推荐、`V2` 模型化增强推荐、`V3` 生态与伙伴推荐。
- 全链路留痕、可解释、可对账、可回放。

## 2. 设计原则

### 2.1 推荐是场景系统，不是单一算法系统

平台必须按推荐位和页面场景定义推荐逻辑，而不是试图先构建一个“大一统推荐模型”。首版推荐位应优先覆盖：

- 首页精选
- 行业专题推荐
- 商品详情页相似推荐
- 商品详情页配套服务推荐
- 卖方主页热门推荐
- 买方工作台发现推荐
- 搜索零结果兜底推荐

### 2.2 推荐是链下服务，不写进 chaincode

Fabric 提供可信事件和审计锚点，但推荐本身属于链下发现服务：

- 推荐计算在链下完成
- 推荐解释、审计和策略版本可做摘要上链
- 推荐结果不写入链上主状态
- 推荐不直接驱动交易主状态变更

### 2.3 推荐结果不能绕过业务放行

推荐系统即使返回候选，也必须在最终返回前经过：

`可见性校验 -> 主体资格校验 -> Usage Policy / 合同约束校验 -> 风控/冻结校验 -> PostgreSQL 最终业务校验`

### 2.4 推荐不单独引入地域限制逻辑

当前阶段不单独为推荐模块扩展额外的地域限制规则。推荐模块只复用平台既有的：

- 可见性规则
- 主体准入规则
- 商品状态规则
- 风控冻结规则

如果后续需要补充地域/司法辖区限制，则应复用平台统一放行结果，而不是在推荐域中另起一套规则。

## 3. 架构结论

平台推荐正式采用：

- `PostgreSQL`：商品主数据、行为事件、画像快照、推荐结果、推荐配置、审计主存
- `OpenSearch`：推荐候选召回与相似检索读模型
- `Redis`：推荐缓存、热门榜单缓存、已看集合缓存、短期去重状态
- `Kafka`：推荐行为流、商品变更流、信誉变化流、推荐结果回流总线
- `recommendation-service`：推荐编排与排序服务

统一结论：

**`PostgreSQL 主数据权威 + OpenSearch 候选召回 + Redis 缓存 + Kafka 事件同步 + recommendation-service 编排排序 + PostgreSQL 最终业务校验`**

## 4. 与搜索模块的关系

推荐模块与搜索模块共享商品/卖方搜索投影，但职责不同：

- 搜索：处理明确查询意图
- 推荐：处理发现、补充发现和无明确查询时的候选编排

推荐可以复用：

- `search.product_search_document`
- `search.seller_search_document`
- `search.model_search_document`
- `search.partner_search_document`
- `search.search_signal_aggregate`

但不能直接把搜索排序当成推荐排序。推荐必须有独立的：

- 推荐位定义
- 行为事件
- 候选召回链路
- 排序配置
- 结果解释
- 审计对象

## 7. 候选召回策略

推荐采用“多路召回 + 硬过滤 + 综合重排 + 最终业务校验”。

### 7.1 V1 候选召回源

- 热门召回
- 内容相似召回
- 聚合协同行为召回
- 新品探索召回
- 配套/捆绑召回
- 卖方主体召回

### 7.2 热门召回

基于以下行为或结果聚合：

- 曝光
- 点击
- 详情查看
- 样例查看
- 询价
- POC 申请
- 下单
- 支付成功
- 验收通过

### 7.3 内容相似召回

基于以下特征：

- 标签
- 类目
- use case
- 行业
- 交付方式
- SKU 类型
- 价格带
- 商品 embedding
- 卖方行业与能力标签

推荐的实现路径：

- `OpenSearch` 倒排 + 向量混合召回
- `pgvector` 作为 fallback 或轻量相似度能力

### 7.4 聚合协同行为召回

首版不直接按个人用户做强个性化，而以**组织级画像和 cohort** 为主：

- 同行业组织偏好
- 同主体类型偏好
- 同使用场景偏好
- 看过 X 的组织也常看 Y
- 采购 X 的组织也常采购 Y

必须满足：

- 聚合阈值保护
- 不直接泄漏单一买方行为

### 7.5 新品探索召回

用于解决新商品冷启动：

- 元数据完整
- 审核已通过
- 风险未阻断
- 卖方信誉达标
- 符合平台基础发现边界

### 7.6 配套召回

适合平台特性：

- 数据商品 -> 配套服务
- 数据商品 -> 报告服务
- 数据商品 -> API / 沙箱
- 模型 -> 推理服务
- 服务 -> 相关数据商品

## 8. 行为事件模型

推荐域必须独立建模行为事件，而不是完全依赖链码事件。

### 8.1 核心事件类型

#### 曝光事件

- `recommendation_panel_viewed`
- `recommendation_item_exposed`

#### 交互事件

- `recommendation_item_clicked`
- `recommendation_item_dismissed`
- `seller_recommendation_clicked`

#### 兴趣事件

- `product_detail_viewed`
- `service_detail_viewed`
- `seller_profile_viewed`
- `sample_preview_viewed`
- `usage_policy_viewed`
- `quote_requested`
- `poc_requested`

#### 转化事件

- `order_submitted`
- `payment_succeeded`
- `delivery_accepted`
- `subscription_renewed`

#### 负向事件

- `not_interested`
- `risk_blocked_after_click`
- `refund_or_dispute_after_order`

### 8.2 事件来源

推荐系统同时消费：

- 应用行为事件
- 订单/支付/验收等业务结果事件
- Fabric 摘要确认事件

Fabric 事件用于补强可信商业结果，不替代前端/后端产生的应用行为流。

## 9. 排序与重排

### 9.1 V1 采用规则型排序

`V1` 不强制上线复杂模型，而采用可解释综合评分：

```text
final_score =
  w_intent * intent_match_score
+ w_similarity * similarity_score
+ w_hotness * hotness_score
+ w_quality * quality_score
+ w_reputation * seller_reputation_score
+ w_freshness * freshness_score
+ w_conversion * conversion_score
+ w_bundle * bundle_affinity_score
- w_risk * risk_penalty
- w_repeat * repetition_penalty
```

### 9.2 V1 排序治理要求

- 先过滤，后排序
- 支持多样性控制
- 支持探索比例控制
- 支持新卖方/新品冷启动扶持
- 支持推荐解释码

### 9.3 推荐解释码

至少支持：

- `popular_overall`
- `popular_in_industry`
- `similar_to_current_item`
- `same_seller_more_items`
- `often_purchased_together`
- `new_and_qualified`
- `trusted_seller_boost`
- `service_bundle_match`

## 10. 开源中间服务策略

### 10.1 当前推荐

首版正式路线：

- 自建 `recommendation-service`
- 使用 PostgreSQL / OpenSearch / Redis / Kafka
- 推荐算法以规则 + 检索 + 聚合行为为主

### 10.2 可选开源组件

#### Gorse

定位：

- 候选召回增强器
- 协同推荐/相似推荐引擎

使用边界：

- 可接入用于候选生成
- 不作为最终放行与最终排序的唯一依据

#### LibRecommender

定位：

- `V2` 模型训练与在线推荐增强
- 更适合离线训练 + 线上服务化

#### RecBole

定位：

- 算法实验和离线评估
- 不建议首版直接作为线上依赖

### 10.3 当前不建议引入

- `Feast`
- `Flink`
- `Milvus`
- 图数据库

这些组件属于 `V2/V3` 之后的可选平台化能力，不是首版阻塞项。

## 12. 同步与一致性

推荐域遵循平台已冻结的双层权威模型：

- PostgreSQL 是推荐主数据与结果留痕权威
- Kafka 是行为和同步总线
- OpenSearch 是召回读模型
- Redis 是缓存

### 12.1 商品/卖方变更同步

```text
业务表写 PostgreSQL
  -> 刷新 search.* 投影
  -> 写 outbox_event(search.* changed)
  -> Kafka
  -> Search Indexer 更新 OpenSearch
  -> recommendation-service 后续读取最新搜索投影
```

### 12.2 行为事件同步

```text
行为事件入库 recommend.behavior_event
  -> 写 outbox_event(recommend.behavior_event)
  -> Kafka
  -> 特征聚合器更新 cohort_popularity / similarity / subject_profile_snapshot
  -> 可选投递给 Gorse / LibRecommender
```

### 12.3 推荐请求链路

```text
请求 recommendation-service
  -> 读取 placement_definition / ranking_profile
  -> 多路候选召回
  -> 硬过滤
  -> 综合重排
  -> PostgreSQL 最终业务校验
  -> 返回 recommendation_result
  -> 写曝光事件和审计
```

## 14. 审计与存证

推荐模块必须全程审计：

- 推荐请求
- 推荐位
- 候选来源
- 应用的过滤规则
- 排序配置版本
- 最终返回结果
- 点击/转化回流

不建议上链：

- 全量点击流
- 画像原文
- 原始推荐候选全量明细

可上链摘要：

- 推荐策略版本 hash
- 排序配置版本 hash
- 每日推荐审计摘要 hash

## 15. V1 / V2 / V3 结论

### 15.1 V1

- 完整规则型推荐
- 不做复杂千人千面
- 可选接入 `Gorse` 作为候选召回增强器
- 最终排序和放行由平台自有 recommendation-service 完成

### 15.2 V2

- 引入模型化推荐
- `Gorse` / `LibRecommender` 可接入
- 支持模型商品推荐与更强个性化

### 15.3 V3

- 页面级推荐编排
- 伙伴/生态推荐
- 多目标优化与更复杂实验能力

## 16. 闭环结论

新增本设计后，平台推荐闭环为：

```text
商品/服务/卖方主数据
  -> search.* 投影
  -> Kafka 同步
  -> OpenSearch 候选召回
  -> recommendation-service 多路召回与重排
  -> PostgreSQL 最终业务校验
  -> recommendation_result
  -> 曝光/点击/转化行为回流
  -> 画像/热度/相似度更新
  -> 审计与可回放
```

这与平台现有的搜索、双层权威模型、支付、审计、权限和数据库设计可以形成闭环。
