# 发布级基线文档硬化未决事项

本文件用于记录 `全集成文档/` 当前已经识别、但尚未彻底收口的发布级文档工程问题。

这些问题不改变产品方向、业务规则或阶段范围，但如果不继续处理，会影响：

- 研发把全集成文档当唯一冻结基线时的可执行性
- 后续 AI Agent / 自动化工具对基线正文的稳定读取
- 多人协作时对“冻结正文”和“边界说明”的理解一致性

---

## 1. 当前结论

当前已经完成：

- 四份全集成文档的统一结构收拢
- 分阶段内容切分与大部分发布级清洗
- 绝对路径、旧主 PRD 引用、旧岗位型角色的大规模收敛
- 角色、权限、数据库种子与页面访问角色的主线同步

当前尚未彻底完成的，不是“业务口径冲突”，而是“冻结正文是否已经足够硬”。

---

## 2. 当前仍应继续跟踪的问题

### 2.1 冻结正文仍残留 TODO / 占位 / 后补实现语句

当前状态：

- `全集成基线-全阶段`
- `全集成基线-V1`
- `全集成基线-V2`
- `全集成基线-V3`

中仍可见以下类型表述：

- `实现阶段再补`
- `TODO`
- `占位`
- `恢复后补写摘要`
- `可按部署时实际模型调整`

问题本质：

- 这类表述适合草稿或实施说明，不适合继续留在“冻结基线正文”
- 会导致读者无法判断这条规则是正式裁决，还是实现时仍可重议

建议收口：

- 正式冻结的规则，直接改写成硬约束
- 尚未冻结的内容，统一迁入“未决事项 / 实施偏差清单”

### 2.2 分阶段基线中的边界说明仍需继续压缩

当前状态：

- `V1 / V2` 版已经做过一轮阶段纯度清理
- 但正文中仍存在少量“后续阶段能力边界说明”与“当前阶段冻结正文”混排

问题本质：

- 这不一定造成业务冲突
- 但会降低阶段版作为“直接开发输入”的清晰度

建议收口：

- `V1` 仅保留会影响当前实现的 `V2/V3` 边界
- `V2` 仅保留会影响当前实现的 `V3` 边界
- 其余说明统一压缩到“本版排除事项”或附录

### 2.3 汇编文档与源文档之间仍存在少量兼容层表述

当前状态：

- 页面说明、权限设计、原始专题稿中仍保留少量“历史职责映射说明”
- 全集成文档为了可读性，仍保留了少量来源追溯信息与兼容注释

问题本质：

- 当前不阻塞开发
- 但会影响将来把全集成文档直接发布为唯一正式版

建议收口：

- 后续若决定将全集成文档作为唯一正式发布版，应再做一轮“去兼容层痕迹”整理

---

## 3. 进入最终发布版前应完成的动作

- 把全集成文档中的 TODO / 占位 / 后补实现语句全部迁出冻结正文
- 给四份全集成文档补一页明确的“本版新增 / 本版继承 / 本版排除”
- 再做一轮阶段纯度复核，确保 `V1/V2` 不混入后续阶段实现级正文
- 再做一轮发布级清洗，继续压缩历史兼容说明

---

## 4. 触发回写正式文档体系的条件

满足以下任一条件时，应将本文件中的事项迁回正式文档体系并关闭：

- 决定把 `全集成文档/` 作为唯一正式发布版
- 开始多人并行研发并以全集成文档作为唯一冻结输入
- 开始接入 AI Agent / 自动化文档解析作为开发前置流程
- 准备对外评审、立项答辩或进入正式交付审查

---

## 5. 一句话结论

当前全集成文档已经足够支撑研发进入下一步开发准备；  
剩余问题主要是“发布级硬化”和“基线正文去软化”，不再是产品方向或核心规则冲突。
