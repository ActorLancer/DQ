# 工程约束与上线前补齐事项

本文件用于记录已经识别、但当前不阻塞 `V1` 起步开发的工程问题。

这些事项不是“不要做”，而是“可以在主 PRD 已冻结最小工程约束的前提下，延后到更接近正式上线或规模扩展时补齐”。

---

## 1. 当前已冻结的最小结论

主 PRD 已明确以下内容：

- `V1` 权利枚举、标准 SKU、统一状态字段、一致性字段和字段字典唯一源
- 外部接口与回调的幂等、验签、`DLQ`、重放约束
- `V1` 租户隔离、对象存储路径、密钥层级、`break-glass` 基线
- 连接器与执行环境的最小生命周期
- 数据库迁移、索引 alias 切换、版本回填与回滚要求
- `V1` 容量模型、压测口径、自动化测试矩阵和固定审批节点

因此当前缺的已经不是“有没有规则”，而是“哪些高阶能力可以在后续阶段再平台化”。

---

## 2. 不阻塞 V1 起步开发、但上线前必须补齐的事项

### 2.1 每租户专属密钥体系

当前状态：

- `V1` 采用“环境主密钥 + 租户逻辑密钥域 + 对象级 `DEK`”已经足够支撑起步开发

尚未彻底完成：

- 每租户专属 `CMK`
- `BYOK`
- `HSM` 全量接管
- 密钥多方审批与分权托管自动化

建议收口：

- `V2` 开始评估 `BYOK / HSM`
- 正式生产扩大前完成高敏租户专属密钥方案

### 2.2 多协议共享连接器框架

当前状态：

- `V1` 已允许只读共享，但可按单后端/单协议/单连接器试点

尚未彻底完成：

- `Snowflake / BigQuery / Delta Sharing` 多协议统一接入
- 统一 revoke 回执
- 统一 attestation 与兼容矩阵平台

建议收口：

- `V2` 建共享连接器框架
- `V3` 再补跨平台责任链与证据互认

### 2.3 搜索索引平台化重建

当前状态：

- 主 PRD 已冻结版本索引、alias 切换、回填顺序

尚未彻底完成：

- 一键蓝绿重建平台
- 自动回填编排
- 大规模索引 schema 演进工具链

建议收口：

- `V1` 允许脚本化执行
- `V2` 再做平台化自动编排

### 2.4 容量二次校准

当前状态：

- 主 PRD 已给出 `V1` 开发与联调基线

尚未彻底完成：

- 基于真实交易量、真实对象存储日增量、真实审计吞吐的二次容量评估
- 压测结果反推 Kafka 分区、OpenSearch 分片、冷热存储分层参数

建议收口：

- 内测期收集一轮真实指标后回写正式容量模型

### 2.5 容灾与混沌演练平台化

当前状态：

- 主 PRD 已冻结 `RPO/RTO` 目标和最小故障演练矩阵

尚未彻底完成：

- 跨地域容灾演练
- 常态化混沌测试平台
- 自动故障注入与恢复报告

建议收口：

- `V1` 先做手工演练和脚本化验证
- 正式生产前补齐自动化体系

### 2.6 复杂审批流引擎

当前状态：

- `V1` 已有固定审批节点矩阵，足够避免研发各自实现一套审批逻辑

尚未彻底完成：

- 可视化工作流设计器
- 条件分支、加签、委托、会签、超时自动流转

建议收口：

- `V1` 继续坚持固定节点 + 简单规则
- `V2/V3` 再评估是否引入工作流引擎

---

## 3. 触发正式收敛的条件

满足以下任一条件时，应将对应事项迁回正式文档体系：

- 开始输出正式技术架构方案
- 开始做生产级部署方案
- 开始做高敏租户或监管租户落地
- 开始做多协议共享连接器 PoC
- 开始做大规模搜索/推荐/审计集群容量评审

---

## 4. 一句话结论

当前最关键的工程约束已经可以支撑 `V1` 开发起步；  
真正后置的是“如何把这些最小规则升级成平台化、自动化、规模化能力”。
