## 2. 历史问题起点（归档）

问题发现时，配置项和资源命名在文档、初始化脚本、runbook、本地样例和运行时入口之间存在多处漂移。

问题发现时已确认的典型现象：

1. 数据库在冻结文档里仍出现 `PG_HOST / PG_PORT / PG_DATABASE / PG_USER / PG_PASSWORD`，但本地 compose、`.env.local` 和脚本实际使用 `POSTGRES_*` 与 `DATABASE_URL`
2. MinIO 文档口径与运行时入口没有区分 `MINIO_ROOT_*`（bootstrap）和 `MINIO_ACCESS_KEY / MINIO_SECRET_KEY`（客户端）
3. Keycloak 文档仍把 `KEYCLOAK_ADMIN_USERNAME` 写成正式主配置名，但本地 stack 实际读取的是 `KEYCLOAK_ADMIN`
4. MinIO bucket、OpenSearch alias、compose 主入口和 Redis key 的收口要求已经提出，但没有和上述环境变量映射一起冻结成同一套命名字典

这意味着：

- 运维按文档配置环境时容易直接配错
- 本地 compose 能启动，但脚本或应用可能仍按旧名读不到配置
- MinIO bucket 与 OpenSearch alias 的初始化和运行时校验可能对不上
- compose 主入口口径不统一会影响本地联调和 runbook
- 搜索缓存失效和运维动作缺乏统一 key 命名规范

## 3. 正确冻结口径

当前阶段应满足以下正式口径：

### 3.1 数据库、MinIO、Keycloak 的变量必须分层

数据库、对象存储、Keycloak 的配置必须区分以下 3 层：

1. 本地基础设施 bootstrap 变量
2. 应用 / 脚本运行时入口变量
3. CLI / 临时调试兼容变量

正式映射如下：

- 数据库
  - bootstrap：`POSTGRES_DB`、`POSTGRES_USER`、`POSTGRES_PASSWORD`、`POSTGRES_PORT`
  - 运行时入口：`DATABASE_URL`
  - CLI 兼容：`PGHOST`、`PGPORT`、`PGDATABASE`、`PGUSER`、`PGPASSWORD`
  - 废弃为正式主配置名：`PG_HOST / PG_PORT / PG_DATABASE / PG_USER / PG_PASSWORD`
- MinIO
  - bootstrap：`MINIO_ROOT_USER`、`MINIO_ROOT_PASSWORD`
  - 运行时入口：`MINIO_ENDPOINT`、`MINIO_ACCESS_KEY`、`MINIO_SECRET_KEY`
  - 本地默认允许 `MINIO_ACCESS_KEY = MINIO_ROOT_USER`、`MINIO_SECRET_KEY = MINIO_ROOT_PASSWORD`
- Keycloak
  - bootstrap：`KEYCLOAK_ADMIN`、`KEYCLOAK_ADMIN_PASSWORD`
  - 运行时入口：`KEYCLOAK_BASE_URL`、`KEYCLOAK_REALM`、`KEYCLOAK_CLIENT_ID`、`KEYCLOAK_CLIENT_SECRET`
  - 废弃为正式主配置名：`KEYCLOAK_ADMIN_USERNAME`

### 3.2 配置与资源命名

- 配置清单
- 初始化脚本
- `.env.local`
- runbook
- smoke / seed / 本地工具脚本

必须共享同一套 env 名与资源名，不允许出现“文档一套、脚本一套、样例一套”的默认主路径。

### 3.3 compose 主入口

当前本地主入口应只有一个正式答案，相关 runbook、环境文档、启动说明必须一致。

### 3.4 Redis key 命名

`Redis` 不只是推荐缓存，还要覆盖搜索缓存与其失效策略，因此 key runbook 必须同时定义：

- 推荐缓存 key
- 搜索缓存 key
- 相关失效、重建、运维动作的 key 命名规则

## 5. 任务目标

把环境配置与资源命名收敛为单一正式答案，确保：

1. 数据库 / MinIO / Keycloak 的 bootstrap 与运行时入口有正式映射表
2. env 名、bucket 名、index alias 名在文档和运行时一致
3. compose 主入口只有一套正式口径
4. 搜索缓存 key 与推荐缓存 key 都有正式 runbook
5. 运维、脚本、样例、runbook 能共享同一命名字典

## 6. 强约束

1. 不能只改文档，不改 `.env.local`、compose 与脚本说明
2. 不能继续保留 `PG_HOST / PG_PORT / PG_DATABASE / PG_USER / PG_PASSWORD` 作为正式主配置名
3. 不能继续保留 `KEYCLOAK_ADMIN_USERNAME` 作为正式主配置名
4. 不能把 `MINIO_ROOT_*` 与 `MINIO_ACCESS_KEY / MINIO_SECRET_KEY` 混成同一层语义
5. 不能只补推荐缓存 key，不补搜索缓存 key
6. 不能继续让 compose 主入口同时存在“当前已确定”和“建议后续新增”两套表述

## 9. 验收标准

### 9.1 静态收口

以下状态必须成立：

- 数据库 / MinIO / Keycloak 的正式映射规则只有一套答案
- bucket env 命名只剩一套正式默认值
- OpenSearch alias env 命名只剩一套正式默认值
- compose 主入口只有一套正式表述
- `redis-keys.md` 同时覆盖搜索缓存与推荐缓存

### 9.2 联调可用性

至少验证：

1. `infra/docker/.env.local` 同时给出本地 bootstrap 与运行时入口样例
2. 本地 runbook、smoke、seed 脚本能按同一套名字读取数据库、MinIO、Keycloak
3. MinIO / OpenSearch 初始化脚本与运行时默认值一致
4. 搜索缓存与推荐缓存 key 命名可按 runbook 对上

### 9.3 运维可证明性

修复后应能明确回答：

- 当前数据库运行时入口是什么
- 当前 MinIO 运行时入口与 bootstrap 变量是什么
- 当前 Keycloak realm 正式变量是什么
- 当前 MinIO bucket env 名是什么
- 当前 OpenSearch alias env 名是什么
- 当前 compose 主入口是什么
- 当前搜索缓存 key 命名规范是什么

若这些问题仍有多套答案，则视为未完成收口。
