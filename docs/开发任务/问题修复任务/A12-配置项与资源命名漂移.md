# A12 配置项与资源命名漂移

## 1. 任务定位

- 问题编号：`A12`
- 严重级别：`medium`
- 关联阶段：`cross-stage`
- 关联任务：`ENV-011`、`ENV-013`、`ENV-016`、`ENV-017`、`ENV-058`、`AUD-022`、`SEARCHREC-006`
- 处理方式：先统一配置项、bucket、index alias、compose 主入口和 Redis key 命名，再继续搜索缓存、运维入口和 runbook 扩展

## 2. 问题描述

当前配置项和资源命名在文档、初始化脚本、运行时代码之间存在多处漂移。

当前已确认的典型现象：

1. 配置清单文档仍使用一套 `MINIO_BUCKET_* / OPENSEARCH_INDEX_*` 命名
2. 运行时代码和初始化脚本实际使用的是 `BUCKET_* / INDEX_ALIAS_*`
3. 本地主入口文档存在“建议后续新增 compose 文件”与“当前正式入口已明确”的冲突
4. Redis key runbook 只覆盖推荐缓存，没有覆盖搜索缓存 key

这意味着：

- 运维按文档配置环境时容易直接配错
- MinIO bucket 与 OpenSearch alias 的初始化和运行时校验可能对不上
- compose 主入口口径不统一会影响本地联调和 runbook
- 搜索缓存失效和运维动作缺乏统一 key 命名规范

## 3. 正确冻结口径

当前阶段应满足以下正式口径：

### 3.1 配置与资源命名

- 配置清单
- 初始化脚本
- 运行时代码
- runbook

必须共享同一套 env 名与资源名。

### 3.2 compose 主入口

当前本地主入口应只有一个正式答案，相关 runbook、环境文档、启动说明必须一致。

### 3.3 Redis key 命名

`Redis` 不只是推荐缓存，还要覆盖搜索缓存与其失效策略，因此 key runbook 必须同时定义：

- 推荐缓存 key
- 搜索缓存 key
- 相关失效、重建、运维动作的 key 命名规则

## 5. 任务目标

把环境配置与资源命名收敛为单一正式答案，确保：

1. env 名、bucket 名、index alias 名在文档和运行时一致
2. compose 主入口只有一套正式口径
3. 搜索缓存 key 与推荐缓存 key 都有正式 runbook
4. 运维、脚本、代码、runbook 能共享同一命名基线

## 6. 强约束

1. 不能只改代码，不改文档
2. 不能只改文档，不改初始化脚本与运行时默认值
3. 不能继续保留两套 env 名作为默认主路径
4. 不能只补推荐缓存 key，不补搜索缓存 key
5. 不能继续让 compose 主入口同时存在“当前已确定”和“建议后续新增”两套表述

## 7. 建议修复方案

### 7.1 先冻结统一 env 与资源命名字典

建议先建立一份正式字典，至少覆盖：

- MinIO bucket env
- OpenSearch alias env
- compose 主入口文件
- Redis 搜索缓存 key 前缀
- Redis 推荐缓存 key 前缀

该字典应成为：

- 配置清单
- 初始化脚本
- 运行时代码
- runbook

的统一执行基线。

### 7.2 统一 MinIO 与 OpenSearch 命名

必须在以下两者中收敛出唯一答案：

- 文档中的 `MINIO_BUCKET_* / OPENSEARCH_INDEX_*`
- 代码和脚本中的 `BUCKET_* / INDEX_ALIAS_*`

无论保留哪套，都必须同步收敛：

- 配置清单
- 初始化脚本
- 运行时读取逻辑
- runbook

### 7.3 收口 compose 主入口口径

必须明确：

- 当前本地主入口就是 [docker-compose.local.yml](/home/luna/Documents/DataB/infra/docker/docker-compose.local.yml)

并移除或修正文档中“建议后续新增 compose 主入口”的历史表述。

### 7.4 补齐 Redis key 规范

`docs/04-runbooks/redis-keys.md` 应至少补齐：

- 搜索结果缓存 key
- facet / 自动补全短缓存 key
- 搜索缓存失效 key / 记录方式
- 推荐缓存 key 与搜索缓存 key 的边界

### 7.5 对运行时做一致性校验

建议在应用启动或 smoke 中补一层一致性检查，至少验证：

- bucket env 是否齐备
- alias env 是否齐备
- compose/runbook 中引用的主入口是否一致

## 9. 验收标准

### 9.1 静态收口

以下状态必须成立：

- bucket env 命名只剩一套正式默认值
- OpenSearch alias env 命名只剩一套正式默认值
- compose 主入口只有一套正式表述
- `redis-keys.md` 同时覆盖搜索缓存与推荐缓存

### 9.2 动态验证

至少验证：

1. 运行时能读取与文档一致的 bucket env
2. 运行时能读取与文档一致的 alias env
3. MinIO / OpenSearch 初始化脚本与运行时默认值一致
4. 搜索缓存与推荐缓存 key 命名可按 runbook 对上

### 9.3 运维可证明性

修复后应能明确回答：

- 当前 MinIO bucket env 名是什么
- 当前 OpenSearch alias env 名是什么
- 当前 compose 主入口是什么
- 当前搜索缓存 key 命名规范是什么

若这些问题仍有多套答案，则视为未完成收口。

## 10. 输出要求

修复该问题的 Agent 在处理时应输出：

1. 变更文件清单
2. 最终 env / bucket / alias / compose / Redis key 字典
3. 文档与运行时对齐结果
4. 启动或 smoke 一致性验证结果

## 11. 一句话结论

`A12` 的核心问题不是“名字不好看”，而是配置项和资源命名已经发生文档/脚本/运行时漂移；如果不先收口，运维会直接按错误口径配置环境，搜索缓存与 alias 运维也会继续各说各话。
