## 2. 历史问题起点（归档）

问题起点是推荐阶段曾经基本停留在 schema / seed 层，缺少真正可执行的运行时主链路；本任务用于把该缺口收缩为单一正式实现。

问题发现时已确认的典型现象：

1. `recommendation` 模块曾为空壳
2. 曾经没有正式推荐接口实现
3. 曾经没有正式推荐 OpenAPI
4. 曾经没有正式推荐测试用例文档
5. 曾经没有 recommendation-service 或等价运行时编排能力
6. `recommend.behavior_event` 的 outbox 触发曾按表名自动派生 topic，而不是冻结口径的 `dtp.recommend.behavior`

这意味着：

- `SEARCHREC-008 ~ SEARCHREC-017` 没有可执行主链路
- 推荐接口、行为回流、ops 配置、重建能力无法被证明
- 一旦开始接行为流，topic 口径会先天漂移

## 3. 正确冻结口径

以接口协议和 PRD 为冻结基线，推荐阶段至少应具备以下正式能力：

### 3.1 推荐请求与行为接口

- `GET /api/v1/recommendations`
- `POST /api/v1/recommendations/track/exposure`
- `POST /api/v1/recommendations/track/click`

### 3.2 推荐运维接口

- `GET /api/v1/ops/recommendation/placements`
- `PATCH /api/v1/ops/recommendation/placements/{placement_code}`
- `GET /api/v1/ops/recommendation/ranking-profiles`
- `PATCH /api/v1/ops/recommendation/ranking-profiles/{id}`
- `POST /api/v1/ops/recommendation/rebuild`

### 3.3 推荐主链路

推荐正式链路应收敛为：

`PostgreSQL 主存 + OpenSearch 候选召回 + Redis 缓存 + Kafka 行为流 + PostgreSQL 最终业务校验 + outbox/Kafka/aggregator 回流`

### 3.4 推荐行为流

推荐行为流正式 topic 固定为 `dtp.recommend.behavior`，不得再按表名自动派生 topic。

## 5. 任务目标

补齐推荐阶段的正式闭环，确保：

1. 推荐模块不再只是 schema / seed
2. 推荐接口与运维接口真实可用
3. 推荐 OpenAPI、测试用例文档、runbook 成套落地
4. 推荐行为流遵循统一事件与 topic 口径
5. 推荐真正符合 `PG 主存 + OpenSearch 召回 + Redis 缓存 + Kafka 行为流 + PG 最终校验`

## 6. 强约束

1. 不能只补 schema，不补运行时实现
2. 不能只补接口，不补 OpenAPI / 测试 / runbook
3. 不能把推荐结果直接建立在 OpenSearch 或 Redis 之上而绕过 `PostgreSQL` 最终校验
4. 不能继续让推荐行为流走自动派生 topic
5. 不能只做“推荐占位页/占位接口”来冒充阶段完成
6. 不能把 recommendation-service 的职责完全省略不实现

## 7. 建议修复方案

### 7.1 先补最小正式推荐运行时

至少建立并接入：

- `apps/platform-core::recommendation`
- 或明确的等价推荐编排服务/模块

要求：

- 能读取 placement / ranking profile
- 能做最小候选召回
- 能执行硬过滤
- 能在返回前做 `PostgreSQL` 最终业务校验

### 7.2 先补最小正式接口族

优先补齐：

- `GET /api/v1/recommendations`
- `POST /track/exposure`
- `POST /track/click`
- `GET/PATCH /api/v1/ops/recommendation/placements*`
- `GET/PATCH /api/v1/ops/recommendation/ranking-profiles*`
- `POST /api/v1/ops/recommendation/rebuild`

### 7.3 明确 recommendation-service 边界

必须明确：

- 当前阶段是否在 `platform-core` 内实现推荐编排
- 还是新增独立 `recommendation-service`

但无论采用哪种组织方式，都必须满足冻结能力，不允许继续只有 schema。

### 7.4 修正推荐行为流 topic 产生方式

重点处理：

- `recommend.behavior_event` 当前仍复用 `common.tg_write_outbox()`

要求：

- 推荐行为流正式 topic = `dtp.recommend.behavior`
- 不再依赖按表名自动派生 topic
- 行为事件应进入统一 outbox / event route 收口体系
- 不再重新打开“推荐行为事件应发往哪个 topic”的选择空间

### 7.5 同步补齐契约和测试资产

至少新增或补齐：

- `packages/openapi/recommendation.yaml`
- `docs/02-openapi/recommendation.yaml`
