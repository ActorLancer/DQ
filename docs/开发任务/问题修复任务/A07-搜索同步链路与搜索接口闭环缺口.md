# A07 搜索同步链路与搜索接口闭环缺口

## 1. 任务定位

- 问题编号：`A07`
- 严重级别：`blocker`
- 关联阶段：`SEARCHREC`
- 关联任务：`SEARCHREC-001` 至 `SEARCHREC-007`、`AUD-022`
- 处理方式：先把搜索同步链路、搜索 API、search-indexer、ops 搜索运维接口收敛成一条正式闭环，再继续推荐与后续增强

## 2. 问题描述

当前搜索同步链路没有真正收缩，至少同时存在三套事件/契约口径：

1. `search.sync`
2. `dtp.search.sync`
3. `search.product.changed`

同时，搜索主链路的关键运行时组件仍未落地：

1. `search-indexer` 仍为空骨架
2. `GET /api/v1/catalog/search` 尚未实现
3. `ops` 搜索同步/重建/缓存失效/排序相关接口尚未形成正式入口

当前已确认的典型现象：

- `catalog` 仍手工写 `search.product.changed`
- `SEARCHREC-001` 自身又写成“消费 `search.sync` 主题或 DB 作业表”
- `search` 模块仍为空壳
- `workers/search-indexer` 仍只有 `.gitkeep`
- 主 router 未挂载搜索正式接口

这意味着：

- Search worker、搜索 API、ops 接口和 catalog 事件写法无法对成一条链
- 很难证明已实现的是冻结设计中的“正式搜索闭环”
- 后续推荐模块会建立在不稳定搜索基线之上

## 3. 正确冻结口径

以任务清单、PRD 和接口协议为冻结基线，V1 搜索链必须收敛为：

`PostgreSQL 主数据/搜索投影 -> outbox -> Kafka -> search-indexer -> OpenSearch -> Redis -> PostgreSQL 最终业务校验`

并且至少应具备以下正式能力：

### 3.1 搜索接口

- `GET /api/v1/catalog/search`

### 3.2 搜索运维接口

- `GET /api/v1/ops/search/sync`
- `POST /api/v1/ops/search/reindex`
- `POST /api/v1/ops/search/cache/invalidate`
- `GET /api/v1/ops/search/ranking-profiles`
- `PATCH /api/v1/ops/search/ranking-profiles/{id}`

### 3.3 搜索同步约束

- 搜索投影权威源在 `PostgreSQL`
- 搜索同步正式 topic 固定为 `dtp.search.sync`
- `search.sync` 与 `search.product.changed` 只允许作为历史说明或迁移映射残留，不再作为运行时默认 topic
- 同步总线应使用正式冻结 topic
- `search-indexer` 写 `OpenSearch`
- `Redis` 只做缓存
- 搜索结果返回前必须回 `PostgreSQL` 做最终状态/可见性/可售性校验

### 3.4 运行模式与 fallback 约束

- `staging / production` 必须启用 `OpenSearch`，并以其作为正式候选召回源
- `local / demo` 允许只用 `PostgreSQL` 搜索投影表运行，不得因为缺少 `OpenSearch` alias / index 就无法启动搜索链路
- `PostgreSQL fallback` 只替代候选集来源，不替代最终业务校验
- 无论使用 `OpenSearch` 还是 `PostgreSQL fallback`，搜索结果返回前都必须回 `PostgreSQL` 做最终状态/可见性/可售性校验

## 5. 任务目标

补齐并收敛搜索正式链路，确保：

1. 搜索同步事件契约只剩一套正式口径
2. `search-indexer` 成为真实可运行进程
3. `GET /api/v1/catalog/search` 正式可用
4. `ops` 搜索同步/重建/缓存失效/排序配置接口可用
5. 搜索链真正符合 `PG -> Kafka -> indexer -> OpenSearch -> Redis -> PG final check`

## 6. 强约束

1. 不能只补搜索 API，不补 `search-indexer`
2. 不能只补 `search-indexer`，不补搜索 API
3. 不能继续允许 `search.sync`、`dtp.search.sync`、`search.product.changed` 三套契约并存
4. 不能继续让 `catalog` 手工写搜索事件而不经过统一收口
5. 不能把 OpenSearch 或 Redis 当成搜索结果最终真相源
6. 不能把 “消费 DB 作业表” 和 “消费 Kafka topic” 同时保留为默认主方案而不明确唯一正式路径

## 7. 建议修复方案

### 7.1 先冻结唯一搜索同步契约

必须直接收口为：

- 搜索同步正式 topic = `dtp.search.sync`
- 搜索同步正式 envelope 与全局 canonical envelope 一致，遵循 `A02`
- `catalog` / 其他上游模块只允许通过统一事件路由产生搜索同步事件

要求：

- 搜索同步与全局事件口径一致
- 不再继续保留历史搜索事件名作为主默认路径
- 修复任务本身不再重新打开 `search.sync / dtp.search.sync / search.product.changed` 的选择空间

### 7.2 实现正式 `search-indexer`

`workers/search-indexer` 至少应具备：

- 消费正式搜索同步事件
- 从 `PostgreSQL` 读取最新搜索投影
- 写入 `OpenSearch`
- 更新同步状态
- 失效相关 `Redis` 缓存

### 7.3 实现正式搜索 API

`GET /api/v1/catalog/search` 至少应具备：

- 查 `Redis`
- 未命中则查 `OpenSearch`
- 返回候选结果后回 `PostgreSQL` 做最终业务校验
- 过滤冻结、不可见、不可售对象

补充要求：

- `staging / production` 默认使用 `OpenSearch candidate -> PostgreSQL final check`
- `local / demo` 必须允许 `PostgreSQL search projection candidate -> PostgreSQL final check`
- 不允许把“本地默认必须先初始化 OpenSearch”继续当作唯一合法口径

### 7.4 补齐运行模式分流与启动检查

至少明确并落地：

- 搜索候选源运行模式配置
- `OpenSearch required` 与 `PG fallback` 两种模式
- 启动自检按运行模式决定是否强制 `OpenSearch` alias / index 存在

要求：

- 生产和联调环境继续强制 `OpenSearch`
- `local / demo` 允许在未初始化 `OpenSearch` 的情况下启动并运行搜索接口
- 不能把 `PG fallback` 实现成直接绕过搜索投影、直扫业务主表的旁路方案

### 7.4 补齐 ops 搜索运维接口

至少实现：

- 搜索同步状态查看
- 重建
- 缓存失效
- 排序配置读取/更新

这些能力也是 `AUD-022` 的依赖面，不能继续缺席。

### 7.5 清理 catalog 旁路写法

重点处理：

- `catalog` 手工写 `search.product.changed`

要求：

- 上游模块不再各自定义私有搜索事件旁路
- 搜索同步统一进入正式搜索链路

## 9. 验收标准

### 9.1 静态收口

以下状态必须成立：

- `search` 模块不再为空壳
- `workers/search-indexer` 不再只有 `.gitkeep`
- 主 router 已挂载 `GET /api/v1/catalog/search`
- ops 搜索接口已挂载
- `catalog` 不再默认手工写私有搜索事件旁路

### 9.2 动态验证

至少验证：

1. 商品变更写入搜索同步事件
2. `search-indexer` 消费后写入 `OpenSearch`
3. 相关 `Redis` 缓存失效
4. `GET /api/v1/catalog/search` 返回结果
5. 结果返回前已做 `PostgreSQL` 最终校验
6. `POST /api/v1/ops/search/reindex` 和 `POST /api/v1/ops/search/cache/invalidate` 可用
7. `local / demo` 在未启用 `OpenSearch` 时，仍可通过 `PostgreSQL fallback` 返回搜索结果
8. `staging / production` 仍以 `OpenSearch` 作为正式候选召回源

### 9.3 契约一致性

修复后应能明确回答：

- 搜索同步正式 topic 是什么
- 答案必须唯一为 `dtp.search.sync`
- 搜索 API 正式入口是什么
- `search-indexer` 正式消费什么
- ops 搜索运维接口有哪些

若仍存在多套默认答案，则视为未完成收口。

1. 变更文件清单

## 11. 一句话结论

`A07` 的核心问题不是“少了一个搜索接口”，而是整个搜索同步链路、搜索 API 和 search-indexer 尚未收敛成一条正式闭环；如果不先统一这条链，`SEARCHREC` 阶段后续所有实现都会建立在漂移基线之上。
