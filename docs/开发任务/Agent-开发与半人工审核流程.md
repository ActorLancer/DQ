# Agent 开发与半人工审核流程

## 1. 目的

本流程用于约束后续 AI Agent 按 `V1-Core` 开发任务清单实施时的执行方式、记录方式、人工审批方式和未实现能力的标注方式，避免以下问题：

- 偏离平台目标和 `V1-Core` 范围
- 跳过前置任务直接实现后续模块
- 做了代码但没有完整覆盖任务要求
- 当前阶段未实现的能力没有被明确记录
- `V2/V3` 预留能力缺少代码注释和清单汇总

本流程是 `开发任务/` 目录下所有执行任务的强制规则。

## 2. 适用范围

- 当前只适用于 `V1-Core`
- 适用于单 Agent 顺序执行模型
- 适用于“Agent 实施 + 人工审批”的交替推进方式

## 3. 单一事实源

执行时必须遵守以下优先级：

1. `开发任务/v1-core-开发任务清单.csv`
2. `开发任务/Agent-开发与半人工审核流程.md`
3. `开发任务/AI-Agent-执行提示词.md`
4. `开发任务/v1-core-开发任务清单.md`
5. `全集成文档/数据交易平台-全集成基线-V1.md`
6. `开发准备/*.md`

若出现冲突：

- 任务分发、依赖、范围以 `CSV` 为准
- 执行方式、记录方式、暂停条件以本流程文件为准
- 业务设计和工程约束以 `全集成文档/V1` 与 `开发准备` 为准

补充说明：

- `开发任务/问题修复任务/A*.md` 若被任务清单引用，只能作为“问题治理与剩余边界说明”使用。
- 其中 `## 2. 历史问题起点（归档）` 与 `## 4. 历史问题证据（归档）` 用于说明问题发现时的起点，不得直接视为当前实现状态。
- 当前执行状态、当前运行规则和后续实现义务，仍以 `CSV / TODO / README / runbook / scripts` 为准。

## 4. 参与角色

### 4.1 执行 Agent

负责：

- 选择当前要执行的任务批次
- 基于冻结文档实现代码、配置、脚本、文档
- 完成验证
- 更新实施日志
- 更新 TODO 与预留清单
- 提交待人工审批的信息

不负责：

- 擅自修改 `V1/V2/V3` 边界
- 擅自裁决冻结文档冲突
- 在审批前继续推进下一批任务

### 4.2 人工审批者

负责：

- 审核当前批次是否真正完成
- 判断是否偏离目标
- 判断是否需要返工
- 判断 TODO 和预留说明是否充分
- 明确是否允许进入下一批任务

## 5. 批次执行规则

### 5.1 批次大小

每次只能提交一个“实现批次”进入审批。

实现批次的合法粒度：

- 1 个复杂任务
- 或 2 到 4 个连续的简单任务

复杂任务通常包括：

- 涉及数据库迁移
- 涉及接口契约
- 涉及状态机
- 涉及权限校验
- 涉及事件发布或消费
- 涉及跨模块联动
- 涉及支付、审计、授权、交付、结算、争议

简单任务通常包括：

- 文档收口
- 枚举补齐
- 小范围配置
- 单点页面骨架
- 单点只读查询

### 5.2 批次约束

一个批次内的多个任务必须同时满足：

- 属于同一 `wave` 或相邻 `wave`
- 共享主要前置依赖
- 不跨越完全不同的业务域
- 能在一次人工审批中判断是否完成

禁止把下列内容混成一个批次：

- 身份登录 + 支付结算
- 数据库迁移 + 大量页面开发
- 搜索索引 + 交易状态机 + 审计回放

## 6. 每批次标准流程

### 6.0 建立批次计划

在开始当前批次前，执行 Agent 必须先通过 `开发任务/V1-Core-实施进度日志.md` 确认当前活动分卷，再在该分卷中写一条“计划中”记录，至少包含：

- 批次编号
- 当前任务编号列表
- 当前批次目标
- 前置依赖核对结果
- 预计涉及文件
- 预计验证方式

未写“计划中”记录，不得开始编码。

### 6.1 选择任务批次

执行 Agent 必须先明确：

- 当前批次任务编号
- 当前批次目标
- 当前批次前置依赖
- 当前批次涉及的冻结文档

### 6.2 开始实现前检查

实现前必须检查：

- 前置任务是否已完成
- 当前仓库已有产物是否应复用
- 当前批次是否会修改数据库、接口、事件、权限、状态机
- 是否需要为 `V2/V3` 做明确预留

### 6.3 实现

实现时必须：

- 只实现当前批次范围内的内容
- 同步修改必要的代码、配置、脚本、测试、文档
- 不得把 `V2/V3` 正式实现混入 `V1-Core`

### 6.4 记录实施日志

每个批次有两次强制日志动作：

1. 开始前写“计划中”
2. 完成后写“待审批”

日志入口固定为：

- `开发任务/V1-Core-实施进度日志.md`

日志正文必须按入口页规则追加写入当前活动分卷；流程文件、README 和审批模板统一只引用入口页，不直接把某个 `P1/P2/P3` 分卷写成长期固定入口。

日志必须包含：

1. 批次编号
2. 当前任务编号列表
3. 当前批次目标
4. 前置依赖核对结果
5. 已实现功能
6. 涉及文件
7. 验证步骤
8. 验证结果
9. 覆盖的冻结文档条目
10. 覆盖的任务清单条目
11. 未覆盖项
12. 新增 TODO / 预留项
13. 待人工审批结论

### 6.5 记录 TODO / 预留项

如果当前阶段未实现，但必须为后续预留，必须同时做两件事：

1. 在代码中写清晰注释
2. 在汇总文件中登记

汇总文件固定为：

- `开发任务/V1-Core-TODO与预留清单.md`

### 6.6 人工审批

每个批次完成后，必须暂停，等待人工审批。

人工审批结果必须写入：

- `开发任务/V1-Core-人工审批记录.md`

人工审批只能给出三种结论：

- `通过`
- `需补充后复审`
- `驳回并返工`

### 6.7 审批通过后才能进入下一批

如果人工审批未通过，执行 Agent 不得继续推进下一批任务。

### 6.8 批次返工规则

若人工审批结论为：

- `需补充后复审`
- `驳回并返工`

则执行 Agent 必须：

1. 回到当前批次，不得新开批次
2. 在实施进度日志中追加“返工”记录
3. 明确本轮返工只修哪些问题
4. 返工后重新提交人工审批

## 7.1A 阶段预留与当前缺口的区分规则

必须区分以下三类未完成项：

- `V1-gap`
  - 本阶段本来就应该完成，但当前批次尚未完成
  - 原则上应尽快补齐，不能长期堆积
- `V2-reserved`
  - 本阶段不实现，但必须预留明确扩展点
- `V3-reserved`
  - 本阶段不实现，但必须预留明确扩展点

禁止把本应属于 `V1-gap` 的缺口伪装成 `V2/V3-reserved`。

## 7. TODO 注释规则

### 7.1 代码注释格式

统一使用以下格式：

```text
TODO(V1-gap, TASK-ID): 说明当前未完成项、原因、补齐条件、验收方式。
TODO(V2-reserved, TASK-ID): 说明为 V2 预留的接口/扩展点，当前不实现的原因，后续接入方式。
TODO(V3-reserved, TASK-ID): 说明为 V3 预留的接口/扩展点，当前不实现的原因，后续接入方式。
```

### 7.2 TODO 最小要求

每个 TODO 必须写清：

- 对应任务编号
- 未实现的具体内容
- 当前不实现的原因
- 后续应在哪个阶段补齐
- 未来补齐时需要修改的模块或文件

禁止只写：

- `TODO`
- `later`
- `fix me`

## 8. TODO 汇总清单规则

`开发任务/V1-Core-TODO与预留清单.md` 中每条记录必须包含：

1. 编号
2. 对应任务编号
3. 类型：`V1-gap / V2-reserved / V3-reserved / tech-debt`
4. 模块
5. 文件路径
6. 当前状态
7. 原因
8. 后续补齐条件
9. 是否阻塞继续开发

## 9. 必做验证

每个批次至少执行以下验证中的适用项：

- 编译通过
- 数据库迁移通过
- 接口单测通过
- 事件契约校验通过
- 权限校验通过
- 页面交互或 API 手测通过
- 审计记录可追踪
- TODO 汇总已同步

若当前批次涉及以下内容，验证不可省略：

- 数据库迁移：必须跑迁移验证
- 接口：必须至少做接口级验证
- 权限：必须验证未授权/已授权两类路径
- 事件：必须验证事件落库或发布
- 状态机：必须验证状态推进和非法状态拦截
- TODO 预留：必须验证代码注释和 TODO 汇总都存在

## 10. 强制暂停条件

出现以下任一情况，必须暂停并等待人工指示：

- 前置任务未完成
- 当前批次目标不清晰
- 冻结文档互相冲突
- 实现结果无法覆盖任务要求
- 数据库迁移失败
- 接口或事件契约不一致
- 权限校验缺失
- 关键状态机行为未闭环
- 当前任务需要跨到 `V2/V3` 才能完成

## 11. 批次完成定义

一个批次只有同时满足以下条件才算完成：

- 当前批次任务全部实现
- 当前批次验证完成
- 当前批次日志已追加
- TODO 与预留项已同步
- 人工审批记录已填写
- 审批结论为“通过”

## 12. 建议的执行节奏

- 每个复杂任务单独成批
- 每批结束就停
- 人工审批不通过时只修当前批，不并入下一批
- TODO 数量持续上升时，应优先安排收敛批，不要无限堆积

## 13. Agent 换手规则

如果中途更换执行 Agent，新的 Agent 在继续前必须先阅读：

1. `开发任务/v1-core-开发任务清单.csv`
2. `开发任务/Agent-开发与半人工审核流程.md`
3. `开发任务/V1-Core-实施进度日志.md`（入口页；再进入当前活动分卷）
4. `开发任务/V1-Core-TODO与预留清单.md`
5. `开发任务/V1-Core-人工审批记录.md`

新的 Agent 不得跳过历史记录直接继续编码。

## 14. 人工审批的最小核对项

人工审批每次至少检查：

1. 当前任务编号是否正确
2. 前置依赖是否真的满足
3. 已实现内容是否覆盖任务要求
4. 是否偏离 `V1-Core` 范围
5. 是否错误引入 `V2/V3` 正式实现
6. 是否补齐了必要的测试或验证
7. 是否补齐了必要的审计、权限、错误码、事件或数据库同步
8. TODO 是否充分且分类正确
9. 是否允许进入下一批

## 15. 不允许的行为

以下行为一律视为违规执行：

- 跳过前置任务
- 批次未审批通过就继续实现
- 只改代码，不更新实施日志
- 只写 TODO 注释，不登记 TODO 汇总
- 把 V1 应做内容标成 V2/V3 预留
- 擅自改动冻结口径
- 在未说明影响范围的前提下重构已有骨架
