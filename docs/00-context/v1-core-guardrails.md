# V1-Core 守则冻结（CTX-002）

## 1. 目标

冻结 `V1-Core` 实施的不可变基础原则，确保后续模块、接口、事件和联调实现遵循同一事实源与状态权威。

## 2. 业务状态与技术边界

1. `PostgreSQL` 是业务主状态权威。
2. `Fabric` 只承担摘要登记、可信确认和可验证证明，不承载业务主状态机。
3. `Kafka` 是事件分发与异步编排总线，不是业务真值。
4. `OpenSearch` 与推荐缓存仅为读模型和召回层，不可绕过主库校验直接放行业务结果。
5. 链回执、支付回执、索引同步结果都必须回写并通过 `PostgreSQL` 规则校验后才可影响业务对象状态。

## 3. 实现约束

1. 关键动作必须同事务落：
   - 业务主对象
   - 审计记录
   - `outbox_event`
2. 业务接口不得同步阻塞等待链确认；链动作走异步路径（`outbox -> Kafka -> adapter/listener -> 回写`）。
3. 外围进程（adapter/worker/provider）不得直接定义订单、合同、授权、交付、结算最终状态。
4. 搜索、推荐、报表、链镜像状态均需回主库做可见性与权限校验。

## 4. 违规判定（即偏离 V1-Core）

- 把 `Kafka/OpenSearch/Redis/Fabric` 的状态当成主事实源。
- 在 handler 内硬编码外部调用并跳过审计/outbox。
- 让外围进程直接推进生命周期核心对象最终状态。
- 用 `V2/V3` 能力替代 `V1` 最小闭环应实现路径。
