# 当前仓库现状与目标差距分析（CTX-014）

## 1. 现状输入（基于 CTX-022/023/024）

当前仓库已存在：

- 单仓多应用骨架（`apps/*`、`packages/*`、`infra/*`、`scripts/*`、`docs/*`）
- `apps/platform-core` 最小服务（Rust + `/healthz`）
- 本地部署资产（`部署脚本/docker-compose.local.yml`、迁移与环境检查脚本）
- 完整的任务与冻结文档目录

## 2. 与目标基线的主要差距

1. 应用实现深度不足：
   - 除 `platform-core` 最小服务外，多数应用仍为 `README` 占位。
2. 运行时收口不足：
   - 本地 compose 可运行，但 `alertmanager`、`otel-collector`、topic/bucket 初始化、Fabric 脚本化能力未收口。
3. 工程治理不足：
   - 缺少 `.github/` CI 流程与自动化验收管线。
4. 文档与执行记录需要收口：
   - 进度日志混合模板/示例/实记录，后续需在文档收口任务中规整。
5. OpenAPI 与 runbook 仍待完善：
   - 当前冻结文档齐全，但面向实现的接口归档和正式 runbook 需在后续任务中落地。

## 3. 复用与迁移策略

1. 优先复用现有目录和脚本，不做无意义重建。
2. 对路径不一致问题采用“迁移 + 兼容”策略逐步收敛（例如 `部署脚本/` 到 `infra/docker/`）。
3. 以 CSV `depends_on` 为准串行推进，避免跨域并行改造造成重复实现。

## 4. 结论

仓库已具备增量实施基础，不需要推倒重建；后续重点应放在：

- 按任务清单补齐可运行能力
- 收口运行时与契约资产
- 建立 CI 与正式验收路径
