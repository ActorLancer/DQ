## 2. 协议原则

- `product` 表示商品聚合。
- `product_type` 表示商品主类型，统一使用：`data_product / service_product / result_product`。
- `sku_type` 表示标准 SKU 真值。
- `sku_code` 表示商品内实现型商业套餐编码。
- `trade_mode` 表示交付/计费路径，不替代标准 SKU 真值。
- `delivery_mode` 统一使用：`file_download / revision_push / share_grant / api_access / template_grant / sandbox_workspace / report_delivery`。
- `pricing_mode` 统一使用：`one_time / subscription / pay_per_use / project_fee`。
- 若数据库持久化层使用 `catalog.product.delivery_type / price_mode`，应与本协议中的 `delivery_mode / pricing_mode` 一一映射。
- `SHARE_RO / QRY_LITE / SBX_STD` 必须分别配置、分别绑定模板、分别开通。

### 2.1 逻辑字段与持久化字段映射

- 商品、页面、搜索、合同、鉴权和审计联查统一使用逻辑字段名，不直接暴露数据库列名
- 当前冻结映射如下：
  - `delivery_mode -> catalog.product.delivery_type`
  - `pricing_mode -> catalog.product.price_mode`
  - `current_state -> trade.order_main.status`
  - `order_amount -> trade.order_main.amount`
  - `metered_quantity -> billing.billing_event.units`
- 当商品接口返回订单、账单、支付或交付摘要时：
  - 应返回 `current_state`，不得直接返回无语义上下文的 `status`
  - 应返回 `order_amount`，不得直接返回无语义上下文的 `amount`
  - 计量型账单摘要应返回 `metered_quantity`，不得直接以 `units` 对外暴露

## 3. 标准 SKU 真值

`V1` 仅允许：

- `FILE_STD`
- `FILE_SUB`
- `SHARE_RO`
- `API_SUB`
- `API_PPU`
- `QRY_LITE`
- `SBX_STD`
- `RPT_STD`

## 4. 公共请求头

- `X-Request-Id`
- `X-Idempotency-Key`
- `Authorization: Bearer <access_token>`

## 5. V1 接口

### 5.1 商品

- `POST /api/v1/products`
- `PATCH /api/v1/products/{id}`
- `POST /api/v1/products/{id}/submit`
- `GET /api/v1/products/{id}`

### 5.2 SKU

- `POST /api/v1/products/{id}/skus`
- `PATCH /api/v1/skus/{id}`

请求关键字段：

- `sku_type`
- `sku_code`
- `trade_mode`
- `billing_mode`
- `acceptance_mode`
- `refund_mode`
- `quota_json`
- `sla_json`

校验规则：

- `sku_type` 必须命中标准 SKU 真值
- `sku_code` 必须在 `product_id` 内唯一
- `trade_mode` 必须与 `sku_type` 匹配
- `SHARE_RO / QRY_LITE / SBX_STD` 必须带各自专属字段

### 5.3 模板绑定

- `POST /api/v1/products/{id}/bind-template`
- `POST /api/v1/skus/{id}/bind-template`
- `GET /api/v1/templates/{id}`

模板绑定规则：

- `SHARE_RO` 只允许绑定共享模板族
- `QRY_LITE` 只允许绑定模板查询模板族
- `SBX_STD` 只允许绑定沙箱模板族

### 5.4 元信息与数据契约

- `PUT /api/v1/products/{id}/metadata-profile`
- `POST /api/v1/assets/{versionId}/field-definitions`
- `POST /api/v1/assets/{versionId}/quality-reports`
- `POST /api/v1/assets/{versionId}/processing-jobs`
- `POST /api/v1/skus/{id}/data-contracts`
- `GET /api/v1/skus/{id}/data-contracts/{contractId}`

### 5.5 原始接入与可交付对象

- `POST /api/v1/assets/{assetId}/raw-ingest-batches`
- `POST /api/v1/raw-ingest-batches/{id}/manifests`
- `POST /api/v1/raw-object-manifests/{id}/detect-format`
- `POST /api/v1/raw-object-manifests/{id}/extraction-jobs`
- `POST /api/v1/assets/{versionId}/preview-artifacts`
- `POST /api/v1/assets/{versionId}/objects`
- `PATCH /api/v1/assets/{assetId}/release-policy`

### 5.6 查询面

- `POST /api/v1/products/{id}/query-surfaces`
- `POST /api/v1/query-surfaces/{id}/templates`

## 6. 错误码建议

- `SKU_TYPE_INVALID`
- `SKU_CODE_DUPLICATED`
- `TRADE_MODE_MISMATCH`
- `TEMPLATE_SKU_TYPE_MISMATCH`
- `QUERY_SURFACE_INVALID`
- `PRODUCT_SUBMIT_FIELDS_INCOMPLETE`

## 7. 一句话结论

目录与商品接口必须把“商品聚合、标准 SKU 真值、商业套餐编码、模板绑定、查询面配置”统一成一套明确语义，不能再混用。
