# 数据交易平台数据库设计总说明

## 1. 设计范围

本说明对应当前全部文档的统一数据库实现基线，覆盖以下核心域：

1. 身份与主体
2. 角色、权限与作用域
3. 商品、版本、模板、策略
4. 订单、交付、验收、账单、争议
5. 支付、资金流、收费规则、对账与清结算
6. 保证金、质押、惩罚、奖励、分润
7. 审计、证据、日志、开发调试
8. 模型、算法、受控计算、联邦任务、证明
9. 联盟链摘要、公链锚定、凭证
10. 跨链请求、见证、ack、补偿
11. 图风控、监管穿透、冻结治理、生态互联

## 2. 数据库分层

建议使用以下 schema：

- `common`：公共函数与通用触发器
- `core`：组织、用户、部门、应用、证书、环境、连接器
- `authz`：角色、权限、角色权限、主体角色绑定
- `catalog`：数据资产、版本、商品、SKU、标签、结构化小样本
- `contract`：模板、合同、策略、签署与授权
- `trade`：询单、订单、订单行、状态历史
- `delivery`：对象、交付、令牌、回执、API 凭证、沙箱、报告
- `billing`：账单、托管、钱包、台账、保证金、退款、赔付、惩罚、奖励、分润
- `payment`：支付渠道、支付意图、交易、Webhook、打款、对账、结算路由
- `support`：争议、证据、裁决
- `risk`：评分、信誉、黑名单、风险告警、图风控
- `audit`：审计事件、证据包、监管查询
- `chain`：链事件投影、公链锚定、凭证
- `ml`：模型、算法、任务、轮次、证明、执行结果
- `crosschain`：跨链请求、ack、见证、网关、补偿
- `ecosystem`：合作伙伴、连接器版本、互认关系
- `developer`：测试应用、测试钱包、mock 绑定
- `search`：倒排与向量检索文档
- `ops`：outbox、dead letter、作业、系统日志

## 3. 设计原则

- 业务主状态在链下数据库
- 链上存摘要、签名、承诺、状态变更和关键证据哈希
- 全部高增长表都预留分区策略或独立索引策略
- 搜索采用 PostgreSQL 全文检索 + `pg_trgm`
- 向量检索采用 `pgvector`
- 图风控优先使用关系边表建模，必要时外接图计算引擎

## 4. 关键对象落库策略

### 4.1 大对象与训练数据

- 原始交易数据、密文数据集、模型文件、证据原文、训练日志原文默认存对象存储
- PostgreSQL 保存：
  - 元数据
  - 哈希
  - 路径
  - 索引字段
  - 生命周期状态
  - 审计与访问记录

### 4.2 小型结构化数据

对于小型交易数据集、样本集、标签集、统计维表，可在 PostgreSQL 中落以下模式：

- `catalog.asset_structured_dataset`
- `catalog.asset_structured_row`

用于：

- 小样本展示
- 轻量聚合查询
- 训练样本预览
- 规则验证

## 5. 搜索与向量设计

### 5.1 倒排检索

主要使用：

- `tsvector`
- `GIN`
- `pg_trgm`

索引对象：

- 商品目录
- 模型目录
- 审计联查摘要
- 风险告警摘要

### 5.2 向量检索

使用 `vector(1536)` 作为默认 embedding 维度占位，可按部署时实际模型调整。

建议索引对象：

- 商品语义搜索
- 模型语义搜索
- 风险文本摘要聚类

## 6. 触发器与自动化

建议统一使用以下触发器：

- `updated_at` 自动更新时间
- 搜索文档自动同步
- 关键状态变更写历史表
- 关键业务表写 outbox_event

## 7. 迁移执行顺序

### 7.1 V1

1. 扩展、schema、公共函数
2. 身份、权限与主体
3. 商品、模板与策略
4. 订单、交付
5. 账单、质押、争议、信誉
6. 审计、搜索、开发调试、运维
7. V1 角色与权限种子数据

### 7.2 V2

1. 模型、算法、受控计算、联邦任务、证明
2. 分润、公链增强与凭证
3. V2 角色与权限种子数据

### 7.3 V3

1. 跨链请求、ack、见证、补偿、连接器与合作伙伴
2. 图风控、监管、冻结治理与互认
3. V3 角色与权限种子数据

## 8. 命名规范

- 主键统一使用 `uuid`
- 主表命名使用单数语义，便于和领域对象直接对应
- 状态字段统一命名为 `status`
- 时间戳统一使用 `timestamptz`
- JSON 扩展字段统一使用 `jsonb`
- 所有业务主表统一包含：
  - `created_at`
  - `updated_at`

## 9. 一句话结论

这套数据库设计不是把 `V1` 做成一套短期模型，而是用 `V1` 先落稳定骨架，让 `V2/V3` 在相同 schema 体系和对象主键体系上自然扩展，不再返工。

## 10. 存储与信任边界落库策略

### 10.1 新增核心对象

- `catalog.asset_custody_profile`
- `catalog.asset_trust_evidence`

### 10.2 新增核心快照字段

- `trade.order_main.storage_mode_snapshot`
- `trade.order_main.delivery_route_snapshot`
- `trade.order_main.trust_boundary_snapshot`
- `delivery.delivery_record.source_binding_id`
- `delivery.delivery_record.executor_type`
- `delivery.delivery_record.trust_boundary_snapshot`

### 10.3 模型与跨平台扩展字段

- `ml.model_asset.artifact_storage_mode`
- `ml.model_asset.weight_custody_mode`
- `ml.training_task.training_data_residency_mode`
- `ml.compute_result.result_export_policy`
- `ecosystem.partner.partner_storage_capability`
- `ecosystem.partner.partner_key_governance_capability`
- `crosschain.cross_chain_request.trust_scope_snapshot`

### 10.4 设计原则补充

- 平台数据库必须能表达“平台不接触明文”的正式模式。
- 平台数据库必须能表达“平台托管但不独占解密能力”的正式模式。
- 所有交付、执行、跨平台互联都必须保留信任边界快照，避免后续策略变更导致历史交易无法追溯。

## 11. 支付、资金流与轻结算落库策略

### 11.1 新增 schema

- `payment`：支付渠道、支付意图、支付交易、Webhook、打款、对账、结算路由

### 11.2 新增核心对象

- `payment.provider`
- `payment.provider_account`
- `payment.payment_intent`
- `payment.payment_transaction`
- `payment.payment_webhook_event`
- `payment.refund_intent`
- `payment.payout_instruction`
- `payment.reconciliation_statement`
- `payment.reconciliation_diff`
- `billing.fee_rule`
- `billing.fee_rule_version`
- `billing.fee_preview`
- `developer.mock_payment_case`

### 11.3 分阶段增量

- `V1`：真实支付适配层、收费规则、费用快照、Mock 支付、人工打款、对账差异。
- `V2`：自动打款、渠道分账/子商户扩展、周期扣费。
- `V3`：多币种、FX 快照、数字资产/交易所结算路由。
