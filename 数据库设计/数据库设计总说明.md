# 数据交易平台数据库设计总说明

## 1. 设计范围

本说明对应当前全部文档的统一数据库实现基线，覆盖以下核心域：

1. 身份与主体
2. 角色、权限与作用域
3. 商品、版本、模板、策略
4. 订单、交付、验收、账单、争议
5. 支付、资金流、收费规则、对账与清结算
6. 保证金、质押、惩罚、奖励、分润
7. 审计、证据、日志、开发调试
8. 模型、算法、受控计算、联邦任务、证明
9. 联盟链摘要、公链锚定、凭证
10. 跨链请求、见证、ack、补偿
11. 图风控、监管穿透、冻结治理、生态互联

## 2. 数据库分层

建议使用以下 schema：

- `common`：公共函数与通用触发器
- `core`：组织、用户、部门、应用、证书、环境、连接器
- `authz`：角色、权限、角色权限、主体角色绑定
- `catalog`：数据资产、版本、商品、SKU、标签、结构化小样本
- `contract`：模板、合同、策略、签署与授权
- `trade`：询单、订单、订单行、状态历史
- `delivery`：对象、交付、令牌、回执、API 凭证、沙箱、报告
- `billing`：账单、托管、钱包、台账、保证金、退款、赔付、惩罚、奖励、分润
- `payment`：支付渠道、司法辖区、走廊策略、收款偏好、支付意图、交易、Webhook、打款、对账、结算路由
- `support`：争议、证据、裁决
- `risk`：评分、信誉、黑名单、风险告警、图风控
- `audit`：审计事件、证据包、监管查询
- `chain`：链事件投影、公链锚定、凭证
- `ml`：模型、算法、任务、轮次、证明、执行结果
- `crosschain`：跨链请求、ack、见证、网关、补偿
- `ecosystem`：合作伙伴、连接器版本、互认关系
- `developer`：测试应用、测试钱包、mock 绑定
- `search`：倒排与向量检索文档
- `ops`：outbox、dead letter、作业、系统日志

## 3. 设计原则

- 业务运营主状态在链下数据库
- Fabric 负责关键承诺与可信证明确认层
- 支付 / CA / 对象存储等外部系统负责其来源事实回执
- 链上存摘要、签名、承诺、状态变更和关键证据哈希
- 全部高增长表都预留分区策略或独立索引策略
- 搜索采用 PostgreSQL 全文检索 + `pg_trgm`
- 向量检索采用 `pgvector`
- 图风控优先使用关系边表建模，必要时外接图计算引擎

## 4. 关键对象落库策略

### 4.1 大对象与训练数据

- 原始交易数据、密文数据集、模型文件、证据原文、训练日志原文默认存对象存储
- PostgreSQL 保存：
  - 元数据
  - 哈希
  - 路径
  - 索引字段
  - 生命周期状态
  - 审计与访问记录

### 4.2 小型结构化数据

对于小型交易数据集、样本集、标签集、统计维表，可在 PostgreSQL 中落以下模式：

- `catalog.asset_structured_dataset`
- `catalog.asset_structured_row`

用于：

- 小样本展示
- 轻量聚合查询
- 训练样本预览
- 规则验证

## 5. 搜索与向量设计

### 5.1 倒排检索

主要使用：

- `tsvector`
- `GIN`
- `pg_trgm`

索引对象：

- 商品目录
- 模型目录
- 审计联查摘要
- 风险告警摘要

### 5.2 向量检索

使用 `vector(1536)` 作为默认 embedding 维度占位，可按部署时实际模型调整。

建议索引对象：

- 商品语义搜索
- 模型语义搜索
- 风险文本摘要聚类

## 6. 触发器与自动化

建议统一使用以下触发器：

- `updated_at` 自动更新时间
- 搜索文档自动同步
- 关键状态变更写历史表
- 关键业务表写 outbox_event
- `ops.outbox_event`、`ops.dead_letter_event` 生产环境建议按 `created_at` 归档或分区

## 7. 迁移执行顺序

### 7.1 V1

1. 扩展、schema、公共函数
2. 身份、权限、会话与主体
3. 商品、模板与策略
4. 订单、交付
5. 账单、质押、争议、信誉
6. 审计、搜索、开发调试、运维
7. 审计、证据链与回放硬化
8. 双层权威模型与一致性增强
9. V1 角色与权限种子数据

### 7.2 V2

1. 模型、算法、受控计算、联邦任务、证明
2. 分润、公链增强与凭证
3. 双层权威模型增量
4. V2 角色与权限种子数据

### 7.3 V3

1. 跨链请求、ack、见证、补偿、连接器与合作伙伴
2. 图风控、监管、冻结治理与互认
3. 双层权威模型增量
4. V3 角色与权限种子数据

## 8. 命名规范

- 主键统一使用 `uuid`
- 主表命名使用单数语义，便于和领域对象直接对应
- 状态字段统一命名为 `status`
- 时间戳统一使用 `timestamptz`
- JSON 扩展字段统一使用 `jsonb`
- 所有业务主表统一包含：
  - `created_at`
  - `updated_at`

## 9. 一句话结论

这套数据库设计不是把 `V1` 做成一套短期模型，而是用 `V1` 先落稳定骨架，让 `V2/V3` 在相同 schema 体系和对象主键体系上自然扩展，不再返工。

### 10.4 设计原则补充

- 平台数据库必须能表达“平台不接触明文”的正式模式。
- 平台数据库必须能表达“平台托管但不独占解密能力”的正式模式。
- 所有交付、执行、跨平台互联都必须保留信任边界快照，避免后续策略变更导致历史交易无法追溯。

- `audit.replay_result`
- `audit.legal_hold`
- `audit.access_audit`

### 13.2 核心字段增强

- `audit.audit_event` 增加：`event_schema_version`、`event_class`、`actor_org_id`、`session_id`、`trusted_device_id`、`application_id`、`error_code`、`before_state_digest`、`after_state_digest`、`previous_event_hash`、`event_hash`、`evidence_manifest_id`、`retention_class`、`legal_hold_status`
- `ops.system_log` 增加：`traceparent`、`span_id`、`previous_log_hash`、`log_hash`、`retention_class`

### 13.3 设计原则

- 审计事件和系统日志默认按追加写处理
- 原始证据正文继续放对象存储或可信存储，数据库保存元数据、清单、hash 和保全状态
- 审计访问本身也必须落库

## 14. 双层权威模型与一致性落库策略

### 14.1 V1 核心对象

- `ops.event_route_policy`
- `ops.outbox_publish_attempt`
- `ops.consumer_idempotency_record`

### 14.2 核心字段增强

- `trade.order_main` / `contract.digital_contract` / `delivery.delivery_record` / `payment.payment_intent` / `billing.settlement_record` 增加：
  - `authority_model`
  - `proof_commit_state`
  - `proof_commit_policy`
  - `external_fact_status`
  - `reconcile_status`
- `ops.outbox_event` 增加：
  - `request_id`
  - `trace_id`
  - `target_bus`
  - `target_topic`
  - `payload_hash`
  - `max_retries`
- `ops.dead_letter_event` 增加：
  - `failure_stage`
  - `reprocess_status`

### 14.3 设计原则

- Kafka 是事件总线，不承担主状态权威。
- 数据库事务内必须同时写业务对象、审计事件和 outbox。
- 外部事实型对象必须保留来源回执映射。
