## 2. 存储策略总览

## 2.1 核心原则

- PostgreSQL 保存平台的主业务状态、元数据、索引、审计、关系和可检索摘要。
- 原始交易数据、加密数据包、模型二进制、证明原文、证据包原文、训练日志原文默认不直接存 PostgreSQL。
- 大对象统一落对象存储或受控存储系统，数据库保存其 `URI`、`HASH`、摘要、生命周期状态和审计引用。
- 链上只存摘要、承诺、签名、关键状态和可验证凭证；链下数据库是业务运营主状态机，Fabric 负责可信证明确认层。

## 2.2 “交易的数据”怎么存

交易的数据不是一张“data_blob”大表直接塞库，而是拆成三层：

1. `catalog.data_asset`
   - 定义数据资产本体
   - 存资产归属、类型、敏感级别、描述、业务元数据

2. `catalog.asset_version`
   - 定义资产版本
   - 存版本号、schema 版本、schema/hash、样本 hash、全量 hash、区域限制、状态

3. `catalog.asset_storage_binding`
   - 定义实际数据体存储位置
   - 存对象存储 URI、对象 hash、加密算法、WORM 标记等

补充层：

- `catalog.asset_sample`
  - 存样本 URI 或样本 payload，用于目录展示和预览
- `catalog.asset_structured_dataset`
  - 定义可直接放入 PostgreSQL 的小型结构化数据集
- `catalog.asset_structured_row`
  - 存小样本或轻量结构化行，支持后续 embedding

结论：

- 交易平台真正成交的大数据文件，设计上是“链下对象存储 + PostgreSQL 元数据引用”
- PostgreSQL 并不承载大体量原始数据本体
- 只有小样本、小型结构化集、目录展示样本、索引摘要会进库

## 2.3 训练后的模型怎么存

模型采用“元数据 + 版本 + 文件地址”的方式：

1. `ml.model_asset`
   - 模型资产主对象
   - 存模型名称、家族、所有者、来源任务、摘要指标、哈希、状态

2. `ml.model_version`
   - 模型版本
   - 存版本号、模型文件 URI、模型哈希、状态

3. `ml.algorithm_artifact`
   - 算法/镜像/运行工件
   - 存镜像 URI、镜像 digest、白名单状态、审核状态

4. `ml.compute_result`
   - 任务输出结果
   - 存结果 URI、结果 hash、导出状态、审核状态

结论：

- 训练后的模型文件本体默认不进 PostgreSQL
- PostgreSQL 存模型元数据、版本关系、工件路径、哈希、指标、权限与审计关联

## 2.4 搜索、倒排、向量怎么存

搜索不是直接扫业务主表，而是采用“PostgreSQL 搜索投影主表 + OpenSearch 搜索读模型 + Redis 缓存”的分层方式：

- `search.product_search_document`
  - 商品/服务全文索引投影
  - PostgreSQL 中保存权威搜索投影和索引同步状态

- `search.seller_search_document`
  - 卖方主体搜索投影
  - 支撑卖方主页和卖方维度过滤

- `search.model_search_document`
  - 模型全文索引投影
  - 同时支持 `tsvector` 和 `pgvector`

- `search.partner_search_document`
  - V3 伙伴/生态搜索投影

- `search.search_signal_aggregate`
  - 曝光、点击、成交、热度等排序信号聚合

- `search.ranking_profile`
  - 综合排序权重与过滤规则

- `search.index_alias_binding`
  - OpenSearch 读写别名绑定

- `search.index_sync_task`
  - 索引同步、reindex、修复任务

结论：

- 倒排：`tsvector + GIN`
- 模糊：`pg_trgm`
- 向量：`vector(1536) + ivfflat`
- PostgreSQL 是搜索投影主表和 fallback 检索基线
- OpenSearch 是商品/卖方/模型/伙伴搜索读模型
- Redis 只做结果和 facet 缓存

## 2.5 链上摘要、凭证和跨链怎么存

- 联盟链/链事件投影：
  - `chain.contract_event_projection`
  - `chain.chain_anchor`

- 公链增强：
  - `chain.public_anchor_batch`
  - `chain.credential_token`
  - `chain.credential_status_history`

- 跨链治理：
  - `crosschain.cross_chain_request`
  - `crosschain.cross_chain_ack`
  - `crosschain.witness_record`
  - `crosschain.request_status_history`
  - `crosschain.compensation_task`

结论：

- PostgreSQL 存链下编排状态、映射关系、摘要和回执
- 链本身不是主业务数据库

## 2.6 新增的“存储与信任边界”怎么存

新增采用以下对象：

1. `catalog.asset_custody_profile`
   - 描述资产或版本采用的平台托管、卖方自持、受控执行或第三方可信存储模式
   - 保存 `storage_mode / payload_location_type / custody_mode / key_control_mode / retention_policy / destroy_policy`

2. `catalog.asset_trust_evidence`
   - 保存卖方自持证明、第三方托管证明、执行边界证明、销毁承诺等信任证据

3. `trade.order_main`
   - 保存 `storage_mode_snapshot / delivery_route_snapshot / trust_boundary_snapshot`

4. `delivery.delivery_record`
   - 保存 `source_binding_id / executor_type / delivery_route / trust_boundary_snapshot`

5. `ml.model_asset / ml.training_task / ml.compute_result`
   - 保存模型和训练任务的托管边界、驻留边界和结果导出边界

6. `ecosystem.partner / crosschain.cross_chain_request`
   - 保存跨平台托管能力、密钥治理能力和责任边界快照

## 3. 通用字段与约定

### 3.1 通用主键

- 业务主表默认使用 `uuid`
- 角色和权限使用业务编码主键：
  - `authz.role_definition.role_key`
  - `authz.permission_definition.permission_code`
- 代币主数据使用自然编码：
  - `billing.token_asset.token_code`

### 3.2 通用时间字段

- `created_at`：创建时间
- `updated_at`：更新时间
- `started_at / finished_at`：任务起止
- `issued_at / revoked_at`：证书/凭证生命周期
- `effective_from / effective_to`：授权或角色生效周期

### 3.3 通用 JSON 字段

- `metadata jsonb`：扩展元数据
- `attrs jsonb`：用户/角色绑定附加属性
- `*_json jsonb`：复杂规则快照
- `payload jsonb`：消息或作业载荷

### 3.4 通用触发器

- `common.tg_set_updated_at`
  - 自动维护 `updated_at`
- `common.tg_order_status_history`
  - 自动记录订单状态变更
- `common.tg_dispute_status_history`
  - 自动记录争议状态变更
- `common.tg_write_outbox`
  - 关键业务表写入 outbox 事件
- `search.tg_refresh_product_search_document`
  - 商品变更自动刷新商品/服务搜索投影
- `search.refresh_seller_search_document_by_id`
  - 卖方主体搜索投影刷新
- `search.refresh_model_search_document_by_id`
  - 模型搜索投影刷新
- `search.refresh_partner_search_document_by_id`
  - 伙伴搜索投影刷新

## 6. 数据存储边界结论

### 6.1 应该放 PostgreSQL 的

- 组织、用户、角色、权限、应用、连接器、环境
- 数据资产元信息、版本、策略、模板、商品目录、SKU
- 订单、合同、授权、交付状态、账单、保证金、惩罚、赔付
- 审计事件、证据包摘要、日志、异步消息、状态历史
- 搜索文档、向量索引
- 模型元数据、模型版本、训练任务、证明、分润结果
- 链上锚定摘要、公链凭证元数据、跨链请求和补偿状态

### 6.2 不应该直接放 PostgreSQL 的

- TB 级原始交易数据
- 大规模密文数据包
- 模型完整二进制权重文件
- 完整训练日志原文
- 大型证据原文包
- 大文件报告附件

这些内容应放：

- 对象存储
- 受控文件存储
- 专用模型仓库
- 归档存储

数据库只保留：

- URI
- HASH
- 摘要
- 元信息
- 状态
- 审计引用

## 7. 是否已经完整设计

如果你的问题是：

- “交易平台需要交易的数据本体、元信息、模型、搜索索引、日志、证据、角色权限有没有设计？”

答案是：**有，而且已经形成完整分层。**

但要注意这个“完整”是架构上的完整，不是说 PostgreSQL 要存一切原始内容。当前设计是：

- 用 PostgreSQL 管理平台核心状态与元数据
- 用对象存储承载大对象和大文件
- 用 `pgvector` 承担轻量向量检索
- 用链上系统承载公开验证和凭证

这是正确的数据库边界，不是缺失。
