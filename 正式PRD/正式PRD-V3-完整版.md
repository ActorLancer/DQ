## 1. 文档目标

V3 的目标是把平台从“单平台可信交易与协作系统”升级为：

**跨域互联、跨平台协同、可监管、可扩展的可信数据空间基础设施**

本文件完整展开：

- 跨链请求治理
- 连接器互联
- 数据空间互认
- 图风控
- 高级监管穿透
- 生态协作开放

---

## 2. 版本定位与目标

### 2.1 V3 的平台定位

V3 不是简单功能叠加，而是平台形态升级：

- 从单平台交易 -> 多平台协作
- 从本地规则放行 -> 跨域规则协同
- 从单平台证据链 -> 跨平台证据链
- 从单点风控 -> 图风控与关系风控

### 2.2 核心目标

1. 实现跨平台连接器互联。
2. 实现跨区域/跨联盟链协同。
3. 建立跨链请求治理与确认机制。
4. 落地图风控、异常关系识别与高级风险治理。
5. 打通监管只读穿透视图与冻结处置能力。
6. 形成生态合作与能力开放底座。
7. 支持数据资产登记、融资、质押辅助服务的底层能力。

### 2.3 非目标

V3 仍不追求：

- 公链替代联盟链主状态机
- 原始高敏数据自由交易
- 匿名开放式代币化数据市场

## 6. 核心业务流程

### 6.1 跨链授权同步链路

1. 本地授权生效
2. 生成跨链请求对象
3. 本地签名
4. 发送至目标链或目标平台
5. 目标侧按 request_id 幂等处理
6. 目标侧返回 ack
7. 本地更新 final_status
8. 审计留痕

### 6.2 跨链结算确认链路

1. 本地账单结算完成
2. 生成结算确认请求
3. 发送目标侧
4. 目标侧确认
5. 回写 ack 与 final_status
6. 异常进入重试/补偿/人工介入

### 6.3 冻结同步链路

1. 本地风险命中
2. 执行冻结
3. 生成冻结同步请求
4. 目标侧同步冻结
5. 更新同步状态
6. 审计留痕

### 6.4 图风控链路

1. 汇总主体关系、设备关系、IP 关系、订单关系、评分关系、授权传播关系
2. 构建关系图
3. 识别异常群体、异常链路、黑名单绕过、洗订单、串谋评分
4. 生成告警
5. 转工单
6. 人工处置与冻结同步

### 6.5 监管穿透链路

1. 监管主体发起只读查询
2. 系统按监管权限聚合订单、合同、授权、跨链请求、证据对象
3. 输出穿透视图
4. 记录监管访问日志

## 7. 功能需求明细

### 7.1 跨链请求治理

必须支持：

- 唯一 request_id
- source_chain_id / target_chain_id
- request_type
- payload_hash
- nonce
- timestamp
- gateway_signature
- witness_signature_set
- ack_hash / ack_status
- final_status
- timeout
- retry_count
- terminate_flag
- compensate_flag

### 7.2 幂等与补偿

必须支持：

- 目标侧按 request_id 幂等处理
- 重复回执不得导致重复授权、重复结算、重复放行
- 超时重试
- 人工终止
- 补偿任务
- 死信事件池

### 7.3 图风控

必须支持图谱输入字段：

- 主体关系
- 设备关系
- IP 关系
- 订单关系
- 评分关系
- 授权传播关系

必须支持图风控输出：

- 风险分
- 告警类型
- 告警对象集合
- 可疑链路说明
- 建议处置动作

### 7.4 监管穿透

必须支持：

- 监管只读视图
- 穿透至订单、合同、授权、账单、跨链请求、证据对象
- 冻结/解冻记录
- 导出监管证明材料

### 7.5 生态互联

必须支持：

- 标准连接器接入规范
- 连接器身份校验
- 连接器版本与兼容性校验
- 互认关系登记
- 合作伙伴能力标识

---

## 8. 规则与治理

### 8.1 跨链治理规则

- 跨链成功不能只以“消息已发出”为准，必须有目标侧 ack 或等价确认
- 跨链失败不得污染本地核心状态
- 涉及授权同步、结算确认、冻结同步的跨链请求必须纳入审计视图与风险工单

### 8.2 图风控规则

必须识别：

- 串谋评分
- 洗订单
- 异常转售链路
- 高频异常下载群体
- 关联主体绕过黑名单或额度限制

### 8.3 监管规则

- 监管穿透仅限只读
- 监管查看权限与业务查看权限分离
- 监管查询必须留痕

## 10. 技术架构与边界

### 10.1 跨链治理边界

- 跨链请求是业务对象，不是简单消息
- 必须具备幂等、回执、超时和补偿
- 不能因为跨链失败污染本地主状态

### 10.2 连接器边界

- 连接器必须具备独立身份
- 必须经过版本、兼容性、策略校验
- 不得绕过本地策略引擎

### 10.3 公链边界

- 公链仍然只做公开验证与增强层
- 不承担核心仲裁和核心结算主状态

### 10.4 预言机 / 见证人边界

- 必须有独立服务身份
- 必须有签名留痕
- 不得共享平台管理员口令

## 17. 数据本体存储与信任边界

### 17.1 V3 扩展目标

`V3` 在前两阶段基础上，正式支持：

- 第三方可信存储
- 跨平台连接器互联
- 跨域托管与执行能力互认
- 面向监管的存储责任链和销毁责任链

### 17.2 跨域互联的边界约束

- 跨平台互联不等于平台接管合作伙伴的明文数据。
- 跨链与跨平台只同步必要的元数据、摘要、策略快照、回执、证据和责任边界。
- 对接方必须暴露其托管能力、密钥治理能力、受控执行能力、保留/销毁承诺和信任证据。

### 17.3 V3 必记字段与证据

跨平台/跨域对象至少记录：

- `partner_storage_capability`
- `partner_key_governance_capability`
- `partner_execution_boundary_capability`
- `cross_platform_trust_scope`
- `external_destroy_attestation`
- `retention_obligation_snapshot`

### 17.4 V3 治理原则

- 跨链治理必须同时携带业务状态与信任边界状态。
- 监管穿透必须能看到“数据放在哪里、谁能解密、谁实际执行、保留多久、何时销毁”。
- 互认关系必须以能力和责任互认为前提，而不是单纯的链间连通。

## 18. 支付、资金流与轻结算

### 18.1 V3 扩展目标

- 支持多币种、跨境、数字资产/交易所结算路由。
- 支持跨平台结算确认、结算补偿与监管穿透。

### 18.2 新增能力

- 结算路由治理
- 汇率/报价快照
- 数字资产转账记录
- 跨平台结算确认与差异处理

### 18.3 边界原则

- 数字资产结算是后期增强轨道，不得替代订单主状态和账务主状态。
- 交易所/跨境能力必须纳入更严格的合规、风控和监管导出机制。
