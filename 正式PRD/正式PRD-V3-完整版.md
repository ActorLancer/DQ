## 1. 文档目标

V3 的目标是把平台从“单平台可信交易与协作系统”升级为：

**跨域互联、跨平台协同、可监管、可扩展的可信数据空间基础设施**

本文件完整展开：

- 跨链请求治理
- 连接器互联
- 数据空间互认
- 图风控
- 高级监管穿透
- 生态协作开放

---

## 2. 版本定位与目标

### 2.1 V3 的平台定位

V3 不是简单功能叠加，而是平台形态升级：

- 从单平台交易 -> 多平台协作
- 从本地规则放行 -> 跨域规则协同
- 从单平台证据链 -> 跨平台证据链
- 从单点风控 -> 图风控与关系风控

### 2.2 核心目标

1. 实现跨平台连接器互联。
2. 实现跨区域/跨联盟链协同。
3. 建立跨链请求治理与确认机制。
4. 落地图风控、异常关系识别与高级风险治理。
5. 打通监管只读穿透视图与冻结处置能力。
6. 形成生态合作与能力开放底座。
7. 支持数据资产登记、融资、质押辅助服务的底层能力。

### 2.3 非目标

V3 仍不追求：

- 公链替代联盟链主状态机
- 原始高敏数据自由交易
- 匿名开放式代币化数据市场

## 6. 核心业务流程

### 6.1 跨链授权同步链路

1. 本地授权生效
2. 生成跨链请求对象
3. 本地签名
4. 发送至目标链或目标平台
5. 目标侧按 request_id 幂等处理
6. 目标侧返回 ack
7. 本地更新 final_status
8. 审计留痕

### 6.2 跨链结算确认链路

1. 本地账单结算完成
2. 生成结算确认请求
3. 发送目标侧
4. 目标侧确认
5. 回写 ack 与 final_status
6. 异常进入重试/补偿/人工介入

### 6.3 冻结同步链路

1. 本地风险命中
2. 执行冻结
3. 生成冻结同步请求
4. 目标侧同步冻结
5. 更新同步状态
6. 审计留痕

### 6.4 图风控链路

1. 汇总主体关系、设备关系、IP 关系、订单关系、评分关系、授权传播关系
2. 构建关系图
3. 识别异常群体、异常链路、黑名单绕过、洗订单、串谋评分
4. 生成告警
5. 转工单
6. 人工处置与冻结同步

### 6.5 监管穿透链路

1. 监管主体发起只读查询
2. 系统按监管权限聚合订单、合同、授权、跨链请求、证据对象
3. 输出穿透视图
4. 记录监管访问日志

## 7. 功能需求明细

### 7.1 跨链请求治理

必须支持：

- 唯一 request_id
- source_chain_id / target_chain_id
- request_type
- payload_hash
- nonce
- timestamp
- gateway_signature
- witness_signature_set
- ack_hash / ack_status
- final_status
- timeout
- retry_count
- terminate_flag
- compensate_flag

### 7.2 幂等与补偿

必须支持：

- 目标侧按 request_id 幂等处理
- 重复回执不得导致重复授权、重复结算、重复放行
- 超时重试
- 人工终止
- 补偿任务
- 死信事件池

### 7.3 图风控

必须支持图谱输入字段：

- 主体关系
- 设备关系
- IP 关系
- 订单关系
- 评分关系
- 授权传播关系

必须支持图风控输出：

- 风险分
- 告警类型
- 告警对象集合
- 可疑链路说明
- 建议处置动作

### 7.4 监管穿透

必须支持：

- 监管只读视图
- 穿透至订单、合同、授权、账单、跨链请求、证据对象
- 冻结/解冻记录
- 导出监管证明材料

### 7.5 生态互联

必须支持：

- 标准连接器接入规范
- 连接器身份校验
- 连接器版本与兼容性校验
- 互认关系登记
- 合作伙伴能力标识

## 8. 规则与治理

### 8.1 跨链治理规则

- 跨链成功不能只以“消息已发出”为准，必须有目标侧 ack 或等价确认
- 跨链失败不得污染本地核心状态
- 涉及授权同步、结算确认、冻结同步的跨链请求必须纳入审计视图与风险工单

### 8.2 图风控规则

必须识别：

- 串谋评分
- 洗订单
- 异常转售链路
- 高频异常下载群体
- 关联主体绕过黑名单或额度限制

### 8.3 监管规则

- 监管穿透仅限只读
- 监管查看权限与业务查看权限分离
- 监管查询必须留痕

## 10. 技术架构与边界

### 10.1 跨链治理边界

- 跨链请求是业务对象，不是简单消息
- 必须具备幂等、回执、超时和补偿
- 不能因为跨链失败污染本地主状态

### 10.2 连接器边界

- 连接器必须具备独立身份
- 必须经过版本、兼容性、策略校验
- 不得绕过本地策略引擎

### 10.3 公链边界

- 公链仍然只做公开验证与增强层
- 不承担核心仲裁和核心结算主状态

### 10.4 预言机 / 见证人边界

- 必须有独立服务身份
- 必须有签名留痕
- 不得共享平台管理员口令

- 跨平台 revision 订阅、版本游标、对象授权和到期撤权协同
- 跨平台 clean room / 受控执行 / 结果证明与证据映射

## 17. 数据本体存储与信任边界

### 17.1 V3 扩展目标

`V3` 在前两阶段基础上，正式支持：

- 第三方可信存储
- 跨平台连接器互联
- 跨域托管与执行能力互认
- 面向监管的存储责任链和销毁责任链

### 17.2 跨域互联的边界约束

- 跨平台互联不等于平台接管合作伙伴的明文数据。
- 跨链与跨平台只同步必要的元数据、摘要、策略快照、回执、证据和责任边界。
- 对接方必须暴露其托管能力、密钥治理能力、受控执行能力、保留/销毁承诺和信任证据。

### 17.3 V3 必记字段与证据

跨平台/跨域对象至少记录：

- `partner_storage_capability`
- `partner_key_governance_capability`
- `partner_execution_boundary_capability`
- `cross_platform_trust_scope`
- `external_destroy_attestation`
- `retention_obligation_snapshot`

### 17.4 V3 治理原则

- 跨链治理必须同时携带业务状态与信任边界状态。
- 监管穿透必须能看到“数据放在哪里、谁能解密、谁实际执行、保留多久、何时销毁”。
- 互认关系必须以能力和责任互认为前提，而不是单纯的链间连通。

### 17.5 V3 元信息与数据契约跨平台扩展

- `V3` 必须让跨平台商品、连接器商品和跨域协作对象携带统一的数据契约摘要、责任边界摘要和可验证元信息快照。
- 跨平台搜索、授权、结算、审计和争议时，平台必须能够区分“本平台声明字段”和“对端平台声明字段”。
- 跨域对象至少补充：互认能力声明、外部质量证明、外部加工责任链、跨平台撤权回执和外部销毁承诺。
- `V3` 不允许只同步业务状态而不同步契约口径，否则监管穿透和跨平台争议无法闭环。

## 18. 支付、资金流与轻结算

### 18.1 V3 扩展目标

- 在新加坡起步和法币走廊底座上，支持多币种、跨境、数字资产/交易所结算路由。
- 支持跨平台结算确认、结算补偿与监管穿透。

### 18.2 新增能力

- 结算路由治理
- 汇率/报价快照
- 数字资产转账记录
- 跨平台结算确认与差异处理

### 18.3 边界原则

- 数字资产结算是后期增强轨道，不得替代订单主状态和账务主状态。
- 交易所/跨境能力必须纳入更严格的合规、风控和监管导出机制。
- 即使进入 `V3`，平台也仍以“路由编排 + 账务镜像 + 持牌伙伴执行”为主，不自建交易所式全球清结算主体。

## 19. 身份认证、注册登录与会话管理

### 19.1 V3 扩展目标

- 在 V1/V2 身份底座上，支持跨平台身份联邦、自适应风险认证和合作伙伴身份互认。
- 让跨域连接器、合作伙伴节点、监管协同动作和跨链见证都挂接到统一的高 assurance 身份治理体系上。

### 19.2 新增能力

- 跨平台身份断言与外部身份绑定
- 更强的设备信任与风险登录识别
- 合作伙伴身份能力与证书互认
- 自适应认证和跨平台 step-up 策略

### 19.3 治理原则

- 跨域合作身份不能绕过本平台的作用域、合规、风控和审计。
- 合作伙伴断言、外部证书、跨链见证与平台主体/成员必须能追溯到统一审计链。
- 即使进入 `V3`，平台仍不把外部证书或跨平台断言直接当作本平台永久登录凭证，而是纳入受治理绑定关系。

## 20. 审计、证据链与回放

### 20.1 V3 扩展目标

- 把 V1/V2 审计底座扩展到跨链、跨平台、图风控、监管穿透和生态协同。
- 让跨域状态、跨域证据和跨域监管查询都能形成统一时间线与统一责任链。

### 20.2 新增审计对象

- `CrossChainRequest / Ack / Compensation`
- 合作伙伴断言与互认证据
- 监管查询与监管导出包
- 冻结/解冻/治理工单
- 图风控样本、告警和处置记录

### 20.3 新增规则

- 跨链请求必须保留 nonce、防重放结果、ack 回执、补偿动作与最终归档摘要
- 监管穿透查询必须保留访问理由、导出范围、脱敏级别和导出摘要
- 合作伙伴断言、外部证书和结算确认必须进入统一审计链，不得只留在对方系统
- `V3` 回放必须支持跨平台、跨链和监管视角

## 21. 双层权威模型增量

### 21.1 V3 扩展目标

- 在 V1/V2 双层权威模型上，把跨链、跨平台结算、监管协同和合作伙伴回执纳入统一权威矩阵。

### 21.2 关键规则

- 数据库继续承担业务运营主状态机职责。
- Fabric 继续承担关键承诺与可信证明确认层。
- 目标侧 `ack`、合作伙伴回执、监管协同回执、跨平台结算确认都属于外部事实来源。
- Kafka 继续负责事件编排与隔离，不承担跨链或跨平台主状态权威。

## 22. 搜索增量

- 增加合作伙伴 / 跨平台主体搜索
- 排序特征可纳入跨平台信誉、图风控与互认范围
- 搜索结果在展示前继续经过 PostgreSQL 本地状态校验

## 23. 推荐增量

### 23.1 V3 目标

在 `V2` 推荐基础上，支持生态伙伴、连接器和跨平台对象发现，并逐步支持页面级推荐位编排。

### 23.2 增量能力

- 伙伴/连接器推荐
- 生态主体推荐
- 页面级推荐位编排
- 更复杂的多目标优化和策略实验

### 23.3 边界

- 生态推荐仍不替代交易与可见性校验
- 跨平台推荐只负责发现，不自动放行跨平台交易

## 24. 日志、可观测性与告警增量

- 跨链请求、图风控、监管穿透、生态互联必须共享统一 trace 和告警链路。
- 图风控告警、跨链失败、伙伴信任边界异常应能直接转 incident。
- `V3` 允许在不改变观测主链的前提下补充更多安全分析或生态监控能力。
