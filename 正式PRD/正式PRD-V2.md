## 1. 文档目标

V2 的目标是在 V1 的可信交易底座之上，正式引入“可用不可见”的高级能力，并将公链展示从演示级提升为生产增强能力。

V2 重点回答：

- 如何把模型服务、C2D、受控训练、联邦/MPC/TEE 等高级能力正式产品化
- 如何在不暴露原始数据的前提下完成计算型交易
- 如何把 Solana 公开验证页和凭证体系落地为生产增强层

---

## 2. 版本目标

### 2.1 核心目标

1. 将模型/算法服务产品正式纳入商品体系。
2. 上线 C2D / 受控计算商品。
3. 支持受控模型训练和结果级授权。
4. 接入 FL / MPC / TEE 的正式能力边界。
5. 上线 Solana 生产级锚定、公开验证页与凭证体系。
6. 上线自动分润、复杂多方收益拆分与贡献度驱动分润能力。

### 2.2 非目标

V2 不承担：

- 跨平台数据空间互认
- 跨链结算确认正式上线
- 大规模外部开发者市场
- 复杂多方生态协同与开放插件体系

1. 项目发起

### 8.2 受控计算

- 任务级计费

### 8.3 联邦 / MPC / TEE

必须支持：

- 参与方登记
- 聚合策略配置

### 8.4 公链展示增强

- 交易凭证签发/吊销

## 9. 合规、风控与审计

### 9.1 合规规则

V2 允许以下能力进入人工增强审批：

- 受控模型训练
- 结果级商业输出权
- V2 计算型商品
- 联邦协作训练

必须阻断：

- 未审算法执行
- 输出越界
- 缺证明材料
- 高敏数据脱离受控环境输出

### 9.2 风控规则

新增风控重点：

- 算法投毒
- 模型输出越界
- 联邦轮次异常
- 结果级导出滥用
- 公链锚定失败或异常回执
- 贡献度作弊与异常分润

### 9.3 审计证据

V2 证据对象新增：

- 算法审计结果
- Compute Task 结果摘要
- 联邦轮次摘要
- TEE 远程证明引用
- 公链凭证签发/吊销记录

## 10. 技术架构与边界

### 10.1 总原则

- 继续保持链下主执行、链上主摘要
- 计算环境必须受控
- 公链展示不得反向驱动核心业务主状态

### 10.2 技术组件新增

- 算法审核服务
- Compute Task 编排器
- 联邦训练协调器
- 证明材料管理服务
- 公链凭证服务

### 10.3 边界

- 原始数据不写公链
- 训练过程不直接上链
- 证明材料只存引用和摘要

## 12. 验收标准

### 12.1 功能验收

- 至少完成 1 类 C2D 交易闭环
- 至少完成 1 类联邦或 MPC 协作闭环
- 模型 API 商品可完成正式交易与计费
- 公链验证页可用于真实客户或合作方场景
- 自动分润与贡献度分润至少完成 1 条真实闭环演练

### 12.2 合规验收

- 受控训练与受控计算均有审批、日志和证据
- 输出边界与证明材料可追溯

### 12.3 技术验收

- 任务编排、结果审查、账单生成可联查
- 公链签发失败不影响主链路

## 14. 结论

V2 的本质是把“V1 的标准交易平台”升级为“可用不可见的数据协作平台”。  
它要求在不破坏 V1 主交易闭环稳定性的前提下，把模型服务、受控计算、公链凭证等高价值能力正式产品化。

---

## 15. 数据本体存储与信任边界

- `V2` 在 `V1` 的平台托管 / 卖方自持 / 买方拉取基础上，正式引入 `受控执行` 与 `模型托管/模型服务` 的信任边界。
- 平台必须支持训练数据不出卖方域、模型权重不默认交给平台、平台仅编排任务和记录证明的模式。
- 模型与算法交付应区分：
  - 模型文件交付
  - 模型服务调用
  - 受控训练结果交付
- `V2` 必须记录模型与计算对象的 `artifact_storage_mode`、`weight_custody_mode`、`result_export_policy`、`execution_visibility_level`。
- TEE / 证明材料用于增强“按指定边界执行”的可信度，但不替代最小明文接触原则。

## 16. 支付、资金流与轻结算

- `V2` 在新加坡起步司法辖区基础上扩展受控的跨境法币走廊，而不是直接做全球无限制支付。
- `V2` 在 V1 支付底座上新增自动打款、周期扣费、渠道分账扩展和自动分润联动。
- 模型 API、受控计算和联邦任务的收费必须支持按周期、按任务、按贡献度三类扩展口径。
- 自动分润结果必须能回写到支付/打款编排层，而不是只停留在账务侧。

## 17. 身份认证、注册登录与会话管理

- `V2` 在 V1 身份骨架上新增 `SAML 2.0`、`SCIM`、Passkey-first 和更完整的企业身份联邦。
- 模型、受控计算、联邦任务、证明服务必须引入更强的服务身份、执行环境身份和机器到机器鉴权。
- `V2` 必须支持更强的企业声明映射、JIT 建档控制、服务身份隔离和证书生命周期治理。
- 高价值任务和高敏结果导出应支持更细的 assurance 等级与 step-up 策略。

## 18. 审计、证据链与回放

- `V2` 必须把受控计算、训练、模型服务、联邦轮次、证明材料和分润结果全部纳入统一审计主链。
- `V2` 证据对象必须覆盖：任务输入摘要、执行环境证明、训练轮次摘要、结果边界摘要、贡献记录与分润依据。
- `V2` 必须支持按任务、模型、训练轮次和证明对象进行联查与回放。
- 自动分润、自动打款、训练结果导出不得绕过强审计和证据清单。

## 19. 双层权威模型增量

- `V2` 继续沿用数据库业务权威 + Fabric 可信证明确认层。
- 模型训练、受控计算、分润结果等复杂流程继续走 `outbox -> Kafka -> consumer` 链路，不把 Kafka 或链事件投影当作唯一主状态。
- 训练结果、证明验证、分润打款等外部事实型结果，必须保留来源回执并回写 `external_fact_status / reconcile_status`。

## 20. 搜索增量

- `V2` 将模型商品纳入搜索范围
- 支持混合检索与排序配置化
- 仍保持 PostgreSQL 投影为权威源、OpenSearch 为读模型
