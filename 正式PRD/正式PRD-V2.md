## 1. 文档目标

V2 的目标是在 V1 标准交易底座之上，正式引入：

- 模型 / 算法服务商品
- C2D / 受控训练 / 受控计算
- 联邦 / MPC / TEE / ZKP 的正式能力边界
- 公链生产级锚定、验证页与凭证体系
- 自动分润、多方收益拆分、贡献度分润

本文件将上述能力按业务、规则、对象、技术、审计、验收完整展开。

## 2. 版本定位与目标

### 2.1 V2 的产品定位

V2 的平台定位是：

**从标准化数据交易平台升级为可用不可见的数据协作平台**

V2 的核心不是新增几个商品，而是把“买副本”升级为：

- 买模型能力
- 买受控计算权
- 买协作训练能力
- 买证明和结果输出能力

### 2.2 核心目标

1. 模型/算法服务正式纳入商品体系。
2. 受控计算任务正式纳入商品体系。
3. 正式支持受控模型训练权。
4. 建立联邦/MPC/TEE/ZKP 的正式产品边界。
5. 上线 Solana 生产级锚定与公开验证页。
6. 正式上线自动分润、多方收益拆分和贡献度驱动分润。

### 2.3 非目标

V2 不承担：

- 跨平台数据空间互认
- 跨链授权同步正式上线
- 跨链结算确认正式上线
- 图风控正式闭环
- 开放式生态插件市场
- 公链替代联盟链主状态

## 3. 承接关系

### 3.1 继承 V1 的能力

V2 保留 V1 全部能力：

- 主体与账户
- 文件快照/版本订阅、只读共享、API/服务、模板查询 lite、查询沙箱、固定报告/结果产品
- 标准合同、授权、验收、退款、计费模板
- Billing Event 模型
- 保证金 / 风险托管
- 合规规则、风控规则、审计证据包
- 开发者通道

并继续沿用以下 `V1` 工程冻结约束：

- `Authorization / Delivery / Settlement / Dispute` 作为正式生命周期对象继续保留
- 统一一致性状态字段族继续适用于 `Order / Contract / Authorization / Delivery / Settlement / Dispute / Billing Event`
- `V1-Core / V1-Extended / V1-Reserved` 属于实施分组，不改变 `V1` 范围结论；`V2` 继续在此基础上扩展，不得回退为“大范围同时开工”模式

### 3.2 新增能力

- 模型 API 商品
- 模型服务期商品
- C2D 商品
- 受控训练与算法审核
- 联邦学习项目
- MPC 结果商品
- ZKP 证明商品
- Solana 生产锚定与公开验证页
- 交易凭证、供应商认证凭证、数据产品护照 NFT
- 自动分润
- 贡献度驱动分润

### 3.3 生命周期对象扩展要求

`V2` 新增模型、受控计算、联邦、证明和分润能力时，不得重新发明第二套交易生命周期对象。新增能力必须挂接到以下对象之一，或显式形成其扩展对象：

- `Authorization`
- `Delivery`
- `Settlement`
- `Dispute`
- `Billing Event`

例如：

- 计算任务授权应落在 `Authorization`
- 任务开通、环境就绪、结果交付应落在 `Delivery`
- 自动分润、贡献度结算、收益拆分应落在 `Settlement`
- 算法争议、结果争议、证明争议应落在 `Dispute`
- 算力费、阶段费、分润补记应落在 `Billing Event`

## 6. 商品模型与 SKU

### 6.1 模型类商品

#### 模型 API 商品

- 形式：模型 API
- 权利：访问权、结果获取权
- 默认禁止：权重复制、逆向、无授权商业输出

#### 模型服务期商品

- 形式：服务期订阅 + 调用量
- 权利：访问权、受限商业输出权
- 需要：额外商业输出审查

### 6.2 受控计算 / clean room 商品

#### `SKU-C1` 受控计算任务包

- 权利：计算权 + 结果获取权
- 交付：任务执行权
- 验收：任务完成 + 结果模板匹配 + 证明材料齐全
- 定价：任务费 + 算力费

#### `SKU-C2` 模板查询 / clean room 包

- 权利：查询权、结果获取权
- 交付：analysis rule / query template / TVF 模板授权
- 验收：模板生效 + 首次成功执行 + 输出边界匹配
- 定价：席位费 / 查询费 / 周期费

### 6.3 联邦 / MPC / 证明商品

#### 联邦学习项目包

- 权利：参与训练权、结果获取权
- 交付：联合训练任务
- 验收：阶段验收
- 定价：项目费 / 阶段费

#### MPC 结果包

- 权利：结果获取权
- 交付：多方结果输出
- 验收：结果生成成功
- 定价：按任务

### 6.4 V2 对 V1 遗留能力的正式补齐

`V2` 除了新增模型、计算和联邦类商品，还必须对 `V1` 中已冻结但仅按最小路径落地的能力完成正式补齐：

- 只读共享：从单后端/单协议试点升级为受治理的共享连接器框架
- 模板查询：从 `lite` 模式升级为完整 `clean room / analysis rule / TVF / output policy`
- 版本订阅：从固定周期升级为更完整的 revision 发布、历史授权、周期治理与交付回执
- 售后规则：把文件快照、共享、API、模板查询、结果产品的授权/验收/退款/赔付补齐为正式规则体系

#### ZKP 证明包

- 权利：证明验证权、结果获取权
- 交付：证明回执
- 验收：验签成功
- 定价：按次

## 20. 数据本体存储与信任边界

### 20.1 V2 扩展目标

`V2` 在 `V1` 的平台托管、卖方自持、API 拉取基础上，正式引入：

- 受控执行
- 模型权重与模型服务分离
- 训练数据不出域
- 结果级交付与证明绑定

### 20.2 模型与训练对象的信任边界

- 模型文件可交付，但不要求平台默认持有权重明文。
- 模型服务可开通，但平台可仅管理服务权限、调用日志和账单，而不是掌握模型底层文件。
- 训练语料、原始样本、参与方局部参数默认不进入平台明文域。
- 训练任务、计算任务和证明材料必须显式记录执行环境、可见性级别和结果导出边界。

### 20.3 V2 必记字段

模型、算法、训练、计算对象至少增加：

- `artifact_storage_mode`
- `weight_custody_mode`
- `training_data_residency_mode`
- `execution_visibility_level`
- `result_export_policy`
- `proof_binding_required`

### 20.4 V2 信任增强

- 允许通过 TEE/远程证明/证明材料增强“按指定环境执行”的可信度。
- 不允许把证明材料误当成替代授权、审计、策略和密钥治理的万能方案。
- 自动分润、贡献度结算必须绑定任务来源、结果边界和证据链，避免对不可见数据做无法追责的收益分配。

### 20.4A V2 存储与查询层扩展

- `V2` 在 `V1` 的控制面/数据面/信任面基础上，正式增强查询/执行面，不再只把对象存储当作下载仓。
- 模型工件、训练输入、结果产物必须进入分层存储治理，至少区分 `training / model / result / preview / archive` 等逻辑区域。
- 查询/执行面必须只暴露经过治理的 `curated / product / result` 区对象，不得直接把原始训练区当外部读取面。

### 20.4B V2 查询与执行面扩展

- `V2` 必须把 `V1` 的 `template_query_lite / sandbox_query` 升级为正式 `clean room / analysis rule / TVF / output policy` 能力。
- 查询面定义、模板版本、分析规则、输出边界、导出限制、执行运行记录必须分层建模，不能继续把全部语义压缩在单一授权对象里。
- `V2` 必须支持按查询面、模板版本、结果边界、配额策略和证明要求进行正式审核与变更管理。
- 结果交付必须区分“查询结果对象”“报告对象”“模型结果对象”，并在账单、审计、争议和回放中分别可追踪。

### 20.5 V2 元信息与数据契约扩展

- `V2` 必须把 `V1` 的十大元信息域扩展到模型商品、受控计算商品、联邦任务产出物和结果产品。
- 模型/计算类商品在数据契约中至少新增：算法来源、执行环境、证明要求、结果导出边界、商业使用边界和责任回滚条件。
- 结果产品与模型服务不能只复用文件商品契约模板，必须形成独立的验收、争议、赔付和审计字段集。
- `V2` 新增对象仍需保留字段结构摘要、质量摘要和加工责任链摘要，只是其对象从原始数据扩展到训练/计算/结果链路。

### 20.6 V2 数据原样处理与产品化加工扩展

- `V2` 继承 `V1` 的六段式加工主链，并把模型工件、训练数据输入、受控执行结果、评估报告和结果产品包装纳入同一治理框架。
- 对受控训练、FL/MPC/TEE 任务，平台可以不直接持有原始训练数据，但必须记录任务输入摘要、执行环境、证明材料和输出边界。
- 模型/结果产品的预览不等同于原始样本，必须生成模型卡、输入输出契约、性能摘要、风险提示和结果导出限制摘要。
- `V2` 的加工责任链不再只描述“清洗/脱敏/标注”，还要覆盖训练、评估、推理、结果封装和责任回滚条件。

## 21. 支付、资金流与轻结算

### 21.1 V2 增量目标

- 把支付域从“可收可退”扩展为“可自动打款、可周期扣费、可与分润联动”。
- 在新加坡起步司法辖区基础上，扩展受控的跨境法币走廊与伙伴路由。

### 21.2 新增能力

- 自动打款批次
- 周期扣费
- 渠道分账/子商户扩展抽象
- 走廊扩展与收款偏好增强
- 分润结果到打款链路的自动编排
- 任务型/模型型商品的复杂计费口径

### 21.3 关键规则

- 模型 API、训练任务、受控计算结果必须支持费用快照和计费事件联动。
- 自动分润不得直接跳过人工复核与风险校验。
- 渠道原生分账能力属于适配层增强，不能绑死为单渠道实现。
- 跨境法币走廊仍必须以伙伴能力和走廊白名单为前提，不允许平台自营清结算替代持牌能力。

## 22. 身份认证、注册登录与会话管理

### 22.1 V2 扩展目标

- 在 V1 本地账号、OIDC SSO、MFA、会话治理和 Fabric 绑定的基础上，补齐企业联邦、服务身份和执行环境身份能力。
- 让模型、受控计算、联邦和证明能力都挂到统一身份体系上，不再依赖各模块自行实现“临时认证”。

### 22.2 新增能力

- `SAML 2.0`
- `SCIM`
- Passkey-first 策略增强
- 更强的企业声明映射与 JIT 建档规则
- 服务身份、连接器身份、执行环境身份增强
- Fabric 证书模板、属性模板和续签治理增强

### 22.3 关键规则

- 企业联邦身份不得绕过平台作用域、风控和审计规则直接放权。
- 模型训练、受控任务执行、联邦聚合、证明服务必须使用独立服务身份，不得复用人工管理员账号。
- Fabric 身份、服务身份、会话和任务执行记录必须可联查，能够定位到“谁触发、谁执行、哪张证书签名、哪次会话授权”。

## 23. 审计、证据链与回放

### 23.1 V2 扩展目标

- 把 V1 审计底座扩展到模型、训练、受控计算、联邦和证明域。
- 让不可见数据协作场景也能形成可核验、可回放、可追责的证据链。

### 23.2 新增审计对象

- 训练轮次摘要
- 聚合结果摘要
- 执行环境证明
- TEE / ZKP / 证明材料引用
- 模型服务调用摘要
- 贡献记录与分润依据

### 23.3 新增规则

- 任务创建、执行、取消、失败、结果导出都必须进入强审计
- 证明材料只能作为增强证据，不能替代授权、策略和回放链路
- 分润和打款联动必须绑定任务来源、结果边界和证据清单
- 高敏结果导出必须保留 `step_up`、执行环境、策略版本和结果摘要

## 24. 双层权威模型增量

### 24.1 V2 扩展目标

- 在 V1 双层权威模型上，把训练、证明、分润、自动打款和模型服务结果全部纳入同一一致性模型。

### 24.2 关键规则

- 数据库继续是业务运营主状态机。
- Fabric 继续承担关键承诺和可信证明确认层。
- Kafka 继续作为事件分发与解耦总线，不得被当作主状态机。
- 训练结果、证明验证、自动打款和分润执行结果必须保留来源回执，并回写 `external_fact_status / reconcile_status`。

## 25. 搜索增量

- 模型商品与结果包纳入统一搜索架构
- 支持混合检索与更强的自动补全 / 同义词 / 拼写纠错
- 排序权重配置化
- 搜索对象继续以 PostgreSQL 投影为权威源，OpenSearch 负责读模型

## 26. 推荐增量

### 26.1 V2 目标

在 `V1` 规则型推荐基础上，引入模型化推荐与更强的个性化发现，但仍保持推荐最终放行由平台业务侧决定。

### 26.2 增量能力

- 模型商品推荐
- `为你推荐`
- `同类机构常买`
- `看过 X 也看过`
- 个性化排序增强
- 推荐实验与变体配置
- 可选接入 `Gorse` / `LibRecommender`

### 26.3 边界

- 模型/开源引擎只负责候选召回或排序增强
- 最终结果仍需通过 PostgreSQL 业务校验
- 推荐策略配置、实验分配和模型结果需要审计

## 27. 日志、可观测性与告警增量

- 模型、训练、联邦、分润、公链增强链路必须全部纳入统一 trace。
- V2 可对模型/训练/推荐实验建立专用 dashboard 与告警规则。
- 继续沿用 `OpenTelemetry + Prometheus + Alertmanager + Grafana + Loki + Tempo`。
- 仍不把 `ELK / OpenSearch` 作为日志主后端硬依赖。

## 28. 敏感数据高阶能力补齐

### 28.1 `V2` 必须补齐的能力

在 `V1` 最小闭环基础上，`V2` 必须把敏感数据治理升级为正式能力：

- 完整 `clean room / analysis rule / TVF / output policy`
- `TEE C2D` 与远程证明引用
- 联邦分析 / 联邦训练中的敏感输入治理
- `MPC / PSI` 结果型交付
- 差分隐私预算台账与预算消耗审查
- 结果导出审批、结果披露复核与销毁/保留证明联动

### 28.2 `V2` 关键对象

- `SensitiveExecutionPolicy` 继续作为受控执行总入口
- `AttestationRecord` 扩展为远程证明、证明材料、环境证明统一记录
- `PrivacyBudgetLedger` 正式启用
- `ResultDisclosureReview` 与 `ComputeResult`、`ProofArtifact` 联动

### 28.3 `V2` 关键规则

- `clean room` 查询、受控计算、联邦训练和结果导出都必须落在“策略先定义、执行再放行、结果后复核”的顺序中。
- 证明材料用于增强可信度，但不替代处理依据、合同快照、输出边界和审计留痕。
- 差分隐私预算一旦启用，任何预算消耗都必须可追溯到具体任务、查询或结果对象。

## 29. 交易链监控、公平性与信任安全增量

### 29.1 V2 扩展目标

- 把 `V1` 的交易链监控底座扩展到训练、受控计算、联邦、模型服务、分润和自动打款。
- 让“不可见数据协作”同样具备可监控、可阻断、可回放、可仲裁能力。

### 29.2 扩展监控对象

- `TradeLifecycleCheckpoint` 扩展到：
  - 任务创建
  - 任务开跑
  - 环境校验通过
  - 证明回执到达
  - 结果审查通过
  - 分润完成
  - 自动打款完成
- `ExternalFactReceipt` 扩展到：
  - 训练执行证明
  - TEE/远程证明引用
  - 分润确认回执
  - 自动打款回执
- `FairnessIncident` 扩展到：
  - 任务执行中断
  - 结果绕过边界
  - 分润迟迟未发
  - 打款完成但状态未回写

### 29.3 关键规则

- 训练或计算完成不等于结果可结算，必须额外完成证明/边界/结果审查。
- 分润计算完成不等于资金可打款，必须额外具备支付编排回执与审计证据。
- 结果产品与模型产品继续遵守“策略先定义、执行再放行、结果后复核”的顺序。

### 29.4 页面与联查要求

- `V2` 监控页必须支持按 `task_id / model_id / reward_record_id / payout_instruction_id / trace_id` 联查。
- 可观测性页面必须显示任务异常、公链增强异常、分润异常的统一时间线。
- 审计页面必须支持按任务回放并展示缺失证明、缺失回执和状态差异。
