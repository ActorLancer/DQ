## 1. 文档目标

本文件是 V1 阶段的完整版正式 PRD，用于把五份基线文件中的 V1 内容完整展开为可执行版本文档。
它不是战略总纲，也不是数据库设计稿，而是面向本阶段落地的完整业务、规则、角色、架构、开发调试和验收基线。

本文件回答以下问题：

1. V1 必须交付哪些能力。
2. V1 明确不交付哪些能力。
3. V1 的角色、商品、交易链路、审查规则、技术边界、开发调试方式如何落地。
4. V1 的验收标准、测试要求、里程碑和退出标准是什么。

## 2. 版本定位与目标

### 2.1 V1 的产品定位

V1 的产品定位是：

**面向 B2B/B2G/G2B 场景的可信数据交易底座**

不是：

- 面向自然人自由交易的数据市场
- 公链资产化交易平台
- 完整数据空间互联平台
- 联邦学习/多方计算为主的高级协作平台

### 2.2 V1 的核心目标

V1 必须实现：

1. 在工业制造/供应链、零售/本地生活两个首批行业中，跑通 5 条标准交易链路。
2. 将 `FILE_STD / FILE_SUB / SHARE_RO / API_SUB / API_PPU / QRY_LITE / SBX_STD / RPT_STD` 八个标准 SKU 沉淀为标准化产品模型、模板和账单事件。
3. 实现“注册准入 -> 上架 -> 交易 -> 交付 -> 验收 -> 结算 -> 争议 -> 审计”的闭环。
4. 建立双层权威模型 + 联盟链摘要存证 + 演示级公链锚定的可信执行模式。
5. 建立可本地调试、可联调、可演示的开发与调试支持体系。

### 2.3 V1 非目标

V1 不交付：

- 模型/算法服务商品正式上线
- 受控模型训练
- C2D / FL / MPC / TEE / ZKP 正式交易
- 生产级 Solana 公开验证页与凭证/NFT
- 跨链请求编排与跨链结算确认
- 自动分润、复杂多方收益拆分、贡献度分润
- 完整 BPM 审批引擎
- 自然人自由供给市场
- 外部开发者市场与插件市场

## 4. 首批行业与标准场景

### 4.1 首批行业

#### 行业 A：工业制造 / 供应链协同

- 典型供方：设备厂商、工业 SaaS、MES/ERP 服务商、物流协同方
- 典型买方：制造企业、供应链协同企业、设备运维服务商
- 主要产品：工业指标 API、质量汇总文件包、供应链查询沙箱

#### 行业 B：零售 / 本地生活经营分析

- 典型供方：零售数据服务商、商圈分析服务商、平台型服务商
- 典型买方：连锁品牌、代理运营商、选址与经营分析团队
- 主要产品：经营分析 API、固定报告、商圈选址查询服务

### 4.2 标准链路

1. 工业设备运行指标 API 订阅
2. 工业质量与产线日报文件包交付
3. 供应链协同查询沙箱
4. 零售门店经营分析 API / 报告订阅
5. 商圈/门店选址查询服务

### 4.3 场景排除项

- 医疗高敏明细
- 金融高敏明细
- 原始敏感个人信息副本
- 跨境自动交付
- 联邦训练正式协作

- Department 作用域在 V1 保留对象与字段预留，V2 完整启用。
- 权限放行采用：

  - Scope
  - 合同 + Usage Policy 联合判定

## 6. 数据对象模型

### 6.1 核心对象

V1 必须落地以下对象：

- Tenant
- Party
- Department
- User
- Application
- Connector
- Execution Environment
- Data Resource
- Data Product
- Product SKU
- Digital Contract
- Usage Policy
- Authorization
- Delivery
- Settlement
- Dispute
- Order
- Billing Event
- Audit Event
- Public Anchor Batch

### 6.2 关键对象职责

#### Tenant / Party

- 表示租户与法律主体
- 支撑合同、账单、风控、责任归属

#### User / Application

- User 用于人工操作与责任留痕
- Application 用于 API 调用与系统账号

#### Data Product / SKU

- Product 用于目录与审核
- SKU 用于可下单的权利包

#### Digital Contract / Usage Policy

- 合同负责法律与业务条款
- Policy 负责系统级访问与使用控制

#### Authorization / Delivery / Settlement / Dispute

- Authorization 表示可审计授权实例，不得仅由策略规则临时推导
- Delivery 表示交付或开通实例，承载回执、开通状态与重试
- Settlement 表示可对账、可分账、可打款的结算实例
- Dispute 表示争议阻断、处理过程和恢复动作实例

#### Order / Billing Event

- Order 表示交易实例
- Billing Event 表示可计费事件源

#### Audit Event

- 负责形成业务日志、安全日志、证据日志的桥接对象

### 6.3 工程冻结规则

- 对象名统一使用英文单数聚合名，字段名统一使用英文 `snake_case`。
- 中文术语只用于文档标题、页面展示和合同解释，不作为工程字段名。
- `bind_*_ids / allowed_* / supported_*` 在 PRD 中属于聚合视图；工程落库优先拆关系表或快照表，数组字段只允许作为缓存和投影。
- `current_version` 只表示当前指针；进入审核、交易、授权、结算的内容必须固化到不可变版本或快照。
- `status_version` 只用于状态并发控制，不得与商品版本、合同版本、策略版本混用。

## 7. 商品体系与 SKU

### 7.1 V1 标准产品类型

仅正式商品化：

1. 文件快照 / 版本订阅产品
2. 只读共享产品
3. API / 服务产品
4. 模板查询 lite 产品
5. 查询沙箱产品
6. 固定报告 / 结果产品

### 7.2 权利边界

V1 正式支持：

- 使用权
- 访问权
- 查询权
- 结果获取权
- 有限内部共享权

V1 默认禁止：

- 所有权转让
- 模型训练权
- 转授权权
- 再分发权
- 无约束商业输出权

### 7.3 V1 标准 SKU 体系

#### 7.3.1 标准 SKU 真值

V1 的标准 SKU 统一冻结为：

- `FILE_STD`
- `FILE_SUB`
- `SHARE_RO`
- `API_SUB`
- `API_PPU`
- `QRY_LITE`
- `SBX_STD`
- `RPT_STD`

这些编码是商品、合同、授权、验收、退款、计费、状态机和搜索索引的统一真值源。

#### 7.3.2 预置商业套餐示例

以下编码只表示标准 SKU 下的实现型套餐示例，不是标准 SKU 真值源：

- `SKU-F1` 文件单次下载包，对应 `FILE_STD`
- `SKU-F2` 文件分批交付包，对应 `FILE_STD`
- `SKU-F3` 版本订阅包，对应 `FILE_SUB`
- `SKU-H1` 只读共享月包，对应 `SHARE_RO`
- `SKU-H2` 版本共享订阅包，对应 `SHARE_RO`
- `SKU-A1` API 基础订阅包，对应 `API_SUB`
- `SKU-A2` API 基础订阅 + 超额计费，对应 `API_SUB / API_PPU`
- `SKU-A3` API 阶梯档位包，对应 `API_SUB / API_PPU`
- `SKU-S1` 单席位月包，对应 `SBX_STD`
- `SKU-S2` 多席位团队包，对应 `SBX_STD`
- `SKU-S3` 按次查询包，对应 `SBX_STD`
- `SKU-S4` 模板查询 lite 包，对应 `QRY_LITE`
- `SKU-R1` 标准报告包，对应 `RPT_STD`
- `SKU-R2` 定制报告项目包，对应 `RPT_STD`
- `SKU-R3` 合成/特征结果包，对应 `RPT_STD`

#### 7.3.3 实施约束

- `SHARE_RO` 必须作为独立商品编码存在，不得并入文件交付或 API 类
- `QRY_LITE` 与 `SBX_STD` 必须分别建模、分别授权、分别验收
- 任何页面、接口、索引、数据库若只识别 `SKU-F1 / SKU-A1 / SKU-S1` 这类实现套餐编码，而不识别标准 SKU 编码，视为未对齐主 PRD

### 7.4 各类型默认规则

#### 文件快照 / 版本订阅

- 仅卖使用权，不卖所有权
- 必须带版本号
- 必须带水印或指纹
- 默认禁止再次分发与训练
- 推荐等级：P0 / P1 / P2；P3 仅在脱敏充分且审批通过时考虑

#### 只读共享

- 仅提供只读共享，不直接开放底层写权限
- 共享对象、recipient、撤权时间必须与订单绑定
- 默认按版本、订阅周期或共享窗口计费
- 推荐等级：P1 / P2 / 部分 P3 聚合或受限共享结果

#### API / 服务

- 交付单位是 Application 级访问权
- 默认需要配额、QPS、IP 限制
- 默认禁止明细拉平形成副本库
- 推荐等级：P1 / P2 / 部分 P3 聚合结果

#### 模板查询 lite

- 默认按模板授权授予
- 默认限制模板参数、结果行数、字段范围和导出次数
- 默认禁止原始明细表导出
- 推荐等级：P2 / P3

#### 查询沙箱

- 默认按席位或项目空间授予
- 默认限制结果行数、字段范围、导出次数和运行时长
- 默认禁止原始明细表导出
- 推荐等级：P2 / P3

#### 固定报告 / 结果产品

- 视为结果产品
- 默认不开放底层数据
- 默认以签收/确认作为交付依据
- 合成结果、特征结果或聚合结果包默认按“结果产品”口径治理

### 7.5 V1 最小可交付边界

当前已设计完成并要求 `V1` 落地：

- 标准产品骨架、8 个标准 SKU、状态机、基础交付链路
- `revision_subscription`、`data_share_grant`、`template_query_grant` 等授权与交付对象
- 文件快照、只读共享、API、模板查询、结果产品的最小差异化授权与验收口径
- 模板查询的白名单、输出边界、导出限制、审计留痕

### 7.6 V1 实施分组

该分组只用于实施顺序与验收批次管理，不改变 `V1 / V2 / V3` 范围结论。

- `V1-Core`
  - 主体与账户
  - 八个标准 SKU
  - 上架、合同、支付锁定、交付、验收、结算、争议
  - 文件交付、只读共享、API 开通、模板查询 lite、查询沙箱、报告交付
  - 联盟链存证、审计日志、基础对账、回放、Mock PaymentProvider
- `V1-Extended`
  - 搜索与推荐增强
  - 企业 OIDC 深化、设备信任治理
  - SDK、Webhook、外部系统对接
  - 监管视图、重点交易监控、风险事件上报
- `V1-Reserved`
  - Solana 演示锚定与公开验证页
  - 合作伙伴控制台、第三方应用审批
  - 复杂共享连接器增强与深度联邦增强

### 7.7 状态机粒度冻结

- `FILE_STD / FILE_SUB` 共用文件类主状态机，但必须保留订阅交付批次差异
- `SHARE_RO` 必须拥有独立共享开通状态机
- `API_SUB / API_PPU` 共用 API 主状态机，但必须保留订阅与按量账单事件差异
- `QRY_LITE` 必须拥有独立模板授权与模板执行状态机
- `SBX_STD` 必须拥有独立沙箱开通与席位状态机
- `RPT_STD` 必须拥有独立报告交付与签收状态机

当前仅按最小路径实现、不要求 `V1` 一次彻底完成：

- 只读共享的完整异构协议矩阵与多连接器治理
- 自由 SQL、完整 clean room、analysis rule / TVF 全能力
- 复杂 revision 发布策略、历史授权、跨平台游标与回执协同
- 面向所有交付模式的完整退款/赔付/争议规则引擎

后续收口要求：

- `V2` 必须把共享连接器框架、完整 clean room、完整 revision 治理、完整差异化售后规则补齐
- `V3` 必须把跨平台共享、跨平台订阅、跨域受控执行和责任链证据补齐

## 8. 模板体系

### 8.1 模板类型

V1 必须支持：

- 合同模板
- 授权模板
- 验收模板
- 退款模板
- 计费模板
- 赔付模板

### 8.2 模板基线

#### 文件包模板

- 合同：`CONTRACT_FILE_V1`
- 授权：`LICENSE_FILE_USE_V1`
- 验收：`ACCEPT_FILE_V1`
- 退款：`REFUND_FILE_V1`

固定字段：

- 资源名称
- SKU 名称
- 版本号
- 交付批次
- 下载次数限制
- 有效期
- 水印策略

#### API 模板

- 合同：`CONTRACT_API_SUB_V1 / CONTRACT_API_PPU_V1`
- 授权：`LICENSE_API_SUB_V1 / LICENSE_API_PPU_V1`
- 验收：`ACCEPT_API_SUB_V1 / ACCEPT_API_PPU_V1`
- 退款：`REFUND_API_SUB_V1 / REFUND_API_PPU_V1`

固定字段：

- 接口名称
- 应用绑定范围
- 配额
- QPS
- 返回字段说明
- 错误码规范

#### 沙箱模板

- 合同：`CONTRACT_SANDBOX_V1`
- 授权：`LICENSE_SANDBOX_USE_V1`
- 验收：`ACCEPT_SANDBOX_V1`
- 退款：`REFUND_SANDBOX_V1`

固定字段：

- 沙箱名称
- 可访问数据集列表
- 允许角色清单
- 会话并发限制
- 导出规则
- 日志保留周期

#### 报告模板

- 合同：`CONTRACT_REPORT_V1`
- 授权：`LICENSE_RESULT_USE_V1`
- 验收：`ACCEPT_REPORT_V1`
- 退款：`REFUND_REPORT_V1`

固定字段：

- 交付件类型
- 报告目录
- 适用范围
- 使用期限
- 引用限制

### 8.3 模板原则

- 固定字段优先
- 可配字段有限开放
- 不支持动态模板设计器
- 不支持复杂条款自由编排
- `FILE_SUB / SBX_STD / RPT_STD` 的授权模板分别固定为 `LICENSE_FILE_USE_V1 / LICENSE_SANDBOX_USE_V1 / LICENSE_RESULT_USE_V1`
- `QRY_LITE`、`SBX_STD`、`RPT_STD` 不得共用同一开通或验收模板族

### 8.3A 数据契约基线

- `V1` 必须把商品元信息、质量摘要、字段结构摘要、加工责任链摘要固化为可签署的数据契约快照。
- 数据契约不是普通商品描述，而是订单签约、交付验收、争议裁决和审计回放的正式依据。
- 同一商品可存在多个 SKU 契约版本，但每笔订单只能绑定一个确定的数据契约版本与摘要。
- 文件快照、版本订阅、只读共享、API/服务、模板查询 lite、结果产品在数据契约中的交付义务和验收规则必须分别定义。

## 10. 核心业务流程

### 10.1 统一主链路

1. 主体注册与审核
2. 数据资源治理与产品化
3. 产品审核与挂牌
4. 搜索与撮合
5. 合同与授权
6. 支付与锁定
7. 交付
8. 验收
9. 结算与账单归档
10. 持续使用监管与售后

### 10.2 标准链路说明

#### 文件副本一次性交易

挂牌 -> 审核 -> 下单 -> 合同 -> 全款锁定 -> 下载令牌签发 -> 下载校验 -> 验收 -> 结算 -> 审计归档

#### 版本订阅交易

挂牌 -> 审核 -> 下单 -> 合同 -> 首期费用锁定 -> 订阅游标签发 -> 按版本推送 / 拉取 -> 每批次回执 -> 周期账单 -> 到期断权

#### 只读共享交易

挂牌 -> 审核 -> 下单 -> 合同 -> 支付锁定 -> recipient / 共享对象绑定 -> share grant 签发 -> 首次成功访问 -> 周期账单 -> 撤权 / 到期断权

#### API 订阅交易

挂牌 -> 审核 -> 下单 -> 合同 -> 首期费用锁定 -> 应用绑定 -> API 凭证签发 -> 健康检查 -> 首次成功调用 -> 周期账单 -> 续费/停用

#### 模板查询交易

挂牌 -> 审核 -> 下单 -> 合同 -> 支付锁定 -> 模板授权 -> 模板查询 -> 受限结果导出 -> 验收 -> 结算

#### 查询沙箱交易

挂牌 -> 审核 -> 下单 -> 合同 -> 支付锁定 -> 沙箱开通 -> 指定查询 -> 受限导出 -> 验收 -> 结算

#### 定制报告服务

需求提交 -> 人工报价 -> 合同 -> 付款锁定 -> 供方生成报告 -> 平台交付 -> 买方验收 -> 结算

#### 查询 + 订阅混合包

签约 -> API Key + 沙箱席位同时开通 -> 账单按基础订阅 + 超额用量计算 -> 到期断权

## 11. 订单、验收与账单

### 11.1 通用订单状态

`draft -> quoted -> approval_pending -> contract_pending -> contract_effective -> payment_locked -> delivery_in_progress -> acceptance_pending -> accepted/rejected/disputed -> settled -> closed`

### 11.1.1 单一状态源规则

- `current_state` 是订单流程推进唯一主状态，也是默认展示主状态。
- `payment_status / delivery_status / acceptance_status / settlement_status / dispute_status / invoice_status` 是业务域子状态，只能由对应领域服务维护。
- `proof_commit_state / external_fact_status / reconcile_status` 是技术与审计镜像状态，不单独替代业务主状态。
- 子状态若触发流程推进，必须由订单编排服务同步推进 `current_state`；任何服务不得仅凭局部子状态把订单视为“已完成”。
- `dispute_status` 只阻断交付、结算、开票和断权，不直接覆写 `current_state`；页面可叠加“争议中”提示。

### 11.2 文件产品

- 交付完成条件：令牌签发成功、文件包生成完成、可校验哈希
- 自动验收条件：成功下载 + 哈希一致，或 3 个工作日无异议
- 拒收条件：文件损坏、字段不符、缺失批次、加密包无法解密
- 账单事件：文件包数量、版本号、下载成功回执

### 11.3 API 产品

- 交付完成条件：应用绑定成功、Key 生效、健康检查通过
- 自动验收条件：首次成功调用，或 5 个工作日无异议
- 拒收条件：接口不可用、字段契约不符、鉴权失败、SLA 未达标
- 账单事件：订阅周期、调用量、超额调用、峰值并发

### 11.4 共享 / 沙箱 / 报告 / 结果产品

- 共享交付完成条件：recipient 绑定成功、share grant 生效、首次成功访问
- 模板查询完成条件：模板授权生效、样例查询通过、输出边界检查通过
- 自动验收条件：首次成功访问 / 首次成功查询，或 5 个工作日无异议
- 拒收条件：共享对象不可访问、权限范围不符、模板不可执行、输出边界不符、结果明显异常
- 账单事件：共享周期、共享对象数、版本推送数、席位天数、查询次数、报告份数、结果包批次

### 11.5 账单事件口径

- 文件：文件包数、版本数、分批交付次数
- 共享：共享周期、共享对象数、recipient 数、版本推送次数
- API：订阅周期、调用次数、超额调用量、峰值并发档位
- 沙箱 / 模板：席位数、席位天数、查询次数、导出次数、模板次数
- 报告 / 结果：报告份数、结果包批次、结果对象数

### 11.6 结算实现边界

- V1 必须支持应结金额计算
- V1 必须支持账单生成
- V1 必须支持人工结算指令
- V1 可支持半自动分账
- V1 不要求自动分润引擎

## 12. 保证金、退款与赔付

### 12.1 保证金 / 风险托管

V1 必须支持：

- 高风险、高金额、低信誉订单可配置要求保证金
- 买方保证金、卖方履约保证金或等价风险托管
- 保证金与订单状态、争议状态、赔付责任、自动验收结果联动

保证金状态包括：

- 已冻结
- 已释放
- 已扣罚
- 已退回
- 待裁决

### 12.2 争议责任

#### 典型责任方

- 商品描述与实际不符：卖方主责
- 数据质量低于承诺：卖方主责
- 平台系统导致无法交付：平台主责
- 买方误操作/超范围使用：买方主责
- 监管要求阻断：非单方责任

### 12.3 证据优先级

1. 链上交易与授权摘要
2. 电子签章合同与授权版本
3. 交付回执 / API 调用日志 / 沙箱日志
4. 质量检测 / 验收报告
5. 平台系统日志 / 告警日志
6. 双方补充说明

### 12.4 SKU 默认争议规则

#### 文件包

- 未下载前可全额退款
- 下载后 48 小时内证明确有质量问题，可补交付或退款
- 买方本地环境问题原则上不退款

#### API

- 未开通成功可全额退款
- 24 小时内持续不可用按未开通处理
- 低于 SLA 阈值按档位赔付

#### 沙箱

- 未开通全额退款
- 晚开通补偿服务期
- 卖方未按约提供数据集可补开或部分退款

#### 报告

- 未交付可退款
- 可修复优先补交付
- 口径明显不符且无法修复可部分或全额退款

### 12.5 SLA 赔付档位

- ≥ 99.5%：无赔付
- 99.0%-99.5%：补偿 3 天服务期
- 98.0%-99.0%：补偿 7 天服务期或代金券
- < 98.0%：补偿 15 天服务期或部分现金赔付

## 16. 状态一致性与事件流

### 16.1 原则

- 数据库是业务运营主状态机
- Fabric 是关键承诺与可信证明确认层
- 外部支付 / CA / 存储回执拥有来源权威
- Kafka 是事件分发总线，不承担业务权威
- 数据库事务内写业务对象 + AuditEvent + `outbox_event`
- 幂等
- 重试

## 25. 数据本体存储与信任边界

### 25.1 V1 总体要求

1. `V1` 必须把“平台不是默认明文托管者”写成显式产品规则。
2. 卖方可以选择：
   - 平台托管
   - 卖方自持
   - API/买方拉取
   - 受限结果交付
3. 平台必须统一管理元数据、哈希、版本、授权、交付令牌、审计证据和生命周期状态。
4. 平台不得把“数据已登记”理解成“平台已获得任意明文使用权”。

### 25.2 V1 必做的存储模式

- 平台托管：平台保存密文对象，交付通过下载票据、短时链接、密钥封装完成。
- 卖方自持：平台保存元数据、哈希、连接方式、交付工单、策略和审计，不默认接收原始明文。
- API/买方拉取：平台签发凭证、控制配额与审计，底层数据源可在卖方侧。
- 受限结果交付：对高敏场景允许只交付查询结果、聚合结果或报告。

### 25.3 V1 关键字段要求

资产或资产版本必须有：

- `storage_mode`
- `payload_location_type`
- `custody_mode`
- `key_control_mode`
- `platform_plaintext_access`
- `platform_unilateral_decrypt_allowed`
- `default_delivery_route`
- `requires_controlled_execution`
- `retention_policy`
- `destroy_policy`

订单和交付必须快照：

- `storage_mode_snapshot`
- `delivery_route_snapshot`
- `trust_boundary_snapshot`
- `source_binding_id`
- `executor_type`
- `receipt_hash`

### 25.4 V1 信任机制

- 平台托管时必须实现数据与密钥分离。
- 卖方自持时必须支持连接器/API 网关交付，不得把它做成线下人工特例。
- 高敏资产默认优先通过 API、沙箱、报告、结果交付而不是明文副本。
- 所有下载、解密、导出、调用、断权、销毁必须留痕。
- `ZKP` 在 `V1` 不是平台信任基础，不承担“证明平台从未看过数据”的职责。

### 25.5 V1 生命周期治理

`V1` 至少追踪以下生命周期：

- 资产
- 版本
- 授权
- 交付
- 凭证
- 审计
- 存储保留与销毁

平台的生命周期治理目标是可追踪、可撤销、可断权、可审计、可证明，而不是无限期托管明文。

### 25.6 V1 分层存储闭环

- `V1` 必须采用四层分工：`PostgreSQL 控制面`、`MinIO/S3 兼容对象存储数据面`、`最小查询/执行面`、`Fabric 信任面`。
- 控制面保存目录、版本、契约、对象绑定、生命周期与审计；对象存储保存原始文件、产品包、预览、交付临时对象和证据原文。
- 平台至少划分 `raw / curated / product / preview / delivery / archive / evidence` 七类分层区域。
- `raw` 区只用于接入与加工，不得绕过治理直接上架；目录、订单和交付只能引用 `product` 区对象或受控执行入口。
- 查询/执行面默认读取 `curated / product` 等受控区域，不允许把 `raw` 当作默认查询面。
- Fabric 只锚定对象摘要、版本摘要、订单摘要、验收摘要和证据批次根，不保存大对象本体。

### 25.7 V1 新增正式存储对象

- `StorageNamespace`
- `StoragePolicyProfile`
- `AssetStorageBinding`
- `StorageObject`
- 分层存储区标识与生命周期策略

### 25.8 V1 查询与执行面闭环

- `V1` 必须把查询/执行面建成正式商品能力，而不是交付后的脚本附属品。
- 查询/执行面至少支持三类入口：`template_query_lite`、`sandbox_query`、`report_result`。
- `template_query_lite` 必须绑定正式模板版本、参数 schema、输出边界、导出限制和配额策略。
- `sandbox_query` 必须绑定正式执行环境、会话时效、导出策略和最小网络边界，不允许直接发放底库账号。
- 每次查询执行都必须形成 `QueryExecutionRun` 记录，至少留存请求摘要、模板引用、结果摘要、计费单位、策略命中和审计引用。
- 查询/执行面默认只读取 `curated / product / result` 等受控区域，不允许把 `raw` 区当默认对外查询面。
- `V1` 不承诺自由 SQL 与完整 clean room，只承诺白名单模板、受控沙箱和受限结果交付。

### 25.9 V1 查询与执行面新增正式对象

- `QuerySurface`
- `QueryTemplate`
- `QueryGrant`
- `QueryExecutionRun`
- `ResultObject / ReportResultObject`

## 26. 数据原样处理与产品化加工

### 26.1 V1 总体要求

1. 平台必须支持“原始格式保留 + 治理层统一”的产品化加工模式，不得要求所有对象先转为同一种格式。
2. 平台必须将交易前加工链路显式拆成：原始接入、格式识别、类内标准化、加工处理、产品包装、交付准备六个区域。
3. 平台提供加工处理能力，但不能默认取得卖方全部原始明文的自由使用权。
4. 同一商品必须同时保留原始对象引用、加工责任链、预览策略、质量摘要和数据契约绑定关系。

### 26.2 V1 交易前主流程

1. 主体准入与资质校验。
2. 原始接入登记：登记来源、采集时间、权利链、对象清单和哈希。
3. 格式识别与分类：识别结构化、半结构化、文档、图像、音视频、数据库转储、模型/Notebook 等对象族。
4. 分类分级与初步质量画像：确认敏感等级、覆盖范围、缺失率、更新时间和异常摘要。
5. 预处理决策：决定走平台托管直交付、卖方自持、自持 API、受控执行或结果产品路径。
6. 类内标准化与加工：抽取 schema、文本、字幕、标签、预览图、质量报告和加工责任链。
7. 产品包装：生成 `dataset + asset + manifest + sample/preview + contract`。
8. 上架审核与可交付性证明：完成对象可用性校验、授权边界校验和样例展示策略校验。

### 26.3 V1 交易后主流程

1. 订单成立与资金锁定。
2. 买方预验收只查看 schema、sample、quality report、contract snapshot 和可交付性证明。
3. 平台按交付模式开通下载票据、共享授权、API token、模板查询会话或结果交付权限。
4. 验收完成后进入结算、分润、撤权和审计归档。
5. Fabric 只锚定关键摘要、状态迁移和证据哈希，不承担原始数据加工执行。

### 26.4 类别化处理要求

- 表格/关系型数据：支持 schema 抽取、主键/时间字段识别、缺失率/重复率统计和标准化导出。
- 半结构化数据：支持 `JSON/JSONL/Avro` 等可校验结构描述和字段路径摘要。
- 文档文本：保留原始文件，同时生成抽取文本、章节索引、摘要和受控预览。
- 图片/音视频：保留原始对象，生成 sidecar manifest、标签、转码摘要、封面图或时长摘要。
- 数据库转储：保留 dump，同时抽取 schema、表清单、采样统计和版本摘要。
- 模型/代码/Notebook：保留工件本体、运行依赖、模型卡和输入输出契约，完整能力延展到 `V2/V3`。

### 26.5 新增正式对象

- 原始接入批次
- 原始对象清单
- 格式识别结果
- 抽取/标准化任务
- 预览或样例工件
- 加工责任链任务及输入来源

### 26.6 关键约束

- 原始对象、加工结果、预览工件和正式交付对象必须可追溯。
- 文件快照类交付一旦下载，不得伪装为可随时撤回模式。
- 共享/API/模板查询类商品必须把撤权、过期、输出边界和审计回放作为正式规则，而不是运维约定。
- 任何加工任务都必须带责任主体、输入来源、输出摘要和证据哈希。

### 26.3 V1 支持的支付方式

- 商品标价默认建议使用 `USD`；实际付款和实际收款币种必须通过走廊策略判定。

- 支付宝

- 重复 webhook 不得导致重复记账、重复退款、重复打款。
- 费用规则必须在合同生效前冻结到订单。
- 账务镜像、渠道结果和订单状态必须可联查。

### 26.5 V1 页面与调试要求

- 新增支付中心、收费规则页、对账中心、Mock 支付调试页。
- 新增司法辖区与走廊策略页、收款偏好页。
- 开发者通道必须支持支付状态联查、Mock webhook 和对账差异演练。

```text
定价 -> 费用预估 -> 支付意图 -> 渠道支付/Mock支付 -> 回调幂等确认
    -> 司法辖区与走廊校验 -> 账务镜像 -> 交付 -> 验收 -> 结算 -> 退款/赔付/打款 -> 对账 -> 审计
```

## 27. 身份认证、注册登录与会话管理

### 27.1 V1 设计目标

- 把主体注册、成员建档、登录、MFA、会话、设备、Fabric 身份和证书治理纳入统一身份体系。
- 明确“平台 IAM 身份”与“Fabric 链上身份”分层，避免普通登录与链上成员资格混用。
- 让支付、交付、审计、风控和链上签名都能复用同一套身份与证据链。

### 27.2 V1 新增能力

- 企业主体注册与邀请制成员注册
- 本地登录
- 企业 `OIDC` SSO
- `MFA`、恢复码、设备治理
- 会话管理、refresh token 轮换与撤销
- 高风险 step-up
- Fabric CA registry、Fabric 身份绑定、证书签发/吊销

### 27.3 认证与会话基线

- 浏览器门户：服务端会话 + `HttpOnly Secure Cookie`
- 开放 API / SDK：短期 access token + refresh token
- 高风险角色默认强制 MFA
- 新设备、新国家/地区、新 IP 段登录必须触发额外校验或告警
- 支付、退款、赔付、证书治理、高敏导出等动作必须触发 step-up

### 27.4 Fabric 身份边界

- Fabric 身份仅用于链上成员表达、链上签名、链上节点和证书责任链
- 不是所有平台成员都必须拥有 Fabric 身份
- 证书中只保存最小必要属性或不透明引用，不得放实名证号原文
- 证书吊销、续签、异常停用必须与平台角色、会话和风控联动

### 27.5 V1 页面与对象要求

- 新增注册页、登录页、MFA 绑定页、会话与设备页、企业 SSO 配置页、Fabric 身份与证书页
- 数据模型必须预留：`Invitation`、`IdentityProof`、`MfaAuthenticator`、`TrustedDevice`、`UserSession`、`RefreshTokenFamily`、`SsoConnection`、`FabricIdentityBinding`、`CertificateRecord`

### 27.6 与现有链路的闭环

```text
主体注册/KYC-KYB -> 成员邀请/SSO 建档 -> MFA/设备绑定 -> 登录建立会话
  -> 角色授权 -> 交易/交付/支付/审计操作 -> 高风险 step-up
  -> 必要时调用 Fabric 身份完成链上签名 -> 会话、令牌、证书、审计证据联查
```

## 28. 审计、证据链与回放

### 28.1 V1 设计目标

- 把审计、证据和回放从“日志功能”提升为独立业务域。
- 让交易、支付、交付、争议、证书、监管查询都进入统一审计主链。
- 形成“可联查、可导出、可回放、可对账、可锚定”的强审计底座。

### 28.2 五层结构

- `L1` 业务审计事件
- `L2` 证据对象与证据清单
- `L3` 完整性哈希链与批次根
- `L4` Fabric 摘要存证
- `L5` 联查、回放、对账与监管导出

### 28.3 V1 必记字段

- `actor_id / actor_org_id`
- `session_id / device_id / application_id`
- `request_id / trace_id`
- `action_name / result_code / error_code`
- `before_state_digest / after_state_digest`
- `tx_hash / evidence_manifest_id`
- `previous_event_hash / event_hash`
- `retention_class / legal_hold_status / sensitivity_level`

### 28.4 V1 必做对象

- `AuditEvent`
- `EvidenceItem`
- `EvidenceManifest`
- `EvidencePackage`
- `AnchorBatch`
- `ReplayJob`
- `ReplayResult`
- `RetentionPolicy`
- `LegalHold`
- `AuditAccessRecord`

### 28.5 V1 上链规则

必须上链：

- 合同/授权快照哈希
- 订单关键状态摘要
- 支付/保证金/退款/赔付/结算批次摘要
- 争议立案/裁决摘要
- 证据清单摘要
- 审计批次根
- 监管导出摘要
- 证书签发/吊销摘要

严禁上链：

- 原始交易数据正文
- 原始支付凭证
- 原始证据正文
- 会话令牌、密码、密钥
- 模型权重全文和原始训练语料

### 28.6 V1 回放边界

- 支持取证回放、状态回放、对账回放
- 默认只允许 `dry-run`
- 有副作用的 replay 必须 step-up，并限制在白名单幂等动作

### 28.7 V1 页面与权限要求

- 新增审计回放页、证据保全与保留策略页、锚定批次页
- 审计摘要查看、原始证据查看、脱敏视图查看、原文视图查看、证据导出、回放执行、`legal hold` 管理必须分权
- 审计查看权限与业务查看权限必须分离

### 28.8 与现有逻辑的闭环

```text
业务动作 -> AuditEvent -> EvidenceItem / EvidenceManifest
  -> retention / legal_hold / WORM
  -> AnchorBatch -> Fabric 摘要存证
  -> 联查 / 导出 / 回放 / 对账
  -> 结果再次写回审计链
```

## 31. 敏感数据受控交付补充基线

### 31.1 V1 正式范围

`V1` 对敏感数据只正式承诺以下最小闭环：

- `report_result`
- `template_query_lite`
- `sandbox_query`
- `seller_hosted_api`

以下能力在 `V1` 只保留架构预留，不作为正式交付承诺：

- 完整 `clean room`
- `TEE C2D`
- 联邦学习 / 联邦分析
- `MPC / PSI`
- 差分隐私预算引擎

### 31.2 V1 必备治理对象

敏感商品一旦进入可售状态，必须至少绑定以下对象或等价快照：

- 敏感处理策略档案
- 处理依据证明
- 安全预览工件
- 敏感执行策略快照
- 结果披露审查记录
- 销毁/保留证明

### 31.3 V1 关键规则

- 含敏感个人信息、高敏行业明细、重要数据相关商品默认不得走原始副本下载。
- 平台即使不持有原始明文，也必须保留版本承诺、结构承诺、可交付性证明和受限试算证据。
- 安全预览必须替代“直接给样本文件”的粗放方式，至少区分可展示字段、掩码字段、不可展示字段和仅统计展示字段。
- 任何敏感结果导出都必须经过输出边界校验；命中阈值的结果必须进入结果披露审查。
- 到期、撤权、争议或监管指令触发时，平台必须能回收授权、冻结结果对象并留存销毁/保留证明。

## 32. 交易链监控、公平性与信任安全

### 32.1 V1 目标

- 把“平台如何持续监控完整交易链并保证交易公平、信任安全”定义为正式产品能力。
- 让订单、支付、交付、验收、结算、争议、敏感执行和链上确认形成统一监控时间线。
- 让监控结果能够直接驱动风控阻断、人工复核、争议升级、结算冻结和审计回放。

### 32.2 六层监控视图

- `L1` 业务状态视图：订单、合同、授权、交付、验收、账单、争议。
- `L2` 交付执行视图：下载、共享、API、模板查询、沙箱、敏感执行、结果导出。
- `L3` 证据视图：审计事件、证据清单、锚定批次、保留策略、回放任务。
- `L4` 公平性视图：超时、拒收、绕过、重复、回执缺失、冻结/补偿建议。
- `L5` 观测视图：日志、trace、告警、事件工单、SLO。
- `L6` Fabric 视图：提交状态、链码事件、区块确认、锚定状态、投影缺口。

### 32.3 核心对象

- `TradeLifecycleCheckpoint`
  - 表达交易检查点，如“支付锁定完成”“交付开通完成”“验收超时”“撤权完成”。
- `ExternalFactReceipt`
  - 表达支付回调、打款回执、执行证明、销毁证明、合作方 ack 等外部事实。
- `FairnessIncident`
  - 表达迟交付、恶意拒收、重复交付、结果绕过、越权导出、外部事实长期缺失等事件。
- `MonitoringPolicyProfile`
  - 表达检查点阈值、升级规则、自动动作与角色通知策略。
- `ChainProjectionGap`
  - 表达链上已确认但链下未投影、链下已期望但链上未确认等缺口。

### 32.4 关键规则

- 关键状态不允许由任何单一子系统单边判定完成。
- 数据库业务状态继续是运营主状态，Fabric 继续是可信确认层，外部回执继续是外部事实来源。
- Kafka、工作流执行成功、链提交成功都只能作为证据输入，不能单独成为全局唯一主状态。
- 敏感商品的监控重点必须从“文件是否被下载”切换为“是否按执行边界被受控使用、是否越权输出、是否完成撤权和销毁/保留证明”。

### 32.5 页面与联查要求

- `V1` 必须新增或正式交付以下页面：
  - 交易链监控总览页
  - 生命周期检查点页
  - 外部事实回执页
  - 公平性事件页
  - 链投影缺口页
- 各页面必须能按 `request_id / trace_id / order_id / payment_intent_id / tx_hash` 互相跳转联查。

### 32.6 自动动作

- 命中规则后可自动执行：
  - 阻断交付或导出
  - 冻结结算
  - 转风险工单
  - 自动发起争议建议
  - 自动要求补充回执或证明
  - 自动写审计摘要并进入锚定批次

### 32.7 验收要点

- 至少完成 1 条文件型交易链监控回放。
- 至少完成 1 条 API/模板查询型交易链监控回放。
- 至少完成 1 条敏感数据受控交付链监控回放。
- 至少完成 1 次“回执缺失 -> 告警 -> 冻结 -> 补充回执 -> 解除”的演练。
