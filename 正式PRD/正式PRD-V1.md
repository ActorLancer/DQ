## 1. 文档目标

本文件是面向研发、测试、实施和验收的 V1 正式 PRD。  
它只回答 V1 当前版本必须交付什么，不覆盖 V2/V3 的正式交付内容。

V1 的目标不是做完整的数据空间平台，而是先跑通：

- 企业/机构级主体准入
- 文件/API/沙箱/报告四类标准产品交易
- 上架、下单、合同、支付锁定、交付、验收、账单、争议、审计闭环
- 联盟链关键摘要存证
- 开发与调试支持底座

---

## 2. 版本目标

### 2.1 核心目标

V1 必须完成以下目标：

1. 在工业制造/供应链、零售/本地生活两个首批行业中，跑通 5 条标准交易链路。
2. 将平台能力收敛为标准化 SKU、标准模板和标准交易状态机。
3. 实现“链下主交易闭环 + 联盟链关键摘要存证 + 公链演示级可选锚定”的可信交易模式。
4. 形成可验收、可审计、可扩展的交易底座。

### 2.2 非目标

V1 不承担以下目标：

- 模型/算法类商品正式上线
- C2D / FL / MPC / TEE / ZKP 正式交易
- 生产级 Solana 公开验证页与 NFT/凭证签发
- 跨链请求编排与跨链结算确认
- 自动分润引擎
- 复杂 BPM 审批引擎
- 面向自然人自由供给的市场化产品

## 5. V1 标准产品与 SKU

### 5.1 标准产品类型

V1 仅支持以下产品类型：

1. 文件副本产品
2. API / 订阅产品
3. 查询沙箱产品
4. 固定报告产品

### 5.2 标准 SKU

V1 标准 SKU 包括：

- 文件类
  - `SKU-F1` 文件单次下载包
  - `SKU-F2` 文件分批交付包
- API 类
  - `SKU-A1` API 基础订阅包
  - `SKU-A2` API 基础订阅 + 超额计费
  - `SKU-A3` API 阶梯档位包
- 沙箱类
  - `SKU-S1` 单席位月包
  - `SKU-S2` 多席位团队包
  - `SKU-S3` 按次查询包
- 报告类
  - `SKU-R1` 标准报告包
  - `SKU-R2` 定制报告项目包

### 5.3 V1 权利边界

V1 正式支持：

- 使用权
- 访问权
- 查询权
- 结果获取权
- 有限内部共享权

V1 默认禁止：

- 所有权转让
- 模型训练权
- 转授权权
- 再分发权
- 无约束商业输出权

## 7. 核心业务流程

### 7.1 统一主链路

1. 主体注册与审核
2. 数据资源治理与产品化
3. 产品审核与挂牌
4. 搜索与撮合
   - PostgreSQL 搜索投影
   - OpenSearch 商品/卖方搜索读模型
   - Redis 搜索缓存
5. 合同与授权
6. 支付与锁定
7. 交付
8. 验收
9. 结算与账单归档
10. 持续使用监管与售后

### 7.2 各产品的交付方式

- 文件产品：下载令牌 + 文件包 + 哈希校验
- API 产品：Application 绑定 + Key/OAuth + 配额
- 沙箱产品：席位开通 + 模板查询 + 导出限制
- 报告产品：下载链接/发送 + 签收回执

### 7.3 订单状态机

通用状态：

`draft -> quoted -> approval_pending -> contract_pending -> contract_effective -> payment_locked -> delivery_in_progress -> acceptance_pending -> accepted/rejected/disputed -> settled -> closed`

## 10. 合规、风控与安全规则

### 10.1 V1 自动阻断

- 原始敏感个人信息副本交易
- 境外主体申请下载副本
- V1 文件/API/沙箱产品申请模型训练
- 沙箱产品申请原始明细导出
- 缺少来源或权利证明

### 10.2 V1 人工复核

- 公共数据授权运营产品
- 高风险行业或重要数据
- 商业输出权申请
- 大额订单

### 10.3 安全基线

- 全链路 TLS
- 敏感字段静态加密
- MFA
- KMS/HSM 或等价可审计能力
- 日志防篡改
- 零信任服务身份

## 11. 链上链下技术方案

### 11.1 总原则

- 双层权威模型
- 联盟链存摘要
- Solana V1 默认关闭，仅演示级锚定

### 11.2 链上允许写入

- 合同哈希
- 授权状态
- 订单状态摘要
- 交付回执摘要
- 账单摘要
- 审计摘要

### 11.3 链下必须保存

- 原始数据
- 文件包
- API 详细调用日志
- 全量审计日志
- 模板原文
- 证据原件

### 11.4 状态一致性机制

- 数据库是业务运营主状态机
- Fabric 是关键承诺与可信证明确认层
- 支付 / CA / 存储等外部系统对其事实回执拥有来源权威
- 数据库事务内同时写业务对象、审计事件和 `outbox_event`
- publisher worker 发布 Kafka
- 下游消费者异步完成链写、通知、搜索、风控、对账
- 失败重试 + `dead_letter_event` + Kafka `DLQ`

## 16. 结论

V1 的正式交付目标是把“可信交易底座”做成一个可真实运行、可测试、可演示、可审计的系统，而不是在首期同时完成所有高级能力。  
V1 只交付四类标准商品，但会为 V2/V3 保留完整的对象模型、权限模型、调试模型与技术边界。

## 17. 数据本体存储与信任边界

- `V1` 不要求所有卖方先把明文数据上传到平台，平台必须支持 `平台托管`、`卖方自持`、`买方拉取/API` 三类正式模式。
- 平台交易的是权利和可验证交付，不是默认交易明文副本。
- 平台必须统一记录：`storage_mode`、`payload_location_type`、`key_control_mode`、`platform_plaintext_access`、`delivery_route_snapshot`。
- 平台托管场景下，数据对象与解密能力必须分离；卖方自持场景下，平台只保存元数据、哈希、工单、令牌和审计证据。
- 高敏资产默认优先通过 API、沙箱、结果交付等方式使用，不鼓励直接下载原始明文副本。

## 18. 支付、资金流与轻结算

### 18.1 V1 目标

- 接入真实支付渠道时不重写订单逻辑。
- 以新加坡作为唯一起步司法辖区上线真实支付与真实结算。
- 支持货款、保证金、退款、赔付、人工打款和基础对账。
- 提供可调试的 `MockPaymentProvider`。

### 18.2 V1 必做能力

- 支付渠道适配层
- 新加坡起步司法辖区配置与走廊白名单
- 支付意图与支付状态管理
- 渠道回调验签与幂等处理
- 收费规则与费用预估
- 内部账务镜像
- 人工退款、人工打款、差异对账
- Mock 支付接口

### 18.3 V1 边界

- 未命中新加坡起步走廊的真实支付不得进入生产主链路
- 不要求自动分润引擎
- 不要求复杂子商户体系
- 不要求数字货币/交易所结算

## 19. 身份认证、注册登录与会话管理

- `V1` 正式支持企业主体注册、成员邀请注册、本地登录、企业 OIDC SSO、MFA、会话与设备治理。
- 平台登录身份与 Fabric 链上身份必须分层，Fabric 身份只用于链上成员资格和链上签名，不直接替代普通登录。
- 浏览器门户优先服务端会话；API/SDK 使用短期 access token + refresh token。
- 高风险动作必须 step-up：合同确认、退款、赔付、打款、角色提升、Fabric 证书签发/吊销。
- `V1` 必须落证书绑定、证书吊销、设备撤销、会话联查与审计留痕骨架。

## 20. 审计、证据链与回放

- `V1` 必须把审计做成独立业务域，不得只依赖普通系统日志。
- 统一记录：actor、org、session、device、request_id、trace_id、action、result、before/after digest、tx_hash、evidence_manifest、event_hash。
- 平台必须显式管理：审计事件、证据项、证据清单、证据包、锚定批次、回放任务、保留策略、`legal hold`。
- Fabric 只保存审计批次根、关键状态摘要和证据清单摘要，不保存原始证据正文。
- `V1` 必须支持审计联查、证据包导出、审计回放 dry-run、支付/交付/争议/证书强审计。
