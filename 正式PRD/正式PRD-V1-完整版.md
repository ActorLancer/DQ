## 1. 文档目标

本文件是 V1 阶段的完整版正式 PRD，用于把五份基线文件中的 V1 内容完整展开为可执行版本文档。

## 2. 版本定位与目标

### 2.1 V1 的产品定位

V1 的产品定位是：

**面向 B2B/B2G/G2B 场景的可信数据交易底座**

不是：

- 面向自然人自由交易的数据市场
- 公链资产化交易平台
- 完整数据空间互联平台
- 联邦学习/多方计算为主的高级协作平台

### 2.2 V1 的核心目标

V1 必须实现：

1. 在工业制造/供应链、零售/本地生活两个首批行业中，跑通 5 条标准交易链路。
2. 将文件/API/沙箱/报告四类产品沉淀为标准化产品模型、SKU、模板和账单事件。
3. 实现“注册准入 -> 上架 -> 交易 -> 交付 -> 验收 -> 结算 -> 争议 -> 审计”的闭环。
4. 建立链下主状态机 + 联盟链摘要存证 + 演示级公链锚定的可信执行模式。
5. 建立可本地调试、可联调、可演示的开发与调试支持体系。

### 2.3 V1 非目标

V1 不交付：

- 模型/算法服务商品正式上线
- 受控模型训练
- C2D / FL / MPC / TEE / ZKP 正式交易
- 生产级 Solana 公开验证页与凭证/NFT
- 跨链请求编排与跨链结算确认
- 自动分润、复杂多方收益拆分、贡献度分润
- 完整 BPM 审批引擎
- 自然人自由供给市场
- 外部开发者市场与插件市场

## 4. 首批行业与标准场景

### 4.1 首批行业

#### 行业 A：工业制造 / 供应链协同

- 典型供方：设备厂商、工业 SaaS、MES/ERP 服务商、物流协同方
- 典型买方：制造企业、供应链协同企业、设备运维服务商
- 主要产品：工业指标 API、质量汇总文件包、供应链查询沙箱

#### 行业 B：零售 / 本地生活经营分析

- 典型供方：零售数据服务商、商圈分析服务商、平台型服务商
- 典型买方：连锁品牌、代理运营商、选址与经营分析团队
- 主要产品：经营分析 API、固定报告、商圈选址查询服务

### 4.2 标准链路

1. 工业设备运行指标 API 订阅
2. 工业质量与产线日报文件包交付
3. 供应链协同查询沙箱
4. 零售门店经营分析 API / 报告订阅
5. 商圈/门店选址查询服务

### 4.3 场景排除项

- 医疗高敏明细
- 金融高敏明细
- 原始敏感个人信息副本
- 跨境自动交付
- 联邦训练正式协作

### 5.2 身份层级

- 组织身份：合同、账单、责任归属
- 用户身份：登录、操作、审批、留痕
- 服务身份：微服务、网关、连接器、预言机机器鉴权
- 链上身份：链上签名、成员准入、凭证签发

### 5.3 V1 权限原则

- V1 允许采用“系统自动判断 + 管理员确认/人工复核”的简化审批实现。
- Department 审批能力保留在模型中，但 V1 不要求完整 BPM 流转。
- V1 至少支持 Tenant、SKU、Application 三层强制作用域。
- Department 作用域在 V1 保留对象与字段预留，V2 完整启用。
- 权限放行采用：
  - RBAC
  - Scope
  - 合同 + Usage Policy 联合判定

### 5.4 典型角色责任

#### 平台侧

- 平台管理员：系统级配置、冻结/解冻受限操作
- 主体审核员：主体准入审核
- 产品审核员：上架审核
- 合规审核员：高风险规则审查
- 风控专员：风险命中处置
- 法务专员：合同条款审核
- 财务结算专员：人工结算/半自动分账确认
- 争议处理专员：退款/赔付裁定
- 审计管理员：证据联查与导出

#### 租户侧

- Tenant 管理员：用户、部门、应用管理
- 数据治理管理员：产品、SKU、样例、元数据维护
- 采购经理：询价、下单、验收、退款申请

## 7. 商品体系与 SKU

### 7.1 V1 标准产品类型

仅支持：

1. 文件副本产品
2. API / 订阅产品
3. 查询沙箱产品
4. 固定报告产品

### 7.2 权利边界

V1 正式支持：

- 使用权
- 访问权
- 查询权
- 结果获取权
- 有限内部共享权

V1 默认禁止：

- 所有权转让
- 模型训练权
- 转授权权
- 再分发权
- 无约束商业输出权

### 7.3 V1 标准 SKU

#### 文件类

- `SKU-F1` 文件单次下载包
- `SKU-F2` 文件分批交付包

#### API 类

- `SKU-A1` API 基础订阅包
- `SKU-A2` API 基础订阅 + 超额计费
- `SKU-A3` API 阶梯档位包

#### 沙箱类

- `SKU-S1` 单席位月包
- `SKU-S2` 多席位团队包
- `SKU-S3` 按次查询包

#### 报告类

- `SKU-R1` 标准报告包
- `SKU-R2` 定制报告项目包

### 7.4 各类型默认规则

#### 文件副本

- 仅卖使用权，不卖所有权
- 必须带版本号
- 必须带水印或指纹
- 默认禁止再次分发与训练
- 推荐等级：P0 / P1 / P2；P3 仅在脱敏充分且审批通过时考虑

#### API / 订阅

- 交付单位是 Application 级访问权
- 默认需要配额、QPS、IP 限制
- 默认禁止明细拉平形成副本库
- 推荐等级：P1 / P2 / 部分 P3 聚合结果

#### 查询沙箱

- 默认按席位授予
- 默认限制结果行数、字段范围、导出次数
- 默认禁止原始明细表导出
- 推荐等级：P2 / P3

#### 固定报告

- 视为结果产品
- 默认不开放底层数据
- 默认以签收/确认作为交付依据

- 合同：`CONTRACT_API_V1`
- 授权：`LICENSE_API_USE_V1`
- 验收：`ACCEPT_API_V1`
- 退款：`REFUND_API_V1`

固定字段：

- 接口名称
- 应用绑定范围
- 配额
- QPS
- 返回字段说明
- 错误码规范

#### 沙箱模板

- 合同：`CONTRACT_SANDBOX_V1`
- 授权：`LICENSE_SANDBOX_USE_V1`
- 验收：`ACCEPT_SANDBOX_V1`
- 退款：`REFUND_SANDBOX_V1`

固定字段：

- 沙箱名称
- 可访问数据集列表
- 允许角色清单
- 会话并发限制
- 导出规则
- 日志保留周期

#### 报告模板

- 合同：`CONTRACT_REPORT_V1`
- 授权：`LICENSE_RESULT_USE_V1`
- 验收：`ACCEPT_REPORT_V1`
- 退款：`REFUND_REPORT_V1`

固定字段：

- 交付件类型
- 报告目录
- 适用范围
- 使用期限
- 引用限制

### 8.3 模板原则

- 固定字段优先
- 可配字段有限开放
- 不支持动态模板设计器
- 不支持复杂条款自由编排

## 10. 核心业务流程

### 10.1 统一主链路

1. 主体注册与审核
2. 数据资源治理与产品化
3. 产品审核与挂牌
4. 搜索与撮合
5. 合同与授权
6. 支付与锁定
7. 交付
8. 验收
9. 结算与账单归档
10. 持续使用监管与售后

### 10.2 标准链路说明

#### 文件副本一次性交易

挂牌 -> 审核 -> 下单 -> 合同 -> 全款锁定 -> 下载令牌签发 -> 下载校验 -> 验收 -> 结算 -> 审计归档

#### API 订阅交易

挂牌 -> 审核 -> 下单 -> 合同 -> 首期费用锁定 -> 应用绑定 -> API 凭证签发 -> 健康检查 -> 首次成功调用 -> 周期账单 -> 续费/停用

#### 沙箱查询交易

挂牌 -> 审核 -> 下单 -> 合同 -> 支付锁定 -> 沙箱开通 -> 模板查询 -> 受限下载 -> 验收 -> 结算

#### 定制报告服务

需求提交 -> 人工报价 -> 合同 -> 付款锁定 -> 供方生成报告 -> 平台交付 -> 买方验收 -> 结算

#### 查询 + 订阅混合包

签约 -> API Key + 沙箱席位同时开通 -> 账单按基础订阅 + 超额用量计算 -> 到期断权

## 11. 订单、验收与账单

### 11.1 通用订单状态

`draft -> quoted -> approval_pending -> contract_pending -> contract_effective -> payment_locked -> delivery_in_progress -> acceptance_pending -> accepted/rejected/disputed -> settled -> closed`

### 11.2 文件产品

- 交付完成条件：令牌签发成功、文件包生成完成、可校验哈希
- 自动验收条件：成功下载 + 哈希一致，或 3 个工作日无异议
- 拒收条件：文件损坏、字段不符、缺失批次、加密包无法解密
- 账单事件：文件包数量、版本号、下载成功回执

### 11.3 API 产品

- 交付完成条件：应用绑定成功、Key 生效、健康检查通过
- 自动验收条件：首次成功调用，或 5 个工作日无异议
- 拒收条件：接口不可用、字段契约不符、鉴权失败、SLA 未达标
- 账单事件：订阅周期、调用量、超额调用、峰值并发

### 11.4 沙箱 / 报告产品

- 交付完成条件：账号/席位开通、模板可执行、样例查询通过
- 自动验收条件：首次登录且成功查询，或 5 个工作日无异议
- 拒收条件：账号未开通、模板不可用、导出规则不符、结果明显异常
- 账单事件：席位天数、查询次数、报告份数、加购模块

### 11.5 账单事件口径

- 文件：文件包数、版本数、分批交付次数
- API：订阅周期、调用次数、超额调用量、峰值并发档位
- 沙箱：席位数、席位天数、查询次数、导出次数、报告份数

### 11.6 结算实现边界

- V1 必须支持应结金额计算
- V1 必须支持账单生成
- V1 必须支持人工结算指令
- V1 可支持半自动分账
- V1 不要求自动分润引擎

## 12. 保证金、退款与赔付

### 12.1 保证金 / 风险托管

V1 必须支持：

- 高风险、高金额、低信誉订单可配置要求保证金
- 买方保证金、卖方履约保证金或等价风险托管
- 保证金与订单状态、争议状态、赔付责任、自动验收结果联动

保证金状态包括：

- 已冻结
- 已释放
- 已扣罚
- 已退回
- 待裁决

### 12.2 争议责任

#### 典型责任方

- 商品描述与实际不符：卖方主责
- 数据质量低于承诺：卖方主责
- 平台系统导致无法交付：平台主责
- 买方误操作/超范围使用：买方主责
- 监管要求阻断：非单方责任

### 12.3 证据优先级

1. 链上交易与授权摘要
2. 电子签章合同与授权版本
3. 交付回执 / API 调用日志 / 沙箱日志
4. 质量检测 / 验收报告
5. 平台系统日志 / 告警日志
6. 双方补充说明

### 12.4 SKU 默认争议规则

#### 文件包

- 未下载前可全额退款
- 下载后 48 小时内证明确有质量问题，可补交付或退款
- 买方本地环境问题原则上不退款

#### API

- 未开通成功可全额退款
- 24 小时内持续不可用按未开通处理
- 低于 SLA 阈值按档位赔付

#### 沙箱

- 未开通全额退款
- 晚开通补偿服务期
- 卖方未按约提供数据集可补开或部分退款

#### 报告

- 未交付可退款
- 可修复优先补交付
- 口径明显不符且无法修复可部分或全额退款

### 12.5 SLA 赔付档位

- ≥ 99.5%：无赔付
- 99.0%-99.5%：补偿 3 天服务期
- 98.0%-99.0%：补偿 7 天服务期或代金券
- < 98.0%：补偿 15 天服务期或部分现金赔付

## 13. 合规审查与风控

### 13.1 审查时间点

- 主体准入审查
- 产品上架审查
- 订单与授权审查
- 持续使用审查

- 短时高频下载 / 超限调用 -> 限流 + 告警

## 25. 数据本体存储与信任边界

### 25.1 V1 总体要求

1. `V1` 必须把“平台不是默认明文托管者”写成显式产品规则。
2. 卖方可以选择：
   - 平台托管
   - 卖方自持
   - API/买方拉取
   - 受限结果交付
3. 平台必须统一管理元数据、哈希、版本、授权、交付令牌、审计证据和生命周期状态。
4. 平台不得把“数据已登记”理解成“平台已获得任意明文使用权”。

### 25.2 V1 必做的存储模式

- 平台托管：平台保存密文对象，交付通过下载票据、短时链接、密钥封装完成。
- 卖方自持：平台保存元数据、哈希、连接方式、交付工单、策略和审计，不默认接收原始明文。
- API/买方拉取：平台签发凭证、控制配额与审计，底层数据源可在卖方侧。
- 受限结果交付：对高敏场景允许只交付查询结果、聚合结果或报告。

### 25.3 V1 关键字段要求

资产或资产版本必须有：

- `storage_mode`
- `payload_location_type`
- `custody_mode`
- `key_control_mode`
- `platform_plaintext_access`
- `platform_unilateral_decrypt_allowed`
- `default_delivery_route`
- `requires_controlled_execution`
- `retention_policy`
- `destroy_policy`

订单和交付必须快照：

- `storage_mode_snapshot`
- `delivery_route_snapshot`
- `trust_boundary_snapshot`
- `source_binding_id`
- `executor_type`
- `receipt_hash`

### 25.4 V1 信任机制

- 平台托管时必须实现数据与密钥分离。
- 卖方自持时必须支持连接器/API 网关交付，不得把它做成线下人工特例。
- 高敏资产默认优先通过 API、沙箱、报告、结果交付而不是明文副本。
- 所有下载、解密、导出、调用、断权、销毁必须留痕。
- `ZKP` 在 `V1` 不是平台信任基础，不承担“证明平台从未看过数据”的职责。

### 25.5 V1 生命周期治理

`V1` 至少追踪以下生命周期：

- 资产
- 版本
- 授权
- 交付
- 凭证
- 审计
- 存储保留与销毁

平台的生命周期治理目标是可追踪、可撤销、可断权、可审计、可证明，而不是无限期托管明文。

## 26. 支付、资金流与轻结算

### 26.1 V1 设计目标

- 真实支付接入与订单主链路解耦。
- 结构化表达货款、保证金、退款、赔付、人工打款与对账。
- 通过 `MockPaymentProvider` 支撑开发测试与演练。

### 26.2 新增能力

- 支付渠道适配层
- 支付意图、支付交易、回调事件
- 收费规则引擎
- 费用预估与费用快照
- 内部账务镜像
- 退款、人工打款、对账差异工单

### 26.3 V1 支持的支付方式

- 支付宝
- 微信支付
- 银联
- PayPal
- 线下对公转账
- 人工确认收款
- Mock 支付

### 26.4 核心规则

- 支付成功前不得进入交付。
- 回调验签失败不得改订单状态。
- 重复 webhook 不得导致重复记账、重复退款、重复打款。
- 费用规则必须在合同生效前冻结到订单。
- 账务镜像、渠道结果和订单状态必须可联查。

### 26.5 V1 页面与调试要求

- 新增支付中心、收费规则页、对账中心、Mock 支付调试页。
- 开发者通道必须支持支付状态联查、Mock webhook 和对账差异演练。

### 26.6 与现有逻辑的闭环

```text
定价 -> 费用预估 -> 支付意图 -> 渠道支付/Mock支付 -> 回调幂等确认
    -> 账务镜像 -> 交付 -> 验收 -> 结算 -> 退款/赔付/打款 -> 对账 -> 审计
```
