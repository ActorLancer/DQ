# 数据交易平台全量用户角色说明

## 1. 文档定位

本文件用于收敛平台的角色体系，目标是：

- 减少研发和权限设计歧义
- 保留当前阶段真正需要的角色
- 用少量核心角色覆盖 `V1 / V2 / V3` 的职责演进
- 不再把每个专题能力、每个页面、每个未来功能都膨胀成独立角色

本文件是“角色主清单”，不是“岗位名大全”。  
页面差异、专题差异、阶段差异，优先通过以下手段表达：

- 权限点
- 作用域
- 主体状态
- 商品/SKU 类型
- 合同与 Usage Policy
- 风控与合规结果

只有在职责边界、审计责任、双人复核或高风险动作明显不同的情况下，才单独定义角色。

## 2. 角色设计原则

### 2.1 基本原则

- 采用 `RBAC + Scope + Policy` 的组合模型。
- 角色负责表达“职责边界”，不负责承载全部业务差异。
- 人工角色与服务身份分开建模。
- `V1` 只启用最小角色集；`V2 / V3` 在同一骨架上增加职责，不再额外造一批新角色。
- 审核、合规、风控、结算、审计、高风险治理必须职责分离。

### 2.2 简化原则

- 不再把“搜索运营员、推荐运营员、节点运维管理员、公链展示管理员、图风控运营角色、监管协同管理员、生态合作管理员”等分别作为一级核心角色。
- 这些能力统一收敛到少量平台核心角色的权限包中。
- 不再把“Department 审批人、报告接收者、模型使用者、结果接收者”等弱职责对象定义为正式核心角色。
- 这些身份统一视为租户内用户类型或作用域，不再单独作为顶层角色。

### 2.3 当前冻结结论

平台正式保留 4 类身份对象：

1. 平台人工角色
2. 租户人工角色
3. 服务身份
4. 外部只读/协同身份

其中人工核心角色总数控制在最小可用集合内，避免角色爆炸。

## 3. 身份对象分层

### 3.1 组织身份

组织身份用于合同、账单、责任和监管归属，不等于人工角色。

组织身份类型包括：

- 数据供方组织
- 数据买方组织
- 双边参与组织
- 平台运营主体
- 监管/观察主体
- 生态合作主体

### 3.2 人工身份

人工身份用于登录、人工操作、审批、复核、处置和审计留痕。

### 3.3 服务身份

服务身份用于：

- API 调用
- 连接器同步
- 交付执行
- 查询沙箱执行
- 链上适配
- 回调处理
- 批处理与异步任务

### 3.4 外部只读/协同身份

用于监管观察、合作伙伴只读联查、有限协同操作。  
这类身份默认不是平台内部运营角色。

## 4. 核心角色总览

当前正式角色只保留以下 12 个：

### 4.1 平台人工角色

1. 平台管理员
2. 平台审核员
3. 平台风控结算员
4. 平台审计安全员

### 4.2 租户人工角色

5. 租户管理员
6. 卖方运营员
7. 买方运营员
8. 租户开发者
9. 租户审计只读员

### 4.3 服务身份

10. 租户应用身份
11. 平台服务身份

### 4.4 外部身份

12. 监管/合作方只读身份

说明：

- 这 12 个是正式角色/身份骨架。
- `V2 / V3` 新能力继续挂接到这 12 个对象，不再新增一堆专题角色。
- 个别能力如果必须分离，优先通过“权限包”实现，而不是新增角色名。

## 5. 平台人工角色

## 5.1 平台管理员

**定位**

- 平台侧最高管理角色。

**主要职责**

- 平台级配置与开关管理
- 角色、权限包、环境、系统策略管理
- 高风险冻结/解冻最终确认
- 支付、链、审计、可观测性等平台级能力的最终治理

**不负责**

- 日常商品审核
- 日常争议处理
- 普通账单执行

**阶段**

- `V1` 启用

## 5.2 平台审核员

**定位**

- 平台侧面向主体、商品、模板、合规的统一审核角色。

**主要职责**

- 主体准入审核
- 商品、SKU、模板、元信息、数据契约审核
- 合规边界复核
- 敏感数据链路审核
- 需要时发起升级复核

**不负责**

- 风险冻结执行
- 账单/退款/赔付执行
- 审计原文导出审批的最终签发

**阶段**

- `V1` 启用
- `V2 / V3` 扩展到计算、联邦、跨域合规审核

## 5.3 平台风控结算员

**定位**

- 平台侧处理风险、争议、退款、赔付、结算的执行角色。

**主要职责**

- 风险命中处置
- 黑白灰名单与冻结建议执行
- 退款、赔付、结算、人工差错修复
- 争议工单流转与裁决支持
- 交易链监控异常处置

**不负责**

- 商品准入审核
- 平台系统高权限配置
- 审计原文长期保留策略配置

**阶段**

- `V1` 启用
- `V2 / V3` 承接分润、跨链、图风控处置

## 5.4 平台审计安全员

**定位**

- 平台侧处理审计、安全、回放、证据、可观测性联查的控制角色。

**主要职责**

- 审计联查
- 证据包导出与回放审批
- `legal hold`、保留/销毁、break-glass 管理
- 安全事件、密钥治理、会话高风险审查
- 日志、Trace、可观测性与审计联动

**不负责**

- 商品经营
- 普通交易推进
- 普通退款和结算执行

**阶段**

- `V1` 启用

## 6. 租户人工角色

## 6.1 租户管理员

**定位**

- 租户内的最高业务管理角色。

**主要职责**

- 成员与部门管理
- 租户内角色分配
- 应用、连接器、执行环境配置
- 组织级策略与默认模板管理
- 组织级审计只读授权

**不负责**

- 平台侧最终审核
- 平台侧冻结/解冻
- 平台侧审计保留策略

**阶段**

- `V1` 启用

## 6.2 卖方运营员

**定位**

- 卖方侧统一业务执行角色。

**主要职责**

- 原始接入、加工、元信息维护
- 商品、SKU、价格、模板、交付面配置
- 文件/API/共享/模板查询/沙箱/报告交付
- 卖方连接器与存储绑定维护
- 卖方侧验收配合、争议举证、收款偏好维护

**不负责**

- 平台审核
- 平台风控结算最终裁决

**阶段**

- `V1` 启用
- `V2 / V3` 继续承接受控执行、伙伴互联等供给侧职责

## 6.3 买方运营员

**定位**

- 买方侧统一交易执行角色。

**主要职责**

- 搜索、询报价、下单
- 合同确认
- 授权申请与使用
- 验收、退款申请、争议发起
- API 使用、模板查询使用、查询沙箱使用、报告接收

**不负责**

- 卖方侧商品配置
- 平台侧结算执行

**阶段**

- `V1` 启用

## 6.4 租户开发者

**定位**

- 租户侧应用接入和开发调试角色。

**主要职责**

- 应用注册、密钥轮换、SDK/API 调试
- Webhook、回调、测试资产使用
- 开发者通道使用
- 模板查询和 API 接入调试

**不负责**

- 商品审核
- 生产账单、退款、赔付执行
- 平台高权限配置

**阶段**

- `V1` 启用

## 6.5 租户审计只读员

**定位**

- 租户内的只读审计和合规核对角色。

**主要职责**

- 查看本租户订单、合同、授权、交付、账单、审计摘要
- 配合内审、法务、财务核对
- 导出受限只读报表

**不负责**

- 修改商品
- 修改订单状态
- 修改账单
- 导出高敏原文证据

**阶段**

- `V1` 启用

## 7. 服务身份

## 7.1 租户应用身份

**定位**

- 租户拥有的程序化访问身份。

**主要职责**

- API 调用
- 下载令牌换取
- 模板查询调用
- 回调接收

**边界**

- 权限始终受租户作用域、商品/SKU、合同和 Usage Policy 限制
- 不等同于人工管理员

## 7.2 平台服务身份

**定位**

- 平台内部服务和适配器身份的统称。

**覆盖范围**

- Fabric 适配
- 事件消费与回放
- 搜索索引
- 推荐任务
- 通知
- 支付回调处理
- 查询执行面

**边界**

- 只能执行被平台显式授权的系统动作
- 不得绕过人工审计链直接改写高风险业务事实

## 8. 外部身份

## 8.1 监管/合作方只读身份

**定位**

- 平台外部的只读或有限协同观察身份。

**主要职责**

- 监管观察
- 伙伴联查
- 有限范围证据查看
- 合作方只读状态核对

**不负责**

- 平台内部运营
- 商品审核
- 订单推进
- 普通交易处理

**阶段**

- `V1` 默认只读或预留
- `V2 / V3` 可在有限协同场景中增强

## 9. 角色与能力映射

### 9.1 V1 核心能力映射

| 能力域 | 主要角色 |
| --- | --- |
| 主体注册与组织管理 | 平台审核员、租户管理员 |
| 商品与 SKU 配置 | 卖方运营员、租户管理员 |
| 搜索与下单 | 买方运营员 |
| 合同与授权 | 买方运营员、卖方运营员、租户管理员 |
| 交付与使用 | 卖方运营员、买方运营员、租户开发者 |
| 退款、赔付、争议、结算 | 平台风控结算员 |
| 审计、回放、安全治理 | 平台审计安全员 |
| 平台配置与高危治理 | 平台管理员 |
| API/SDK 接入与调试 | 租户开发者、租户应用身份、平台服务身份 |

### 9.2 V2 扩展能力映射

`V2` 不新增顶层角色，主要在既有角色上增加权限包：

- 平台审核员：增加受控执行、联邦、证明审核权限
- 平台风控结算员：增加分润与任务风控权限
- 卖方运营员：增加受控执行供给配置权限
- 买方运营员：增加受控执行任务发起和结果接收权限
- 平台服务身份：增加计算、联邦、证明、模型相关服务权限

### 9.3 V3 扩展能力映射

`V3` 仍不新增大量角色，主要增加跨域权限包：

- 平台管理员：跨链、生态、互认治理
- 平台风控结算员：图风控、跨平台异常处置
- 平台审计安全员：监管穿透、跨平台证据联查
- 监管/合作方只读身份：有限协同与跨平台联查
- 平台服务身份：跨链网关、生态连接器等系统级服务权限

## 11. 职责分离规则

以下动作不应由单一人工角色闭环完成：

- 主体审核 + 商品审核 + 高风险放行
- 商品审核 + 结算执行
- 争议裁决 + 赔付执行 + 审计原文导出
- 平台高权限配置 + 审计保留策略修改
- 风险冻结执行 + 风险冻结解除 + 审计清理

最小分离要求：

- 平台管理员不能替代平台审核员做日常审核
- 平台审核员不能替代平台风控结算员做争议与结算执行
- 平台风控结算员不能替代平台审计安全员做证据原文导出与回放审批

## 12. 高风险动作要求

### 12.1 V1

以下动作必须至少满足“角色权限 + 二次认证 + 审计留痕”：

- 主体冻结/解冻
- 商品下架/冻结
- 退款/赔付执行
- 高敏证据导出
- replay 执行
- break-glass
- 支付通道切换
- 存储与信任边界修改

### 12.2 V2

在 `V1` 基础上增加：

- 受控执行开通
- 联邦任务发起/参与
- 证明结果放行
- 分润规则修改

### 12.3 V3

在 `V2` 基础上增加：

- 跨链/跨平台治理动作
- 监管协同冻结
- 生态互认与信任边界修改

## 13. 角色边界提醒

- 角色是职责边界，不是页面分组。
- 同一角色可因作用域不同只看到不同菜单和对象。
- 不再把“搜索运营、推荐运营、可观测性运营、公链展示、节点运维、图风控、监管协同、生态合作”等分别提升为核心角色。
- 这些能力统一由核心平台角色或平台服务身份承接，再通过权限包区分。
- 机器身份不是“低配人工角色”，必须单独建模、单独审计。

### 14.2 V2 增量启用

- 受控执行权限包
- 联邦/证明权限包

### 14.3 V3 增量启用

- 跨链治理权限包
- 图风控权限包
- 监管协同权限包
- 生态互联权限包

## 15. 一句话结论

这份角色说明的冻结结论是：

- 平台不再采用“每个功能点一个角色”的膨胀式设计。
- 当前正式角色收敛为少量核心人工角色、少量服务身份和少量外部身份。
- `V2 / V3` 的能力扩展优先通过权限包和作用域表达，而不是继续新增大量角色。
