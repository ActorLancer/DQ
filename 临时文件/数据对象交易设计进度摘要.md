# 数据对象交易设计进度摘要

- 更新时间：2026-04-09
- 用途：快速查看“当前围绕数据对象本身及其交易”已经设计到什么程度、哪些只是预留、哪些还明显没设计透。
- 说明：本摘要只围绕数据对象、数据商品、交易主链路、交付、策略、搜索/推荐、审计与数据库落库；不展开 IAM、支付、日志等外围域。

---

## 1. 一句话判断

当前设计已经把“数据对象如何被产品化、如何交易、如何交付、如何审计、如何落库”这条主骨架做出来了。

如果只看 `V1`，已经不是“概念阶段”，而是已经形成：

- `DataAsset -> AssetVersion -> Product -> SKU -> Contract/Usage Policy -> Order -> Delivery -> Acceptance -> Billing/Audit`

但也要明确：

- `V1` 现在冻结的是 **文件快照/版本订阅、只读共享、API/服务、查询沙箱/模板查询 lite、固定报告/结果产品** 五类标准商品；
- 高级能力如 **模型商品、受控计算、联邦训练、MPC、ZKP** 主要还是 `V2/V3`；
- 数据对象“怎么接入、怎么治理、怎么做字段级/行业级规则、怎么做高质量评分和更细颗粒度执行控制”，还没有全部设计完。

---

## 2. 当前已经明确设计好的内容

### 2.1 交易对象已经不再只定义为“数据集”

目前已经把交易对象收敛为 5 类基础 family，并在其上扩展高级对象：

- 文件 / 对象包产品
- 表 / 库 / 视图 / 数据共享产品
- API / 服务产品
- 受限查询 / 受控计算产品
- 结果 / 合成 / 特征 / 模型衍生产品

但正式阶段边界也已经明确：

- `V1` 正式交付：文件快照/版本订阅、只读共享、API/服务、查询沙箱/模板查询 lite、固定报告/结果产品
- `V2` 才进入：计算权、模型服务、联邦训练、MPC、ZKP 等

---

### 2.2 数据对象已经拆成“资产、版本、可交付对象、商品、SKU”多层

当前核心建模已经清楚：

- `DataAsset`
  - 逻辑上的可交易数据资产
- `DataAssetVersion`
  - 某个具体版本、快照、批次、加工结果
- `AssetObjectBinding`
  - 某个版本下真正可交付、可共享、可查询的逻辑对象
- `Product`
  - 面向市场曝光的商品封装
- `ProductSKU`
  - 可下单的最小交易单元

这意味着当前方案已经明确：

- “数据是什么” 和 “卖什么权利” 不是一个对象
- 一个资产可以有多个版本
- 一个版本可以挂多个可交付对象
- 一个版本可以被多个商品封装
- 一个商品可以有多个 SKU

这部分骨架已经比较完整。

---

### 2.3 权利模型与策略模型已经设计出来

目前已经明确支持的核心权利包括：

- 使用权
- 访问权
- 查询权
- 计算权
- 结果获取权
- 有限内部共享权
- 模型训练权
- 商业输出权
- 转授权权

同时已经明确了 `V1` 权利边界：

- `V1` 正式支持：使用权、访问权、查询权、结果获取权、有限内部共享权
- `V1` 默认禁止：所有权转让、无限期训练权、转授权、开放再分发

并且已经用这些对象承接：

- `DigitalContract`
- `UsagePolicy`
- `policy_binding`
- `authorization_grant`

也就是说，当前不是“卖一份文件”，而是已经转成“卖一种受限制的权利包”。

---

### 2.4 数据本体存储与信任边界已经设计清楚

这是当前设计里比较成熟的一块。

已经明确：

- 平台不是默认明文托管者
- 平台交易的是权利和可验证交付，不默认等于交易明文副本
- 原始数据、大对象、模型权重、训练语料、证据原文默认不直接进 PostgreSQL

目前已经支持并建模了 5 类模式：

- 平台托管
- 卖方自持
- 受控执行
- 买方拉取
- 第三方可信存储

并且已经要求在资产/订单/交付中显式保存：

- `storage_mode`
- `payload_location_type`
- `custody_mode`
- `key_control_mode`
- `default_delivery_route`
- `trust_boundary_snapshot`
- `receipt_hash`

这意味着“卖方不信任平台”这个问题已经不是空口原则，而是已进入对象模型和数据库设计。

---

### 2.5 数据交易主链路已经形成闭环

当前统一主链路已经非常明确：

1. 主体注册与审核
2. 数据资源治理与产品化
3. 产品审核与挂牌
4. 搜索与撮合
5. 合同与授权
6. 支付与锁定
7. 交付或执行
8. 验收
9. 结算与账单归档
10. 持续使用监管与售后

并且 `V1` 已按商品类型拆出了标准链路：

- 文件副本一次性交易
- API 订阅交易
- 沙箱查询交易
- 定制报告服务
- 查询 + 订阅混合包

所以“数据对象如何完成一次真实交易”这条主链已经有了。

---

### 2.6 交付与验收已经按商品类型设计

当前不是一个统一的“交付按钮”，而是已经区分：

- 文件交付
  - 下载票据、校验和、回执、密钥封装
- API 交付
  - API Key / OAuth、配额、健康检查、首次成功调用
- 沙箱交付
  - 工作区、席位、模板查询、导出限制
- 报告交付
  - 报告成果物、版本、验收回执

数据库也已经落了这些对象：

- `delivery.storage_object`
- `delivery.delivery_record`
- `delivery.key_envelope`
- `delivery.delivery_ticket`
- `delivery.delivery_receipt`
- `delivery.api_credential`
- `delivery.api_usage_log`
- `delivery.sandbox_workspace`
- `delivery.report_artifact`

---

### 2.7 搜索与推荐已经开始围绕数据对象工作

目前已经不是“只有目录页”。

搜索已支持：

- 搜索数据商品
- 搜索服务商品
- 搜索卖方主体

推荐已支持：

- 推荐数据商品
- 推荐服务商品
- 推荐卖方主体

并且已经为“数据商品被发现和排序”设计了：

- 标签体系
- 搜索投影
- 排序权重
- 热度/信誉/质量等信号
- 推荐位
- 推荐行为事件

这一块说明：数据对象已经不只是“能被上架”，而是已经进入“能被发现、比较、筛选、推荐”的阶段。

---

### 2.8 审计、证据链、一致性已经接到数据交易主链路上

围绕数据对象交易，当前已经明确：

- 关键事件必须留痕
- 合同、授权、交付、验收、结算、争议都进入审计
- 关键摘要进入 Fabric
- 数据库是业务运营主状态机
- Fabric 是可信证明确认层
- 外部系统对支付/CA/存储等事实拥有来源权威

也就是说，“这份数据是什么、谁卖的、卖了什么权利、谁拿到了什么、什么时候交付的、怎么验收的”，理论上都已经能被追踪。

---

### 2.9 数据库层已经形成完整骨架

围绕数据对象交易，目前数据库已经不是只有几张目录表，而是已经至少覆盖：

- 资产域
  - `catalog.data_asset`
  - `catalog.asset_version`
  - `catalog.asset_storage_binding`
  - `catalog.asset_sample`
  - `catalog.asset_structured_dataset`
  - `catalog.asset_structured_row`
- 商品域
  - `catalog.product`
  - `catalog.product_tag`
  - `catalog.product_sku`
- 合同/策略域
  - `contract.usage_policy`
  - `contract.digital_contract`
  - `contract.policy_binding`
- 交易域
  - `trade.order_main`
  - `trade.order_line`
  - `trade.authorization_grant`
- 交付域
  - `delivery.*`
- 搜索域
  - `search.product_search_document`
  - `search.seller_search_document`
  - `search.model_search_document`
- 推荐域
  - `recommend.*`
- V2/V3 模型与受控计算域
  - `ml.*`

因此，从“对象模型 + 数据库骨架”的角度看，围绕数据对象交易已经做到了比较深。

---

## 3. 当前属于“已建模但不是 V1 已完成”的内容

这些内容已经进入全量设计，但不应误判为当前版本已经完全落地：

- 模型资产交易
- 模型文件交付 / 模型服务交付
- 受控计算任务
- 联邦训练
- MPC 结果产品
- ZKP 产品
- 数据产品护照 NFT
- 公链展示增强
- 更复杂的跨平台可信存储 / 互认

换句话说：

- **对象模型有了**
- **数据库预留了**
- **V2/V3 流程也有了**
- 但如果只看 `V1`，这些还不算当前主交付完成项

---

## 4. 当前明显还没设计透或还没冻结的内容

这是你现在最值得继续补的部分。

### 4.1 数据接入与加工治理细节还不够深

虽然已经有“数据资源治理与产品化”的链路，但还缺更细的设计，例如：

- 原始数据接入规范
- 批量导入 / 增量同步 / 流式接入细则
- 数据清洗、脱敏、标准化流水线详细规则
- 样本自动生成与预览生成规则
- 数据版本生成策略和发布策略

当前更像“有业务骨架”，还不是“有完整数据工程方案”。

### 4.2 行业级 / 字段级资产模型还没深化

目前有统一元数据字典，但还缺：

- 工业、零售等首批行业的专用字段模板
- 不同行业的 schema 模板
- 字段级质量规则
- 字段级敏感等级继承规则
- 字段级 / 记录级实际执行方案

也就是说，平台级通用骨架已定，但行业落地模板还不够细。

### 4.3 数据质量评分和卖方信誉如何计算还不够明确

当前已经出现：

- `quality_score`
- `seller_reputation_score`
- `credit_level`
- `risk_level`

但还没看到完全冻结的：

- 质量分计算公式
- 信誉分更新规则
- 降权/加权条件
- 争议、退款、验收失败如何影响分值

这会影响搜索排序、推荐排序和上架治理。

### 4.4 Usage Policy 的执行引擎还没完全细化

现在已经有：

- `Usage Policy` 对象
- 权限放行链
- 合同 + 策略联合判定

但还缺更细的执行设计，例如：

- 字段级裁剪怎么做
- 结果导出限制怎么落
- API 响应脱敏怎么执行
- 沙箱结果逆识别怎么阻断
- 训练/推理/导出在执行时怎样绑定策略

目前这块是“规则对象已建模”，但“策略执行引擎”还不够完整。

### 4.5 数据血缘 / 加工来源 / 衍生关系还不够正式

文档里有血缘和可信证明的方向，但围绕数据对象本身，还缺一个更正式的：

- 原始资产 -> 加工资产 -> 商品 -> 模型 -> 推荐/索引衍生物

的血缘链模型与页面设计。

### 4.6 数据对象运营规则还不够完整

比如还缺更具体的：

- 上架后运营策略
- 精品/特色/专题位规则
- 卖方扶持规则
- 数据商品生命周期运营规则
- 冷门商品的处理规则

搜索和推荐骨架已做，但“商品运营”还没有完全做细。

---

## 5. 当前进度判断

如果只评价“围绕数据对象交易”的设计进度，我的判断是：

- **V1 主骨架：已经比较完整**
- **全量对象模型：已经很完整**
- **数据库骨架：已经完整**
- **行业细节 / 执行引擎 / 高级商品能力：还没完全设计透**

更通俗地说：

- 你现在已经知道“要卖什么、怎么卖、怎么交付、怎么审计、数据库怎么落”
- 但还没有把“数据治理工程、字段级执行、质量算法、行业专用模板、V2/V3 高级交易对象执行细节”完全做完

---

## 6. 如果现在要继续补，建议优先级

建议下一步优先补这 6 类内容：

1. 行业级数据资产模板与字段标准
2. 数据接入 / 清洗 / 脱敏 / 版本发布流程
3. 数据质量评分规则与信誉联动规则
4. Usage Policy 执行引擎设计
5. 数据血缘与衍生关系设计
6. 高级交易对象（模型 / 受控计算 / 联邦训练）的正式交易细则
