（Dead Letter Event（死信事件 / Dead Letter Queue））

技术点,推荐配置,常见坑 & 解决办法
Outbox,同一库事务 + JSONB payload,表膨胀 → 定期归档已处理记录（PARTITION by created_at）
Worker,多实例 + SKIP LOCKED + 批量,脑裂 → 使用分布式锁（Redis）或数据库 Advisory Lock
重试策略,指数退避（1s → 2s → 4s …）,无限重试 → 必须设置 max_retries + Dead Letter
Dead Letter,单独 DLQ Topic + 监控 dashboard,死信堆积 → 设置告警 + 自动/手动 Reprocess
幂等性,事件必须带 event_id / trace_id,重复消费 → 消费者用唯一键防重
延迟,轮询间隔 1~3 秒,太高 → 改用 CDC (Debezium) 或 Postgres LISTEN/NOTIFY
