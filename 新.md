## 先给总判断

你的整体需求已经把**平台方向**写得很清楚了，但还没有把**工程实现路线**彻底拍板。

已经很明确的，是这些事情：

真正还没定下来的，不是“做不做这个平台”，而是：

* 联盟链底座到底选 **Fabric** 还是 **FISCO BCOS**
* 交易状态到底是 **链上权威** 还是 **数据库权威**
* V1 到底是 **模块化单体** 还是 **一开始微服务**
* 身份、权限、策略到底落到哪套具体产品
* 消息总线、对象存储、KMS、沙箱查询引擎、支付托管方式，到底怎么选

---

### 2. 链上链下分工已经确定

从正式 PRD、架构稿、数据库设计、业务流程几个文件交叉看，方向是一致的：

* **链上**：状态摘要、签名、证据 hash、订单/结算关键控制点、审计锚点
* **链下**：原始数据、密文文件、证据原文、模板正文、模型文件、日志全文、检索数据

这点不用再犹豫。平台不是“把原始数据上链卖”。

---

### 3. 主数据底座已经确定是 PostgreSQL 系

你当前数据库脚本、README、测试 compose 都已经统一到了 **PostgreSQL + pgvector + pg_trgm + 分阶段 migration**。而且正式设计明确要求：

* PostgreSQL 承载主业务状态与元数据
* 大对象和密文放对象存储
* 搜索走全文检索和向量检索
* 全部版本支持 upgrade / downgrade

PostgreSQL 18 已正式发布；PostgreSQL 自带全文检索能力，`pg_trgm` 支持相似度检索，而 `pgvector` 官方扩展支持把向量和业务数据放在同一个 Postgres 体系里做近邻检索。这个方向和你的文档是对齐的。([PostgreSQL][1])

---

### 4. 身份模型的大方向已经确定

你的文档已经把身份分成四层：

* 组织身份
* 用户身份
* 服务身份
* 链上身份

这意味着实现上也基本确定：

* **用户登录**不能直接拿链证书替代
* **链上身份**更适合做签名、成员准入、可信凭证
* **服务身份**必须独立于用户身份
* **高风险动作**要跟 MFA、审计、复核绑定

这个方向跟当前主流 IAM 方案是一致的。Keycloak 官方文档支持基于标准协议的 OIDC/OAuth2/SAML，并提供细粒度 Authorization Services；这很适合你做“登录与客户端凭证”的统一入口。([Keycloak][2])

---

### 5. 审计与证据链模型已经确定

你的文档里对审计要求其实非常重，而且数据库里已经落了：

* `audit.audit_event`
* `audit.evidence_package`
* `chain.contract_event_projection`
* `chain.chain_anchor`
* `ops.outbox_event`
* `ops.dead_letter_event`

这说明平台的工程基因不是普通电商，而是**强留痕、强可回放、强对账**的平台。

## 二、现在最重要的：哪些技术还没有定，或者文档里还有冲突

## 2. 联盟链底座还没有定

### Fabric 适合什么

Hyperledger Fabric 官方文档现在最强的是这几块：

* **Private Data Collections**
* **多组织共同批准链码生命周期**
* **更成熟的联盟治理模型**
* **链码可作为外部服务运行**

这很适合“多个机构共同治理、隐私边界复杂、要长期做联盟协同”的平台。([hyperledger-fabric.readthedocs.io][3])

如果你的真实目标是：

* 一开始就要多机构共治
* 多组织联盟链是正式生产形态
* 对隐私通道/联盟治理要求很高

那就直接上 **Fabric**。

### 最终建议

我会这样定：

* **V1 首选：FISCO Fabric**
* **同时强制引入 `ChainAdapter` 抽象层**
* `SolanaAnchorAdapter` 保留接口，默认关闭
* V2/V3 再决定是否追加 `FabricAdapter` 或跨链网关

---

### 什么时候再拆微服务

到 V2/V3 再把这些独立出来：

* FL / C2D 编排
* 图风控
* 公链凭证服务
* 跨链网关
* 监管只读同步
* 高吞吐审计检索

---

## 4. 认证、鉴权、策略引擎的具体产品还没定

你的文档已经很清楚要求：

* V1 至少有 `RBAC + Scope + 合同/Usage Policy 联合判定`
* 全量模型还预留了 `ABAC + PBAC`
* 后端中间件顺序也定义好了：身份 → 主体状态 → 权限 → 作用域 → 策略 → 风控 → 高风险校验 → 审计

### 我的推荐

#### 认证层

* **Keycloak** 作为统一认证中心
* 人登录：OIDC
* 应用接入：Client Credentials
* 以后监管/合作伙伴接入也能走统一协议

Keycloak 官方文档支持 OIDC/OAuth2/SAML，并有授权服务能力。([Keycloak][2])

#### 权限层

* **RBAC + Scope** 放数据库和业务服务里
* 角色、权限点、作用域绑定，你现有 SQL 设计已经能承接

#### 策略层

* V1 不建议一开始就上很重的统一策略平台
* V1 先在应用层实现：

  * 合同快照
  * usage_policy
  * 地域/用途/时效/导出限制
  * 风险阻断

#### V2 再引入 OPA

OPA 官方文档就是典型 policy-as-code 引擎，适合把用途限制、地域限制、输出限制、监管例外规则逐步外置。([openpolicyagent.org][5])

也就是说：

* **V1：Keycloak + 应用内策略判断**
* **V2：Keycloak + OPA**
* 不建议把细粒度业务策略全塞进 IAM 产品本身

---

## 5. 对象存储和密钥治理产品还没定

你的文档已经明确：

* 大对象、密文、模型、证据原文不进 PostgreSQL
* 交付要有 ticket、envelope、receipt_hash
* 对象要版本化
* 证据类对象要支持 WORM / 不可篡改保留

### 我的推荐

#### 对象存储

* 私有化/本地化：**MinIO**
* 云上：**S3 兼容对象存储**

MinIO 官方文档支持 bucket versioning，也支持 object lock / WORM retention，这和你的证据链要求是匹配的。([MinIO AIStor Documentation][6])

#### 加密方式

* 文件内容：AES-GCM
* 每订单或每版本独立 DEK
* DEK 用买方公钥或组织 KMS 公钥封装
* 下载只给短期 ticket，不直接暴露对象真实路径

#### KMS

* 开发/测试：软件 KMS 即可
* 生产：云 KMS / HSM / Vault Transit 三选一
* 关键是**轮换、审计、权限隔离**，不是先纠结品牌

## 6. 搜索技术其实基本能定了

我不建议你现在就上 OpenSearch / Elasticsearch 作为首版硬依赖。

因为你的数据库设计已经非常明确地押在：

* PostgreSQL 全文检索
* `pg_trgm`
* `pgvector`
* `search.product_search_document`

### 我的推荐

#### V1

只用：

* PostgreSQL 全文检索
* `pg_trgm`
* 标签、分类、状态、价格、行业等精确筛选

#### V2/V3

再考虑：

* 向量召回
* 混合检索
* 独立搜索引擎
* 日志检索和商品检索分开

PostgreSQL 当前内置全文检索功能完善，`pg_trgm` 适合相似搜索；`pgvector` 官方扩展则很适合作为“轻量向量能力”，不用首版就引入独立向量数据库。([PostgreSQL][7])

## 7. 消息总线产品还没有必要过早重型化

文档要求的是：

* outbox
* dead letter
* idempotency
* 重试
* 消费者解耦
* 事件投影
* 通知 / 索引 / 风控 / 报表异步化

它要求的是**机制**，不是要求 V1 立刻上 Kafka。

### 我的推荐

#### V1

* **PostgreSQL outbox_event + worker**
* `dead_letter_event`
* 必要时加 **RabbitMQ**
* 不要先上 Kafka

#### V2/V3

当你开始有这些情况时，再换 Kafka / Redpanda：

* 训练事件流
* 跨链回执流
* 高吞吐审计分发
* 风险图谱流式输入
* 多服务独立扩缩容

## 8. 沙箱查询 / 受控计算引擎还没有定型

这是一个明显“需求已存在、实现还不能一次拍死”的部分。

你的文档里已经定义了：

* V1 有沙箱商品
* V2 有 C2D / 受控训练 / FL
* 但首版并不适合直接做成通用计算平台

### 我的建议

#### V1

沙箱不要做成“大而全的通用计算平台”，而做成：

* 固定查询模板
* 只读工作区
* 受限导出
* 席位管理
* 审计记录
* 少量 SQL/指标型查询能力

#### 技术上

如果首版数据量不大：

* 直接基于 PostgreSQL 的只读工作区就够

如果你明确要分析型查询：

* 再考虑引入 Trino / ClickHouse
* 但这不该成为 V1 阻塞项

## 9. 支付 / 托管 / 结算方式还没定

你的文档里已经确定有：

* 买方保证金
* 卖方履约保证金
* 风险托管
* 结算、赔付、扣罚、释放

但还没明确 **资金系统怎么接**。

### 我的建议

#### V1 不做重型真实资金链路

先做：

* 平台内部账务台账
* 线下转账确认
* 手工或半自动结算
* 链上只记托管状态摘要和责任结果

#### 不建议 V1 直接做

* 真正的生产级链上支付清结算
* 复杂自动分润
* 完整财税自动化

因为这会把首版拖进最难的合规区。

## 三、如果今天拍板，我给你的推荐技术栈是这个

## V1 首选实现栈

### 前端

* **TypeScript**
* **React + Next.js**
* 后台组件库可选 **Ant Design**

理由：你页面很多，而且有门户、供方工作台、买方控制台、审计/风控/开发者通道，多角色后台更适合 React 生态。

### 后端

* **Java 21 + Spring Boot**
* **模块化单体**
* 数据访问层建议：

  * 要么 MyBatis
  * 要么 jOOQ
* 不建议首版就强依赖 ORM 自动映射全部复杂模型

理由：你的平台是重状态机、重权限、重审计、重 SQL 的企业系统，Java/Spring 对这类系统更稳，而且如果你选 FISCO，Java 生态衔接也更顺。

### 联盟链

* **FISCO Fabric**
* 合约用 **Solidity**
* 通过 `ChainAdapter` 隔离链差异

### 数据库

* **PostgreSQL 18**
* 扩展：

  * `pgcrypto`
  * `citext`
  * `pg_trgm`
  * `btree_gist`
  * `vector`

你当前脚本和测试镜像已经按这个方向组织好了，继续沿用最省返工。PostgreSQL 18 已正式发布。([PostgreSQL][1])

### 缓存

* **Redis**
* 用于：

  * ticket
  * session
  * rate limit
  * MFA / challenge
  * 热门目录缓存
* 不能作为最终状态源

### 对象存储

* **MinIO / S3**
* 证据桶必须启用：

  * versioning
  * object lock / legal hold
  * access log

### IAM

* **Keycloak**
* 人登录走 OIDC
* 应用走 client credentials
* API 商品可挂 API Key / OAuth

### 权限与策略

* V1：

  * RBAC + Scope + 业务内 policy evaluator
* V2：

  * 引入 OPA

### 观测与审计

* **OpenTelemetry + Prometheus + Grafana**
* 日志可以选 **Loki** 或 ELK
* 审计日志单独 schema / 单独索引口径

### 部署

* 本地：Docker Compose
* 测试/预发/生产：Kubernetes
* 有状态组件走 StatefulSet，无状态服务走 Deployment；StatefulSet 本来就是 Kubernetes 用来管理带持久化和稳定身份的状态服务的标准控制器。([Kubernetes][8])

## 四、如果你从一开始就想做“真正多机构联盟”，那我会改成这套

* 联盟链：**Hyperledger Fabric**
* 链码：**Go**
* 后端：Java 或 Go 都行
* 私有数据：用 Fabric 的 **Private Data Collections**

* 外部运行：必要时链码作为 external service

Fabric 官方文档在多组织链码治理、私有数据集合和 external chaincode service 这些点上更成熟。([hyperledger-fabric.readthedocs.io][3])

这条路线更适合“真的要多个机构长期共治”，但 V1 的实现速度通常会慢于 FISCO 方案。

## 五、我建议你现在立刻冻结的 10 个技术决策

这 10 个不拍板，研发永远进不了稳定状态：

1. **联盟链底座**：FISCO 还是 Fabric
2. **关键状态权威源**：数据库、链上，还是双层权威
3. **V1 部署形态**：模块化单体还是微服务
4. **认证中心**：Keycloak 还是企业现有 IAM
5. **策略引擎路线**：V1 代码内置，V2 再 OPA，还是现在就引入
6. **对象存储**：MinIO / 云 S3
7. **KMS 方案**：软件 KMS、云 KMS、HSM
8. **消息总线路线**：先 outbox-only，还是 RabbitMQ/Kafka
9. **沙箱实现范围**：只读模板查询，还是通用分析工作区
10. **资金链路**：内部台账 + 线下确认，还是接真实支付/托管

## 六、我的最终建议

如果你问我：“现在基于全部需求清单，最合理、最稳、最适合推进的路线是什么？”

我的答案是：

### 先这样定

* **V1：FISCO BCOS + Java Spring Boot + PostgreSQL 18 + Redis + MinIO + Keycloak**
* **架构：模块化单体 + ChainAdapter + Outbox**
* **一致性：双层权威模型**
* **搜索：先 PostgreSQL FTS / pg_trgm，不上独立搜索引擎**
* **策略：V1 内置，V2 再 OPA**
* **公链：只保留 AnchorService，默认关闭**
* **支付：先内部账务与半自动结算，不做真实资金闭环**
* **沙箱：先做受控查询，不做通用计算平台**

### 为什么是这条路线

因为它最符合你现有文档已经定下来的事实：

* V1 要的是**可信交易闭环**
* 不是 V2/V3 的全部能力提前实现
* 你的数据库、对象模型、权限体系、审计模型都已经偏向这条路线
* 而且这条路线返工最少，最容易先跑出可验收的系统

接下来最值得做的，不是继续讨论“大方向对不对”，而是把这 10 个决策写成 ADR，一项项冻结。
