- 避免 V2/V3 因为模型缺口出现大面积返工

## 2. 总体建模原则

### 2.1 建模目标

- 支持数据商品、模型商品、受控计算、联邦协作、跨链治理的统一业务骨架
- 保持“链下主执行、链上主摘要”的一致性建模
- 支持合同、策略、账单、审计、风控、监管的可追溯关系
- 支持未来新增商品类型、执行环境、证明类型、目标链和合作伙伴

### 2.2 关键设计原则

- `业务对象` 与 `链上摘要对象` 分离
- `交易实例` 与 `计费事件` 分离
- `授权对象` 与 `访问凭证` 分离
- `风险事件` 与 `治理动作` 分离
- `运行态对象` 与 `审计证据对象` 分离
- `聚合根内强一致`，聚合间通过事件和引用关联

### 2.3 版本启用策略

```text
V1:
- 启用主体、商品、订单、交付、争议、账单、保证金、审计、开发调试主模型
- 预留训练、模型、分润、跨链、图风控、连接器对象

V2:
- 正式启用模型、算法、受控计算、训练任务、证明、凭证、分润对象

V3:
- 正式启用跨链请求、见证、连接器、风险图谱、治理工单、监管查询对象
```

### 2.4 工程冻结规则

- 对象名统一使用英文单数聚合名，字段名统一使用英文 `snake_case`。
- 中文术语只用于文档解释、页面展示和合同文案，不作为工程字段名。
- `bind_*_ids / allowed_* / supported_*` 在领域文档中视为聚合视图；正式落库优先拆关系表或快照表，数组字段只允许作为缓存与搜索投影。
- `current_version` 只表示当前指针，进入审核、交易、授权、结算的正文必须形成不可变快照。
- `status_version` 只用于状态并发控制，不得与合同版本、商品版本、策略版本混用。

### 2.5 跨版本生命周期冻结规则

- `Authorization / Delivery / Settlement / Dispute / BillingEvent` 是跨 `V1 / V2 / V3` 的正式生命周期对象，不得在后续版本引入与之平行的第二套交易生命周期主对象。
- `V2` 的模型、受控计算、联邦、证明、分润能力必须挂接到上述生命周期对象，或显式作为其扩展对象。
- `V3` 的跨链、监管、图风控和生态互联能力可以新增治理对象，但不得替代本地交易生命周期对象。
- 统一一致性状态字段族继续适用于 `Order / Contract / Authorization / Delivery / Settlement / Dispute / BillingEvent`。
- `V1-Core / V1-Extended / V1-Reserved` 仅属于实施分组，不改变领域边界；后续版本引用 `V1` 对象时必须保留这层来源语义，避免把演示级能力误当作既有生产事实。

## 3. 顶层领域划分

建议将系统拆为 12 个核心领域：

1. `身份与主体域`
2. `目录与商品域`
3. `合同与策略域`
4. `交易与订单域`
5. `交付与执行域`
6. `账单、托管与分润域`
7. `争议与售后域`
8. `信誉与风控域`
9. `审计与证据域`
10. `链上摘要与公链增强域`
11. `跨链与生态互联域`
12. `开发与调试支持域`
13. `搜索与索引同步域`

**职责**

- `origin_region`
- `sensitivity_level`
- `version_status`

### 4.2.2A AssetObjectBinding

**主键**

- `asset_object_id`

**职责**

- 表示某个 `DataAssetVersion` 下真正可交付、可共享、可查询或可授权的逻辑对象
- 让“版本”继续细化为文件对象、表/视图共享对象、API 端点、模板查询对象、结果对象

**典型对象类型**

### 4.2.3 Product

**主键**

- `product_id`

**职责**

- 表示面向市场曝光的商品封装
- 承接目录、审核、搜索、详情展示和上下架状态

**核心字段**

- `product_id`
- `product_type`
- `delivery_mode`
- `usage_mode`
- `sku_template_id`
- `current_version`
- `product_status`

**关系**

- `DataAsset 1 -> N Product`
- `DataAssetVersion 1 -> N Product`
- `Product 1 -> N ProductMetadataProfile`
- `Product / ProductSKU 1 -> N DataContract`

**生命周期**

- `draft -> pending_review -> listed -> suspended -> retired`

**建模规则**

- `Product` 是目录、审核、搜索、详情展示和聚合视图事实源。
- `product_type` 仅表达目录与展示分类，不替代 `sku_type`。
- `delivery_mode` 表达商品默认展示或推荐交付路径，可作为 SKU 初始化默认值，但不替代 SKU 级事实源。
- `usage_mode` 表达推荐使用姿态或默认使用方式，不等同于正式权利集合。
- `sku_template_id` 表达 SKU 模板族默认值或批量生成入口，不等同于单个 SKU 的最终模板绑定。

**职责**

### 4.2.3E RawIngestBatch / RawObjectManifest / FormatDetectionResult / ExtractionJob / PreviewArtifact

**主键**

- `raw_ingest_batch_id`
- `raw_object_manifest_id`
- `format_detection_result_id`
- `extraction_job_id`
- `preview_artifact_id`

**职责**

- `RawIngestBatch`：表示一次原始对象接入登记批次。
- `RawObjectManifest`：表示批次下的原始对象、来源、hash、大小和声明格式。
- `FormatDetectionResult`：表示对象族识别、格式识别和推荐加工路径。
- `ExtractionJob`：表示 schema 抽取、OCR、ASR、转码、文本抽取、预览生成等前置任务。
- `PreviewArtifact`：表示样例、文本预览、图片预览、query preview 等对外展示工件。

**关系**

- `DataAsset 1 -> N RawIngestBatch`
- `RawIngestBatch 1 -> N RawObjectManifest`
- `RawObjectManifest 1 -> N FormatDetectionResult`
- `RawObjectManifest 1 -> N ExtractionJob`
- `DataAssetVersion 1 -> N PreviewArtifact`

### 4.2.4 ProductSKU

**主键**

- `sku_id`

**职责**

- 表示可售最小单元
- 一个 `Product` 下可以有多个 `SKU`
- 承担“标准 SKU 真值 + 商业套餐编码”的双层表达

**核心字段**

- `sku_id`
- `sku_type`
- `sku_code`
- `rights_profile_code`
- `allowed_rights`
- `delivery_mode`
- `pricing_mode`
- `trade_mode`

**字段解释**

- `sku_type`：标准 SKU 真值，`V1` 仅允许 `FILE_STD / FILE_SUB / SHARE_RO / API_SUB / API_PPU / QRY_LITE / SBX_STD / RPT_STD`
- `sku_code`：商品内的实现型商业套餐编码，例如月包、团队包、按次包
- `rights_profile_code`：由正式 `rights_type` 组合生成的派生标签，仅用于模板映射、展示和检索
- `allowed_rights`：正式权利集合，只能引用 `rights_type`
- `delivery_mode`：SKU 级交付路径事实源
- `pricing_mode`：SKU 级对外收费模式事实源
- `trade_mode`：数据库或执行层内部执行路径投影，可由 `sku_type + delivery_mode + pricing_mode` 派生，不构成外部业务主轴
- 页面、接口、模板绑定、账单和搜索若只识别 `sku_code` 而不识别 `sku_type`，视为未对齐主 PRD

**建模规则**

- `ProductSKU` 是下单、合同、授权、交付、验收、账单与结算事实源。
- 若 `Product` 与 `SKU` 在 `delivery_mode`、模板选择或使用姿态上存在差异，均以 `SKU` 快照为准。
- `usage_mode` 与 `allowed_rights` 不得混用：前者表达推荐使用姿态，后者表达正式权利边界。
- `pricing_mode` 是正式对外收费模式；计量、归集与结算仍由 `BillingEvent` 和计费模板驱动。

**SKU 分类**

- `V1`: 文件快照/版本订阅、只读共享、API 服务、模板查询 lite、查询沙箱、固定报告/结果产品
- `V2`: 模型 API、模型服务期、受控计算任务包、clean room 模板包、联邦项目包、MPC 结果包、ZKP 证明包

## 4.4 交易与订单聚合

### 4.4.1 Order

**主键**

- `order_id`

**职责**

- 表示一次完整交易实例

**核心字段**

- `order_id`
- `product_id`
- `sku_id`
- `buyer_org_id`
- `seller_org_id`
- `current_state`
- `status_version`
- `payment_status`
- `delivery_status`
- `acceptance_status`
- `settlement_status`
- `dispute_status`
- `invoice_status`
- `proof_commit_state`
- `external_fact_status`
- `reconcile_status`
- `order_amount`
- `price_snapshot`
- `buyer_deposit`
- `seller_deposit`
- `contract_id`
- `policy_id`
- `request_id`
- `trace_id`

**生命周期**

- `created -> buyer_locked -> seller_delivering -> delivered -> accepted / rejected -> settled / arbitrated -> closed`

**状态规则**

- `current_state` 是订单流程推进唯一主状态，也是默认展示主状态。
- `payment_status / delivery_status / acceptance_status / settlement_status / dispute_status / invoice_status` 是业务域子状态。
- `proof_commit_state / external_fact_status / reconcile_status` 是技术与审计镜像状态，不替代 `current_state`。
- 子状态变化若触发流程推进，必须由订单编排层同步推进 `current_state`。
- `status_version` 只用于并发控制和状态顺序校验，不表达业务阶段。

## 7. 生命周期总表

### 7.1 商品与目录对象

```text
DataAsset:
draft -> active -> archived

DataAssetVersion:
draft -> ready -> bound_to_product -> deprecated -> archived

Product:
draft -> pending_review -> listed -> suspended -> retired

ProductSKU:
draft -> active -> inactive -> archived
```

### 7.2 交易对象

```text

created -> buyer_locked -> seller_delivering -> delivered -> accepted -> settled -> closed
created -> buyer_locked -> seller_delivering -> delivered -> rejected -> dispute_opened
dispute_opened -> auto_decision | manual_review -> resolved -> settled -> closed

Delivery:
prepared -> committed -> available -> consumed | expired

BillingEvent:
pending -> recognized -> billed -> settled | reversed
```

```text

draft -> recruiting -> ready -> running -> aggregating -> rewarded -> completed
draft/recruiting/running -> aborted | failed

draft -> registered -> reviewed -> listed -> active -> suspended -> archived

pending -> calculating -> risk_checking -> payable -> settled
pending/calculating/risk_checking -> frozen -> manual_review -> canceled
```

这套领域模型的核心不是“把未来对象全做出来”，而是：

- 现在就把对象边界、主键、关系方向、生命周期和扩展点定对
- `V1` 只启用必要子集
- `V2/V3` 在原骨架上扩展，不推翻重建

## 15. 支付、资金流与轻结算聚合

### 15.1 PaymentProvider

**职责**

- 描述支付渠道能力、沙箱能力、回调能力、退款能力和打款能力。

### 15.1.1 JurisdictionProfile / CorridorPolicy / PayoutPreference

**职责**

- `JurisdictionProfile`：描述起步司法辖区及后续扩展司法辖区的支付/收款/数字资产能力边界。
- `CorridorPolicy`：描述付款地、收款地、可用币种、是否允许 FX、是否允许数字资产和是否需要人工复核。
- `PayoutPreference`：描述卖方默认收款币种、默认收款渠道和默认出金方式。

### 15.2 PaymentIntent

**职责**

- 表示一次待支付、待确认或待取消的支付意图，是订单和真实支付渠道之间的桥梁。
- 必须快照付款方司法辖区、收款方司法辖区、价格币种和命中的走廊策略。

### 15.3 PaymentTransaction

**职责**

- 记录真实支付、退款、打款、回调确认等渠道级交易事实。

### 15.4 FeeRule / FeeRuleVersion

**职责**

- 表示平台服务费、渠道手续费、保证金规则和增值服务费的版本化定义。

### 15.5 PayoutInstruction / ReconciliationStatement

**职责**

- `PayoutInstruction`：表示待执行或已执行的打款/分账指令。
- `ReconciliationStatement`：表示渠道账单导入与差异核对对象。

## 16. 身份认证、注册登录与会话管理聚合

### 16.1 Invitation

**职责**

- 表示组织成员邀请、邀请令牌、邀请状态和接受结果。

### 16.2 IdentityProof / AuthMethodBinding / MfaAuthenticator

**职责**

- `IdentityProof`：表达主体/成员实名与 assurance 等级。
- `AuthMethodBinding`：表达本地凭证、OIDC/SAML 外部身份、服务凭证等认证绑定。
- `MfaAuthenticator`：表达 WebAuthn、TOTP、恢复码等 MFA 因子。

### 16.3 TrustedDevice / UserSession / RefreshTokenFamily

**职责**

- `TrustedDevice`：表达设备信任状态、最近使用、撤销状态。
- `UserSession`：表达浏览器或 API 会话。
- `RefreshTokenFamily`：表达 refresh token 轮换与整族撤销。

### 16.4 SsoConnection / ExternalIdentityBinding / StepUpChallenge

**职责**

- `SsoConnection`：表达企业 OIDC/SAML 连接及声明映射。
- `ExternalIdentityBinding`：表达平台成员与外部身份的绑定。
- `StepUpChallenge`：表达高风险动作的二次认证过程。

### 16.5 FabricCaRegistry / FabricIdentityBinding / CertificateRecord / CertificateRevocationRecord

**职责**

- `FabricCaRegistry`：表达 Fabric CA、MSP、注册策略和配置。
- `FabricIdentityBinding`：表达平台主体/成员/服务身份与链上身份的绑定。
- `CertificateRecord`：表达证书、有效期、摘要、序列号和状态。
- `CertificateRevocationRecord`：表达吊销原因、吊销来源和吊销证据。

### 16.6 关键关系

- `Organization 1 -> N Invitation`
- `UserAccount 1 -> N AuthMethodBinding`
- `UserAccount 1 -> N MfaAuthenticator`
- `UserAccount 1 -> N TrustedDevice`
- `UserAccount 1 -> N UserSession`
- `TrustedDevice 1 -> N RefreshTokenFamily`
- `Organization 1 -> N SsoConnection`
- `SsoConnection 1 -> N ExternalIdentityBinding`
- `UserAccount / ServiceIdentity / Organization -> N FabricIdentityBinding`
- `FabricCaRegistry 1 -> N CertificateRecord`
- `CertificateRecord 1 -> 0..1 CertificateRevocationRecord`

## 17. 审计、证据链与回放聚合增量

### 17.1 EvidenceItem / EvidenceManifest / EvidenceManifestItem

**职责**

## 21. 敏感数据治理增量对象

### 21.1 `SensitiveHandlingPolicy`

- 表示某个资产版本或商品在敏感场景下的处理策略。
- 关键内容包括：分类分级、是否含 `PI/SPI`、允许交付路径、要求的执行边界、输出边界、撤权和销毁口径。

### 21.2 `LegalBasisEvidence`

- 表示处理依据证明、授权证明、委托处理证明或监管许可证明。
- 是敏感商品可上架、可执行、可导出的前提之一。

### 21.3 `SafePreviewArtifact`

- 表示敏感商品对外展示的安全预览工件。
- 可以是 schema 摘要、脱敏样例、合成样例、模板试算结果或统计摘要。

### 21.4 `SensitiveExecutionPolicy`

- 表示订单、查询面、模板授权或沙箱工作区在敏感场景下的执行策略快照。
- 用于决定是否允许执行、是否要求 step-up、是否要求证明以及导出边界。

### 21.5 `AttestationRecord`

- 表示受控执行回执、远程证明摘要、执行环境证明或卖方自持执行证明。
- `V1` 可记录最小执行回执，`V2` 起扩展到 `TEE / proof artifact`。

### 21.6 `ResultDisclosureReview`

- 表示结果导出前的披露审查。
- 约束“哪些字段、多少粒度、在什么审批条件下可以导出”。

### 21.7 `DestructionAttestation`

- 表示对象到期、撤权、争议或监管指令后形成的保留/销毁证明。

### 21.8 关系补充

- `DataAsset / AssetVersion` 1 对多关联 `LegalBasisEvidence`、`SafePreviewArtifact`
- `AssetVersion` 1 对 1 关联 `SensitiveHandlingPolicy`
- `Order / QuerySurface / TemplateGrant / SandboxWorkspace` 可关联 `SensitiveExecutionPolicy`
- `QueryExecutionRun / ReportArtifact / StorageObject` 可关联 `ResultDisclosureReview`
- `Order / ResultObject / StorageObject` 可关联 `DestructionAttestation`

## 22. 交易链监控、公平性与信任安全对象增量

### 22.1 `TradeLifecycleCheckpoint`

- 表达某个交易对象在完整生命周期中的检查点状态。
- 关键内容包括：检查点编码、预期时间、实际发生时间、来源类型、来源对象、状态、联查主键。

### 22.2 `ExternalFactReceipt`

- 表达来自支付渠道、打款渠道、执行环境、合作伙伴、监管协同或销毁/保留流程的外部事实回执。
- 是 `external_fact_status` 的正式来源对象，而不是备注字段。

### 22.3 `FairnessIncident`

- 表达交易公平性、可信执行和信任边界异常。
- 可以由超时、状态冲突、回执缺失、越权输出、跨链 ack 缺失等规则触发。

### 22.4 `MonitoringPolicyProfile`

- 表达交易链监控规则档案。
- 统一定义检查点阈值、升级条件、告警规则、自动动作和通知策略。

### 22.5 `ChainProjectionGap`

- 表达链上已确认但链下未投影、链下预期应存在但链上未确认、或跨平台镜像差异。
- 是一致性修复与链上链下联查的正式对象。

### 22.6 关键关系

- `trade.order_main 1 -> N ops.trade_lifecycle_checkpoint`
- `trade.order_main 1 -> N ops.external_fact_receipt`
- `trade.order_main 1 -> N risk.fairness_incident`
- `ops.monitoring_policy_profile 1 -> N ops.trade_lifecycle_checkpoint`
- `ops.outbox_event 1 -> N ops.chain_projection_gap`
- `chain.chain_anchor 1 -> N ops.chain_projection_gap`
- `ops.trade_lifecycle_checkpoint / ops.external_fact_receipt -> risk.fairness_incident` 通过来源关联形成“异常由何而来”的责任链
