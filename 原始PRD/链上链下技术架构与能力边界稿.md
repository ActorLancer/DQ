# 链上链下技术架构与能力边界稿

- 文档名称：链上链下技术架构与能力边界稿
- 适用范围：基于区块链技术的数据交易平台 V1 / V2 / V3
- 版本：D1
- 日期：2026-04-07
- 上位输入：PRD-R4《基于区块链技术的数据交易平台需求清单（需求基线增强整合版）》

---

## 1. 文档目标

本文件用于把 PRD 中“联盟链主链路 + 公链展示层 + 链下受控交付/计算”的原则，落成研发可继续拆分的技术架构边界说明，主要回答：

1. 哪些能力必须放在链上，哪些必须留在链下。
2. 联盟链、公链、预言机、连接器、沙箱、执行环境分别负责什么。
3. 不同产品类型的交易事件如何在链上链下协同。
4. V1 和 V2 的技术边界在哪里。
5. 发生故障、降级、回滚时，主链路如何保证不依赖公链继续运行。

---

## 2. 总体结论

本平台采用以下主架构：

> **联盟链主交易底座 + 链下受控存储/交付/计算 + Solana 公链展示与外部验证增强层**

其目标不是把原始数据放到链上，而是把：
- 身份
- 合同摘要
- 授权状态
- 交付回执
- 账单摘要
- 审计证据索引
- 公示锚点

做成可信、可追溯、可验证的记录体系。

---

## 3. 架构设计原则

1. 链上不存原始数据，只存状态、摘要、凭证、索引、签名。  
2. 联盟链是生产级可信证明确认层，公链不是核心依赖；业务运营主状态仍在链下数据库。  
3. 链下是原始数据、模型、日志、执行环境的主要承载层。  
4. 所有链下敏感行为都必须通过受控入口进入：下载令牌、API 网关、沙箱、V2 计算环境。  
5. 公链仅承担锚定、展示、凭证和外部可验证作用。  
6. 主链路必须支持公链关闭、同步失败、锚定延迟情况下继续运行。  
7. 任何外部系统（支付、签章、时间戳、KYC/KYB、TEE 证明）都必须通过可审计的集成层接入。

## 4. 分层架构

## 4.1 应用层 / 门户层
能力：
- 目录搜索
- 上架中心
- 询报价
- 订单与合同
- 账单与工单
- 公开验证页
- 凭证展示页
- 数据产品护照页

不承载：
- 原始数据处理
- 链上共识
- 深度规则计算

## 4.2 交易控制层 / 规则执行层
能力：
- 身份认证
- 权限判定
- 合规审查
- 风控规则
- 合同模板与数字合约
- 订单状态机
- 账单聚合
- 争议处理
- 公链披露策略

这是平台的“业务大脑”，负责链上链下协调。

## 4.3 执行层 / 数据与计算交付层
能力：
- 文件交付
- API 网关
- 沙箱查询
- 安全空间
- V2 计算环境
- 连接器网络
- 数据清洗、脱敏、版本、血缘

该层处理真实数据访问与执行，但不直接承担交易可信证明。

## 4.4 信任层 / 联盟链 / 存证层
能力：
- 成员身份
- 合同摘要
- 授权状态
- 订单关键状态
- 交付回执摘要
- 账单摘要
- 审计摘要
- 冻结/下架/审批结果
- 监管观察节点可见事件

这是生产可信底座。

## 4.5 公链展示层 / Solana 层
能力：
- 事件批次哈希锚定
- 公共时间戳
- 供应商认证凭证
- 不可转让交易凭证
- 数据产品护照 NFT
- 对外公开验证

不承载：
- 核心订单状态机
- 原始合同与隐私数据
- 主交易权限判定

## 4.6 外部服务与生态接口层
能力：
- KYC/KYB
- CA / 电子签章
- 第三方支付 / 发票
- 时间戳服务
- TEE 远程证明
- 安全检测
- KMS / HSM
- SIEM / SOC
- 公链同步服务 / 消息桥

| 能力 | 联盟链 | Solana 公链 | 链下系统 |

| 合同正文 | 否（仅摘要） | 否 | 是 |
| 合同哈希 | 是 | 可锚定批次 | 否 |
| Usage Policy 状态 | 是（摘要/状态） | 否 | 完整策略 |
| 原始数据 | 否 | 否 | 是 |
| 样例数据 | 否 | 否 | 是 |
| 文件交付 | 否（仅回执） | 否 | 是 |
| API 调用 | 否（仅摘要/账单事件） | 否 | API 网关 |
| 沙箱查询 | 否（仅回执/审计摘要） | 否 | 沙箱系统 |

| 供应商认证状态 | 是/链下联合 | 是（展示用） | 原始审核档案 |

## 6. 联盟链能力边界

联盟链在本平台承担以下生产能力：

1. 成员身份与组织准入
2. 合同摘要上链
3. 授权状态变更上链
4. 订单关键状态上链
5. 交付回执摘要上链
6. 账单摘要与结算结果摘要上链
7. 审计事件摘要上链
8. 冻结、下架、审批结论上链
9. 监管观察节点共享可信事件

联盟链不承担：
- 原始数据存储
- API 直接调用
- 沙箱执行
- 批量日志存储
- 模型文件存储
- 大规模实时报价撮合计算

---

## 7. Solana 公链能力边界

## 7.1 适合放到 Solana 的内容
- 联盟链事件批次哈希
- 供应商认证凭证
- 不可转让交易凭证
- 数据产品护照 NFT
- 公开验证页索引

## 7.2 不适合放到 Solana 的内容
- 原始订单状态机
- 原始数据
- 原始合同
- 实名身份
- 敏感日志
- 核心账单与结算明细
- 权限判定逻辑

## 7.3 V1 / V2 边界
- **V1**：默认关闭，仅演示级锚定到测试网/演示环境。
- **V2**：可生产开放公开验证页与凭证/NFT。
- **V3**：再考虑与生态伙伴的更深度展示互认。

## 13. 故障与降级策略

## 13.1 联盟链故障
- 暂停新交易进入“可信生效”阶段
- 允许前台只读浏览
- 缓存待上链事件
- 恢复后补写摘要

## 13.2 Solana 故障
- 不影响核心交易闭环
- 暂停锚定与公开展示
- 标记为 `anchor_pending`
- 恢复后批量补锚定

## 13.3 文件交付系统故障
- 暂停文件型新交付
- 已签约订单进入延迟交付状态
- 不得伪造验收完成

## 13.4 API 网关故障
- SLA 监测触发
- 自动告警
- 进入赔付工单流程

## 13.5 沙箱故障
- 暂停新开席位
- 存量用户提示维护
- 按 SLA 判断退款/赔付

## 15. 身份、密钥与证书边界

1. 登录身份不等于链上身份。  
2. 用户、应用、连接器、跨链服务必须使用不同身份。  
3. 下载令牌、API Key、服务签名密钥必须可轮换可吊销。  
4. KMS/HSM 或等价可审计方案必须承载核心密钥治理。  
5. 平台管理员不得直接复用连接器或公链签发服务密钥。

- 输出限制
- 临时凭证

## 16.3 公链展示层
- 披露策略过滤
- 不可写入原始隐私信息
- 凭证吊销同步
- 回执异常监控

## 16.4 跨链/预言机
- 重放保护
- 回执验签
- 重试与幂等
- 双通道日志审计

## 19. 结论

这套架构的核心不是“上不上链”，而是：

- 哪些信任要由链来保证
- 哪些数据必须留在链下受控环境
- 哪些展示可以通过公链增强
- 哪些能力在 V1 绝不能因为 KPI 需求而破坏主交易闭环

因此，平台的正确架构边界应当始终保持为：

> **联盟链负责可信证明与审计确认，链下负责业务运营主状态、真实数据与受控执行，Solana 负责公开可验证展示与轻量凭证。**

## 20. 支付、资金流与清结算架构边界补充

### 20.1 总原则

- 支付适配层必须独立，不得把支付宝/微信/PayPal SDK 写进订单服务。
- 平台维护业务主状态、账务镜像、清结算指令和审计证据，真实资金优先留在持牌渠道/银行侧。
- 起步司法辖区固定为新加坡；司法辖区配置、走廊策略和伙伴白名单属于链下关键控制面。
- 渠道回调、对账文件、打款回执都属于链下关键证据对象。

### 20.2 新增技术组件

- 支付编排服务
- 支付渠道适配器
- 内部账务镜像服务
- 收费规则引擎
- 对账与清结算服务
- Mock PaymentProvider

### 20.3 链上链下边界

- 支付明细、渠道回调原文、对账文件、退款/打款流水必须保留在链下。
- 链上只允许写入账单摘要、结算摘要、分润摘要和必要的审计摘要。
- 数字资产/交易所结算属于后续扩展，V1 不进入生产主流程。

## 21. 新增：身份认证与 Fabric 证书边界

- 平台 IAM 层负责注册、登录、会话、MFA、企业 SSO、设备治理与 step-up。
- Fabric CA / MSP 层负责链上成员注册、证书签发、续签、吊销和链上签名主体表达。
- 两层必须通过绑定关系打通，但不得把 Fabric 证书直接当作普通控制台登录主凭证。
- 浏览器门户建议采用服务端会话；开放 API 采用短期 access token + refresh token；服务到服务采用独立服务身份。
- 证书属性只放最小必要属性或不透明引用，实名原文保留在链下数据库与审计系统。

## 22. 新增：审计、证据链与回放架构边界

### 22.1 分层要求

- 业务服务负责生成结构化审计事件
- 证据服务负责登记原始证据对象与证据清单
- 锚定服务负责生成批次根并写入 Fabric
- 回放服务负责按订单、支付、任务、争议和监管视角重建时间线
- 监管导出服务负责按权限生成脱敏或原文导出包

### 22.2 Fabric 职责补充

- Fabric 只承载审计批次根、证据清单摘要、关键状态摘要和监管导出摘要
- Fabric 不承载原始日志、原始证据和全量调用明细

### 22.3 链下职责补充

链下必须保存：

- 结构化审计事件
- 原始证据对象引用
- 保留策略与 `legal_hold`
- 回放任务与回放结果
- 审计访问记录

### 22.4 一致性要求

- 审计事件、证据清单、锚定批次之间必须通过 `request_id / trace_id / audit_id / batch_id` 串联
- 审计批次写链失败不得阻塞主交易链路，但必须进入重试或人工介入队列
- 回放服务只能对幂等白名单动作开放补偿重放，默认仅支持 dry-run

## 23. 新增：双层权威模型与一致性边界

### 23.1 权威分层

- 链下数据库：业务运营权威
- Fabric：关键承诺与可信证明确认层
- 支付 / CA / 对象存储：外部事实来源权威
- Kafka：事件分发与解耦总线，不承担业务权威

### 23.2 统一设计要求

- 平台页面、接口和账务镜像默认按数据库状态驱动。
- Fabric 只确认关键摘要、关键回执和可共享证明，不承担高频业务状态投影主源。
- 外部事实对象必须保留来源回执，并以来源系统回执为事实依据。
- 同一对象不得同时把“数据库最终态”和“链事件投影态”都描述为唯一主状态。

### 23.3 一致性链路

```text
数据库事务
  -> 业务对象 + AuditEvent + outbox_event
  -> publisher worker
  -> Kafka
  -> Fabric writer / 通知 / 搜索 / 风控 / 对账消费者
  -> 回写 proof_commit_state / external_fact_status / reconcile_status
  -> 差异进入 dead letter / replay / reconciliation
```

### 23.4 V1 / V2 / V3

- `V1`：事务性 outbox + Kafka + dead letter + 基础回放/对账
- `V2`：更复杂的训练、证明、分润纳入统一模型
- `V3`：跨链、跨平台、监管协同进入统一权威矩阵

## 24. 新增：搜索读模型与索引同步边界

### 24.1 分层要求

- PostgreSQL：商品、卖方、标签、价格、质量、信誉等搜索投影权威源
- OpenSearch：商品搜索与卖方搜索读模型
- Redis：搜索结果与 facet 缓存

### 24.2 核心规则

- 搜索引擎不是新的业务真相源
- 搜索结果只负责召回、聚合与排序
- 最终详情展示、下单、支付、交付前必须回 PostgreSQL 校验当前状态

### 24.3 同步链路

```text
业务写库
  -> 搜索投影刷新
  -> outbox_event
  -> Kafka
  -> Search Indexer
  -> OpenSearch
  -> Redis 失效
```

### 24.4 环境边界

- `local / demo`：允许 PostgreSQL fallback 搜索
- `staging / production`：应启用 OpenSearch

## 25. 新增：推荐服务与发现编排边界

推荐模块正式定位为链下发现服务：

- PostgreSQL：推荐行为事件、画像快照、推荐结果与配置权威
- OpenSearch：商品/服务/卖方候选召回读模型
- Redis：推荐缓存与已看去重状态
- Kafka：行为流、候选更新流、推荐结果回流总线
- recommendation-service：推荐位编排、多路召回、过滤与重排

边界规则：

- 不把推荐逻辑写入 chaincode
- 不把 OpenSearch / Redis / Kafka 当成推荐主状态机
- 推荐结果最终返回前必须回 PostgreSQL 进行可见性、资格、风控与状态校验
- 推荐与搜索共享 `search.*` 投影，但使用独立的推荐配置与结果存储

统一链路：

```text
业务主数据 / 行为事件
  -> PostgreSQL
  -> outbox_event
  -> Kafka
  -> Search Indexer / Feature Aggregator
  -> OpenSearch / recommend.*
  -> recommendation-service
  -> PostgreSQL 最终校验
  -> 推荐结果与审计
```

## 26. 新增：日志与可观测性边界

平台可观测性正式采用：

- `OpenTelemetry`
- `Prometheus`
- `Alertmanager`
- `Grafana`
- `Loki`
- `Tempo`

边界如下：

- `audit.*`：证据与审计主链
- `ops.system_log`：关键结构化系统日志镜像
- `Loki`：原始运行日志后端
- `Tempo`：trace 后端
- `Prometheus / Alertmanager`：指标与告警

明确要求：

- 不将 `ELK / Elasticsearch / OpenSearch` 作为 `V1` 日志主后端硬依赖
- OpenSearch 继续仅服务搜索/召回读模型
- PostgreSQL 只保存观测配置、trace 索引、告警事件、事件工单和关键日志镜像
- 任何运行态联查最终都必须能通过 `request_id / trace_id / object_id` 回到业务对象和审计对象
