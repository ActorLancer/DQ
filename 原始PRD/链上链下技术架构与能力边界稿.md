# 链上链下技术架构与能力边界稿

- 文档名称：链上链下技术架构与能力边界稿
- 适用范围：基于区块链技术的数据交易平台 V1 / V2 / V3
- 版本：D1
- 日期：2026-04-07
- 上位输入：PRD-R4《基于区块链技术的数据交易平台需求清单（需求基线增强整合版）》

---

## 1. 文档目标

本文件用于把 PRD 中“联盟链主链路 + 公链展示层 + 链下受控交付/计算”的原则，落成研发可继续拆分的技术架构边界说明，主要回答：

1. 哪些能力必须放在链上，哪些必须留在链下。
2. 联盟链、公链、预言机、连接器、沙箱、执行环境分别负责什么。
3. 不同产品类型的交易事件如何在链上链下协同。
4. V1 和 V2 的技术边界在哪里。
5. 发生故障、降级、回滚时，主链路如何保证不依赖公链继续运行。

---

## 2. 总体结论

本平台采用以下主架构：

> **联盟链主交易底座 + 链下受控存储/交付/计算 + Solana 公链展示与外部验证增强层**

其目标不是把原始数据放到链上，而是把：
- 身份
- 合同摘要
- 授权状态
- 交付回执
- 账单摘要
- 审计证据索引
- 公示锚点

做成可信、可追溯、可验证的记录体系。

---

## 3. 架构设计原则

1. 链上不存原始数据，只存状态、摘要、凭证、索引、签名。  
2. 联盟链是生产主链路，公链不是核心依赖。  
3. 链下是原始数据、模型、日志、执行环境的主要承载层。  
4. 所有链下敏感行为都必须通过受控入口进入：下载令牌、API 网关、沙箱、V2 计算环境。  
5. 公链仅承担锚定、展示、凭证和外部可验证作用。  
6. 主链路必须支持公链关闭、同步失败、锚定延迟情况下继续运行。  
7. 任何外部系统（支付、签章、时间戳、KYC/KYB、TEE 证明）都必须通过可审计的集成层接入。  

---

## 4. 分层架构

## 4.1 应用层 / 门户层
能力：
- 目录搜索
- 上架中心
- 询报价
- 订单与合同
- 账单与工单
- 公开验证页
- 凭证展示页
- 数据产品护照页

不承载：
- 原始数据处理
- 链上共识
- 深度规则计算

## 4.2 交易控制层 / 规则执行层
能力：
- 身份认证
- 权限判定
- 合规审查
- 风控规则
- 合同模板与数字合约
- 订单状态机
- 账单聚合
- 争议处理
- 公链披露策略

这是平台的“业务大脑”，负责链上链下协调。

## 4.3 执行层 / 数据与计算交付层
能力：
- 文件交付
- API 网关
- 沙箱查询
- 安全空间
- V2 计算环境
- 连接器网络
- 数据清洗、脱敏、版本、血缘

该层处理真实数据访问与执行，但不直接承担交易可信证明。

## 4.4 信任层 / 联盟链 / 存证层
能力：
- 成员身份
- 合同摘要
- 授权状态
- 订单关键状态
- 交付回执摘要
- 账单摘要
- 审计摘要
- 冻结/下架/审批结果
- 监管观察节点可见事件

这是生产可信底座。

## 4.5 公链展示层 / Solana 层
能力：
- 事件批次哈希锚定
- 公共时间戳
- 供应商认证凭证
- 不可转让交易凭证
- 数据产品护照 NFT
- 对外公开验证

不承载：
- 核心订单状态机
- 原始合同与隐私数据
- 主交易权限判定

## 4.6 外部服务与生态接口层
能力：
- KYC/KYB
- CA / 电子签章
- 第三方支付 / 发票
- 时间戳服务
- TEE 远程证明
- 安全检测
- KMS / HSM
- SIEM / SOC
- 公链同步服务 / 消息桥

## 5. 链上链下能力边界总表

| 能力 | 联盟链 | Solana 公链 | 链下系统 |
|---|---|---|---|
| 主体 DID / 凭证摘要 | 是 | 否 | 原始主体资料 |
| 合同正文 | 否（仅摘要） | 否 | 是 |
| 合同哈希 | 是 | 可锚定批次 | 否 |
| Usage Policy 状态 | 是（摘要/状态） | 否 | 完整策略 |
| 原始数据 | 否 | 否 | 是 |
| 样例数据 | 否 | 否 | 是 |
| 文件交付 | 否（仅回执） | 否 | 是 |
| API 调用 | 否（仅摘要/账单事件） | 否 | API 网关 |
| 沙箱查询 | 否（仅回执/审计摘要） | 否 | 沙箱系统 |
| 账单明细 | 否（摘要） | 否 | 账单系统 |
| 审计日志原文 | 否（摘要/索引） | 否 | 审计存储 |
| 冻结/解冻状态 | 是 | 可选锚定 | 控制系统执行 |
| 凭证/NFT | 否 | 是 | 元数据与吊销记录 |
| 供应商认证状态 | 是/链下联合 | 是（展示用） | 原始审核档案 |

## 6. 联盟链能力边界

联盟链在本平台承担以下生产能力：

1. 成员身份与组织准入
2. 合同摘要上链
3. 授权状态变更上链
4. 订单关键状态上链
5. 交付回执摘要上链
6. 账单摘要与结算结果摘要上链
7. 审计事件摘要上链
8. 冻结、下架、审批结论上链
9. 监管观察节点共享可信事件

联盟链不承担：
- 原始数据存储
- API 直接调用
- 沙箱执行
- 批量日志存储
- 模型文件存储
- 大规模实时报价撮合计算

---

## 7. Solana 公链能力边界

## 7.1 适合放到 Solana 的内容
- 联盟链事件批次哈希
- 供应商认证凭证
- 不可转让交易凭证
- 数据产品护照 NFT
- 公开验证页索引

## 7.2 不适合放到 Solana 的内容
- 原始订单状态机
- 原始数据
- 原始合同
- 实名身份
- 敏感日志
- 核心账单与结算明细
- 权限判定逻辑

## 7.3 V1 / V2 边界
- **V1**：默认关闭，仅演示级锚定到测试网/演示环境。
- **V2**：可生产开放公开验证页与凭证/NFT。
- **V3**：再考虑与生态伙伴的更深度展示互认。

---

## 8. 链下能力边界

链下系统必须承载：

1. 原始数据存储
2. 样例数据
3. 文件包生成与下载
4. API 服务
5. 沙箱查询
6. 质量检测
7. 脱敏与清洗
8. 审计原始日志
9. 账单明细与结算明细
10. 证据原件
11. V2 受控计算环境
12. 模型文件与算法容器

链下系统必须满足：
- 加密存储
- 分级隔离
- 水印/指纹
- 可冻结/可归档
- 可追溯版本
- 支持密钥治理

## 10. 事件流设计

## 10.1 文件型交易事件流
1. 下单
2. 合同签署
3. 支付锁定
4. 文件包生成
5. 下载令牌签发
6. 交付回执摘要上链
7. 买方下载
8. 验收成功
9. 账单生成
10. 结算摘要上链
11. 批次锚定到 Solana（可选）

## 10.2 API 订阅事件流
1. 下单与签约
2. Application 绑定
3. Key 签发
4. 首次成功调用
5. 账单事件持续归集
6. 周期账单
7. 结算摘要上链
8. 周期性批次锚定（可选）

## 10.3 沙箱事件流
1. 下单与签约
2. 席位开通
3. 样例查询通过
4. 查询/导出事件写入审计系统
5. 周期账单
6. 风控事件处置
7. 结算摘要上链

## 10.4 公链锚定事件流
1. 联盟链批次事件生成
2. 计算 Merkle Root / Hash Root
3. 生成 Public Anchor Batch
4. 提交到 Solana
5. 回执采集
6. 更新公开验证页
7. 如失败则重试或降级关闭展示

## 13. 故障与降级策略

## 13.1 联盟链故障
- 暂停新交易进入“可信生效”阶段
- 允许前台只读浏览
- 缓存待上链事件
- 恢复后补写摘要

## 13.2 Solana 故障
- 不影响核心交易闭环
- 暂停锚定与公开展示
- 标记为 `anchor_pending`
- 恢复后批量补锚定

## 13.3 文件交付系统故障
- 暂停文件型新交付
- 已签约订单进入延迟交付状态
- 不得伪造验收完成

## 13.4 API 网关故障
- SLA 监测触发
- 自动告警
- 进入赔付工单流程

## 13.5 沙箱故障
- 暂停新开席位
- 存量用户提示维护
- 按 SLA 判断退款/赔付

## 15. 身份、密钥与证书边界

1. 登录身份不等于链上身份。  
2. 用户、应用、连接器、跨链服务必须使用不同身份。  
3. 下载令牌、API Key、服务签名密钥必须可轮换可吊销。  
4. KMS/HSM 或等价可审计方案必须承载核心密钥治理。  
5. 平台管理员不得直接复用连接器或公链签发服务密钥。

## 16. 安全边界

## 16.1 平台侧
- 审计日志与业务日志隔离
- 高风险动作双人复核
- 管理员操作强审计

## 16.2 链下执行环境
- 网络隔离
- 输出限制
- 临时凭证
- 细粒度访问控制

## 16.3 公链展示层
- 披露策略过滤
- 不可写入原始隐私信息
- 凭证吊销同步
- 回执异常监控

## 16.4 跨链/预言机
- 重放保护
- 回执验签
- 重试与幂等
- 双通道日志审计

## 17. 非功能要求（架构相关）

## 17.1 可用性
- 核心交易控制层目标可用性：99.9%
- API 网关目标可用性：99.9%
- 公链展示服务可低于核心服务，但不得影响主交易链路

## 17.2 可观测性
- 链上提交成功率
- 锚定成功率
- 账单事件采集成功率
- 交付成功率
- SLA 违约率
- 风控命中率
- 凭证吊销同步成功率

## 17.3 可扩展性
- V1 支持文件/API/沙箱三类产品
- 通过统一事件模型预留 V2 计算型产品
- 通过 Connector / Environment 抽象预留多环境接入

## 19. 结论

这套架构的核心不是“上不上链”，而是：

- 哪些信任要由链来保证
- 哪些数据必须留在链下受控环境
- 哪些展示可以通过公链增强
- 哪些能力在 V1 绝不能因为 KPI 需求而破坏主交易闭环

因此，平台的正确架构边界应当始终保持为：

> **联盟链负责可信状态与审计，链下负责真实数据与受控执行，Solana 负责公开可验证展示与轻量凭证。**

## 20. 支付、资金流与清结算架构边界补充

### 20.1 总原则

- 支付适配层必须独立，不得把支付宝/微信/PayPal SDK 写进订单服务。
- 平台维护业务主状态、账务镜像、清结算指令和审计证据，真实资金优先留在持牌渠道/银行侧。
- 渠道回调、对账文件、打款回执都属于链下关键证据对象。

### 20.2 新增技术组件

- 支付编排服务
- 支付渠道适配器
- 内部账务镜像服务
- 收费规则引擎
- 对账与清结算服务
- Mock PaymentProvider

### 20.3 链上链下边界

- 支付明细、渠道回调原文、对账文件、退款/打款流水必须保留在链下。
- 链上只允许写入账单摘要、结算摘要、分润摘要和必要的审计摘要。
- 数字资产/交易所结算属于后续扩展，V1 不进入生产主流程。
