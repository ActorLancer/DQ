# 数据商品存储与分层存储设计

- 文档定位：原始 PRD 增补专题稿
- 目标：明确平台对数据商品本体、对象存储、查询/执行面与 Fabric 存证面的分层存储闭环
- 适用阶段：`V1 / V2 / V3`

---

## 1. 设计目标

本文件回答四类问题：

1. 数据商品本体究竟存数据库、对象存储还是查询层。
2. 平台托管、自持交付、API 服务、模板查询、受控计算分别如何走存储闭环。
3. 平台如何用 PostgreSQL、MinIO/S3 兼容对象存储、查询/执行环境、Fabric 联盟链完成职责分层。
4. 从原始接入到交付、验收、保留、销毁，整条存储生命周期如何避免逻辑断裂。

---

## 2. 核心结论

平台必须采用四层分工模型：

1. **控制面：PostgreSQL**
2. **数据面：MinIO / S3 兼容对象存储**
3. **查询/执行面：受控查询与执行环境**
4. **信任面：Fabric 联盟链**

这四层缺一不可，但职责必须严格分离。

平台不应把“有元数据”理解成“持有数据本体”，也不应把“对象在 MinIO”理解成“数据库中不需要治理关系”。

---

## 3. 四层架构职责

### 3.1 控制面：PostgreSQL

控制面存放可查询、可联查、可审计、可驱动业务状态机的对象：

- 组织、用户、角色、应用、证书绑定
- 商品目录、SKU、合同、数据契约、订单、账单、争议
- schema 摘要、字段结构、质量报告、标签、版本
- 原始接入批次、对象清单、格式识别、加工任务、预览工件
- 存储命名空间、存储策略、对象绑定、交付记录、生命周期状态
- API 凭证、模板查询授权、任务状态、审计事件
- Fabric 交易号、对象哈希、链下 URI、对账与一致性状态

控制面的原则是：

- 只存治理所需信息，不存大规模原始数据本体
- 必须支持跨对象联查、筛选、回放、审计和权限控制
- 必须作为链下业务权威源

### 3.2 数据面：MinIO / S3 兼容对象存储

数据面负责保存大对象和二进制制品：

- 原始文件
- 加工后产品包
- 加密包
- 预览文件
- 文档抽取物、OCR/ASR 中间结果、转码结果
- 图片、音频、视频
- 模型文件、Notebook、Manifest、签名包
- 证据原文、导出文件、归档文件

默认建议：

- `V1` 起步使用 `MinIO / S3 兼容抽象`
- 产品设计绑定 `S3-compatible object storage` 能力，不把实现写死在单一厂商
- 本地开发、测试、演示优先用 `MinIO`

对象存储的原则是：

- 承载本体，不承载核心业务关系
- 支持 bucket / prefix / namespace / lifecycle / retention / object lock
- 对数据库暴露 URI、hash、大小、加密信息和保留策略

### 3.3 查询/执行面：受控查询与执行环境

查询/执行面用于承接“不是下载文件，而是查询/计算”的商品：

- 表型共享
- 模板查询
- query sandbox
- clean room lite / 完整 clean room
- 受控统计
- 受控计算
- 模型推理与结果产品

其职责不是替代控制面和对象存储，而是：

- 对 `curated / product` 区对象提供受控读取
- 对高敏数据提供不出域访问
- 把查询、执行、导出边界作为正式能力表达

`V1` 要求：

- 至少支持最小查询/执行抽象，不强制一次落完整 lakehouse
- 可以通过既有 `ExecutionEnvironment + TemplateQueryGrant + Sandbox` 落最小闭环

`V2/V3` 再扩展：

- 完整查询引擎
- 只读共享
- 完整 clean room
- 更复杂的湖仓/数仓协同

### 3.4 信任面：Fabric 联盟链

Fabric 不保存大数据本体，只保存可信承诺与可验证摘要：

- 数据产品登记摘要
- 对象 hash / manifest hash / 批次根
- revision 与版本摘要
- 授权与订单关键状态
- 验收收据
- 结算结果摘要
- 审计批次根与证据清单摘要

Fabric 的原则是：

- 只存摘要，不存原始大对象
- 只承担可信证明确认，不承担主查询与主存储
- 与数据库、对象存储回执和审计事件共同构成可追责闭环

---

## 4. 分层存储区设计

平台必须显式采用分区/分层概念，而不是所有对象混放一个 bucket：

1. `raw`：原始接入区，不直接对外售卖
2. `staging`：处理中间区，供标准化、脱敏、转码、抽取使用
3. `curated`：治理后区，形成可控 schema、质量报告和责任链
4. `product`：正式对外商品交付区
5. `preview`：样例、预览、缩略和摘要区
6. `delivery`：订单交付临时对象区
7. `temp`：短期中间结果区
8. `archive`：归档区
9. `evidence`：证据原文与保全对象区
10. `model`：模型、Notebook、签名包和运行依赖区

其中：

- `raw` 与 `staging` 不得默认对外开放
- `product` 只暴露经过契约、质量、合规和交付策略确认后的对象
- `preview` 必须遵守样例遮罩与最小披露策略
- `delivery` 应带有效期与回收规则
- `evidence` 与 `archive` 应优先启用保留/不可变策略

---

## 5. 闭环逻辑

推荐闭环如下：

```text
原始对象进入 raw 区
  -> 数据库登记 RawIngestBatch / RawObjectManifest
  -> 类内标准化与加工写入 staging / curated
  -> 形成质量报告、字段结构、责任链、数据契约
  -> 合格对象进入 product 区
  -> 样例与预览进入 preview 区
  -> 下载/交付临时对象进入 delivery 区
  -> 对象摘要、订单摘要、验收摘要进入 Fabric
  -> 生命周期结束后进入 archive / destroy
```

这条链路要求：

- 商品目录只引用 `product` 区正式对象
- 买方预验收优先读 `preview` 区和数据库摘要
- 查询/执行面优先读取 `curated / product` 区，不直接绕过治理访问 `raw`
- Fabric 只锚定对象摘要、状态摘要和证据批次根

## 6. 各类商品的存储落点

### 6.1 文件型商品

- 原始文件：`raw`
- 清洗/脱敏结果：`curated`
- 可售交付包：`product`
- 样例/缩略：`preview`
- 一次性交付临时包：`delivery`

### 6.2 表/库/视图型商品

- 大对象本体可仍位于对象存储或卖方数据库
- 数据库保存 schema、对象引用、查询入口和授权信息
- 查询/执行面负责只读共享或模板查询

### 6.3 API / 服务型商品

- 原始底层数据可在卖方侧、平台对象存储或平台代理层
- PostgreSQL 保存 API 入口、配额、策略、调用日志和对账信息
- 对象存储只保存必要样例、文档、OpenAPI 和离线缓存对象

### 6.4 模板查询 / 受控计算商品

- 原始输入可不进入平台
- 查询/执行面读取 `curated` 区或卖方受控环境
- PostgreSQL 保存模板、边界、任务和证明
- Fabric 保存任务与结果摘要

### 6.5 模型/结果型商品

- 模型工件、依赖、签名包：`model`
- 结果集文件：`product` 或 `delivery`
- 模型卡、评估摘要、预览结果：`preview`

---

## 7. V1 / V2 / V3 边界

### 7.1 V1

- PostgreSQL 作为控制面正式落地
- `MinIO / S3 兼容对象存储` 作为默认数据面正式落地
- `raw / curated / product / preview / delivery / evidence / archive` 分层表达正式落地
- 查询/执行面只要求最小抽象，不强制绑定单一大数据产品
- Fabric 只保存摘要、状态与证据批次根

### 7.2 V2

- 引入更正式的查询/执行层
- 支持模型商品、受控计算、完整 clean room 与更多非下载交付模式
- 对象分层进一步扩展到 `model / result / training` 等产物

### 7.3 V3

- 支持跨平台对象存储能力声明与互认
- 支持第三方可信存储和外部对象仓互联
- 支持外部 product 区、外部 preview 区和外部 archive 承诺映射

---

## 8. 关键约束

1. 任何对象一旦成为可售正式交付对象，必须能追溯到来源、版本、质量报告、责任链和契约。
2. PostgreSQL 不得承担大规模原始文件主存储职责。
3. Fabric 不得承担原始数据本体存储职责。
4. `raw` 区对象不得绕过标准化与审核直接对外售卖。
5. `preview` 区不得泄露超出契约许可的原始明文。
6. `delivery` 区对象必须具备过期与回收机制。

---

## 9. 结论

数据商品存储的正确实现方式，不是“把所有数据放数据库”或“把所有文件扔对象存储”，而是：

**数据库管治理、对象存储管本体、查询层管受控访问、Fabric 管可信证明。**

在这个前提下，再叠加前面已经确定的原始接入、产品化加工、支付、审计和双层权威模型，整套平台逻辑才能真正闭环。
