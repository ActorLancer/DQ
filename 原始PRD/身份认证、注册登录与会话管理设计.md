# 身份认证、注册登录与会话管理设计

## 1. 目标

本设计用于补齐平台在以下方面的正式需求：

- 主体注册与成员注册
- 平台登录与企业单点登录
- 多因子认证与高风险二次认证
- 会话、刷新令牌、设备信任与登出治理
- Fabric 联盟链成员身份、证书绑定、吊销与审计
- 平台 IAM 身份、链上身份、服务身份之间的边界

本文件不是技术实现细节稿，而是产品、架构、权限、数据库和后续接口协议统一遵循的身份治理基线。

## 2. 设计原则

### 2.1 平台登录身份与链上身份分层

平台必须严格区分以下身份层：

- 平台主体身份：企业、机构、监管方、合作伙伴等业务主体
- 平台成员身份：登录控制台、审批、签约、验收、治理的自然人账号
- 平台服务身份：应用、连接器、执行环境、微服务、网关等机器身份
- Fabric 链上身份：用于链上成员准入、链上签名、链上节点治理、链上凭证签发的联盟链身份

规则：

- 平台登录身份不等于 Fabric 链上身份
- Fabric 证书不直接作为普通控制台登录主凭证
- 不是所有用户都必须签发 Fabric 身份
- Fabric 身份必须绑定到已实名、已授权、已审计的平台主体或成员

### 2.2 登录协议与会话策略分层

推荐基线：

- 浏览器门户：`OIDC / OAuth 2.0 Authorization Code + PKCE + 服务端会话 + HttpOnly Cookie`
- 开放 API / SDK：短期 `Access Token` + 可轮换 `Refresh Token`
- 企业单点登录：优先 `OIDC`，兼容 `SAML 2.0`
- 服务到服务：`Client Credentials` 或等价机器身份机制

### 2.3 高风险动作必须再认证

以下动作必须触发 `step-up auth`：

- 合同确认与高风险授权生效
- 退款、赔付、人工打款、保证金扣罚
- 权限扩张、角色变更、SSO 连接更新
- Fabric 身份签发、续签、吊销
- 高敏数据导出、训练用途放行、跨域互联放行

### 2.4 证书最小披露

Fabric 证书属性中仅允许放最小必要的身份属性或不透明引用，不得直接写入：

- 身份证号原文
- 护照号原文
- 营业执照号原文
- 银行账户原文

这类信息必须留在链下受控数据库和审计体系中。

## 3. 注册设计

### 3.1 注册模式

平台至少支持以下注册入口：

1. 企业/机构主体注册
2. 组织成员邀请注册
3. 企业 SSO 首次登录自动建档
4. 开发者/测试主体注册

### 3.2 主体注册流程

```text
组织发起注册
  -> 填写主体信息
  -> 校验邮箱/手机
  -> 选择登录方式
  -> 绑定 MFA
  -> 提交 KYC/KYB / 实名材料
  -> 主体审核
  -> 审核通过后激活组织
  -> 才允许上架、交易、支付、申请链上身份
```

主体注册必须与平台既有准入、黑白灰名单、合规审查联动。

### 3.3 成员注册流程

```text
组织管理员发起邀请
  -> 生成邀请令牌
  -> 成员接受邀请
  -> 设置登录标识与首选认证方式
  -> 绑定 MFA
  -> 授予角色
  -> 如需要链上身份，再进入 Fabric 身份申请
```

### 3.4 企业 SSO 首次登录

规则：

- 企业先配置 OIDC/SAML 连接
- 首次企业 SSO 登录可触发 JIT 建档
- JIT 仅创建“待激活成员”，不自动授予高权角色
- 高权限角色必须经组织管理员确认

### 3.5 社交登录边界

生产交易主体不应依赖社交登录作为正式交易身份。  
社交登录仅可用于开发者门户、测试环境或公共内容门户，不得直接绑定财务、合规、证书治理职责。

## 4. 登录与认证设计

### 4.1 本地登录

本地登录至少支持：

- 用户名/邮箱/手机号 + 密码
- WebAuthn / Passkey
- TOTP 二次认证
- 恢复码

密码不是唯一推荐方式，但必须提供可替代的强认证路径。

### 4.2 企业 SSO

`V1` 正式支持：

- 企业 `OIDC` SSO

`V2` 增量支持：

- `SAML 2.0`
- `SCIM` 用户同步

### 4.3 MFA 策略

优先级建议：

1. `Passkey / WebAuthn`
2. `TOTP`
3. 备份恢复码
4. 短信/邮件 OTP 仅作为恢复或低信任兜底

以下角色 `V1` 起强制 MFA：

- 平台超级管理员
- 主体审核员 / 合规审核员 / 风控专员
- 财务结算专员 / 支付渠道管理员
- Tenant 管理员 / 财务经理 / 安全管理员
- Fabric CA 管理相关角色

### 4.4 二维码授权

二维码授权可以作为：

- 桌面登录时的移动端确认
- 高风险操作再确认
- 会话迁移或设备换绑

但不应替代主账号体系或企业 SSO。

## 5. 会话与设备治理

### 5.1 浏览器会话

浏览器登录建议使用：

- 服务端会话
- `HttpOnly + Secure + SameSite` Cookie
- 登录成功后轮换 `session_id`
- 高风险提权后再次轮换 `session_id`

### 5.2 Token 体系

API / SDK 推荐：

- `access token`：5-15 分钟
- `refresh token`：7-30 天
- refresh token 必须支持轮换、复用检测、整族吊销

### 5.3 设备信任

平台必须支持：

- 设备登记
- 设备命名
- 最近登录记录
- 风险设备标记
- 设备撤销
- 新设备、新国家/地区、新 IP 段登录告警

### 5.4 超时与并发策略

建议基线：

- 普通会话 idle timeout：30 分钟
- 普通会话 absolute timeout：12 小时
- 提权会话：10-15 分钟
- 默认允许多设备，但必须支持“踢出其他设备”

### 5.5 会话治理动作

平台必须支持：

- 主动登出
- 撤销单会话
- 撤销设备
- 整族 refresh token 作废
- 风险事件命中后强制失效会话
- 证书吊销后联动阻断相关会话和服务身份

## 6. Fabric 联盟链身份与证书治理

### 6.1 Fabric 身份适用对象

Fabric 身份只发给真正需要链上成员能力的主体：

- 平台服务身份
- 连接器 / 执行环境 / 网关等机器身份
- 企业链上管理员
- 少量具备链上治理职责的人工成员

不应默认给全体终端用户签发链上证书。

### 6.2 Fabric 身份签发流程

```text
成员已完成主体准入和 MFA
  -> 提交 Fabric 身份申请
  -> 校验角色、组织、MSP、affiliation、attrs
  -> 审批通过
  -> 调用 Fabric CA register
  -> 生成密钥或登记密钥托管位置
  -> 调用 enroll / reenroll
  -> 记录证书摘要、序列号、有效期、绑定关系
  -> 激活链上身份
```

### 6.3 Fabric 证书生命周期

必须支持：

- 注册 `register`
- 签发 `enroll`
- 续签 `reenroll`
- 吊销 `revoke`
- 禁用
- 归档

吊销触发场景至少包括：

- 用户离职
- 角色撤销
- 组织退出或冻结
- 密钥疑似泄露
- 设备丢失
- 风险事件命中

### 6.4 Fabric 证书与平台对象绑定

绑定关系必须能回溯到：

- 哪个主体
- 哪个成员或服务身份
- 哪个 MSP / affiliation
- 哪套属性模板
- 哪个审批记录
- 哪次签发 / 续签 / 吊销事件

## 7. 领域对象与数据库预埋要求

平台必须正式建模至少以下对象：

- `Invitation`
- `IdentityProof`
- `AuthMethodBinding`
- `MfaAuthenticator`
- `TrustedDevice`
- `UserSession`
- `RefreshTokenFamily`
- `SsoConnection`
- `ExternalIdentityBinding`
- `StepUpChallenge`
- `FabricCaRegistry`
- `FabricIdentityBinding`
- `CertificateRecord`
- `CertificateRevocationRecord`

这些对象属于平台长期骨架，`V1` 即使未完整启用全部企业联邦能力，也必须预留。

## 8. 分阶段边界

### 8.1 V1

- 本地账号注册/登录
- 组织邀请注册
- 企业 OIDC SSO
- MFA
- 会话与设备管理
- 高风险 step-up
- Fabric 身份绑定模型
- Fabric 证书签发/吊销
- 身份与证书审计

### 8.2 V2

- `SAML 2.0`
- `SCIM`
- Passkey-first 策略增强
- 更强的企业身份映射与自动建档
- 服务身份与执行环境身份增强

### 8.3 V3

- 跨平台身份联邦
- 自适应风险认证
- 更强的设备信任与硬件认证
- 跨域合作伙伴身份互认与治理

## 9. 与整套交易逻辑的闭环

身份体系加入后，完整闭环变为：

```text
主体注册/KYC-KYB
  -> 成员邀请/SSO 建档
  -> MFA 与设备绑定
  -> 登录建立会话
  -> 角色与作用域授权
  -> 交易/交付/支付/审计操作
  -> 高风险动作触发 step-up
  -> 必要时调用 Fabric 身份完成链上签名
  -> 会话、证书、令牌、操作证据全量留痕
```

这条链路与现有：

- 合规审查
- 风控冻结
- 支付与清结算
- 链上存证
- 数据本体信任边界

已经可以形成闭环。

## 10. 结论

平台必须把“注册登录”和“联盟链身份”视为两套绑定但不混用的能力。  
真正的目标不是让所有用户都去直接使用链上证书登录，而是用平台 IAM 管理人和会话，用 Fabric 身份管理链上成员资格、链上签名与证书责任链。

## 11. 与双层权威模型的一致性补充

- 平台会话、MFA、设备信任、SSO 建档与 step-up 结果，以平台 IAM 数据库状态为权威。
- Fabric 证书签发、续签、吊销是否成立，以 Fabric CA / MSP 回执为来源权威。
- Fabric 证书摘要、吊销摘要与高 assurance 身份事件再进入 Fabric 审计/存证链。
- Kafka 可用于身份联邦、审计、风控、通知等异步联动，但不能替代 IAM 主状态机。
