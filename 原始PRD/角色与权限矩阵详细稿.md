# 角色与权限矩阵详细稿

- 文档名称：角色与权限矩阵详细稿
- 适用范围：基于区块链技术的数据交易平台 V1 / V2
- 版本：D1
- 日期：2026-04-07
- 上位输入：PRD-R4《基于区块链技术的数据交易平台需求清单（需求基线增强整合版）》

---

## 1. 文档目标

本文件用于把 PRD 中的“角色分类、权限原则、主体对象模型”落成可执行的权限设计基线，主要回答以下问题：

1. 平台内到底有哪些角色与机器身份。
2. 不同角色分别能操作哪些对象。
3. 权限如何在 Tenant / Department / User / Application / Connector / Environment 五层生效。
4. 哪些动作必须职责分离，不能由单一角色闭环完成。
5. V1 最小可实现的权限模型应该做到什么程度。

---

## 2. 适用原则

1. 采用 **RBAC + ABAC + PBAC + Scope** 混合模型。  
2. 权限授予对象不仅包括人，也包括应用、连接器、执行环境与监管观察身份。  
3. 权限判断必须同时考虑：主体身份、产品类型、数据等级、合同状态、环境策略、风控命中情况。  
4. 默认遵循最小权限、默认拒绝、按需放权、到期自动失效。  
5. 高风险动作必须双重控制：角色权限 + 审批/策略放行。

## 3. 权限对象模型

## 3.1 核心对象

- **Tenant / Organization**：组织工作空间，承载业务边界。
- **Party**：法律主体，负责签约、付款、审计、责任承担。
- **Department**：部门与成本中心。
- **User**：人工操作者。
- **Application**：系统身份，用于 API 调用、自动化任务、第三方集成。
- **Connector**：连接外部数据源、沙箱、执行环境的接入点。
- **Execution Environment**：下载环境、API 服务环境、沙箱、V2 受控计算环境。
- **Data Resource / Product / SKU**：资源、产品与可下单权利包。
- **Contract / Usage Policy / Order / Billing Event / Audit Event**：交易控制对象。
- **Public Anchor Batch / Credential Token / Passport**：公链展示相关对象。

## 3.2 权限作用域

权限必须绑定到以下一个或多个作用域：

- Tenant 级
- Department 级
- Product / SKU 级
- Application 级
- Connector 级
- Execution Environment 级
- Region / Network Zone 级
- 时间窗口级
- 配额级

## 7. 权限判断链路

任一请求均必须按以下顺序判定：

1. Tenant 是否有效。
2. Party 是否完成认证、是否被限制。
3. User / App 是否属于合法 Tenant / Department。
4. Role 是否具备该动作基础许可。
5. Scope 是否覆盖目标对象。
6. Product / SKU 是否允许该动作。
7. Contract / Usage Policy 是否已生效。
8. Connector / Environment 是否允许访问。
9. 是否命中地域、时间窗、配额限制。
10. 是否命中合规/风控阻断。
11. 是否需要额外审批。
12. 是否需要二次认证或双人复核。

## 11. 职责分离（SoD）规则

以下动作不得由同一人独立闭环完成：

1. **供方上架 + 平台审核**  
2. **合同编辑 + 最终审批**  
3. **账单生成 + 结算执行**  
4. **退款申请受理 + 赔付执行**  
5. **高风险规则配置 + 风险命中解除**  
6. **公链凭证签发 + 吊销恢复**  
7. **节点成员配置 + 节点审计确认**  
8. **密钥管理 + 安全事件复核**  

V1 最小要求：
- 合规审批与财务结算至少分离。
- 产品上架与产品审核至少分离。
- 风控冻结与冻结解除至少分离。

## 12. 主体类型与默认权限包

## 12.1 数据供方默认权限包
- 可创建资源、产品、SKU
- 可上传样例与质量说明
- 可查看订单、交付记录、回款记录
- 不可直接放行高风险合规结论

## 12.2 数据买方默认权限包
- 可浏览目录、询价、下单、验收、续费
- 可查看自己购买范围内的合同与账单
- 不可查看供方原始资源底层配置

## 12.3 双边参与方默认权限包
- 同时具备供方与买方能力，但作用域必须分开
- 同一 Tenant 下建议按 Department 分离供方与买方角色

## 12.4 监管观察主体默认权限包
- 只读监管视图
- 可导出审计包
- 不得修改业务状态

## 13. ABAC / PBAC 规则样例

### 13.1 ABAC 样例
- 若 Party 类型 = 境外主体，则禁止下载型高敏 SKU。
- 若 Data Level >= 高敏，则仅允许 Query / Sandbox / V2 Compute。
- 若 User Department 不在合同授权范围内，则拒绝访问。
- 若 Application 来源 IP 不在白名单，则拒绝 API 调用。

### 13.2 PBAC 样例
- 合同未生效：拒绝任何交付动作。
- 账单逾期：暂停续费前的新调用额度发放。
- Usage Policy 未允许导出：即使角色允许，也不得导出结果。
- 风控命中共享凭证：自动临时冻结 Application。

## 15. 审批权限矩阵（目标态）

本节描述的是平台的**完整目标审批能力**。  
对个人项目或 V1 阶段，允许采用“系统自动判断 + 管理员确认/人工复核”的简化实现；  
但 Department 审批、法务审批、财务审批、合规审批等能力对象必须在模型中保留，V2/V3 需要完整实现，不从总体功能蓝图中删除。

## 16. V1 最小实现要求

V1 必须落地以下能力：

1. Tenant / Party / Department / User / Application / Connector / Environment 六类对象建模。
2. 平台角色与租户角色分离。
3. RBAC 基础权限。
4. 至少支持 Tenant、SKU、Application 三层强制作用域；Department 作用域在 V1 可保留对象与字段预留，V2 完整启用。
5. 支持“角色 + 合同 + Usage Policy”联合判定。
6. 支持到期自动断权。
7. 支持 Application 凭证吊销。
8. 支持冻结 Order / SKU / Application / Connector。
9. 支持审计日志按人、按应用、按对象追溯。
10. V1 可用管理员确认替代 Department 审批流实现，但不得删除 Department 审批能力定义。

---

## 17. V2 扩展要求

1. 字段级 / 记录级 / 任务级 ABAC。
2. 算法白名单与 Compute Task 权限。
3. 联邦学习参与方权限。
4. 多租户连接器与跨环境策略。
5. 公链凭证签发与吊销权限。
6. 监管观察节点更细粒度视图。
7. 基于风险级别动态提权/降权。

---

## 18. 不纳入 V1 的复杂能力

1. 完整 BPM 审批引擎。
2. 自然人 C2C 供给权限体系。
3. 复杂多方转授权链。
4. 全局通用数据空间连接器互认。
5. 公链治理投票型权限模型。

---

## 19. 研发落地建议

后续进入设计时，建议继续输出：

1. 页面级权限点清单。
2. 后端 API 权限码清单。
3. 数据库对象所有权模型。
4. 审计字段必填清单。
5. 角色初始化脚本与默认权限包。

---

## 20. 结论

这套权限设计的核心不是“谁能点哪个按钮”，而是把：

- 组织责任
- 合同权利
- 使用策略
- 机器身份
- 风险处置

放进同一个统一权限框架里。  
V1 先保证组织级、SKU 级、应用级权限闭环，并为 Department 级审批与作用域保留完整对象模型；V2 再完整启用 Department 级权限、继续下钻到字段级、任务级和公链凭证治理。

## 21. 支付、资金流与清结算角色补充

### 21.1 平台侧新增角色

- `支付渠道管理员`：管理支付渠道、商户号、回调配置、证书、沙箱参数以及新加坡起步司法辖区和走廊白名单。
- `定价与收费管理员`：管理收费规则、费率版本、参数生效时间。
- `对账专员`：导入账单、核对差异、关闭差异工单。
- `资金结算运营员`：执行人工打款、审核退款、复核赔付。
- `Tenant 管理员 / 财务经理`：维护默认收款币种、默认出金方式与默认收款渠道偏好。

### 21.2 V2 / V3 增量角色

- `自动打款运营员`：管理自动打款批次、失败重试和异常回滚。
- `渠道分账管理员`：管理子商户、分账能力与渠道收益拆分。
- `数字资产结算管理员`：管理多币种、交易所路由和数字资产结算风险。

### 21.3 权限边界原则

- 支付渠道配置与实际打款执行不得由同一人工角色独占。
- 司法辖区/走廊策略配置与实际打款执行不得由同一人工角色独占。
- 收费规则配置与退款执行不得由同一人工角色独占。
- 对账差异关闭与审计证据导出应职责分离。

## 22. 新增：身份认证、注册登录与会话管理角色边界

### 22.1 平台侧新增角色

- `身份与访问管理员`：管理登录策略、MFA 策略、会话基线、设备治理和高风险 step-up 规则。
- `Fabric CA 管理员`：管理 Fabric CA registry、证书签发、续签、吊销、MSP 与属性模板。

### 22.2 租户侧新增角色

- `租户身份管理员`：管理成员邀请、成员状态、设备治理、成员登录策略。
- `企业 SSO 管理员`：管理企业 OIDC/SAML 连接、声明映射、JIT 建档和后续 SCIM 同步。

### 22.3 权限边界原则

- 企业 SSO 配置与高权限角色授予不得由同一人工角色独占。
- Fabric 证书签发与证书吊销审计不得由同一人工角色独占。
- 会话撤销、设备撤销与资金类高风险执行必须具备可追责操作主体。
- 机器身份、连接器身份、执行环境身份不得复用人工管理员账号。

## 23. 新增：审计、证据链与回放角色边界

### 23.1 审计权限分层

审计相关权限至少拆分为：

- 联查摘要
- 原始证据查看
- 脱敏视图查看
- 原文视图查看
- 证据包导出
- 回放执行
- 保留策略管理
- `legal hold` 管理
- 锚定批次查看/管理
- `break-glass` 查看

### 23.2 关键角色职责

- `审计管理员`：联查、导出、回放、保留策略与 `legal hold` 管理
- `审计只读角色`：仅联查与脱敏视图查看
- `监管观察员`：默认只读摘要/脱敏视图，不默认看原文
- `安全管理员`：仅在异常调查或 `break-glass` 场景下查看敏感原文
- `财务结算专员 / 对账专员`：仅查看支付、退款、赔付、打款、对账相关审计对象

### 23.3 强制职责分离

- 证据导出 与 `legal hold` 解除不得由同一人工角色长期独占
- 回放执行 与 回放结果确认不得由同一人工角色长期独占
- 锚定批次生成 与 锚定结果复核不得由同一人工角色长期独占
- 敏感原文查看 与 监管导出审批不得由同一人工角色长期独占

## 24. 新增：双层权威模型与一致性角色边界

### 24.1 新增角色建议

- `一致性运营员`：负责 outbox、Kafka 发布、dead letter、重处理、对账修复和一致性联查。

### 24.2 关键权限

- `ops.consistency.read`
- `ops.consistency.reconcile`
- `ops.outbox.read`
- `ops.outbox.relay`
- `ops.dead_letter.read`
- `ops.dead_letter.reprocess`
- `ops.event_route.manage`

### 24.3 边界原则

- 外部事实对象的最终成立仍以来源系统回执为准。
- Kafka 运维权限与业务放行权限必须分离。

### 27.1 新角色建议

### 27.2 关键权限

- `ops.observability.manage`
- `ops.log.query`
- `ops.log.export`
- `ops.trace.read`
- `ops.alert.read`
- `ops.alert.ack`

- `ops.incident.read`
- `ops.incident.manage`

- 可观测性数据用于运行联查，不得越权修改业务主状态。
