# 数据商品查询与执行面设计

- 文档定位：原始 PRD 增补专题稿
- 目标：明确数据商品在“非下载式交付”场景下的查询面、执行面、模板治理和结果边界设计
- 适用阶段：`V1 / V2 / V3`

---

## 1. 设计目标

本文件回答五个问题：

1. 数据商品在不下载原始文件时，平台如何正式提供查询和执行能力。
2. `V1` 的模板查询 / 沙箱 / clean room lite 到底做到什么程度。
3. 查询面如何与对象存储分层、授权、审计、验收和计费闭环。
4. `V2` 如何从 lite 模式升级到完整 clean room / analysis rule / TVF。
5. `V3` 如何支持跨平台查询面和跨域受控执行协同。

---

## 2. 核心结论

平台必须把“查询/执行面”定义为正式产品能力，而不是交付后的临时附属脚本。

查询/执行面的本质是：

**受控访问入口 + 模板治理 + 输出边界治理 + 审计计费联动**

而不是：

**给买方一段自由 SQL 或一个随意可连的数据库账号**

---

## 3. 四个核心对象

`V1` 起应显式建模以下正式对象：

1. `QuerySurface`
2. `QueryTemplate`
3. `QueryGrant`
4. `QueryExecutionRun`

### 3.1 QuerySurface

表示某个数据商品、资产版本或对象对外暴露的查询/执行入口。

必须表达：

- 面向哪个资产版本
- 运行在哪个执行环境
- 属于哪种查询面类型
- 可读数据范围
- 输出边界
- 频率/配额约束
- 当前状态

### 3.2 QueryTemplate

表示一类被批准的模板查询或受控执行模板。

必须表达：

- 模板名称与版本
- 参数定义
- analysis rule / 风险限制
- 输出结果 schema
- 可导出边界
- 风险规则与审计要求

### 3.3 QueryGrant

表示订单成交后，买方被授予某个查询面的使用权。

必须表达：

- 绑定哪个订单
- 允许哪些模板
- 输出边界快照
- 运行配额
- 授权有效期
- revoke 生命周期

### 3.4 QueryExecutionRun

表示一次实际执行动作。

必须表达：

- 谁发起
- 在哪个 grant / session 下运行
- 使用了哪个模板
- 输入参数摘要
- 输出对象与结果摘要
- 行数/计费单元
- 审计与风控结果

---

## 4. V1 最小查询/执行面

### 4.1 V1 只做三类正式能力

1. 查询沙箱
2. 白名单模板查询
3. 受限结果导出

### 4.2 V1 不做的事情

- 自由 SQL
- 买方直接连底层数据库
- 任意自定义 UDF
- 完整 clean room
- 多方联合分析规则引擎
- 差分隐私全量能力

### 4.3 V1 运行原则

1. 查询面默认只读 `curated / product` 区，不直接读 `raw`。
2. 查询必须通过已审核模板或已审核运行入口。
3. 模板参数必须有 schema 和范围校验。
4. 导出只允许受限结果，不允许底层原始表明细自由导出。
5. 每次执行都必须记录 `QueryExecutionRun` 和结果摘要。

---

## 5. V1 业务闭环

```text
卖方配置 QuerySurface
  -> 绑定执行环境
  -> 配置输出边界和配额
  -> 录入 QueryTemplate
  -> 审核通过后上架
  -> 买方成交获得 QueryGrant
  -> 开通 SandboxWorkspace / Session
  -> 买方执行模板查询
  -> 平台记录 QueryExecutionRun
  -> 结果写 Result Artifact / Report Artifact
  -> 验收、计费、审计、Fabric 摘要联动
```

---

## 6. V1 查询面类型

### 6.1 `template_query_lite`

- 仅允许白名单 SQL / 参数化模板
- 输出必须匹配约定 schema
- 适合 `V1`

### 6.2 `sandbox_query`

- 在受控 workspace 中运行模板
- 会话、导出、并发、时效全受控
- 适合 `V1`

### 6.3 `report_result`

- 不直接开放查询，只开放结果包或固定报告
- 适合高敏场景和轻量交付

---

## 7. 与存储分层的关系

查询/执行面不是独立漂浮层，必须受存储分层约束：

- 默认只能读取 `curated / product / result`
- 不得把 `raw` 当对外查询面
- 预览面只读 `preview`
- 导出结果进入 `delivery / result`
- 归档结果进入 `archive`

查询/执行面必须记录：

- 来源分层区域
- 输出分层区域
- 是否允许导出
- 是否允许二次执行

---

## 8. 与授权、计费、审计的关系

### 8.1 授权

查询权必须作为独立权利表达：

- 哪个用户/应用可执行
- 哪些模板可执行
- 哪些参数范围可用
- 运行多久
- 可导出多少

### 8.2 计费

至少支持以下口径：

- 按席位
- 按查询次数
- 按运行时长
- 按结果导出次数

### 8.3 审计

每次执行必须记录：

- 模板版本
- 参数摘要
- 运行人/应用
- 输出摘要
- 导出动作
- 风控阻断与放行结果

---

## 9. V2 扩展

`V2` 在 `V1` 基础上正式补齐：

- 完整 clean room
- analysis rule
- TVF
- 更复杂的输出策略
- 更严格的结果边界与证明

此时 `QuerySurface` 将升级为：

- `clean_room_surface`
- `analysis_rule_surface`
- `tvf_surface`
- `federated_compute_surface`

---

## 10. V3 扩展

`V3` 增加：

- 跨平台 QuerySurface 映射
- 跨平台模板治理
- 外部执行环境互认
- 外部结果摘要与导出回执互认

平台应能区分：

- 本平台 QuerySurface
- 合作方 QuerySurface
- 本平台执行
- 合作方执行

---

## 11. 数据库落点建议

`V1` 起建议新增：

- `catalog.query_surface_definition`
- `delivery.query_template_definition`
- `delivery.query_execution_run`

并增强：

- `delivery.template_query_grant`
- `delivery.sandbox_workspace`

---

## 12. 结论

数据商品的“查询/执行面”必须被设计成正式产品面，而不是数据库账号或脚本交付的替代品。

在你当前的平台里，最务实的路径是：

- `V1`：模板查询 + 沙箱 + 受限结果
- `V2`：完整 clean room / analysis rule / TVF
- `V3`：跨平台 QuerySurface 与跨域受控执行

这样才能和你前面已经定下来的分层存储、数据契约、审计、Fabric 摘要和双层权威模型保持一致。
