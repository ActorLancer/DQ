# 数据本体存储与信任边界设计

- 文档定位：原始 PRD 增补专题稿
- 目标：明确平台对“交易数据本体、模型文件、受控结果、密钥与信任边界”的统一设计原则
- 适用阶段：`V1 / V2 / V3`

---

## 1. 设计目标

本文件解决四类问题：

1. 平台究竟存什么，哪些数据不应默认进入平台明文域。
2. 当数据本体不在平台时，如何完成文件、API、模型、受控执行等交易闭环。
3. 当卖方不信任平台时，如何通过边界约束、密钥分离、受控执行和审计证据建立信任。
4. 交易达成后，平台如何持续管理资产、授权、交付、调用和销毁生命周期，而不是无限期持有明文。

---

## 2. 核心结论

平台应被定义为：

**交易协调者 + 权利执行者 + 审计/存证者 + 生命周期治理者**

而不是：

**默认明文托管者**

平台可以支持托管，但不能要求所有交易必须先把明文数据上传到平台。

---

## 3. 总体原则

1. 平台交易的是“可验证的数据使用权 / 访问权 / 查询权 / 计算权 / 结果获取权”，不等于默认交易明文副本。
2. 数据资产元信息、授权、交付令牌、审计证据必须在平台统一治理。
3. 原始数据、密文大对象、模型权重、训练语料、证据原文默认不直接落 PostgreSQL。
4. 平台必须同时支持：
   - 平台托管模式
   - 卖方自持模式
   - 受控执行模式
   - 买方拉取模式
   - 第三方可信存储模式
5. 平台信任不依赖单一密码学名词，而依赖：
   - 最小明文接触
   - 密钥分离
   - 受控交付
   - 全量审计
   - 版本与哈希固化
   - 可证明的撤销、过期、销毁

---

## 4. 三层对象拆分

### 4.1 资产元信息层

平台必须保存：

- `asset_id / asset_version_id`
- 标题、分类、标签、描述
- schema 摘要与版本
- 哈希指纹
- 敏感等级
- 权利与限制
- 存储模式
- 交付模式
- 生命周期状态

### 4.2 交付与控制层

平台必须保存：

- 下载令牌
- API 凭证
- 沙箱授权
- 交付工单
- 有效期、配额、次数
- 审计日志
- 授权与断权状态

### 4.3 数据载荷层

数据载荷不一定在平台，允许位于：

- 平台对象存储
- 卖方对象存储
- 卖方数据库
- 卖方 API
- 受控执行环境
- 第三方可信存储
- 模型仓库

---

## 5. 存储模式

### 5.1 平台托管模式

适用：

- 标准文件包
- 低中敏聚合数据
- 卖方愿意托管的平台化交付

要求：

- 平台保存密文对象或受控对象
- 数据与密钥分离
- 交付通过下载票据、解密票据、短时链接完成

### 5.2 卖方自持模式

适用：

- 卖方不信任平台
- 数据高度敏感
- 数据量大或实时更新
- 卖方已有稳定 API / 对象存储 / 连接器

要求：

- 平台只保存元数据、哈希、连接方式、策略与审计
- 平台不默认获得原始明文
- 交易后通过卖方侧连接器/API 网关完成交付

### 5.3 受控执行模式

适用：

- 不允许数据副本出域
- 只卖查询结果、统计结果、模型结果
- 训练任务、计算任务、隐私计算任务

要求：

- 数据保留在卖方或专用执行环境
- 买方提交任务，不直接获取原始底层数据
- 平台记录任务、输入摘要、输出摘要、证明和结算

### 5.4 买方拉取模式

适用：

- API 订阅
- 按次查询
- 实时数据流

要求：

- 平台签发访问权与配额
- 买方按授权拉取
- 数据源可在卖方侧，也可在平台代理层

### 5.5 第三方可信存储模式

适用：

- 需要独立托管方
- 多方共认的对象存储/模型仓库
- 跨平台互联场景

要求：

- 平台记录第三方托管关系、证据、摘要和责任边界

---

## 6. 平台必须掌握什么

平台必须掌握：

- 元数据
- 哈希
- 版本
- 合同与策略
- 授权记录
- 交付记录
- 调用与使用记录
- 风险事件
- 证据包
- 生命周期事件

平台不应默认掌握：

- 原始明细明文
- 模型权重明文
- 原始训练语料
- 独立可长期解密的密钥材料

---

## 7. 交易后如何交付

### 7.1 文件一次性交付

- 平台托管：平台保存密文对象，交易完成后签发下载票据与密钥封装
- 卖方自持：平台下发交付工单，卖方侧生成一次性下载地址或通过连接器投递

### 7.2 按次收费 API

- 平台负责应用、凭证、配额、计费和审计
- 数据源可位于卖方侧、平台代理层或受控服务节点

### 7.3 模型出售

- 交付模型文件：模型本体放对象存储/模型仓库，平台只存元数据、版本、hash、许可证
- 交付模型服务：买方调用推理服务，不直接获取模型权重

### 7.4 受控训练与受控计算

- 平台编排任务
- 数据保留在卖方或专用执行环境
- 平台管理任务、证明、结果摘要和结算

---

## 8. 信任问题怎么解决

### 8.1 不把希望押在 ZKP 上

ZKP 可用于证明某个断言成立，但不能直接证明：

- 平台从未看过明文
- 平台线下从未复制过数据

因此，ZKP 只能作为局部增强能力，不能当作平台信任的唯一基础。

### 8.2 真正有效的五层解法

1. 最小化平台明文接触
2. 数据与密钥分离
3. 高敏资产优先使用受控执行，不交明文
4. 所有解密、交付、调用、导出进入审计证据链
5. 用版本、哈希、回执、证明、销毁记录建立可追责闭环

### 8.3 密钥治理原则

- 平台托管对象不等于平台单独掌控解密能力
- 卖方可保留密钥主控权
- 平台可使用等价可审计的软件 KMS 起步，后续接入完整 KMS/HSM
- 所有密钥封装、签发、轮换、吊销、异常停用必须留痕

---

## 9. 生命周期治理对象

平台追踪的是以下生命周期，而不是单纯追踪明文文件：

1. 资产生命周期
2. 版本生命周期
3. 权利生命周期
4. 交付生命周期
5. 凭证生命周期
6. 风险生命周期
7. 审计生命周期
8. 存储生命周期

每个生命周期至少要具备：

- 状态
- 状态变更原因
- 操作主体
- 时间
- 证据引用
- 是否上链摘要

---

## 10. 必须纳入系统的最小字段

每个可交易资产至少要显式记录：

- `storage_mode`
- `payload_location_type`
- `custody_mode`
- `key_control_mode`
- `platform_plaintext_access`
- `platform_unilateral_decrypt_allowed`
- `default_delivery_route`
- `requires_controlled_execution`
- `trust_evidence_count`
- `retention_policy`
- `destroy_policy`

每个订单/交付至少要记录：

- `storage_mode_snapshot`
- `delivery_route_snapshot`
- `trust_boundary_snapshot`
- `executor_type`
- `source_binding_id`
- `receipt_hash`

---

## 11. 分阶段要求

### 11.1 V1

- 必须支持平台托管、卖方自持、API 拉取三类正式模式
- 必须支持高敏资产“仅沙箱/仅结果交付”的限制表达
- 必须支持密钥分离与下载票据
- 必须在页面、权限、订单、交付、数据库中显式落字段

### 11.2 V2

- 正式支持模型权重、模型服务、训练任务的信任边界建模
- 支持受控训练、受控结果导出、证明材料绑定
- 支持更强的执行环境证明和输出约束

### 11.3 V3

- 支持跨平台/第三方可信存储互认
- 支持跨域连接器互联下的信任边界表达
- 支持监管穿透下的存储责任追踪与保全

---

## 12. 一句话结论

平台不是通过“承诺不会滥用数据”来建立信任，而是通过**让平台默认不必接触明文、即使托管也不独占解密能力、所有关键动作都有证据链**来建立信任。

## 13. 与支付、资金流与清结算的联动补充

- 平台交易的不是默认明文副本，因此支付完成后也必须按存储模式和交付模式执行，不能因为“已付款”就默认平台可明文转移数据。
- 支付状态、交付状态和信任边界快照必须绑定保存，避免后续争议中出现“钱已付但平台无权交付”或“平台越权导出”的问题。
- 卖方自持、受控执行和第三方可信存储模式下，支付成功后应触发的是交付工单、令牌或任务权限，而不是平台内部明文复制。
- 起步司法辖区为新加坡时，真实支付与真实出金只在被允许的新加坡走廊内发生，不影响数据本体可托管、卖方自持或受控执行三种模式。

## 14. 与身份认证、注册登录与会话管理的联动补充

- 数据访问的信任边界不只由存储模式决定，还由主体身份、成员角色、服务身份、会话状态和 step-up 结果共同决定。
- 平台必须能证明“谁在什么设备、什么会话、使用什么令牌或证书，对哪个数据对象执行了什么动作”。
- 高敏下载、密钥解封、训练用途放行、跨域导出必须绑定更高 assurance 等级的成员身份或服务身份。
- Fabric 链上身份用于链上成员和链上签名责任链，不直接放大平台对数据明文的访问能力。
