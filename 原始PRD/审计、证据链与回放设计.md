- 审计事件记录到什么粒度

## 2. 设计原则

1. 审计不是附属日志，而是独立业务域。
2. 业务状态、审计事件、证据对象、链上摘要必须分层建模。
3. 平台必须支持“联查”“导出”“回放”“对账”四种审计使用方式。
4. 审计查看权限与业务查看权限必须严格分离。
5. 原始证据、敏感日志和监管导出必须支持脱敏视图与原文视图双模式。
6. Fabric 只存摘要、根哈希、状态证明和索引，不存原始证据正文。
7. 默认采用链下主状态机 + 审计哈希链 + Fabric 批量锚定，不让链确认阻塞业务主链路。
8. 所有高风险审计动作本身也必须被审计。

## 3. 五层审计体系

### 3.1 L1 业务审计事件层

统一记录：

- 谁做了什么
- 对哪个对象做
- 结果如何
- 请求来自哪里
- 是否触发高风险认证
- 是否涉及链上回执、支付回调、执行环境或监管查询

### 3.2 L2 证据对象层

统一登记以下原文或原文引用：

- 合同快照
- 授权/策略快照
- 交付回执
- 下载回执
- API 首次成功调用回执
- 支付回调原文
- 对账单原文
- 退款/赔付/打款回执
- 争议材料
- 裁决单
- 训练轮次摘要
- 跨链 ack / 见证材料
- 监管导出包

### 3.3 L3 完整性证明层

平台必须为审计和证据对象维护：

- 事件哈希 `event_hash`
- 前序哈希 `previous_event_hash`
- 证据项哈希
- 证据清单哈希 `manifest_hash`
- 审计批次根 `batch_root`

### 3.4 L4 Fabric 存证层

Fabric 负责保存：

- 审计批次摘要
- 合同/授权/支付/结算/争议关键状态摘要
- 证据包 manifest 摘要
- 监管导出摘要
- 证书签发/吊销摘要

### 3.5 L5 回放与对账层

平台必须支持：

- 取证回放
- 状态回放
- 对账回放
- 白名单补偿重放

## 5. 证据模型

平台必须显式建模以下对象：

- `EvidenceItem`
- `EvidenceManifest`
- `EvidenceManifestItem`
- `EvidencePackage`
- `AnchorBatch`
- `AnchorItem`
- `ReplayJob`
- `ReplayResult`
- `LegalHold`
- `RetentionPolicy`
- `AuditAccessRecord`

其中：

- `EvidenceItem` 表示单个证据对象
- `EvidenceManifest` 表示一次取证/导出/监管查询所对应的证据清单
- `EvidencePackage` 表示面向用户或监管的导出包
- `AnchorBatch` 表示需要上链的一批审计摘要
- `ReplayJob` 表示一次联查或回放任务
- `AuditAccessRecord` 表示谁看了哪些敏感审计内容

## 6. 什么必须上链，什么不能上链

### 6.1 必须上 Fabric 的摘要对象

- 主体审核通过/撤销摘要
- 商品版本登记摘要
- 合同快照哈希
- 授权策略哈希
- 订单关键状态摘要
- 支付/保证金/退款/赔付/结算批次摘要
- 争议立案/裁决摘要
- 证据清单摘要
- 审计批次根
- 监管导出包摘要
- Fabric 证书签发/吊销摘要

### 6.2 严禁上链的内容

- 原始交易数据正文
- API 全量调用正文
- 原始支付凭证
- 会话令牌、刷新令牌、密码、密钥
- 证据原件正文
- 模型权重全文
- 原始训练语料

## 7. 保留、保全与销毁

平台必须支持：

- `retention_class`
- `WORM` 保留
- `legal_hold`
- 冷热分层归档
- 到期销毁
- 销毁证明

规则：

- 证据对象默认按策略进入对象存储或可信存储
- 高风险争议、监管查询、重大退款/赔付、证书吊销相关证据应支持 `legal_hold`
- 删除不是直接物理删除，应先校验 retention 与 hold 状态

## 8. 回放设计

### 8.1 回放类型

- `forensic_replay`：取证回放，只重建时间线
- `state_replay`：状态回放，重算对象状态并与现状比对
- `reconciliation_replay`：对账回放，重建支付/打款/账务链路
- `compensation_replay`：补偿重放，仅对白名单幂等动作开放

### 8.2 回放输入

- `order_id`
- `payment_intent_id`
- `payout_instruction_id`
- `case_id`
- `task_id`
- `tx_hash`
- `evidence_package_id`
- `request_id`

### 8.3 回放输出

- 时间线重建
- 状态差异
- 缺失证据清单
- 链上摘要校验结果
- 对账差异结果
- 是否建议补偿

### 8.4 回放安全边界

- 默认只允许 dry-run
- 有副作用的 replay 必须二次认证
- 允许副作用的 replay 必须命中白名单动作
- 回放本身必须生成审计事件和证据

## 9. 支付与结算审计要求

以下对象必须纳入强审计主链：

- `PaymentIntent`
- `PaymentTransaction`
- `RefundIntent`
- `PayoutInstruction`
- `DepositHold`
- `SettlementBatch`
- `ReconciliationStatement`
- `ReconciliationDiff`

要求：

- 渠道回调原文必须留存
- 对账单原文必须可校验 hash
- 差异处理过程必须可回放
- 支付成功、退款成功、打款成功、差异关闭都必须形成高优先级审计事件

## 10. 审计访问权限

必须至少区分以下权限能力：

- 审计摘要查看
- 原始证据查看
- 脱敏视图查看
- 原文视图查看
- 证据包导出
- 回放执行
- 保留策略管理
- `legal hold` 管理
- 上链批次查看
- 上链批次管理
- `break-glass` 查看

规则：

- `audit.trace.read` 只代表联查，不等于原文查看
- 原文查看、导出、回放、解除 hold 必须 step-up
- 监管角色默认只看脱敏或摘要视图
- break-glass 必须强理由、强留痕、事后复核

## 11. 页面与模块建议

V1 应具备：

- 审计联查页
- 证据包导出页
- 审计回放页
- 证据保全与保留策略页
- 锚定批次页

V2 增量：

- 受控计算证明与审计联查页
- 训练/证明/结果边界联合回放页

V3 增量：

- 跨链审计联查页
- 监管批量穿透导出页
- 图风控回放页

## 12. V1 / V2 / V3 边界

- `V1`：统一审计事件、证据清单、证据包、上链批次、回放 dry-run、legal hold 基础能力、支付/交付/争议/证书强审计
- `V2`：受控计算、训练、模型、TEE/ZKP 证明、分润与公链增强纳入强审计
- `V3`：跨链、图风控、监管穿透、生态互联纳入统一审计与回放

## 13. 与整套交易逻辑的闭环

```text
业务动作
  -> 生成 AuditEvent
  -> 生成 EvidenceItem / Manifest
  -> 写入对象存储 / WORM 保全
  -> 批量形成 AnchorBatch
  -> 写 Fabric 摘要
  -> 支持联查 / 导出 / 回放 / 对账
  -> 监管查询与导出
  -> 争议、赔付、冻结、证书吊销等动作继续回写审计链
```

新增这一部分后，平台在交易、支付、身份、证书、争议、监管和跨链等关键域上，都有了统一的强审计和回放基础。

## 14. 与双层权威模型的一致性补充

- 审计原文、证据清单、回放任务与差异结果，以数据库和对象存储中的原文记录为业务权威。
- 审计批次根、关键证据清单摘要、监管导出摘要，以 Fabric 确认结果为可信证明确认层。
- 支付、证书、对象销毁等外部事实事件必须把来源回执作为审计证据对象保存，不得只记平台内部状态。
- Kafka 负责审计事件分发和锚定服务联动，但不是审计真相源。
