# 日志、可观测性与告警设计

## 1. 文档目标

本文件用于补齐平台“日志、指标、追踪、告警、事件处置”的正式设计基线，解决以下问题：

- 普通系统日志、审计日志、证据链、指标、追踪如何分层
- 平台应采用什么观测技术栈
- 关键日志、告警、工单和追踪索引哪些需要进入 PostgreSQL
- Loki、Tempo、Prometheus、Grafana 与平台主数据库如何协作
- 日志查询、告警处置、事件工单、SLO 和审计访问如何治理
- 如何在不破坏现有双层权威模型的前提下建立强留痕、强联查、强告警的运维体系

## 2. 设计原则

1. 审计域与日志域必须分层，不得把普通系统日志当作审计主记录。
2. PostgreSQL 只保存观测配置、关键日志镜像、追踪索引、告警事件、工单和治理状态，不承接全量原始运维日志。
3. 原始应用日志应进入日志后端，原始 trace 应进入 tracing 后端，指标应进入 metrics 后端。
4. 所有观测对象必须可通过 `request_id / trace_id / object_type / object_id` 与交易、支付、交付、审计、推荐链路关联。
5. 观测系统必须遵守平台既有权限、脱敏、保留与 legal hold 规则。
6. 高风险观测动作本身必须审计，包括原始日志导出、告警规则修改、backends 切换和工单强制关闭。

## 3. 观测域与审计域边界

### 3.1 审计域负责

- 业务动作审计事件
- 证据对象与清单
- Fabric 摘要锚定
- 回放与对账
- 监管导出

### 3.2 观测域负责

- 应用日志
- 网关访问日志
- 任务与作业日志
- 节点、链适配器、消息总线运行日志
- 指标与 SLO
- Trace / Span 联查
- 告警与事件工单

### 3.3 统一原则

- 审计是证据域，权威在 `audit.*`
- 观测是运行域，权威在 `ops.* + Loki/Tempo/Prometheus`
- 审计访问与日志访问都必须再审计
- 推荐、搜索、支付、跨链等模块都必须向观测域输出统一链路标识

## 4. V1 正式采用的观测技术栈

### 4.1 采集标准

- `OpenTelemetry`
- 所有服务统一输出 `logs + metrics + traces`
- 所有服务必须传播 `traceparent`

### 4.2 指标与告警

- `Prometheus`：采集指标
- `Alertmanager`：规则告警分发
- `Grafana`：统一看板与查询入口

### 4.3 日志后端

- `Loki`：V1 正式采用的应用/运维日志后端
- 不将 `ELK / Elasticsearch / OpenSearch` 作为 V1 日志主后端硬依赖
- 后续若有安全分析、长期全文日志分析需求，可在 `V2/V3` 再引入独立日志检索引擎

### 4.4 Trace 后端

- `Tempo`：V1 推荐采用的 trace 后端
- 平台至少保存 `trace_id / traceparent / span_id` 索引，便于从业务主键反查 trace

### 4.5 数据库存储职责

PostgreSQL 中仅承接：

- 观测后端配置
- 日志保留策略
- 关键结构化系统日志镜像
- Trace 索引
- 告警规则
- 告警事件
- 事件工单
- SLO 定义与快照

## 5. 日志分类

平台日志至少分为以下 7 类：

1. `application_log`
   - 业务服务、BFF、网关、推荐服务、搜索索引器
2. `access_log`
   - 网关、下载、API、沙箱入口
3. `security_log`
   - 登录、step-up、证书、密钥、令牌、权限提升、break-glass
4. `integration_log`
   - 支付回调、Fabric writer、搜索同步、推荐回流、跨链网关
5. `job_log`
   - reindex、reconcile、replay、批量锚定、推荐重建
6. `infrastructure_log`
   - PostgreSQL、Kafka、Redis、Loki、Tempo、Prometheus、链节点
7. `audit_mirror_log`
   - 审计域关键动作的结构化镜像，不替代审计事件

## 7. 日志脱敏与保留

### 7.1 严禁进入普通日志的内容

- 密码
- refresh token / access token 原文
- 私钥、密钥材料原文
- 支付卡敏感字段
- 原始证件图片
- 原始证据文件正文
- 原始训练语料

### 7.2 保留策略

- `ops_default`
- `security_sensitive`
- `access_short_term`
- `trace_index_default`
- `incident_long_term`

### 7.3 数据保留边界

- 原始日志主存放在 `Loki`
- 原始 trace 主存放在 `Tempo`
- 指标与告警状态主存放在 `Prometheus/Alertmanager`
- PostgreSQL 保存治理配置、事件索引和关键镜像

## 8. 采集与同步链路

### 8.1 应用日志 / 指标 / 追踪

```text
服务进程
  -> OpenTelemetry SDK / Auto Instrumentation
  -> OTel Collector / Grafana Alloy
  -> Loki / Tempo / Prometheus
  -> Grafana 展示
```

### 8.2 关键日志镜像入库

```text
服务进程输出结构化关键日志
  -> OTel Collector / 平台日志适配器
  -> 关键日志选择性镜像至 ops.system_log
  -> 必要时写 outbox_event
  -> 供联查 / 告警 / 审计补充使用
```

### 8.3 告警入库与事件工单

```text
Prometheus rule / Alertmanager webhook / 自定义监控器
  -> ops.alert_event
  -> 触发通知 / 工单
  -> ops.incident_ticket / ops.incident_event
  -> 审计留痕
```

## 9. 告警与事件工单设计

### 9.1 告警来源

- Prometheus 规则告警
- 应用自定义异常阈值
- Fabric writer / gateway 失败
- Kafka / outbox / dead letter 堆积
- 搜索同步滞后
- 推荐缓存重建失败
- 支付 webhook 异常
- 关键登录/安全异常

### 9.2 告警状态

- `open`
- `acknowledged`
- `suppressed`
- `resolved`
- `closed`

### 9.3 工单状态

- `open`
- `investigating`
- `mitigating`
- `resolved`
- `closed`

### 9.4 V1 必备告警

- `dead_letter_event` 积压
- Fabric 提交失败率异常
- 支付回调失败率异常
- 搜索索引同步滞后
- 推荐重建失败
- 新设备/异常登录峰值

## 11. 权限与访问治理

建议新增以下权限族：

- `ops.observability.read`
- `ops.observability.manage`
- `ops.log.query`
- `ops.log.export`
- `ops.trace.read`
- `ops.alert.read`
- `ops.alert.ack`
- `ops.alert.manage`
- `ops.incident.read`
- `ops.incident.manage`
- `ops.slo.read`
- `ops.slo.manage`

规则：

- 原始日志导出必须 step-up
- 原始日志访问必须再审计
- 告警规则修改必须再审计
- 事件工单关闭必须留痕

## 13. 与现有体系的闭环关系

```text
业务服务 / 网关 / 任务 / 节点
  -> OpenTelemetry
  -> Loki / Tempo / Prometheus
  -> 告警 / 工单 / 关键日志镜像入 PostgreSQL
  -> 与 request_id / trace_id / object_id 串联
  -> 与 audit.* / ops.outbox_event / dead_letter_event / consistency trace 联查
```

结论：

- 审计仍是证据主链
- 日志与可观测性是运行主链
- 两者通过 `request_id / trace_id / object_type / object_id` 闭环
- V1 正式采用 `OpenTelemetry + Prometheus + Alertmanager + Grafana + Loki + Tempo`
