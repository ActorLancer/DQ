# 交易链监控、公平性与信任安全设计

- 文档定位：原始 PRD 增补专题稿
- 适用范围：`V1 / V2 / V3`
- 目标：把“平台如何监控完整交易链、保证交易公平、信任安全、敏感数据闭环可验证”落成正式设计基线

---

## 1. 文档目标

本文件回答以下问题：

1. 平台到底监控什么，才算“监控完整交易链”。
2. 平台如何把“公平交易”从口号变成可执行、可观测、可仲裁的状态机。
3. 交易链中的链上事件、链下状态、支付回执、交付回执、执行回执如何统一关联。
4. 敏感数据、受控执行、模板查询、API、沙箱、报告结果这几类不同交付模式如何纳入同一套监控闭环。
5. 平台如何在不读取全部原始数据正文的前提下，仍然形成可信监控、可信争议处理和可信结算。

## 2. 核心结论

### 2.1 平台不能承诺“纯密码学绝对公平”

现实工程里，数字商品交易通常做不到“完全无信任、无中介、无反悔、无 abort 的绝对公平交换”。

平台能真正落地的是：

**受治理状态机 + 乐观型托管/仲裁 + 密码学摘要与执行证明 + 保证金/扣罚 + 强审计 + 自动化监控**

也就是说，平台不追求“任何一方永远不能作恶”的神话，而追求：

- 作恶很难
- 作恶有代价
- 作恶可检测
- 作恶可举证
- 作恶后可仲裁、可回滚、可赔付、可追责

### 2.2 “交易链监控”不是普通技术监控

这里的监控不是只看 CPU、接口延迟和服务存活。

平台必须同时监控五条链：

1. **业务状态链**
   - 订单、授权、验收、结算、争议、撤权状态是否按规则迁移
2. **证据链**
   - 每个关键动作是否形成正式证据对象、证据摘要和可回放记录
3. **执行链**
   - 文件交付、API 调用、模板查询、沙箱执行、受控计算是否真实发生
4. **外部事实链**
   - 支付、打款、对象存储、证书、远程证明、第三方服务回执是否真实到达
5. **可信证明链**
   - Fabric 摘要存证、链码事件、commit status 与链下投影是否一致

只有这五条链联动，平台才能真正做到：

- 可监控
- 可联查
- 可告警
- 可仲裁
- 可上链证明

### 2.3 对敏感数据，平台监控的是“受控使用是否按规则发生”

对敏感数据，平台不应把“是否把明文文件传给买方”作为唯一完成标准。

敏感交易的完成条件应改成：

- 是否完成合法准入
- 是否绑定处理依据
- 是否完成安全预览
- 是否按受控执行策略执行
- 是否形成执行回执和结果回执
- 是否通过结果披露审查
- 是否完成撤权/过期/销毁或保留

## 3. 设计原则

1. **状态机优先**
   - 交易监控首先围绕状态机设计，而不是围绕页面按钮或人工流程设计。
2. **双层权威**
   - 数据库是业务运营主状态机，Fabric 是关键承诺与可信证明确认层。
3. **外部事实独立权威**
   - 支付成功、打款成功、对象销毁、证书吊销、远程证明成功等，以其来源系统回执为准。
4. **监控即控制**
   - 监控不是只报警，还必须驱动阻断、冻结、撤权、补偿、争议和人工接管。
5. **统一标识贯穿全链**
   - 所有链路都必须至少能关联 `request_id / trace_id / order_id / tx_id / event_id`。
6. **敏感链路最小披露**
   - 敏感交易监控中，不记录不必要的原始正文，不把普通日志变成敏感数据二次泄露源。
7. **监控动作本身可审计**
   - 手工放行、手工重试、手工补偿、日志导出、回放、break-glass 都必须反向进入审计链。

## 5. 六层交易链监控模型

## 5.1 L1 业务状态监控层

这是最核心的一层。

平台必须把交易抽象成正式状态机，而不是“下单了就算开始、发货了就算结束”的松散流程。

建议主状态机：

`Draft -> Registered -> Validated -> Listed -> Ordered -> FundsLocked -> DeliveryPrepared -> BuyerPrecheckPassed -> GrantedOrDelivered -> AcceptedOrAutoAccepted -> Settled -> Closed`

异常支路：

- `Rejected`
- `Disputed`
- `Refunded`
- `Compensated`
- `Slashed`
- `Revoked`
- `Expired`
- `Failed`

每个状态必须显式定义：

- 进入条件
- 退出条件
- 必备证据
- 超时规则
- 自动动作
- 人工接管条件

## 5.2 L2 交付与执行监控层

平台必须监控“交付是否真实发生”，不能只看订单状态。

不同交付模式的可验证完成条件：

### 文件型

- 加密对象已预存
- 对象哈希匹配
- 下载令牌已签发
- 下载回执已生成
- 解密密钥发放动作已留痕

### API 型

- 凭证已生成
- 凭证绑定到指定 app / scope
- 首次成功调用达到约定标准
- 网关记录有正式调用回执

### 模板查询 / 沙箱型

- 查询面存在
- 授权已生效
- 模板参数校验通过
- 查询执行成功
- 结果对象生成
- 输出边界校验通过

### 报告 / 结果型

- 报告对象生成
- 版本摘要和哈希固定
- 买方签收或达到自动验收条件

### 敏感受控执行型

- 敏感执行策略存在
- 处理依据存在
- 安全预览存在
- 执行回执或证明存在
- 结果披露审查完成

## 5.3 L3 证据与审计监控层

每个关键状态迁移至少应形成以下之一：

- 审计事件
- 证据对象
- 证据清单
- 证据包
- 上链摘要

平台要持续监控：

- 是否存在“状态迁移成功但没有证据”的孤儿状态
- 是否存在“证据存在但订单未进入对应状态”的悬挂证据
- 是否存在“上链摘要已提交但链下证据包不完整”的不一致

## 5.4 L4 公平性与风控监控层

平台要监控的不只是技术故障，还包括交易公平性风险。

必须覆盖：

- 卖方收款前是否已经满足可交付条件
- 买方拿到结果前是否已锁资
- 是否存在卖方迟迟不交付
- 是否存在买方长期占用资源不验收
- 是否存在反复触发争议、恶意拒收、恶意退款
- 是否存在敏感结果导出绕过审查
- 是否存在异常调用、批量拉取、疑似二次倒卖

## 5.5 L5 运维可观测性监控层

这层使用标准技术栈承接运行事实：

- `OpenTelemetry`
- `W3C Trace Context`
- `Prometheus`
- `Alertmanager`
- `Grafana`
- `Loki`
- `Tempo`

这层负责把服务、数据库、消息、网关、任务、链写、支付回调、查询执行等统一串起来。

## 5.6 L6 Fabric 可信确认层

平台必须监控以下 Fabric 相关事实：

- 交易是否已 submit
- commit status 是否成功
- chaincode event 是否发出
- 区块号和交易号是否记录
- 链下投影是否已消费对应事件
- 是否存在链上已确认但链下未投影
- 是否存在链下已推进但链上摘要未确认

## 6. 全链路统一标识设计

如果没有统一标识，完整交易链监控无法成立。

平台至少统一以下标识：

- `request_id`
- `trace_id`
- `span_id`
- `order_id`
- `inquiry_id`
- `delivery_id`
- `query_run_id`
- `task_id`
- `payment_intent_id`
- `settlement_id`
- `case_id`
- `approval_ticket_id`
- `transaction_id`（Fabric tx）
- `block_number`
- `event_id`

要求：

1. 所有 HTTP / gRPC / MQ / cron / worker / callback / chain adapter 都必须传播 `traceparent`。
2. 所有业务对象写审计时必须带 `request_id`。
3. 所有链适配器必须把 `tx_id` 回填到业务状态、审计事件和证据清单。
4. 所有异步事件必须带 `event_id + trace_id + correlation_id`。

## 7. 平台如何保证交易公平

## 7.1 公平的工程定义

平台不应把“公平”定义为抽象口号，而应定义为：

1. 任一方不能在未承担成本的情况下单方面获益。
2. 任何关键动作都必须留下可验证回执。
3. 任一方违约时，另一方可以获得自动保护或仲裁保护。
4. 争议时平台能根据正式证据和状态机判定责任。

## 7.2 文件型商品的公平闭环

建议采取：

1. 卖方先上传加密包到对象存储。
2. 平台记录对象哈希、大小、manifest、可用性校验结果。
3. 买方下单并锁资。
4. 买方只能先看 preview / contract / quality summary，不拿密钥。
5. 锁资成功且卖方交付条件满足后，平台才释放下载票据或密钥封套。
6. 买方验收通过后结算。
7. 超时未拒收则自动验收。
8. 最终把对象哈希、交付回执、验收摘要、结算摘要上链。

## 7.3 非托管 / 敏感商品的公平闭环

如果卖方不托管原始数据，就不能再用“发密钥”模型。

此时交易完成条件必须改成“受控能力被成功提供并留下正式回执”：

- API：成功返回符合约定的结果，并记录首次成功调用回执
- 模板查询：模板成功执行、输出边界通过、结果摘要生成
- 沙箱：会话成功开通、执行成功、导出受限结果
- 报告：结果对象生成、签收或自动验收
- 敏感执行：attestation / execution receipt / result review 均完成

如果卖方中途 abort、回执缺失、执行失败且超时不修复：

- 自动退款买方
- 扣卖方保证金
- 进入信誉扣分与争议链路

## 7.4 不允许单边“说完成就完成”

以下状态不得由单侧主观决定：

- `FundsLocked`
- `GrantedOrDelivered`
- `Accepted`
- `Settled`
- `Destroyed`

必须绑定：

- 系统回执
- 对手方确认
- 自动验收规则
- 外部事实回执
- 审批票据

## 8. 敏感数据完整交易链监控闭环

## 8.1 监控重点改变

对敏感数据，平台监控目标从“是否下载了文件”改成：

- 是否合法准入
- 是否存在处理依据
- 是否存在安全预览
- 是否按受控路径执行
- 是否命中结果披露审查
- 是否按期撤权
- 是否存在销毁/保留证明

## 8.2 敏感链路必监控对象

- `SensitiveHandlingPolicy`
- `LegalBasisEvidence`
- `SafePreviewArtifact`
- `SensitiveExecutionPolicy`
- `AttestationRecord`
- `ResultDisclosureReview`
- `DestructionAttestation`

## 8.3 敏感链路强告警场景

- 敏感商品上架但无处理依据
- 敏感商品存在副本下载路径
- 敏感执行发生但无执行策略
- 结果导出发生但无披露审查
- 到期后授权未回收
- 结果对象应销毁但未形成销毁证明
- 审计日志中出现疑似敏感正文泄漏

## 10. 监控指标体系

平台必须同时定义 **技术指标、业务指标、风险指标、敏感指标**。

## 10.1 技术指标

- 订单创建接口成功率
- 支付回调成功率
- Fabric commit 成功率
- Fabric commit 延迟
- chaincode event 消费延迟
- outbox 发布成功率
- dead letter 堆积量
- 查询执行成功率
- API 开通成功率

## 10.2 业务指标

- 各状态停留时长
- `FundsLocked -> GrantedOrDelivered` 转化率
- `GrantedOrDelivered -> Accepted` 转化率
- 自动验收比例
- 争议率
- 退款率
- 赔付率
- 保证金扣罚率

## 10.3 公平性与信任指标

- 卖方超时交付率
- 买方超时验收率
- 交付后争议率
- 敏感交易异常导出拦截率
- 证据缺失率
- 链上链下一致性失配率

## 10.4 敏感链路指标

- 无处理依据上架拦截次数
- 安全预览生成成功率
- 结果披露审查通过率
- 到期撤权成功率
- 销毁/保留证明完备率

## 11. 自动化监控动作

监控必须驱动动作，而不是停留在图表。

### 11.1 自动阻断

- 敏感商品命中非法交付路径
- 结果导出无披露审查
- 风险冻结对象继续发放授权

### 11.2 自动撤权

- 订单过期
- 订阅到期
- 支付逆转
- 裁决要求回收访问

### 11.3 自动进入争议

- 超时未交付
- 交付对象缺失
- 执行回执缺失
- 外部事实失败且未补偿

### 11.4 自动补偿/退款建议

- 卖方未履约
- 受控执行超时失败
- 平台执行故障导致买方无法使用

### 11.5 自动锚定

- 合同快照
- 订单关键状态
- 交付回执摘要
- 验收摘要
- 结算摘要
- 争议裁决摘要
- 敏感链路证明摘要

## 12. Fabric 在完整交易链监控中的作用

Fabric 不负责保存全部监控数据正文，但很适合做“关键承诺和关键状态摘要”的可信确认层。

平台应把以下动作映射为链码事件：

- `ProductRegistered`
- `ProductValidated`
- `OrderCreated`
- `FundsLocked`
- `DeliveryPrepared`
- `AccessGranted`
- `AcceptancePassed`
- `SettlementReleased`
- `AccessRevoked`
- `DisputeOpened`
- `DisputeClosed`

应用层收到以下信号后，驱动链下联动：

1. `Submit`
2. `CommitStatus`
3. `ChaincodeEvent`

链下联动对象包括：

- PostgreSQL 业务状态
- MinIO / 对象存储对象状态
- API 网关 token
- 沙箱会话
- 查询面授权
- 审计系统
- 一致性系统

## 13. 可观测性落地要求

## 13.1 Trace 传播要求

平台必须统一采用 `traceparent` 贯穿：

- 网关
- BFF
- 订单服务
- 支付服务
- 交付服务
- 查询/执行服务
- 链适配器
- worker
- Kafka consumer
- 审计服务

## 13.2 语义规范要求

统一采用 `OpenTelemetry` 语义规范记录：

- HTTP
- DB
- Messaging
- Job / Task
- External RPC

对 MQ 链路特别要求：

- 每条 outbox / Kafka 事件必须带 `event_id`
- 每条消息必须带 `trace_id`
- 消费者必须保留 `consumer_group / topic / partition / offset`

## 13.3 告警设计要求

至少对以下场景建立正式告警规则：

- 状态机停滞超时
- 双层权威失配
- dead letter 堆积
- 支付回调连续失败
- Fabric commit 失败率异常
- 关键链码事件缺失
- 敏感结果导出绕过审查
- 销毁证明缺失

## 14. 数据模型建议

在现有设计基础上，建议追加或强化以下对象：

- `TradeLifecycleCheckpoint`
  - 记录订单在关键里程碑上的时间点、责任方、完成证据与超时状态
- `ExternalFactReceipt`
  - 统一表达支付、打款、销毁、证书、远程证明等外部事实回执
- `FairnessIncident`
  - 统一表达迟交付、恶意拒收、结果绕过、执行中断等公平性事件
- `MonitoringPolicyProfile`
  - 定义不同商品类型、敏感等级和交付路径适用的监控策略
- `ChainProjectionGap`
  - 记录链上已确认、链下未投影，或链下已推进、链上未确认的失配

如果短期不单独建表，至少也应以正式 JSON 快照或审计对象表达，不能只靠工单备注。

## 16. 与现有设计的兼容结论

这套方案与当前平台既有设计是兼容的，而且应视为对既有设计的“上位收口”：

1. 与“双层权威模型”兼容：
   - 数据库继续是运营主状态，Fabric 继续是可信确认层。
2. 与“审计、证据链与回放”兼容：
   - 交易链监控把审计对象作为监控闭环的一部分，而不是替代审计域。
3. 与“日志、可观测性与告警”兼容：
   - OTel / Prometheus / Loki / Tempo 负责运行观测，不改变原有选型。
4. 与“敏感数据处理与受控交付”兼容：
   - 把敏感执行策略、结果披露审查、销毁证明正式纳入交易链监控。

因此，这不是一套与原设计冲突的新体系，而是把之前分散在：

- 状态机
- 审计
- 一致性
- 日志
- 敏感数据治理

中的要求收口成一套“可观测、可控制、可仲裁、可证明”的交易链监控总方案。

## 17. 参考实践

以下资料说明这套设计在工程上是可行的：

- W3C Trace Context：统一分布式追踪上下文标准
- OpenTelemetry：统一 traces / metrics / logs 与 context propagation
- Hyperledger Fabric Gateway：支持提交交易、等待 commit status、接收 chaincode events
- Prometheus + Alertmanager：支持规则告警、pending/firing 和通知编排

本平台的做法不是直接照搬任何单一产品，而是把这些成熟能力叠加到你当前已经确定的：

- PostgreSQL 主状态
- Fabric 联盟链可信底座
- Kafka 事件总线
- 审计证据链
- 敏感数据受控交付

之上，形成一套适合数据交易平台的正式治理方案。
