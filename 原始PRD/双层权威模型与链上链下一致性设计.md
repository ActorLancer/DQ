# 双层权威模型与链上链下一致性设计

- 文档名称：双层权威模型与链上链下一致性设计
- 适用范围：基于区块链技术的数据交易平台 V1 / V2 / V3
- 版本：D1
- 日期：2026-04-08
- 上位输入：PRD-R1《基于区块链技术的数据交易平台-需求清单》

---

## 1. 文档目标

本文件用于统一解决“交易状态到底谁说了算”的问题，明确：

1. 业务运行态由谁负责。
2. 可信证明由谁负责。
3. 支付、证书、存储等外部事实由谁负责。
4. 数据库、Fabric、Kafka、worker、dead letter、回放、对账之间如何配合。

---

## 2. 总体结论

平台采用：

> **双层权威模型 + 外部事实来源矩阵 + 事务性 Outbox + Kafka 事件总线 + Dead Letter + 回放/对账/补偿闭环**

统一原则如下：

1. **链下数据库是业务运营主状态机。**
2. **Fabric 是关键承诺与可信证明确认层。**
3. **支付、CA、对象存储等外部系统对其事实结果拥有来源权威。**
4. **Kafka 是事件分发与解耦总线，不是业务权威源。**

---

## 3. 三类权威

## 3.1 业务运营权威

载体：

- PostgreSQL

负责：

- 订单
- 合同正文
- 授权明细
- 支付流程状态
- 交付状态
- 验收状态
- 争议状态
- 会话状态
- 审计原文索引

特点：

- 高频变化
- 允许重试、补偿、人工介入
- 需要支撑联查、调试、运营台和页面实时展示

结论：

- 平台内部页面、接口、账务、交付和调试默认以数据库状态为准。

## 3.2 可信证明权威

载体：

- Hyperledger Fabric

负责：

- 合同快照哈希
- 授权快照哈希
- 订单关键状态摘要
- 交付回执摘要
- 验收结果摘要
- 结算批次摘要
- 争议裁决摘要
- 审计批次根
- 证书签发/吊销摘要

特点：

- 不可篡改
- 可多组织共享
- 用于外部可验证、事后举证和监管/合作方证明

结论：

- 对外可信承诺与不可抵赖证明，以 Fabric 已确认状态为准。

## 3.3 外部事实来源权威

载体：

- 支付渠道 / 银行
- Fabric CA / MSP
- 对象存储 / WORM 存储
- 第三方签章 / 时间戳 / 合规服务

负责：

- 支付成功/失败
- 退款成功/失败
- 打款成功/失败
- 证书签发/吊销
- 对象删除/保留完成
- 第三方签章完成

结论：

- 这些对象的“事实是否成立”以外部回执为准。
- 数据库保留运营镜像，Fabric 保留摘要证明。

## 4. 统一状态模型

关键业务对象必须明确拆分以下状态语义：

## 4.1 业务状态

- 字段可沿用现有 `status` / `payment_status` / `settlement_status`
- 语义：平台当前业务运行态

## 4.2 证明提交状态

建议统一命名：

- `proof_commit_state`

枚举建议：

- `not_required`
- `pending`
- `submitted`
- `confirmed`
- `failed`
- `dead_letter`

## 4.3 外部事实状态

建议统一命名：

- `external_fact_status`

枚举建议：

- `n/a`
- `pending`
- `confirmed`
- `failed`
- `reversed`

## 4.4 证明提交策略

建议统一命名：

- `proof_commit_policy`

枚举建议：

- `none`
- `async_evidence`
- `required_before_effective`
- `required_before_external_visibility`

## 4.5 一致性校验状态

建议统一命名：

- `reconcile_status`

枚举建议：

- `clean`
- `pending_check`
- `mismatch`
- `pending_manual`
- `repaired`

## 4.6 权威模型标签

建议统一命名：

- `authority_model`

默认值：

- `dual_layer`

## 5. 对象分类规则

## 5.1 纯链下业务权威

这类对象不应等待链确认才成立：

- 注册、登录、会话、设备
- 主体资料原文
- 商品草稿
- 报价和询价
- 支付意图
- 下载令牌
- API Key
- 沙箱席位
- 审计原始日志

规则：

- 数据库写入成功即可进入业务有效态。
- 链上仅做摘要或不做链上存证。

## 5.2 链下业务权威 + Fabric 可信确认

这类对象采用双层权威：

- 合同摘要
- Usage Policy 生效摘要
- 订单关键状态摘要
- 交付回执摘要
- 验收结果摘要
- 结算批次摘要
- 争议裁决摘要
- 审计清单摘要

规则：

- 平台运营按数据库推进。
- 对外可信证明按 Fabric 确认结果认定。

## 5.3 外部事实权威 + 平台镜像 + 链上摘要

这类对象不能简单说数据库或链上权威：

- `payment_succeeded`
- `refund_succeeded`
- `payout_succeeded`
- `certificate_revoked`
- `object_destroy_completed`

规则：

- 事实以外部回执为准。
- 数据库保存业务镜像。
- Fabric 写摘要证明和审计引用。

## 6. V1 推荐实现

## 6.1 事务性 Outbox

每个写请求在同一数据库事务中完成：

1. 写业务对象
2. 写审计事件
3. 写 `ops.outbox_event`

只有事务提交成功，业务状态才成立。

## 6.2 Publisher Worker

worker 负责：

- 读取待发布的 `outbox_event`
- 发布到 Kafka
- 更新 `published_at`
- 记录发布尝试
- 失败时退避重试

推荐实现：

- 多实例 worker
- `FOR UPDATE SKIP LOCKED`
- 必要时配合 PostgreSQL advisory lock

## 6.3 Kafka 事件总线

Kafka 负责：

- 跨服务分发
- 解耦链适配器、审计、通知、搜索、风控
- 支持下游重放
- 隔离失败消费者

但 Kafka 不负责：

- 定义业务状态权威
- 替代数据库事务

## 6.4 下游消费者

典型消费者包括：

- Fabric 写链服务
- 审计锚定批次服务
- 搜索索引服务
- 通知服务
- 风控联动服务
- 对账/修复服务

所有消费者都必须：

- 使用 `event_id` / `trace_id`
- 做幂等去重
- 写消费结果

## 6.5 Dead Letter

必须同时保留两层：

- 数据库 `ops.dead_letter_event`
- Kafka DLQ topic

作用：

- 数据库层支撑运营处理、联查、回放、人工修复
- Kafka 层支撑流式消费者失败隔离

## 7. 推荐字段与运行要求

## 7.1 outbox_event

必须支持：

- `event_schema_version`
- `request_id`
- `trace_id`
- `idempotency_key`
- `authority_scope`
- `proof_commit_policy`
- `target_bus`
- `target_topic`
- `partition_key`
- `payload_hash`
- `retry_count`
- `max_retries`
- `available_at`
- `published_at`
- `last_error_code`
- `last_error_message`

生产建议：

- 高吞吐环境按 `created_at` 做分区或归档，避免表膨胀。

## 7.2 dead_letter_event

必须支持：

- `request_id`
- `trace_id`
- `failure_stage`
- `first_failed_at`
- `last_failed_at`
- `reprocess_status`
- `reprocessed_at`

## 7.3 发布尝试与消费幂等

建议显式记录：

- `OutboxPublishAttempt`
- `ConsumerIdempotencyRecord`

目的：

- 追踪发布失败点
- 防止重复消费
- 支撑重放与对账

## 8. Kafka、轮询与 CDC 边界

## 8.1 V1 默认方案

- `DB transaction + outbox + publisher worker + Kafka`

## 8.2 LISTEN / NOTIFY 用法

- 只作为 publisher 唤醒提示
- 不作为可靠消息队列

## 8.3 CDC / Debezium 用法

- 可作为 V2+ 优化方案
- 用于降低轮询延迟和减轻数据库负担
- 不替代事务性 outbox 语义

---

## 9. 分域规则

## 9.1 订单与合同

- `trade.order_main.status` 是业务状态
- `proof_commit_state` 反映订单摘要是否已进入 Fabric
- 高价值合同可切换到 `required_before_effective`

## 9.2 支付与结算

- `payment.payment_intent.status` 是平台支付流程镜像
- `external_fact_status` 以支付渠道回执为准
- `proof_commit_state` 表示支付/结算摘要是否已可信落链

## 9.3 交付与验收

- 交付页和验收页以数据库状态驱动
- 交付回执和验收摘要异步进入 Fabric
- 不允许链失败反向污染已确认的本地交付事实

## 9.4 审计与证据

- 审计原文和证据清单先写数据库和对象存储
- 锚定批次异步写 Fabric
- 失败进入锚定死信与人工介入

## 9.5 证书与身份

- 平台会话和登录以平台 IAM 为权威
- Fabric 证书状态以 Fabric CA / MSP 为来源权威
- 平台保存绑定关系和审计引用

## 9.6 跨链

- `CrossChainRequest` 是业务对象，不是普通消息
- 源侧请求状态以本平台数据库为运行态
- 目标侧 `ack` / witness / gateway receipt 是外部事实
- 可信共享摘要再进入 Fabric / 跨链审计链

---

## 10. 不允许的错误模型

平台不得混用以下错误做法：

1. 把 Kafka 当作业务主状态机
2. 把“消息已发送”当作链上确认成功
3. 把“交易已上链”理解成平台所有业务状态都必须由链事件回放得出
4. 在请求内同时写数据库、发 Kafka、调 Fabric 并假装这是一笔原子事务
5. 让链写失败直接覆盖本地已成立的业务事实
6. 让重试导致重复扣费、重复交付、重复存证

---

## 11. V1 / V2 / V3 边界

- `V1`
  - 双层权威模型落地
  - 事务性 outbox
  - Kafka 事件总线
  - Dead Letter
  - 基础回放、对账、补偿闭环
- `V2`
  - CDC / Debezium 优化
  - 更丰富的事件路由和跨域任务状态一致性
  - 训练、证明、分润纳入统一一致性模型
- `V3`
  - 跨链、跨平台、监管协同进入统一权威矩阵
  - 更复杂的外部事实来源治理

---

## 12. 闭环结论

```text
请求进入
  -> 数据库事务写业务对象 + 审计 + outbox_event
  -> publisher worker 发布 Kafka
  -> 下游消费者执行业务副作用
  -> Fabric / 支付 / CA / 存储返回确认
  -> 数据库回写 proof_commit_state / external_fact_status / reconcile_status
  -> 差异进入 dead letter / replay / reconciliation
  -> 修复结果再次写回数据库与审计链
```

新增本设计后，平台在交易、支付、交付、证书、审计和跨链等域上的状态归属将不再冲突：

- 平台内部运行态以数据库为准
- 对外可信证明以 Fabric 为准
- 外部事实以来源系统回执为准
- Kafka 只负责分发，不负责定义真相
