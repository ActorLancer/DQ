# 商品搜索、排序与索引同步设计

- 文档名称：商品搜索、排序与索引同步设计
- 适用范围：基于区块链技术的数据交易平台 `V1 / V2 / V3`
- 文档定位：搜索、排序、索引与缓存的正式设计基线

---

## 1. 目标

本设计用于统一平台在以下能力上的正式方案：

- 数据商品搜索
- 服务商品搜索
- 卖方主体搜索
- 多条件筛选与排序
- 搜索索引投影
- PostgreSQL 与专业搜索引擎的数据同步
- Redis 搜索缓存与失效
- 搜索一致性、重建、回放与对账

本文件的目标不是让搜索引擎变成新的业务真相源，而是明确：

- 搜索从哪里取数
- 搜索结果怎么排序
- 搜索索引怎么同步
- 索引、缓存和主库不一致时怎么修复

---

## 2. 总体原则

平台搜索体系采用以下边界：

1. **PostgreSQL 是主业务状态和搜索投影权威源**
2. **OpenSearch 是商品搜索专用读模型**
3. **Redis 只做搜索缓存与短时加速**
4. **Kafka 是搜索索引同步事件总线**
5. **交易、详情、下单、支付、交付等最终业务状态一律回 PostgreSQL 校验**

因此本平台的搜索架构不是：

- OpenSearch 直接做商品主库
- Redis 保存完整商品权威索引

而是：

```text
PostgreSQL 主数据
  -> 搜索投影表
  -> outbox_event
  -> Kafka
  -> Search Indexer
  -> OpenSearch 索引
  -> Redis 缓存
```

## 3. 搜索对象范围

平台搜索至少覆盖以下对象：

### 3.1 商品搜索

- 数据商品
- 服务商品
- 结果包 / 报告商品
- API / 文件 / 沙箱 / 服务类 SKU

### 3.2 卖方主体搜索

- 企业
- 机构
- 单位
- 平台合作供方

### 3.3 `V2 / V3` 增量对象

- 模型商品
- 生态合作方
- 跨平台互认主体

---

## 4. 搜索引擎分层

## 4.1 PostgreSQL 层

负责：

- 主数据
- 搜索投影表
- 标签、分类、价格、状态、质量、信誉、风控等主记录
- 搜索同步状态
- 索引回放与重建输入源
- OpenSearch 不可用时的功能降级

## 4.2 OpenSearch 层

负责：

- 全文检索
- faceted search / aggregations
- 多字段相关性排序
- function score
- `V2` 混合检索
- `V2/V3` 自动补全、同义词、拼写纠错、多语言分析

### 4.3 Redis 层

负责：

- 热门查询结果缓存
- facet 缓存
- 自动补全缓存
- 搜索结果 ID 列表短缓存
- 缓存依赖键与失效

Redis 不负责：

- 商品真相源
- 最终业务状态
- 最终搜索文档源数据

---

## 5. V1 正式方案

## 5.1 V1 生产基线

`V1` 生产环境正式采用：

- PostgreSQL 主库
- OpenSearch 商品搜索读模型
- Redis 搜索缓存

但为方便开发与降级：

- `local / demo` 允许只用 PostgreSQL 搜索投影表运行
- `staging / production` 应启用 OpenSearch

## 5.2 V1 搜索能力范围

### 支持

- 关键词搜索
- 精确筛选
- 商品、服务、卖方三类对象搜索
- 综合排序
- 质量 / 信誉 / 热度 / 上架时间排序
- facet 统计
- PostgreSQL fallback 搜索

### 不做

- 个性化推荐
- 跨模态搜索
- 复杂自定义 DSL
- 首页强制向量召回

## 6. 搜索投影设计

## 6.1 商品搜索投影

`search.product_search_document` 需要承载至少以下信息：

- 商品标题、子标题、描述
- 分类、行业、场景、标签集合
- 商品类型：数据商品 / 服务商品
- 卖方名称、卖方类型、卖方国家/地区
- SKU 类型、交付方式、权利类型
- 价格模式、价格区间、币种
- 上架状态、样例可用性、数据分级
- 质量评分
- 卖方信誉、卖方风险等级
- 热度、近期成交量
- 排序特征
- 文档版本与索引同步状态

## 6.2 卖方搜索投影

新增 `search.seller_search_document`：

- 卖方主体名称
- 卖方主体类型
- 国家 / 地区
- 行业标签
- 合规等级
- 信誉 / 风险
- 已上架商品数
- 排序特征
- 文档版本与索引同步状态

## 6.3 标签集合设计

为保证数据商品可搜索、可筛选、可排序，标签体系必须正式建模：

- 标签主数据
- 标签类型
- 标签分组
- 标签别名
- 标签状态
- 父子层级
- 商品-标签映射
- 标签来源与权重

## 7. 排序设计

## 7.1 召回与排序分离

搜索流程必须分为两层：

1. **召回层**
   - 关键词匹配
   - 模糊匹配
   - 标签命中
   - 分类命中
   - `V2` 向量召回
2. **排序层**
   - 综合排序
   - 质量排序
   - 信誉排序
   - 最新排序
   - 价格排序
   - 热度排序

## 7.2 V1 默认综合排序

建议默认公式：

```text
final_score =
  w_text * lexical_score
  + w_quality * quality_score_norm
  + w_reputation * seller_reputation_norm
  + w_hotness * hotness_score_norm
  + w_freshness * freshness_score_norm
  - w_risk * risk_penalty
```

推荐初始权重：

- 文本相关性：`40%`
- 质量评分：`20%`
- 卖方信誉：`15%`
- 热度：`10%`
- 新鲜度：`10%`
- 风险惩罚：`5%`

## 7.3 先过滤再排序

以下对象必须在排序前先被硬过滤：

- 未上架
- 已冻结
- 风险阻断
- 不满足司法辖区要求
- 不满足主体准入要求
- 已下架或已失效

---

## 8. OpenSearch 索引设计

建议索引按对象拆分：

- 商品搜索索引
- 卖方搜索索引
- `V2` 模型搜索索引

日志与审计检索不得与商品检索共用同一索引体系。

建议维护别名：

- `product_search_read`
- `product_search_write`
- `seller_search_read`
- `seller_search_write`

作用：

- 支持零停机重建索引
- 支持版本切换
- 支持故障回退

---

## 9. Redis 缓存设计

Redis 仅做缓存，不做主数据。

### 9.1 建议缓存对象

- 热门查询结果
- facet 聚合
- 自动补全
- 搜索结果 ID 列表

### 9.2 建议失效方式

- 事件驱动失效为主
- TTL 为辅

### 9.3 不允许的做法

- Redis 中保存商品权威状态
- Redis 先于 PostgreSQL 决定可售状态
- 仅依赖 TTL 不做事件失效

---

## 10. 一致性与同步设计

## 10.1 核心原则

- PostgreSQL 是搜索文档的权威源
- OpenSearch 是读投影
- Redis 是缓存
- Kafka 负责传播变更
- 搜索结果最终业务可见性由 PostgreSQL 校验

## 10.2 商品变更链路

```text
业务服务更新 PostgreSQL 主表
  -> 同事务刷新 search.product_search_document / seller_search_document
  -> 同事务写 outbox_event
  -> publisher 发布 Kafka
  -> Search Indexer 消费事件
  -> 从 PostgreSQL 读取最新搜索投影
  -> 写入 OpenSearch
  -> 更新索引同步状态
  -> Redis 失效相关缓存
```

## 10.3 搜索请求链路

```text
搜索 API
  -> 查 Redis
  -> 未命中则查 OpenSearch
  -> 返回候选实体 ID + score + facets
  -> 回 PostgreSQL 校验当前状态
  -> 过滤失效 / 冻结 / 不可见对象
  -> 返回最终结果
```

## 10.4 文档版本

每个搜索投影对象必须具备：

- `document_version`
- `source_updated_at`
- `index_sync_status`
- `indexed_at`
- `last_index_error`

用途：

- 幂等写入
- 丢弃过期事件
- 重建与对账

## 10.5 回放与修复

必须支持：

- 索引重建
- 单对象重投递
- 缓存失效重放
- PostgreSQL 与 OpenSearch 版本对账
- `dead letter` 重处理

## 13. 与现有架构的闭环关系

本设计与现有体系不冲突，关系如下：

- 与“双层权威模型”兼容：搜索引擎不是权威源
- 与 `outbox + Kafka + dead letter` 兼容：搜索同步走同一事件机制
- 与审计体系兼容：索引修复、别名切换、缓存失效都要留痕
- 与支付、交付、风控兼容：搜索结果最终可售性由 PostgreSQL 当前状态决定

因此，新增本设计后，平台在搜索领域形成闭环：

`主数据 -> 搜索投影 -> 事件同步 -> 搜索引擎 -> 缓存 -> 查询 -> PostgreSQL 最终校验 -> 审计/对账/修复`

---

## 14. 一句话结论

平台搜索正式采用：

**`PostgreSQL 主数据权威 + OpenSearch 商品搜索读模型 + Redis 缓存 + Kafka/outbox 同步 + PostgreSQL 最终业务校验`**

这既满足首版商品搜索能力，也为 `V2/V3` 的混合检索和更复杂排序留下了稳定演进路径。
