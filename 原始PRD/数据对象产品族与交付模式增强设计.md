## 1. 文档目标

本文件用于把“平台到底交易什么样的数据对象、采用哪些交付模式、哪些进入 V1 正式商品化、哪些保留到 V2/V3”重新收敛为统一口径。

本文件优先解决四个问题：

1. 数据对象不应再被简单理解为“一个文件包”。
2. 平台必须同时支持“副本交付、只读共享、API 服务、模板查询、结果交付”等不同形态。
3. 交易类型要和对象类型、交付方式、验收方式、策略约束一一对应。
4. 分阶段实现时，必须保留全量对象模型，但不能把 V1 做成所有高级能力的大杂烩。

---

## 2. 设计结论

### 2.1 五类基础数据对象 family

平台围绕数据对象的业务建模应统一收敛为五类基础 family：

1. 文件 / 对象包产品
2. 表 / 库 / 视图 / 数据共享产品
3. API / 服务产品
4. 受限查询 / 受控计算产品
5. 结果 / 合成 / 特征 / 模型衍生产品

这五类是“顶层对象家族”，不是版本阶段。  
`V1` 只把其中适合标准化闭环的部分正式商品化，`V2/V3` 再扩展到更复杂的执行形态。

### 2.2 七类标准交易方式

平台应统一支持以下七类交易方式：

1. 一次性文件快照出售
2. 版本订阅 / 周期更新
3. 零拷贝只读共享
4. API / 服务按调用或按周期计费
5. 模板查询 / clean room lite / 受限结果交付
6. 联邦学习 / MPC / 受控任务协作
7. 结果产品 / 合成数据 / 特征结果 / 模型结果交付

### 2.3 分阶段落地原则

- `V1` 正式商品化：
  - 文件快照 / 版本订阅
  - 零拷贝只读共享
  - API / 服务
  - 查询沙箱 / 模板查询 lite
  - 固定报告 / 结果产品
- `V2` 正式商品化：
  - 完整 clean room / 模板查询 / TVF / analysis rule
  - 模型 API / 模型服务
  - 受控计算任务
  - 联邦学习
  - MPC 结果
- `V3` 正式商品化：
  - 更复杂的多方协作结果
  - 更复杂的跨平台共享
  - 更复杂的模型与结果编排

### 2.4 当前完成度与后续收口要求

当前已设计完成并冻结：

- 五类基础对象 family
- 七类标准交易方式
- `V1` 正式商品化边界
- 数据库和领域模型骨架
- 基础交易链路、权限点和页面入口

当前尚未彻底设计完成、但已明确必须在 `V2/V3` 补齐：

- 只读共享的完整异构协议矩阵与多连接器治理
- 模板查询 `lite` 到完整 `clean room` 的规则、执行和证据体系
- 版本订阅的复杂 revision 治理与跨平台协同
- 不同交付模式下完整的授权/验收/退款/赔付规则引擎

## 3. 五类基础对象家族

## 3.1 文件 / 对象包产品

**定义**

- 交付固定版本文件、副本对象包或结构化导出包。

**典型对象**

- CSV
- Excel
- JSON
- Parquet
- 图片 / 音频 / 视频
- 压缩包
- 数据库导出包
- 脱敏导出包
- 合成数据文件包
- 特征文件包

**适用场景**

- 历史快照
- 公开衍生包
- 标签数据
- 标注数据集
- 批量对接前的离线导出

**主要风险**

- 副本扩散
- 再分发
- 越权训练
- 无法撤回

**设计要求**

- 强版本号
- 哈希 / 指纹 / 水印
- 下载令牌
- 有效期 / 次数限制
- 明确禁止再分发与默认训练

## 3.2 表 / 库 / 视图 / 数据共享产品

**定义**

- 交付的是只读数据共享对象，不是原始文件包。

**典型对象**

- 只读表
- 只读视图
- secure view
- linked dataset
- Delta share
- read-only schema
- 共享 topic / stream 的只读订阅入口

**适用场景**

- 持续可查的结构化数据
- 双方本来就在数仓 / 湖仓 / 数据共享体系内
- 卖方希望零拷贝、可撤权、少导出

**主要风险**

- 共享对象越权扩散
- 读侧权限配置错误
- subscriber 侧复制外流
- 共享对象生命周期和订单脱钩

**设计要求**

- 只读共享
- 可撤权
- 共享对象和订单绑定
- 共享协议快照
- subscriber / recipient 定向授权
- 到期自动断权

## 3.3 API / 服务产品

**定义**

- 买方购买调用能力、查询能力或持续服务，不直接获得底层全量数据。

**典型对象**

- REST / GraphQL / RPC / MQ / WebSocket
- 评分 API
- 检索 API
- 指标查询 API
- 画像服务
- 结果回调服务

**适用场景**

- 实时数据
- 持续更新数据
- 使用权强于复制权
- 卖方希望严格控制用途

**主要风险**

- 凭证共享
- 超配额调用
- 批量拉平形成副本库
- SLA 争议

**设计要求**

- `Application` 级授权
- API Key / OAuth / OIDC
- 配额 / QPS / 限流 / IP 白名单
- 调用日志、计费事件、断权
- 明确是否允许批量明细拉取

## 3.4 受限查询 / 受控计算产品

**定义**

- 卖的不是数据副本，而是在受控环境下执行指定查询、模板分析或算法任务的能力。

**典型对象**

- 查询沙箱
- SQL 模板
- TVF 模板
- clean room lite
- query template
- 白名单分析任务
- 受限导出结果

**适用场景**

- 买方只需要分析结论，不需要底层全量数据
- 数据敏感、不能直接复制出域
- 需要严格限制输出形态和导出范围

**主要风险**

- 查询模板越界
- 输出逆向识别
- 结果泄漏原始明细
- 算法未审

**设计要求**

- 模板白名单
- 输出边界
- 行列数限制
- 导出次数限制
- 审计和回放
- 高级计算任务进入 `V2`

## 3.5 结果 / 合成 / 特征 / 模型衍生产品

**定义**

- 交付的是分析结论、聚合结果、合成数据、特征结果、模型服务或模型工件，而不是原始数据主体本身。

**典型对象**

- 固定报告
- 聚合统计包
- 匹配结果
- 去标识化结果
- 合成数据集
- 特征结果包
- Embedding / 向量结果
- 训练好的模型
- 模型 API / 模型服务期

**适用场景**

- 买方只要结果不要明文
- 卖方希望最大限度减少原始数据暴露
- AI / 风控 / 推荐 / 匹配等能力输出

**主要风险**

- 结果可逆
- 结果再出售
- 模型逆向
- 权重泄漏

**设计要求**

- 明确原始数据关系
- 明确可逆性和推断风险
- 明确是否允许导出、商用、二次训练
- 结果类 `V1` 可以先做报告 / 聚合 / 合成 / 特征文件包
- 模型服务、模型工件正式商品化进入 `V2`

## 4. 七类标准交易方式

## 4.1 一次性文件快照出售

**定义**

- 卖方交付某一时点固定版本的数据包。

**适用**

- 历史档案
- 标注数据
- 合成数据包
- 静态研究数据

**V1**

- 正式支持

## 4.2 版本订阅 / 周期更新

**定义**

- 买方不是买单次下载，而是买一组持续更新的 revision / release。

**适用**

- 风险名单
- 指标库
- 周期目录
- 数据 revision feed

**V1**

- 正式支持
- 先支持“按版本顺序交付”和“到期断权”

## 4.3 零拷贝只读共享

**定义**

- 卖方共享表 / 视图 / linked dataset / datashare / read-only schema。

**适用**

- 结构化共享
- 长期只读查询
- 双方已有数仓 / 湖仓能力

**V1**

- 正式支持，但限于已接入共享协议或连接器能力成熟的对象

## 4.4 API / 服务按调用计费

**定义**

- 买方购买调用能力或服务期限。

**适用**

- 实时查询
- 持续更新
- 指标服务
- 评分服务

**V1**

- 正式支持

## 4.5 模板查询 / clean room lite

**定义**

- 买方只允许执行被批准的查询模板，并获得受限结果。

**适用**

- 交叉客群
- 经营分析
- SQL 模板查询
- 轻量 clean room

**V1**

- 正式支持 `lite` 版本
- 以查询沙箱、模板 SQL、结果边界控制为主

**V2**

- 才进入完整 clean room、analysis rule、TVF、差分隐私等高级能力

## 4.6 联邦学习 / 安全计算任务

**定义**

- 买方提交协作训练或多方安全计算任务，不接触原始数据。

**V2**

- 正式支持

## 4.7 结果产品 / 合成数据 / 特征结果

**定义**

- 交付的是结果、合成集、特征包、报告或模型输出。

**V1**

- 正式支持结果类和衍生文件类

**V2**

- 扩展到模型服务和模型工件

## 5. 和当前平台架构的兼容性判断

这套设计与当前已冻结架构兼容，原因如下：

1. **与数据本体存储与信任边界兼容**
   - 文件包可走平台托管 / 卖方自持
   - 只读共享可走卖方自持 / 第三方可信存储
   - API 产品可走卖方 API / 平台代理
   - 模板查询可走查询沙箱 / 受控执行
   - 结果产品可走对象存储、报告交付、服务返回

2. **与双层权威模型兼容**
   - 业务主状态仍在 PostgreSQL
   - Fabric 只锚定关键承诺和摘要
   - 共享授权、API 凭证、模板查询授权、结果交付都可作为关键摘要上链

3. **与现有支付与结算兼容**
   - 文件/结果产品：一次性或分批
   - 订阅/共享/API：周期账单
   - 模板查询：按席位 / 按次 / 按模板
   - 联邦/安全计算：项目制 / 任务制

4. **与搜索、推荐、审计兼容**
   - `product_type`、`sku_type`、`delivery_mode`、标签体系足以承接多类型发现
   - 推荐仍然是链下发现服务，不是交易真相源
   - 审计与证据链可覆盖共享授权、订阅交付和模板查询

## 6. 对当前设计的修正

## 6.1 现有设计里应该保留的部分

- 文件副本、API/订阅、查询沙箱、固定报告四类 `V1` 标准闭环
- `DataAsset -> AssetVersion -> Product -> SKU -> Contract / UsagePolicy -> Order -> Delivery`
- 平台不是默认明文托管者
- 高级计算、联邦、MPC、模型服务放 `V2`

## 6.2 现有设计里需要增强的部分

1. `V1` 不能只理解为“文件/API/沙箱/报告”四个死分类，而应理解为：
   - 文件快照 / 版本订阅
   - 只读共享
   - API / 服务
   - 查询沙箱 / 模板查询 lite
   - 固定报告 / 结果产品

2. `DataAssetVersion` 之下需要显式描述可交付对象：
   - 文件对象
   - 表 / 视图 / 共享对象
   - API endpoint
   - 查询模板
   - 结果对象

3. 订单交付模型需要补齐：
   - 共享授权
   - revision 订阅
   - 模板查询授权

4. 结果产品不能再只等同于“固定报告”：
   - 还应覆盖聚合结果、合成数据、特征结果、匹配结果

## 8. 数据库与对象模型最小增量

为了支撑本轮增强设计，数据库与领域模型至少要补齐以下对象：

1. `catalog.asset_object_binding`
   - 描述某个 `AssetVersion` 下可交付的逻辑对象
   - 支持 `file / table / view / secure_view / linked_dataset / delta_share / api_endpoint / sql_template / tvf_template / result_dataset / synthetic_dataset / feature_set`

2. `delivery.data_share_grant`
   - 描述零拷贝共享授权、recipient、share locator、到期撤权

3. `delivery.revision_subscription`
   - 描述版本订阅、交付游标、最后成功版本、下一次计划交付

4. `delivery.template_query_grant`
   - 描述模板查询授权、模板摘要、输出边界、执行配额

5. `catalog.product_sku.trade_mode`
   - 显式标记 `snapshot_sale / revision_subscription / read_only_share / api_metered / template_query / result_delivery / controlled_compute`

---

## 9. 推荐冻结口径

平台应正式冻结以下判断：

1. “交易对象”不再等于“文件数据集”。
2. “结果产品”是正式一等商品，不只是固定报告的备注。
3. “零拷贝共享”是正式交易方式，不是例外场景。
4. “模板查询 lite” 是 `V1` 能力，“完整 clean room” 是 `V2` 能力。
5. `V1` 不上联邦学习、MPC、模型服务正式交易，但对象模型必须从一开始预留。

---

## 10. 结论

这次增强后的统一口径是：

**平台交易的是“可验证的数据使用权 / 共享权 / 访问权 / 查询权 / 结果权”，而不是单一的文件副本。**

对 `V1` 来说，应正式商品化的不是原来狭义的四类，而是五类：

1. 文件快照 / 版本订阅
2. 只读共享
3. API / 服务
4. 查询沙箱 / 模板查询 lite
5. 固定报告 / 结果产品

这既保留了原先架构里可控的 `V1` 范围，又把后续最容易返工的“共享、revision、结果类商品”一次性纳入统一骨架。
