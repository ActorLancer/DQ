## 2. 技术选型总原则

- 业务主状态权威在 `PostgreSQL`，不是在 `Kafka`、`OpenSearch`、缓存或链上回执。
- `Fabric` 是关键承诺、状态摘要和可信证明确认层，不保存原始数据本体。
- 搜索、推荐、支付、审计、链适配都必须服从“最终回 PostgreSQL 业务校验”的原则。
- 多语言允许存在，但必须分工明确，不得多门语言重复实现一套交易主链路。
- `V1` 优先完成标准交易闭环；`V2/V3` 的隐私计算、联邦学习、受控执行增强按阶段引入，不抢跑。

## 3. 平台核心技术底座

### 3.1 必须采用

- 区块链：`Hyperledger Fabric`
- 主数据库：`PostgreSQL`
- 对象存储：`MinIO / S3 兼容对象存储`
- 事件总线：`Kafka`
- 缓存：`Redis`
- 搜索读模型：`OpenSearch`
- 身份认证：`Keycloak`
- 可观测性：`OpenTelemetry + Prometheus + Alertmanager + Grafana + Loki + Tempo`
- 开发态支付：`Mock Payment Provider`

### 3.2 职责边界

- `PostgreSQL`：控制面主库，保存主数据、生命周期对象、配置、审计索引、搜索投影和推荐主存
- `MinIO`：原始文件、交付对象、预览文件、报告结果、模型制品、证据原文
- `Kafka`：outbox 分发、搜索同步、推荐行为流、审计联动、死信隔离
- `Redis`：热点缓存、推荐短缓存、幂等辅助、限流辅助
- `OpenSearch`：商品/服务/卖方/模型搜索读模型与候选召回
- `Fabric`：订单摘要、授权摘要、验收摘要、证据批次根、可信确认结果

## 4. 语言分工

### 4.1 Rust

`Rust` 是平台核心后端主语言，负责：

- `api-gateway`
- `catalog-service`
- `trade-service`
- `billing-service`
- `audit-service`
- `search-service`
- `recommendation-service`
- `consistency-service`
- `storage-gateway`
- 平台主状态机、生命周期对象编排、支付编排、权限校验

### 4.2 Go

`Go` 主要用于 `Fabric` 域和部分中间件适配：

- chaincode
- `fabric-adapter`
- `fabric-event-listener`
- `fabric-ca-admin`
- 链上摘要写入、链码事件监听、证书与 CA 治理
- 部分对接型适配服务

### 4.3 Python

`Python` 用于离线与数据处理：

- 数据清洗、标准化、质量分析
- 报表任务
- 推荐离线任务
- 模型处理与分析作业

### 4.4 TypeScript

`TypeScript + React/Next.js` 用于：

- 门户前台
- 卖方工作台
- 买方工作台
- 运营后台
- 审计/监管视图
- 可观测性与开发者页面

### 4.5 Java

`Java` 允许引入，但限定为：

- 接入 JVM 生态开源项目
- 数据空间连接器、元数据治理、专项规则或集成服务
- 未来 `V2/V3` 的扩展型治理能力

不建议让 `Java` 再实现交易主链路。

### 4.6 Haskell

`Haskell` 可保留为实验性工具语言，仅适用于：

- 规则 DSL 原型
- 离线一致性校验器
- 合同/策略形式化验证原型

不进入 `V1` 核心交易链路，不掌握订单、合同、授权、交付、结算主状态。

## 6. 本地与联调环境

### 6.1 本地开发

建议使用 `Docker Compose` 启动：

- `PostgreSQL`
- `Redis`
- `Kafka`
- `MinIO`
- `OpenSearch`
- `Keycloak`
- `Prometheus`
- `Alertmanager`
- `Grafana`
- `Loki`
- `Tempo`
- `Mock Payment Provider`
- `Fabric` 测试网络

### 6.2 联调 / 测试

建议迁移到：

- `Kubernetes`
- `Helm`
- 独立数据库、消息总线、搜索、对象存储、认证中心
- `Fabric` 测试链网络

## 7. 隐私计算、联邦学习与沙箱

### 7.1 V1 建议

- 查询沙箱 / 模板执行隔离：优先评估 `gVisor`
- `OPA` 只预留接入边界，不作为 `V1` 正式运行时硬依赖

### 7.2 V2 重点 PoC

- `SecretFlow`：隐私计算、MPC、HE、TEE、PSI、联合分析
- `OpenFL`：联邦学习训练协作
- `Kata Containers`：高隔离执行环境

### 7.3 当前不建议优先引入

- `PySyft` 作为主集成对象
- `MPC / TEE / FL / ZKP` 正式交易闭环提前进入 `V1`

## 8. 外部开源项目引入判断

### 8.1 值得评估

- `Eclipse EDC`
  - 用于未来数据空间连接器和跨组织共享交换
- `DataHub`
  - 用于未来元数据治理、目录发现和采集增强

### 8.2 主要作为参考

- `Hyperledger FireFly`
  - 作为链上链下流转和多方流程编排参考
- `CKAN`
  - 作为数据目录与门户信息架构参考
- `Ocean Protocol`
  - 作为 `Compute-to-Data` 思路参考

## 9. 当前正式定稿建议

**主组合：**

- 核心后端：`Rust`
- Fabric 与链码：`Go`
- 数据处理与离线：`Python`
- 前端：`TypeScript`
- JVM 生态能力承载：`Java`
- 实验型规则工具：`Haskell（可选）`

**基础设施：**

- `PostgreSQL + MinIO + Kafka + Redis + OpenSearch + Keycloak + OpenTelemetry + Prometheus + Alertmanager + Grafana + Loki + Tempo + Fabric`

## 10. 一句话结论

当前阶段最稳妥的技术路线是：

**`Rust 承担平台主业务，Go 承担 Fabric 与链交互，Python 承担数据处理与离线，TypeScript 承担前端，Java 只在确有 JVM 开源项目依赖时引入，Haskell 只保留为可选实验工具。`**
