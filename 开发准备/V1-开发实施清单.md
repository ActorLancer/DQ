## 2. 开发启动前必须冻结的单一事实源

以下内容在进入数据库设计、接口设计、页面开发和联调前必须视为只读真值：

- 权利枚举：`use / access / query / result_get / internal_share_limited`
- 标准 SKU：`FILE_STD / FILE_SUB / SHARE_RO / API_SUB / API_PPU / QRY_LITE / SBX_STD / RPT_STD`
- 默认授权模板固定值：
  - `FILE_SUB -> LICENSE_FILE_USE_V1`
  - `SBX_STD -> LICENSE_SANDBOX_USE_V1`
  - `RPT_STD -> LICENSE_RESULT_USE_V1`
- API 模板命名：
  - `API_SUB -> *_API_SUB_V1`
  - `API_PPU -> *_API_PPU_V1`
- 生命周期主对象：
  - `Order`
  - `Digital Contract`
  - `Authorization`
  - `Delivery`
  - `Settlement`
  - `Dispute`
  - `Billing Event`
- 统一一致性状态字段族：
  - `authority_model`
  - `proof_commit_policy`
  - `proof_commit_state`
  - `external_fact_status`
  - `reconcile_status`
  - `idempotency_key`
  - `request_id`
  - `trace_id`
  - `status_version`

## 3. V1-Core 开发顺序建议

### 3.1 第一批：主数据与目录骨架

- Tenant / Party / Department / User / Application / Connector / Execution Environment
- Data Resource / Data Product / Product SKU
- 商品元信息、标签、数据契约、质量报告
- 8 个标准 SKU 的商品配置页与模板绑定页

验收口径：

- 可以创建标准产品
- 可以创建标准 SKU
- 模板绑定规则能拦截错误模板族

### 3.2 第二批：交易闭环骨架

- Inquiry / Order / Digital Contract / Usage Policy
- Payment Intent / Escrow / Deposit
- Authorization / Delivery / Acceptance / Settlement / Dispute
- Billing Event

验收口径：

- 可以从商品下单到结算跑通一条标准链路
- 统一状态字段族能在订单、合同、授权、交付、结算、争议、账单对象上落地
- `current_state` 和各业务域子状态可联动，不出现双主状态机

### 3.3 第三批：交付与控制面

- 文件交付
- API 开通
- 只读共享开通
- 模板查询授权
- 查询沙箱开通
- 报告/结果产品交付

验收口径：

- `SHARE_RO / QRY_LITE / SBX_STD / RPT_STD` 都能以独立 SKU 进入交付
- 不同 SKU 的验收模板与退款模板独立可配
- 到期断权覆盖用户、应用、连接器和环境

### 3.4 第四批：审计、一致性与支付

- Audit Event / Evidence Package / 回放
- 双层权威一致性
- Payment Provider 抽象 / Mock PaymentProvider
- 基础对账与人工打款

验收口径：

- 外部事实能回写 `external_fact_status / reconcile_status`
- 链上摘要能回写 `proof_commit_state`
- 重复回调和重放不会重复扣费、重复交付、重复结算

## 4. 接口冻结前检查清单

- 商品接口只把 `product` 当作目录和展示事实源
- 交易接口只把 `sku` 当作下单、授权、交付、验收、账单和结算事实源
- 不对外并列暴露正式 `billing_mode`
- 对外只使用 `pricing_mode`
- 生命周期摘要接口只返回对象化字段：
  - `authorization_status`
  - `delivery_status`
  - `settlement_status`
  - `dispute_status`
- 不再使用旧模板名：
  - `CONTRACT_API_V1`
  - `LICENSE_API_USE_V1`
  - `ACCEPT_API_V1`
  - `REFUND_API_V1`

## 5. 数据库冻结前检查清单

- `Product` 与 `ProductSKU` 分表且职责分离
- `Authorization / Delivery / Settlement / Dispute` 作为正式实体存在
- 统一一致性状态字段族已全部落库，或通过同名视图映射落库
- `Billing Event` 与 `Settlement` 分离，且有明确关系表或归集规则
- 周期型 SKU 允许多笔 `Settlement`
- 执行或交付快照不可被后续模板或策略静默改写

## 6. 页面冻结前检查清单

- 交付页已拆开：
  - 文件交付页
  - API 开通页
  - 只读共享开通页
  - 模板查询开通页
  - 查询沙箱开通页
  - 报告/结果产品交付页
- 订单详情页同时显示：
  - `current_state`
  - `payment_status`
  - `delivery_status`
  - `acceptance_status`
  - `settlement_status`
  - `dispute_status`
- 审计联查页、一致性联查页、交易链联查页显示统一一致性状态字段族

## 7. 联调前必须覆盖的最小场景

### 7.1 首批 5 条标准场景

- 工业设备运行指标 API 订阅
- 工业质量与产线日报文件包交付
- 供应链协同查询沙箱
- 零售门店经营分析 API / 报告订阅
- 商圈/门店选址查询服务

### 7.2 必测 SKU 覆盖

- `FILE_STD`
- `FILE_SUB`
- `SHARE_RO`
- `API_SUB`
- `API_PPU`
- `QRY_LITE`
- `SBX_STD`
- `RPT_STD`

### 7.3 必测异常

- 模板绑定错误
- 重复支付回调
- 支付成功但链上摘要失败
- 链上摘要成功但数据库投影失败
- 授权已撤销但仍可调用
- 争议冻结后仍然继续结算

## 8. 当前不应提前展开的内容

以下内容不应在进入 `V1-Core` 前抢跑实现：

- V2 计算权商品化
- 联邦学习、MPC、TEE、ZKP 正式交易链路
- 生产级 Solana 展示闭环
- 自动分润引擎
- 跨链结算确认
- 复杂 BPM 审批引擎

## 9. 一句话结论

当前文档包已经足以进入 `V1-Core` 的数据库、接口、页面和联调拆解阶段；后续研发只应围绕冻结口径拆任务，不应再从实现层反向修改上位规则。
