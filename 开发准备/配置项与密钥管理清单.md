# 配置项与密钥管理清单

## 1. 文档定位

本文件用于冻结当前阶段平台配置项分类、密钥分类、存放位置、注入方式、轮换要求和禁止事项。

本文件服务于以下工作：

- 本地开发配置管理
- 测试环境配置管理
- 后续生产环境密钥治理
- 证书、令牌、下载密钥和外部集成密钥管理

本文件不替代 KMS/HSM 实现方案，不替代具体 Vault 或 Secret Manager 选型文档。

- 本文件只冻结 `V1` 的配置分层、敏感信息边界和本地/测试环境管理要求。

## 2. 总原则

### 2.1 当前冻结原则

- 配置与密钥必须分离管理
- 数据对象与长期解密能力不得由同一平面永久独占
- 证书、令牌、下载授权、服务签名密钥必须支持轮换、吊销、过期和异常停用
- 高风险配置修改必须进入审计并支持 step-up

### 2.2 当前阶段现实约束

当前阶段允许：

- 采用可审计的软件密钥管理方式
- 在本地和测试环境使用 `.env` 或容器 secret 文件

当前阶段不允许：

- 将真实生产密钥硬编码进代码仓
- 将生产证书私钥直接放在前端或普通容器镜像中
- 将数据库密码、支付签名密钥、Fabric 私钥直接打印到日志

## 3. 配置分类

统一把配置分成 4 类：

1. 普通运行配置
2. 敏感配置
3. 密钥与证书材料
4. 可派生临时凭证

## 4. 普通运行配置

## 4.1 定义

不直接导致越权、签名、解密或支付风险的配置。

示例：

- `APP_ENV`
- `APP_PORT`
- `LOG_LEVEL`
- `BASE_URL`
- `PG_HOST`
- `PG_PORT`
- `KAFKA_BROKERS`
- `MINIO_ENDPOINT`
- `OPENSEARCH_ENDPOINT`
- `KEYCLOAK_BASE_URL`

## 4.2 存放方式

允许使用：

- `.env.local`
- Docker Compose 环境变量
- Kubernetes ConfigMap

## 5. 敏感配置

## 5.1 定义

虽不属于私钥或证书材料，但泄漏后可导致环境被接管或外部系统被滥用的配置。

示例：

- `PG_PASSWORD`
- `REDIS_PASSWORD`
- `OPENSEARCH_PASSWORD`
- `KEYCLOAK_CLIENT_SECRET`
- `KEYCLOAK_ADMIN_PASSWORD`
- `MOCK_PAYMENT_SIGNING_SECRET`
- 外部支付通道 `api_key / client_secret`

## 5.2 存放方式

当前建议：

- 本地：`.env.local.secret` 或容器 secret 文件
- 测试：Docker secret / 密码文件 / 私有 CI secret
- 生产：Secret Manager / Vault / KMS 集成

禁止：

- 提交到 Git
- 放进前端构建产物
- 写入截图、工单正文、普通日志

## 6. 密钥与证书材料

## 6.1 定义

可直接用于签名、解密、证书认证或生成下游高权限访问能力的材料。

示例：

- `FABRIC_PRIVATE_KEY_PATH`
- `FABRIC_SIGN_CERT_PATH`
- `FABRIC_TLS_CERT_PATH`
- 下载密钥封装材料
- 服务签名私钥
- 渠道签名证书或私钥
- 未来 KMS 主密钥引用

## 6.2 当前处理原则

- 只能以文件或 secret volume 形式注入
- 不允许以内联字符串硬编码到配置文件
- 不允许经前端传递
- 不允许直接进入普通数据库字段明文

## 6.3 数据库存放边界

数据库允许存：

- `certificate_digest`
- `secret_ref`
- `client_secret_ref`
- 密钥材料引用标识
- 加密后的 key envelope

数据库不允许存：

- 明文私钥
- 明文渠道签名密钥
- 明文长期下载解密密钥

## 7. 临时凭证

## 7.1 定义

生命周期短、可吊销、用于特定高风险或交付动作的临时凭证。

示例：

- `X-Step-Up-Token`
- 下载票据
- 共享开通临时 token
- 模板查询授权 token
- 沙箱会话 token

## 7.2 管理原则

- 允许落库为摘要或受控记录
- 必须有失效时间
- 必须支持主动吊销
- 必须留审计

## 9. 环境文件建议

建议分层：

- `.env.example`
  - 只放键名与非敏感默认值
- `.env.local`
  - 本地普通运行配置
- `.env.local.secret`
  - 本地敏感配置
- `secrets/`
  - 本地证书和私钥文件目录，不进 Git

## 10. 注入方式建议

## 10.1 本地开发

允许：

- `.env.local`
- `.env.local.secret`
- 挂载 `secrets/` 目录

## 10.2 Docker Compose

建议：

- 普通配置走 `environment`
- 敏感配置优先走 `env_file` 或 secret file
- 证书和私钥走 volume mount

## 10.3 测试与生产

建议：

- 普通配置：ConfigMap / 环境变量
- 敏感配置：Secret Manager / Vault / Secret volume
- 密钥材料：独立 secret mount 或专用密钥服务

## 11. 轮换与吊销规则

## 11.1 最低要求

- 核心密钥至少每 `90` 天轮换一次
- 证书必须支持吊销与失效
- 下载授权、step-up token、共享 token 必须具备短时效

## 11.2 触发轮换的情况

- 人员变更
- 环境泄漏怀疑
- 签名失败异常
- 证书异常吊销
- 审计命中高风险事件

## 12. 日志与审计要求

必须审计：

- 密钥轮换
- 证书签发/吊销
- 高风险配置修改
- 支付通道密钥变更
- 下载密钥封装策略修改

日志中必须脱敏：

- 数据库密码
- 支付密钥
- 客户端密钥
- 证书私钥路径后的文件内容
- 下载票据完整值

## 13. 当前禁止事项

- 把真实生产密钥写入 `.env.example`
- 把 `client_secret`、支付签名密钥、Fabric 私钥提交进 Git
- 在代码里直接写死 `PG_PASSWORD`、`KEYCLOAK_CLIENT_SECRET`、`MOCK_PAYMENT_SIGNING_SECRET`
- 在接口返回中暴露密钥原文
- 在审计导出包中默认带出私钥材料

## 14. 后续演进建议

当前阶段先采用可审计的软件密钥管理；后续建议逐步演进到：

- KMS/HSM
- 统一 Secret Manager
- 更细粒度的证书签发审批
- 自动轮换流水线

## 15. 一句话结论

当前阶段应统一理解为：

**配置可进环境，敏感配置进 secret，私钥与证书材料只走文件或专用密钥管理路径；所有高风险密钥动作必须可审计、可轮换、可吊销。**
