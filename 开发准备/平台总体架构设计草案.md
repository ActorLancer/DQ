# 平台总体架构设计草案

## 1. 文档定位

本文件用于给当前平台提供一份可直接进入工程设计阶段的总体架构草案。  
架构风格以**模块化单体主应用 + 少量外围适配进程 / Worker + 独立基础设施**为核心，不提前进入全面微服务化。

本文件目标：

- 覆盖当前所有已设计能力，避免遗漏功能
- 统一模块边界、依赖方向、事件边界和状态归属
- 为后续数据库、接口、页面、联调和部署设计提供单一架构入口

本文件不改变既定产品范围，不新增业务能力。

## 2. 架构目标与约束

### 2.1 架构目标

- 支撑 `V1` 标准交易闭环：上架、搜索、下单、合同、授权、交付、验收、结算、争议、审计、上链摘要
- 支撑八个标准 SKU：`FILE_STD / FILE_SUB / SHARE_RO / API_SUB / API_PPU / QRY_LITE / SBX_STD / RPT_STD`
- 支撑强审计、强回放、强对账、链上链下双层权威模型
- 支撑本地开发、联调、演示、生产前冻结四类环境
- 允许 `V2/V3` 在不破坏 `V1` 主状态模型的前提下扩展 C2D、FL、MPC、TEE、ZKP、公链展示增强和外部连接器

### 2.2 架构约束

- `PostgreSQL` 是业务运营主状态权威
- `Fabric` 是关键承诺、状态摘要和可信证明确认层
- `Kafka` 是事件总线，不是业务主状态机
- `OpenSearch` 是搜索/召回读模型，不是商品事实源
- `Redis` 是缓存和辅助状态，不是主数据源
- 不允许业务请求同步阻塞在链确认上
- 生命周期对象必须统一围绕：
  - `Order`
  - `Digital Contract`
  - `Authorization`
  - `Delivery`
  - `Settlement`
  - `Dispute`
  - `Billing Event`

## 3. 总体架构风格

### 3.1 选型结论

当前正式采用：

- **架构风格**：模块化单体
- **主应用**：`platform-core`
- **外围进程**：链适配、链监听、异步任务、索引同步、通知、Mock
- **基础设施**：数据库、缓存、消息、对象存储、搜索、认证、观测、联盟链

### 3.2 为什么不是微服务

当前阶段不采用全面微服务，原因如下：

- 交易主链路强依赖事务性边界，适合在单应用内收敛
- 强审计、强一致性、双层权威模型下，过早拆服务会放大状态分叉风险
- 当前更需要冻结模块边界，而不是放大部署边界
- 对个人开发和首版交付而言，模块化单体的调试、重构、发布成本明显更低

### 3.3 为什么不是传统大单体

虽然采用单体主应用，但必须满足以下条件，否则视为无效架构：

- 模块边界显式定义
- 模块依赖显式限制
- 模块事件显式定义
- 模块数据库访问边界可追踪
- 模块测试可以按模块单独运行

### 4.2 主应用与外围进程关系

- `platform-core` 承担业务主状态和主编排
- 外围进程承担外部适配、链事件监听、长任务、索引同步等技术职责
- 外围进程不得反向定义业务主状态，只能输入外部事实或消费集成事件

## 7. 模块依赖规则

### 7.1 允许的高层依赖方向

```text
iam / party / access
    ↓
catalog / contract_meta / listing / provider_ops
    ↓
inquiry / order / contract / authorization / delivery / billing / dispute
    ↓
audit / consistency / fairness / search / recommend / ops / developer
```

### 7.2 依赖限制

- `iam / party / access` 是基础模块，可被大多数模块依赖
- `catalog / contract_meta / listing` 不得反向依赖 `billing / dispute`
- `order` 可调用 `contract / authorization / billing / delivery`
- `delivery` 不得绕过 `order` 任意推进主状态
- `billing` 不得绕过 `order / settlement` 直接宣布交易完成
- `audit` 记录事实，不直接裁定业务逻辑
- `search / recommend` 只能读投影或读接口，不得直接改业务主表
- `fairness / ops` 只能提出阻断动作或人工处理建议，不得直接篡改业务事实
- `integration` 只做适配，不自定义业务规则

### 7.3 数据访问规则

- 每个模块优先只访问本模块聚合和仓储
- 跨模块读取优先使用查询接口或投影视图
- 不允许在模块外直接更新其他模块主表
- 数据库虽然共享，但代码层必须显式维护边界

## 8. 模块事件边界

### 8.1 事件分层

- 领域事件：
  - 模块内业务变化事件
- 集成事件：
  - 通过 `outbox_event` 出站，供 Kafka 消费者使用
- 审计事件：
  - 所有关键动作留痕

### 8.2 事件原则

- 同一数据库事务内必须写：
  - 业务对象
  - 审计事件
  - `outbox_event`
- 不允许先提交业务，再异步补审计和事件
- 链确认、支付回调、对象存储回执、证书回执统一作为外部事实回写

### 8.3 典型事件流

```text
业务命令
  -> platform-core 更新主对象
  -> 记录 audit_event
  -> 写 outbox_event
  -> publisher worker
  -> Kafka
  -> 下游：
       search-indexer
       notification-worker
       fabric-adapter
       recommendation pipeline
       consistency checker
```

## 9. 状态与数据归属

### 9.1 核心事实源

- `Product`
  - 目录、展示、审核、搜索事实源
- `ProductSKU`
  - 下单、合同、授权、交付、验收、账单、结算事实源
- `Order`
  - 主业务流程推进源
- `Authorization / Delivery / Settlement / Dispute`
  - 正式生命周期对象
- `Billing Event`
  - 事件型计费源
- `Audit Event`
  - 审计事实

### 9.2 技术状态归属

- `proof_commit_state`
  - 来自链适配与链确认
- `external_fact_status`
  - 来自支付、证书、对象存储等外部回执
- `reconcile_status`
  - 来自一致性与对账修复逻辑

这些状态不得替代业务主状态，只能作为技术状态族存在。

## 13. 核心业务数据流

### 13.1 商品上架流

```text
卖方提交商品
  -> catalog / contract_meta / listing
  -> 审核
  -> 生成 product / sku / 模板绑定
  -> 写 audit_event + outbox_event
  -> search-indexer 更新搜索投影
```

### 13.2 下单到交付流

```text
买方下单
  -> order
  -> contract 快照
  -> billing 生成 payment_intent
  -> authorization 开通
  -> delivery 开通
  -> acceptance
  -> settlement
  -> audit / consistency / fabric 摘要联动
```

### 13.3 外部事实回写流

```text
支付回调 / Fabric commit / 对象存储回执 / 证书回执
  -> integration / fabric-event-listener
  -> 回写 external_fact_status / proof_commit_state / reconcile_status
  -> 必要时触发人工处理、重试或阻断
```

### 13.4 搜索与推荐流

```text
业务主表更新
  -> 搜索投影刷新
  -> outbox_event
  -> Kafka
  -> search-indexer 写 OpenSearch
  -> Redis 失效
  -> 搜索/推荐请求
  -> OpenSearch 召回
  -> PostgreSQL 最终业务校验
```

## 14. 后续拆服务路径

### 14.1 可以优先独立的模块

- `search`
- `recommend`
- `audit`
- `billing`
- `fabric integration`
- `data-processing`

### 14.2 最晚拆分的模块

- `order`
- `contract`
- `authorization`
- `delivery`

原因：

- 它们共同构成当前最核心的交易主状态机
- 过早拆分最容易导致状态漂移

## 15. 架构执行建议

当前阶段的推荐执行顺序：

1. 先按本文件完成模块边界冻结
2. 再输出服务清单与服务边界正式版
3. 再冻结接口清单与 OpenAPI schema
4. 再冻结事件模型与 Topic 清单
5. 再进入数据库和页面实现拆解

## 16. 一句话结论

当前平台最合适的一版总体架构是：

**以 `platform-core` 作为模块化单体主应用，围绕已冻结的生命周期对象承载全部业务主状态；将 Fabric 适配、链事件监听、搜索索引、数据处理、通知和 Mock 支付等能力作为外围独立进程；以 `PostgreSQL + Kafka + MinIO + OpenSearch + Redis + Keycloak + Fabric + 可观测性栈` 构成统一底座。**
