# 数据交易平台角色权限矩阵正式版

## 1. 文档说明

本矩阵基于：

- [全量用户角色说明](/home/luna/Documents/DataB/用户角色说明/全量用户角色说明.md)
- [权限点清单](/home/luna/Documents/DataB/权限设计/权限点清单.md)

说明：

- 本矩阵按角色给出“默认权限包”
- 最终放行仍需叠加作用域、策略、风控和合同判断
- 矩阵中的 `R` 表示只读，`RW` 表示可读写/执行，`RV` 表示可审核/复核，`-` 表示默认无权

## 7. 职责分离补充

以下权限组合不应落到同一人工账号：

- `catalog.product.submit` + `review.product.review`
- `template.contract.manage` + `trade.contract.review`
- `billing.settlement.review` + `billing.refund.execute`
- `dispute.case.review` + `dispute.case.resolve`
- `model.algorithm.whitelist_manage` + `compute.task.execute`
- `profitshare.rule.manage` + `profitshare.reward.execute`
- `crosschain.request.create` + `crosschain.witness.manage`
- `risk.freeze.execute` + `risk.freeze.release`

## 7.1 SKU 与模板职责分离补充

- `catalog.sku.create/update` 的执行责任属于数据治理管理员、产品经理/上架管理员
- `review.product.review` 负责审核 SKU 是否正确使用标准 SKU 真值，而不是仅检查套餐文案
- `template.contract.bind / template.policy.bind` 负责模板兼容性，不得把 `SHARE_RO / QRY_LITE / SBX_STD` 绑定到错误模板族
- `delivery.template_query.enable` 与 `delivery.sandbox.enable` 必须视为两类不同开通动作，不应由同一页面或同一工单步骤模糊替代

## 8. 一句话结论

角色权限矩阵正式版定义的是“默认许可边界”。真正落地时，前端菜单、页面按钮、接口鉴权和审计日志都应以本矩阵为基线，再叠加作用域和业务规则做最终放行。
