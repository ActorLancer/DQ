# 数据对象交付与协作能力未决事项

本文件用于归档当前已经识别、但尚未在 `V1` 阶段彻底解决的数据对象交付与协作能力问题。

这些问题不推翻现有需求设计，但如果后续不继续收口，容易在开发中被误认为“已经完整解决”。

---

## 1. 当前结论

当前正式文档已经冻结以下内容：

- 五类基础数据对象 family
- 七类标准交易方式
- `V1` 五类正式商品化范围
- `V2/V3` 的扩展方向
- 对应的对象模型、权限模型、页面入口和数据库骨架

当前尚未彻底完成的，不是“是否要做”，而是“何时把它做完整”。

---

## 2. V1 已允许，但仅按最小路径落地的能力

### 2.1 只读共享

当前状态：

- `V1` 已允许只读共享作为正式商品
- 已有 `share grant`、recipient、revoke、生命周期和权限设计

未彻底解决：

- 不承诺一次完整支持 `Snowflake / BigQuery / Delta Sharing` 等异构协议矩阵
- 不承诺多连接器统一治理、统一 revoke 回执与跨系统证据闭环

当前建议：

- `V1` 仅支持单后端 / 单协议 / 单连接器试点实现

后续收口：

- `V2` 补齐共享连接器框架
- `V3` 补齐跨平台共享与责任链协同

### 2.2 模板查询 / clean room lite

当前状态：

- `V1` 已允许模板查询 `lite`
- 已要求模板白名单、参数校验、输出边界、导出限制和审计留痕

未彻底解决：

- 不支持自由 SQL
- 不支持完整 `analysis rule / TVF / differential privacy / egress control`

当前建议：

- `V1` 仅支持白名单模板和受限结果交付

后续收口：

- `V2` 补齐完整 clean room 规则与执行体系
- `V3` 补齐跨平台受控执行与结果证明协同

### 2.3 版本订阅 / revision 治理

当前状态：

- `V1` 已允许版本订阅
- 已有订阅关系、周期更新、交付游标和版本发布对象

未彻底解决：

- 不承诺复杂历史 revision 授权
- 不承诺动态授权窗口、复杂订阅策略、跨平台 revision cursor 协同

当前建议：

- `V1` 仅支持固定周期和标准 revision 发布策略

后续收口：

- `V2` 补齐完整 revision 治理
- `V3` 补齐跨平台 revision 协同和责任边界映射

### 2.4 差异化授权 / 验收 / 退款 / 赔付规则

当前状态：

- 文档已区分文件快照、只读共享、API、模板查询、结果产品
- 已明确这些对象不应走同一套交付逻辑

未彻底解决：

- 尚未形成完整的“按交付模式分流”的规则引擎
- 尚未把复杂争议、赔付、证据判定全部细化到可配置层

当前建议：

- `V1` 先按标准模板落地最小分流规则

后续收口：

- `V2` 补齐正式规则体系
- `V3` 补齐跨平台争议、责任链和监管穿透

---

## 3. 进入正式技术方案前必须再次拍板的事项

- `V1` 只读共享到底先选哪一种共享后端 / 协议
- `V1` 模板查询 `lite` 的模板语法、执行引擎和导出边界
- `V1` 版本订阅的固定周期、版本发布与补发规则
- `V1` 不同交付模式的最小验收/退款/赔付模板
- `V2` 共享连接器框架与 clean room 的技术路径
- `V3` 跨平台共享与责任链互认的证据模型
