# IAM 与 PDP 技术选型未决事项

## 1. 归档目的

本文件用于记录平台在以下两类技术问题上的当前结论与未决事项：

- `IAM` 技术接入方案
- 策略决策接口与 `PDP` 运行时方案

这两个问题已经形成了明确的需求边界，但尚未冻结最终产品选型与部署方案，因此暂不回写为全局正式定稿。

---

## 2. 当前已确定的内容

以下内容已经在正式需求体系中成立：

- 浏览器门户采用 `OIDC / OAuth 2.0 Authorization Code + PKCE + 服务端会话 + HttpOnly Cookie`
- API / SDK 采用短期 `Access Token` + 可轮换 `Refresh Token`
- 企业联邦 `V1` 支持 `OIDC`
- `V2` 增量支持 `SAML 2.0 / SCIM`
- 服务到服务采用 `Client Credentials` 或等价机器身份机制
- 最终放行链路固定为：
  - 身份
  - 主体状态
  - 权限
  - 作用域
  - 合同 / `Usage Policy`
  - 风控
  - 高风险校验
  - 审计
- 平台采用 `RBAC + ABAC + PBAC + Scope` 混合模型
- `IAM` 不能替代平台业务授权与策略主数据

---

## 3. 当前推荐但未冻结的方案

### 3.1 IAM

当前推荐路线：

- `V1`：统一认证中心优先评估 `Keycloak`
- 人登录：`OIDC`
- 机器身份：`Client Credentials`
- 企业联邦：`OIDC`，`V2` 再扩 `SAML / SCIM`

当前状态：

- 这是推荐技术接入方案
- 不是已经冻结的唯一实现

### 3.2 PDP

当前推荐路线：

- `V1`：应用内 `PDP`
- `V2`：如策略复杂度明显上升，再评估引入 `OPA`

当前状态：

- 这是推荐演进路径
- 不是已经冻结的唯一实现

---

## 4. 为什么现在不全量同步到所有正式文档

原因：

- 这次新增的是“技术接入与实现边界”
- 不是新的产品需求
- 不是新的业务规则
- 如果现在全量同步，容易把候选实现误写成正式选型

因此当前处理原则是：

- 需求边界继续以现有正式文档为准
- 具体 `IAM / PDP` 产品选型先放在未决事项中跟踪

---

## 5. 后续必须回头冻结的事项

未来进入正式技术方案阶段时，至少要拍板以下内容：

### 5.1 IAM 侧

- 是否正式采用 `Keycloak`
- `realm / tenant / org` 映射方式
- 浏览器登录接法：
  - 纯前后端分离
  - `BFF / session`
- 服务账号与机器身份建模方式
- `OIDC discovery / JWKS / introspection` 使用方式
- 企业 `OIDC / SAML / SCIM` 管理边界
- 会话注销、令牌撤销、设备治理的最终实现

### 5.2 PDP / 策略侧

- `V1` 应用内 `PDP` 是否单独做共享组件
- `PDP` 输入模型是否冻结
- `PDP` 输出模型是否冻结
- 是否在 `V2` 引入 `OPA`
- 若引入 `OPA`：
  - bundle 发布方式
  - decision log
  - 缓存策略
  - 降级策略
  - `fail-open / fail-closed` 原则

---

## 6. 触发正式收敛的条件

满足以下任一条件时，应将本问题迁出并同步回正式文档体系：

- 开始输出正式技术架构方案
- 开始输出认证域或权限域开发文档
- 开始定义统一网关 / BFF / SDK 接入规范
- 开始进行 `IAM` 产品 PoC 或选型评审
- 开始实现统一 `PDP` / 策略执行模块

## 8. 一句话结论

当前平台在认证、鉴权、策略判断这条主链上，需求边界已经够用；  
真正还没定的是“是否选 `Keycloak`”以及“何时引入 `OPA`”。  
这两个问题应作为技术选型事项继续跟踪，不应提前当作全局正式定稿。
