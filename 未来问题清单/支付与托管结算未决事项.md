# 支付与托管结算未决事项

## 1. 归档目的

本文件用于记录数据交易平台在支付、托管、清结算领域中已经识别、但当前阶段尚未最终冻结实施方案的问题。

这些问题已经有产品级和架构级方向，但暂不视为当前阻塞开发的最大问题；后续进入支付接入、清结算实现、法务合规落地阶段时，必须回到本文件逐项收敛。

## 2. 当前已明确的结论

以下内容已在正式文档体系中确定，不属于未决问题：

- 支付、资金、轻结算是独立业务域，不能直接耦合进订单主流程
- 平台默认不自建真实资金池，不默认直接持有客户真实资金
- 真实资金流优先通过外部持牌支付机构、银行或托管合作方完成
- 平台负责维护业务主状态、账务镜像、结算指令、对账和审计证据
- 已建模的资金对象包括：订单货款、买方保证金、卖方履约保证金、风险托管、退款、赔付、扣罚、结算打款、分润预留
- 已确定需要支付渠道适配层和 `MockPaymentProvider`
- `V1` 起步司法辖区已收敛为新加坡

## 3. 当前未最终冻结的问题

以下问题已经识别，但尚未最终定稿：

### 3.1 真实支付伙伴未最终确定

尚未最终确定：

- `V1` 真实接入哪一家或哪几家新加坡持牌支付/银行/托管合作方
- 不同渠道分别负责收款、退款、打款、对账还是由单一合作方统一承接
- 法币跨境付款和卖方出金是否由同一合作方处理

### 3.2 托管与结算实施路线未完全拍板

尚未最终确定：

- `V1` 采用支付伙伴原生托管/子商户能力，还是平台侧业务托管镜像 + 人工/半自动打款
- 风险托管余额与真实账户体系如何映射
- 保证金冻结、释放、扣罚是否完全依赖支付伙伴能力，还是由平台账务镜像主导

### 3.3 真实运营参数未冻结

尚未最终确定：

- 结算周期：`T+0 / T+1 / T+N`
- 平台服务费费率或阶梯规则
- 渠道手续费承担方
- 保证金比例、风险分级阈值
- 退款与赔付的人工审核阈值

### 3.4 失败恢复和异常运营细则未完全细化

尚未最终确定：

- 支付成功但回调失败时的最终确认规则
- 渠道重复回调、部分退款、退款失败、打款失败的处理策略
- 对账不平时的差异归因、补记账、人工工单和关账规则

### 3.5 数字货币与交易所结算暂不进入 `V1`

当前只保留未来扩展方向，尚未冻结：

- 数字资产结算的开放条件
- 允许的币种范围
- 上链资产、稳定币、交易所出入金、合规伙伴之间的衔接方式

## 4. 为什么当前不视为最大阻塞问题

当前平台在需求、流程、领域模型、权限和数据库层面，已经明确了支付与结算域的业务边界和对象模型，因此：

- 主交易链路已经闭环
- 订单、保证金、风险托管、赔付、退款、结算对象已经完成建模
- 后续可以先基于 `MockPaymentProvider` 推进设计和开发
- 真实渠道接入、法务确认、持牌合作方选择可在后续支付专项中收敛

因此，这部分问题属于：

- 重要
- 必须解决
- 但不一定是当前开发前最先阻塞的事项

出现以下任一情况时，应立即回到本文件逐项收敛，并同步回正式文档体系：

- 开始真实接入支付渠道
- 开始选择新加坡起步合作伙伴
- 开始开发退款、赔付、打款、对账功能
- 开始确定平台收费参数
- 开始讨论跨境收付款或数字资产结算
