# 8. Phase 3：跨域互联、监管增强与风险治理系统设计

## 8.1 阶段目标

Phase 3 面向生态级协作。此时平台不再只是一个单体联盟链系统，而是多个平台、多个联盟链、多个监管主体和多个行业节点参与的数据流通基础设施。平台必须具备跨链消息、跨域授权、风险识别、冻结与回滚、监管穿透和数据谱系追踪能力。

**Phase 3 关键能力**

| **能力**           | **说明**                                                         | **必要性**         |
|--------------------|------------------------------------------------------------------|--------------------|
| 跨链/跨平台互认    | 支持来自其他交易平台或联盟链的消息、授权与结算确认。             | 解决单平台孤岛。   |
| 预言机与事实见证   | 把链下事实（下载完成、监管审批、外部签章）转化为链上可验证事件。 | 解决链上链下断裂。 |
| 图谱化风控         | 识别串谋、洗订单、批量异常下载、异常授权路径。                   | 解决静态规则不足。 |
| 监管节点与审计接口 | 提供只读穿透、按单查询、按主体查询和风险告警订阅。               | 满足合规与治理。   |
| 冻结/隔离/受控回滚 | 对高风险资产或主体实施暂停、冻结、隔离和后续恢复。               | 实现事故可控。     |

## 8.2 跨链与预言机设计

跨链设计不应把桥接系统做成新的单点信任。建议采用“跨链请求对象 + 网关签名 + 预言机见证 + 目标链确认”的四段式模型。源平台发起 CrossChainRequest，生成请求摘要和 nonce；网关使用组织密钥签名；预言机或见证人验证链下事实；目标链接收并回写 ack；源平台在收到足够签名后将请求状态 finalized。

**跨链控制点**

| **控制点** | **设计要求**                                 | **备注**                 |
|------------|----------------------------------------------|--------------------------|
| 消息唯一性 | 必须包含 nonce、timestamp、source_chain_id。 | 防重放。                 |
| 签名验证   | 至少验证源链网关签名和见证签名。             | 避免伪造消息。           |
| 幂等处理   | 同一 ccr_id 在目标链重复接收不应重复执行。   | 使用 request ledger。    |
| 回执确认   | 目标链必须生成 ack 并返回摘要。              | 避免单向成功、双向失败。 |
| 失败补偿   | 支持超时重试、人工介入和终止。               | 不能无上限重试。         |

## 8.3 异常交易识别与图风控

Phase 3 的风险识别建议采用“规则引擎 + 图分析 + 风险工单”的组合。规则引擎适合处理显式阈值，例如短时高频下载、同一主体异常下单、黑名单命中；图分析适合处理串谋关系、评分联盟、异常转售链路、密钥共享群体和多跳传播。

| 下载异常 | 节点：主体、IP、设备、对象；边：下载行为     | 发现批量下载、共享凭证。   |

## 8.4 监管与审计设计

监管接口必须遵循最小披露原则：默认只开放订单摘要、主体身份、审计轨迹、风险标签和证据摘要；只有在授权场景下才可展开查看更多内容。监管节点不应拥有修改业务状态的能力，除非由平台治理规则明确允许冻结或隔离操作。

**监管视图能力**

| **能力**     | **输入**              | **输出**                                       |
|--------------|-----------------------|------------------------------------------------|
| 按订单穿透   | order_id              | 订单状态轨迹、关联主体、交付摘要、争议与裁决。 |
| 按主体穿透   | org_id / user_id      | 主体交易概览、信誉历史、风险标签、关联案件。   |
| 按资产穿透   | asset_id / product_id | 版本谱系、授权链、跨链同步记录。               |
| 风险告警订阅 | 风险规则或主体清单    | 告警事件流、处置结果。                         |

## 8.5 冻结、隔离与受控回滚

冻结和回滚必须是治理能力而不是日常能力。推荐设计三级控制：第一，冻结主体，暂停其创建新订单、下载和任务参与；第二，冻结商品或资产版本，阻止继续上架或交付；第三，受控回滚链下结果，例如撤销下载令牌、撤销后续 API 访问和通知相关平台。链上历史记录不可删除，但可以通过新事件标记状态失效。

- 冻结必须有明确的触发源：风控高危、监管指令、人工复核确认。

- 冻结操作必须双人审批并写审计。

- 回滚只对未来可控权限生效，不承诺抹除已离线持有的数据副本，因此需结合水印、合同和法律手段。

## 8.6 Phase 3 验收标准

- 至少打通一条跨链授权或跨链结算确认链路。

- 至少落地一类图风控任务并能输出风险告警。

- 提供监管只读视图和按单穿透查询。

- 支持冻结主体、冻结商品和冻结任务三类风险处置操作。
