# 2. 需求分析：角色、场景、能力边界与约束

## 2.1 参与角色

**角色定义**

| **角色**        | **职责**                                               | **系统权限**                                         |
|-----------------|--------------------------------------------------------|------------------------------------------------------|
| 数据卖方        | 发布数据商品、维护版本、交付密文、提交证据、参与仲裁。 | 创建商品、上架、下架、查看自身订单、提交交付与申诉。 |
| 数据买方        | 浏览商品、下单、支付、验收、评分、发起争议。           | 查询目录、创建订单、下载密文、确认收货、评分、申诉。 |
| 平台运营方      | 负责目录治理、规则配置、风控运营、客服与流程调优。     | 配置参数、管理审核流程、查看运营报表、启动人工复核。 |
| 区块链管理方    | 负责链网络治理、证书签发、通道管理、合约升级。         | 组织管理、节点管理、链码部署与升级。                 |
| 监管/审计方     | 依法查看审计记录、风险告警和证据摘要。                 | 只读审计视图、风险告警订阅、监管接口访问。           |
| 联邦参与节点    | 在二期参与模型训练、提交参数更新与贡献证明。           | 创建或加入训练任务、提交模型更新、接收奖励。         |
| 跨链网关/预言机 | 在三期承担跨链消息证明和链下事实见证。                 | 接收跨链请求、校验签名、记录确认。                   |

## 2.2 典型业务场景

- 静态数据集交易：卖方上架一批脱敏后的数据集，买方购买后通过密文下载并验真。

- 数据 API 服务交易：买方购买的是访问次数、时间窗口或带宽配额，而非文件副本。

- 数据结果交付：卖方仅提供计算结果或统计结果，平台记录计算任务与结果凭证。

- 联邦训练任务：多个机构不共享原始数据，仅共享加密参数或模型更新，按贡献获得奖励。

- 跨域互联交易：交易发生在两个独立平台或联盟链之间，需要跨链授权与结算确认。

- 监管审计与异常追踪：平台对异常行为、串谋评分、违规转售、异常下载模式进行识别。

## 2.3 功能性需求

**功能需求分层**

| **层级**   | **必须能力**                                                                       | **阶段归属** |
|------------|------------------------------------------------------------------------------------|--------------|
| 基础交易层 | 注册、认证、商品目录、版本管理、订单、保证金、交付、验真、仲裁、评分、结算、审计。 | Phase 1      |
| 隐私协作层 | 策略授权、训练任务、联邦学习、贡献评估、奖励发放、模型水印、策略模板。             | Phase 2      |
| 互联治理层 | 跨链通信、预言机、异常识别、风险评分、冻结/回滚、监管接口、数据谱系。              | Phase 3      |

## 2.4 非功能性需求

**非功能目标（建议基线）**

| **类别** | **设计目标**                                               | **说明**                               |
|----------|------------------------------------------------------------|----------------------------------------|
| 可用性   | 核心交易路径全年可用性不低于 99.9%                         | 通过多副本、故障转移、链节点冗余实现。 |
| 一致性   | 交易状态以链上事件为准，链下数据库最终一致                 | 采用事件驱动补偿机制和幂等设计。       |
| 安全性   | 最小权限、全链路加密、可追踪密钥、不可抵赖审计             | 链上签名 + 链下密文 + KMS + 审计。     |
| 可扩展性 | 支持分阶段新增联邦学习、跨链、监管视图                     | 对象模型和合约事件预留扩展字段。       |
| 性能     | 目录查询与下单操作应满足秒级响应；交付与验真视数据大小而定 | 链上不承载大文件。                     |
| 可运维性 | 必须提供监控、告警、日志、链健康探针与审计报表             | 避免链只可用不可运维。                 |

## 2.5 非目标与约束

- Phase 1 不追求跨链互通，不处理复杂多方联合计算，不直接处理高复杂度版权仲裁。

- 平台默认不对数据内容“真实性”做绝对担保，平台保障的是交易过程可信和证据可审计；内容质量需通过样本、信誉、第三方认证和争议机制综合保障。

- 不建议把通用公有链作为第一阶段底座。Phase 1 到 Phase 3 均应优先使用联盟链或许可链，以满足性能、权限和监管要求。

- 对于超大文件、流式数据、模型参数和高频 API 数据，应采用链下对象存储或消息总线承载，链上只记录摘要与状态。
