# 9. 通用基础能力：身份、密钥、存储、日志、监控与运维

## 9.1 身份与访问控制

平台应采用“组织身份 + 用户身份 + 链上身份 + 服务身份”四层身份模型。组织身份用于业务归属、合同和审计；用户身份用于人机交互与审批；链上身份用于签名和共识；服务身份用于微服务之间的鉴权。

**访问控制模型**

| **层次** | **控制方式**                           | **建议实现**                           |
|----------|----------------------------------------|----------------------------------------|
| 组织级   | 组织是否具备某类资产、任务或监管权限   | 组织属性、行业属性、合规状态。         |
| 用户级   | 角色、审批、职责分离                   | RBAC + ABAC 混合。                     |
| 资源级   | 商品、订单、任务、模型、证据、审计事件 | Owner/Participant/Regulator 访问矩阵。 |
| 操作级   | 查看、下载、交付、审批、冻结、导出     | 细粒度权限点，统一审计。               |

| **推荐做法：**统一接入 OIDC/OAuth2.1 或企业 IAM 作为登录层，链上证书/DID 作为可信签名层，避免把链身份直接暴露为最终用户登录方式。 |
|-----------------------------------------------------------------------------------------------------------------------------------|

## 9.2 密钥与证书管理

密钥管理建议采用集中 KMS + 链节点证书服务 + 客户端公钥封装的组合。对象加密密钥（DEK）由卖方或平台代理生成；DEK 不直接长期保存明文；DEK 使用买方公钥或组织 KMS 公钥封装为 envelope；链节点证书用于交易签名；平台服务密钥用于消息签名和跨链网关。

**密钥类型**

| **密钥/证书** | **用途**               | **生命周期要求**                           |
|---------------|------------------------|--------------------------------------------|
| 组织链证书    | 链上交易签名、成员认证 | 支持吊销与轮换。                           |
| 用户签名密钥  | 高风险审批、证据签名   | 与 MFA 绑定。                              |
| DEK           | 文件或对象加密         | 一次交易一把或一次版本一把，严禁长期复用。 |
| KEK/KMS Key   | 封装 DEK 或密钥加密    | 由 KMS 托管，支持审计。                    |
| 服务签名密钥  | 下载令牌、跨链消息签名 | 严格隔离环境与权限。                       |

## 9.3 数据与存储策略

**存储分工**

| **存储介质** | **承载内容**                                   | **说明**                       |
|--------------|------------------------------------------------|--------------------------------|
| 联盟链账本   | 状态、摘要、签名、证据 Hash、奖励与惩罚记录    | 高可信、低容量、不适合大对象。 |
| 关系数据库   | 业务查询模型、工作流状态、报表索引             | 以链上事件为准做投影。         |
| 对象存储     | 密文数据文件、证据文件、模型文件、下载回执原文 | 必须开启版本化与加密。         |
| 缓存         | 会话、令牌、热点目录、临时状态                 | 不得作为最终状态源。           |
| 搜索索引     | 商品搜索、日志检索、审计检索                   | 敏感字段需脱敏后入索引。       |

## 9.4 审计与可观测性

必须建设统一可观测体系：应用日志、审计日志、链事件日志、对象存储访问日志、训练任务日志、跨链消息日志。日志按用途分层：应用日志用于故障诊断；审计日志用于追责；指标用于容量与 SLO；链事件同步用于一致性对账。

**核心监控指标**

| **类别** | **指标**                                   | **目标/用途**      |
|----------|--------------------------------------------|--------------------|
| 交易指标 | 下单成功率、交付成功率、验真失败率、争议率 | 衡量主链路健康度。 |
| 链指标   | 交易提交成功率、区块确认时延、背书失败率   | 发现链网络异常。   |
| 存储指标 | 下载成功率、对象读取时延、签名令牌失败率   | 发现交付链路问题。 |
| FL 指标  | 轮次完成率、聚合耗时、异常更新率           | Phase 2 使用。     |
| 风控指标 | 高危告警数、误报率、冻结恢复时长           | Phase 3 使用。     |

## 9.5 运维控制要求

- 链节点、数据库、对象存储、API 网关和消息队列必须具备健康探针和自动告警。

- 所有生产变更必须走灰度发布或滚动发布，不得在高峰直接替换链码和核心服务。

- 合约升级必须经过测试网和预生产网演练，并提供升级前后兼容性说明。

- 证书吊销、密钥轮换、对象生命周期清理和日志归档必须纳入周期运维任务。
